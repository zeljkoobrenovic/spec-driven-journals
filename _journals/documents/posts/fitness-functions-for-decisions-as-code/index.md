> Fitness functions are objective automated checks, written in code, that verify decisions are being maintained — making decisions testable, assurable, and continuously enforced.

**Fitness functions** are objective automated checks — written as code — that verify a decision is being maintained. They make decisions testable and assurable, and they help with quality assurance, regulatory processes, and governance.

## How fitness functions connect to decisions

A decision record *documents* the decision. A fitness function *assures* it.

- **Example decision.** We use event sourcing for audit requirements.
- **Example fitness function.** A continuous-integration check that fails the build when a state change is committed without producing an event.

## Why fitness functions help

- **Objective measurement.** A fitness function passes or fails. The status of a decision is visible, not a matter of opinion.
- **Continuous use.** Fitness functions are living rules. They run on every commit and every build, not at quarterly review time.
- **Confidence to refactor.** If a refactor breaks a decision, the fitness function catches it before the change merges.
- **Scalable governance.** Standards are enforced by tooling rather than by a review bottleneck.

## Can fitness functions use AI?

Yes — fitness functions can lean on LLMs to evaluate work against a set of decisions. A simple prompt template:

```txt
IMPORTANT: Prefer retrieval-led reasoning over pre-training-led reasoning.
IMPORTANT: Turn on extended thinking. Turn on expert advice. Turn on search.

This is a fitness function to evaluate if our work is
using all our decisions, and is correct and accurate.

- Our decisions are here: {url}
- Our work to evaluate is here: {url}

Explain any errors, problems, gaps, weaknesses. Be direct. Be decisive.
```

This works best for soft constraints — design alignment, naming consistency, decision coverage — where a deterministic check would be brittle.

## Architecture unit testing

For hard constraints, dedicated architecture-testing libraries enforce structural rules from inside the existing test framework:

- [ArchUnit](https://www.archunit.org/) — architecture rules for Java code, expressed as plain JUnit tests.
- [ArchUnitTS](https://github.com/LukasNiessen/ArchUnitTS) — architecture rules for TypeScript and JavaScript, integrating with Jest, Vitest, Jasmine, and others.
