> A product domain is the unit of memory in Spec-Driven Product Architecture: one folder that keeps customer value, strategy, delivery, product bricks, teams, roadmap, and evidence in a form AI agents can inspect and change.

The central authoring unit in Spec-Driven Product Architecture is the product domain.

A product domain is not just a market category and not just a software bounded context. It is the durable modeling boundary for a coherent product area: the customers it serves, the jobs it supports, the product strategy it pursues, the capabilities it needs, the product bricks that make those capabilities real, and the teams and roadmap that move it forward.

In the source project, domains live under [`_config/product-domains/<domain-id>/`](https://github.com/zeljkoobrenovic/spec-driven-product-architecture/tree/main/_config/product-domains).

That folder is the source of truth. The generated HTML under [`docs/product-domains/<domain-id>/`](https://github.com/zeljkoobrenovic/spec-driven-product-architecture/tree/main/docs/product-domains) is the review and publishing surface.

The next articles use Ride Sharing Marketplace as a running example. Its [source specification folder](https://github.com/zeljkoobrenovic/spec-driven-product-architecture/tree/main/_config/product-domains/ride-sharing-marketplace) shows the authored model, and its [generated documentation](https://zeljkoobrenovic.github.io/spec-driven-product-architecture/product-domains/ride-sharing-marketplace/start/index.html) shows the review surface produced from that model.

## Why The Domain Folder Matters

AI agents need a place to return to. Without a stable source-of-truth folder, every prompt becomes a new negotiation about context. The agent may remember the previous conversation poorly, invent structure, or optimize one artifact while breaking another.

The domain folder gives the agent a working memory that is:

- **structured** - files have expected names and schemas.
- **inspectable** - future sessions can read the same source before making changes.
- **cross-linked** - customers, capabilities, bricks, teams, and plans can refer to one another.
- **validatable** - JSON can be parsed and references can be checked.
- **publishable** - generated documentation lets humans review the model without reading raw JSON.

This is why the repository guidance says to start in [`_config/**`](https://github.com/zeljkoobrenovic/spec-driven-product-architecture/tree/main/_config), not in generated pages.

The exact set can evolve, but the discipline should not: product architecture lives in source files, not in one-off prose.

## Source First, Docs Second

Generated documentation is still important. It makes the model readable, searchable, and easier to review with stakeholders who do not want to inspect JSON.

But generated docs are not the place to make strategic changes.

The workflow should be:

1. Edit the source model.
2. Validate the source model.
3. Generate the documentation.
4. Review the rendered documentation.
5. Return to source if the rendered story exposes a gap.

This matters because direct edits to generated pages cannot preserve the model. They may look good, but the next generation run overwrites them and the agent cannot reason from them reliably.

## Stable IDs Are Part Of The Spec

The repository insists on lowercase stable IDs. That can feel like a mechanical rule, but it is a product-architecture rule.

An ID is how one part of the model can refer to another part without depending on display text. A capability can point to a brick. A team can own a brick. A roadmap item can reference a target. A data dependency can identify the module that owns or uses a data asset.

Names can improve over time. IDs should survive those wording changes.

For AI agents, stable IDs reduce ambiguity. The agent can change "Trip Request and Intent Capture" to a better display name while preserving references to `trip`. That is the difference between editing a model and rewriting a document.

## The Domain Is Not A Prompt

Prompts are useful for creation. They are not sufficient for maintenance.

A prompt can say, "Create a product architecture model for a ride-sharing marketplace." The domain source files record what the agent actually decided: which rider groups exist, which jobs matter, which capabilities are needed, which bricks support those capabilities, which teams own them, and where assumptions remain.

That difference is essential. The prompt starts the work. The domain model preserves the work.

## What Good Looks Like

A strong domain model has a recognizable shape:

- Customer groups are materially different, not duplicate personas with renamed labels.
- Jobs to be done describe customer progress, not internal feature usage.
- KPIs are measurable and tied to product outcomes.
- Product capabilities are outcome-based.
- Product bricks are buildable and ownable.
- Teams can plausibly own the bricks.
- Roadmap and delivery objects point back to customers, capabilities, bricks, or teams.
- Evidence and assumptions are visible enough for future reviewers.
