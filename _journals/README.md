# Journals

`_journals/` contains the source material for every generated journal site.
Each immediate child directory with a `config.yaml` is treated as one journal by
`python3 _wiring/build.py`.

## Structure

```text
_journals/<journal>/
  config.yaml
  assets/
  posts/
```

The generated output for a journal is `docs/<journal>/`. During a build, that
output directory is removed and recreated, then journal assets and per-post
assets are copied into it.

## `config.yaml`

The config file is the table of contents for a journal:

```yaml
title: ORG Engineering Principles
description: Durable engineering principles that guide Organization's work.
logo: assets/images/example.jpg
logo_credit: Image credit

sections:
  - title: Design
    description: How systems should be shaped.
    posts:
      - small-and-simple/index.md
```

Post paths are relative to `_journals/<journal>/posts/`. Current journals use
the per-post folder layout, so config order is preserved by the build.

## Post Layout

Use this layout for new posts:

```text
posts/<optional-section>/<slug>/
  index.md
  spec.md
  assets/
    images/
    icons/
```

`index.md` is the published post. `spec.md` is the working contract for
non-trivial posts. When `spec.md` exists, the build writes
`docs/<journal>/<permalink>.spec.html` and adds a "View spec" link to the post.

Per-post assets are merged into the generated journal-level `assets/`
directory, so post content should keep using paths such as
`assets/images/<name>.jpeg`.

## Specs

Use `_wiring/spec-template.md` when creating a spec. Its front matter is small:

```markdown
---
status: draft
revised: 2026-05-20
---
```

Supported spec statuses are `draft`, `accepted`, `drifted`, and `superseded`.
The build renders the status on the spec page, and marks the post's "View spec"
link when a spec is `drifted` or `superseded`.

Update `revised:` whenever the spec changes or when drift is discovered.

## Front Matter

Posts use Jekyll-style front matter:

```markdown
---
id: ORG-PRIN-SMALL-AND-SIMPLE
title: "Design Principle: Small and Simple"
date: 2026-05-20
author: Organization Tech
excerpt: A short summary for the index card.
permalink: small-and-simple
timetoread: 6 min read
status: draft:gray
tags: design, architecture
icon: assets/icons/small-and-simple.png
logo: assets/images/small-and-simple/logo.jpeg
logo_credit: Image generated by Google Gemini
---
```

Important fields:

| Field | Notes |
| --- | --- |
| `title` | Used on index cards, post pages, browser titles, and cross-link text. |
| `permalink` | Stable output slug. Produces `docs/<journal>/<permalink>.html`. |
| `id` | Stable record identifier for ADR-shaped posts. |
| `status` | Editorial status, conventionally `<state>:<color>`. |
| `excerpt` | Summary shown on index cards. |
| `tags` | Comma-separated chips and filters. |
| `icon` | Index-card image path relative to the journal root. |
| `logo` | Optional post hero image path relative to the journal root. |

Do not change `permalink` or `id` for already-published records unless the URL
or identifier change is intentional.

## Body Conventions

ADR-shaped and principle-shaped posts usually open with a short highlight:

```markdown
> **Status**: DRAFT
>
> **Decision**: One paragraph that states the decision.
```

For principles, use `**Principle**` instead of `**Decision**`.

Use `[[permalink]]` for cross-record links. The build resolves links across
journals and prefers an in-journal target when one exists. If a target does not
exist, the literal `[[permalink]]` remains visible in the rendered page.

## Supported Content Blocks

Normal Markdown stays in `markdown` blocks. The build also recognizes these
custom fences:

| Type | Opening fence | Closing fence |
| --- | --- | --- |
| Mermaid | `---begin mermaid---` | `---end mermaid---` |
| Force graph | `---begin force-graph---` | `---end force-graph---` |
| Bubble chart | `---begin bubble-chart---` | `---end bubble-chart---` |
| Wardley map | `---begin wardley-map---` | `---end wardley-map---` |

The bubble-chart fence also accepts the historical typo `buble-chart`.

## Add a Post

1. Create `_journals/<journal>/posts/<slug>/index.md`.
2. Create `_journals/<journal>/posts/<slug>/spec.md` for non-trivial work.
3. Add the post to the right section in `_journals/<journal>/config.yaml`.
4. Run `python3 _wiring/build.py`.
5. Confirm the generated HTML exists under `docs/<journal>/`.

## Add a Journal

1. Create `_journals/<journal>/config.yaml`.
2. Add `posts/` and optional `assets/`.
3. Add at least one post path under `sections:`.
4. Run `python3 _wiring/build.py`.
5. Add the journal to `_start/_config/apps.json` if it should appear on the
   start page.
6. Run `python3 _start/generate-docs.py`.
