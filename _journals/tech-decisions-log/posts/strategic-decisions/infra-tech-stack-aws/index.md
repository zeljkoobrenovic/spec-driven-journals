> **Status**: DRAFT
>
> **Decision**: Organization commits to AWS as its strategic public cloud and will consolidate backend workloads there under AVIV Group's AWS commercial contract and account configuration. Organization will use AVIV Group's central AWS functions for generic, non-differentiating topics such as procurement, baseline FinOps, and AWS relationship management. Workload-level engineering, architecture, security posture, and product ownership remain with Organization.

## Decision

Adopt **AWS** as Organization's strategic public cloud provider for backend workloads.

Concretely:

* New and migrated backend workloads default to AWS.
* Organization uses AVIV Group's AWS commercial contract rather than negotiating an independent master agreement.
* Organization accounts sit inside AVIV Group's AWS Organization and account structure.
* AVIV Group central functions own the generic AWS relationship: procurement, commercial negotiation, baseline FinOps, consolidated billing, support relationship, and high-level AWS engagement.
* Organization owns workload architecture, service-level decisions, security posture inside its accounts, product reliability, and workload-level cost behavior.

This decision defines the cloud provider and commercial operating model. It does not decide which AWS services every workload should use.

## How to Use This Decision

Use this ADR when a team asks where a backend workload should run or how AWS responsibility is split:

| Question | Default answer |
| --- | --- |
| Is this an Organization backend workload? | Run it on AWS unless there is an explicit exception. |
| Does Organization need its own AWS contract? | No. Use the AVIV Group contract and central commercial functions. |
| Does AVIV own workload architecture? | No. Organization owns architecture, reliability, security choices, and product outcomes inside its accounts. |
| Is every AWS service automatically approved? | No. Service choice still needs architecture, security, cost, and operating-model review. |
| Does a different cloud or multi-cloud pattern fit? | Only with a documented decision and clear ownership model. |

The decision is meant to stop provider-level fragmentation. It should move discussion from "which cloud?" to "which AWS pattern is appropriate for this workload?"

## Why This Default

AWS is the pragmatic default because Organization is part of an AVIV Group estate that already has AWS scale, contract terms, central functions, and inherited workloads. Using that structure gives Organization:

* better unit economics than it would likely achieve alone;
* less procurement and contract overhead;
* access to group-level AWS support, escalation, and relationship management;
* a simpler migration path for inherited AWS-aligned workloads;
* one strategic cloud skill base rather than multiple parallel provider models.

The decision is not a claim that AWS is best for every possible workload. It is a consolidation decision: one strategic public cloud is easier to govern, staff, secure, and optimize than several.

## Responsibility Boundary

| Area | AVIV Group central AWS functions | Organization |
| --- | --- | --- |
| Commercial contract | Own master AWS agreement, group commitments, support tier, and commercial negotiations. | Provide demand forecasts and material workload plans that affect commitments. |
| Account structure | Provide the organization model, baseline account patterns, and shared guardrails. | Request and use accounts according to workload, environment, and ownership needs. |
| Baseline FinOps | Provide cost reporting, commitment management, tagging standards, and anomaly detection. | Own workload-level cost drivers, capacity choices, rightsizing, and cost-aware design. |
| Security guardrails | Provide organization-level controls and shared security tooling. | Own application security, data protection choices, IAM usage, and secure configuration inside accounts. |
| AWS relationship | Manage group-level AWS relationship, TAM access, and strategic escalations. | Own incident context, workload diagnostics, and product impact assessment. |
| Service selection | Define group-level restrictions or preferred service patterns where they exist. | Choose concrete AWS services within guardrails and document meaningful risks. |

This boundary needs active maintenance. Most friction will come from unclear ownership, not from AWS itself.

## Consequences

### Positive

* Organization gets access to AVIV Group scale, negotiated terms, commitment pools, and consolidated billing.
* Organization avoids staffing a full AWS procurement, contract, and baseline FinOps function.
* Inherited AVIV-aligned workloads can stay within the same provider and commercial envelope.
* Consolidating on one cloud reduces skills sprawl and architecture fragmentation.
* Organization engineering capacity stays focused on workload-level architecture, product features, and reliability.

### Negative

* Organization becomes commercially dependent on AVIV Group's AWS contract and cloud strategy.
* Provider lock-in to AWS is reinforced.
* Some negotiation and escalation flows must go through AVIV rather than directly from Organization.
* Group-level standards for tags, accounts, guardrails, reporting, and baseline controls become inputs to Organization's environment.
* If AVIV changes its strategic cloud direction later, Organization inherits the re-evaluation question.

## What This Means for Teams

* Start backend workload planning from AWS.
* Use approved infrastructure-as-code, account, tagging, security, and deployment patterns.
* Treat cost visibility as an engineering responsibility, not a finance-only responsibility.
* Review AWS service choices for maturity, operational fit, data handling, security model, cost behavior, quotas, and lock-in.
* Make exceptions to AWS, group guardrails, or standard account models rare, explicit, owned, and time-bound.
* Do not assume AVIV central functions own product reliability. They provide shared cloud capabilities; teams own their workloads.

## Exception Criteria

A proposal to use another provider, an independent AWS contract, or a multi-cloud pattern should explain:

* the user, product, regulatory, or engineering need that cannot be met inside the AWS baseline;
* whether the issue is caused by AWS itself, the AVIV contract model, the account model, missing platform capability, or local implementation quality;
* migration and integration impact for networking, identity, observability, security, data, and deployment pipelines;
* who owns procurement, provider relationship, cost management, support, security posture, and incident response;
* how the workload will meet the same reliability, audit, and security expectations as AWS workloads;
* when the exception will be revisited.

## Considered Options

| Option | Assessment |
| --- | --- |
| **AWS via AVIV Group contract and central functions** | Chosen. Best fit for inherited workloads, commercial leverage, and operating-model simplicity. |
| **AWS with an independent Organization contract** | More autonomy, but higher overhead and likely weaker unit economics. |
| **Different primary public cloud** | Possible in theory, but creates migration cost and discards group AWS leverage. |
| **Multi-cloud from the start** | Reduces provider concentration in theory, but multiplies skills, governance, security, and cost complexity now. |

## Scope and Revisiting

In scope:

* strategic public cloud provider selection;
* AVIV Group AWS commercial and central-function model;
* workload-level ownership boundary;
* backend workloads, inherited and new.

Out of scope:

* account vending and AWS foundation tooling, covered by [[infra-tech-stack-aws-foundation]];
* AWS networking, covered by [[infra-tech-stack-networking]];
* backend compute, covered by [[infra-tech-stack-compute]];
* backend messaging, covered by [[infra-tech-stack-messaging]];
* detailed service-by-service AWS architecture.

Revisit this decision if:

* AVIV Group changes its primary cloud strategy;
* AWS contract terms materially affect Organization cost or autonomy;
* central AWS functions no longer meet Organization's baseline needs;
* a regulatory or product requirement makes another provider materially better;
* Organization's workload profile changes enough that AWS is no longer the obvious consolidation target.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Public cloud provider** | The primary cloud platform where Organization runs backend workloads. |
| **AWS Organization** | The group-level AWS account structure under which Organization accounts are managed. |
| **AWS account** | A boundary for workloads, permissions, costs, quotas, and blast radius. |
| **Workload** | An application, service, data platform, or product capability running on AWS. |
| **Central AWS function** | AVIV Group capability that handles common AWS topics across companies or domains. |
| **FinOps** | Shared engineering, product, and finance practice for managing cloud spend. |
| **Commitment discount** | Commercial discount mechanism such as Savings Plans or reserved capacity. |
| **Guardrail** | A policy, standard, or control that shapes what teams can do in AWS accounts. |
