> **Status**: DRAFT
>
> **Decision**: For the next 6-12 months, Organization will use AVIV Group's central AWS foundation for account creation, account lifecycle management, landing-zone guardrails, and access management. Organization will not stand up parallel tooling, fork the central setup, or significantly diverge during this period. The expected future state is simpler, more conventional AWS-native tooling with more country-level ownership for Organization.

## Decision

For the next **6-12 months**, Organization will use **AVIV Group's existing central AWS foundation** as its baseline for account and access management.

Concretely:

* AWS accounts for Organization are created and lifecycle-managed through the central AVIV Group setup.
* AWS Organizations, AWS Control Tower, and AFT remain the inherited foundation path.
* Human access remains managed through the central AVIV access model, including current custom-built access scripts.
* Organization will not build parallel account-vending, organization, landing-zone, or access-management tooling during this period.
* Organization will not fork the central tooling during this period.
* Organization will prepare for a later shift toward more conventional AWS-native tooling and more delegated country-level ownership.

This decision sits under the public cloud decision, [[infra-tech-aws]]. It defines the foundation layer only. Workload compute, networking, messaging, and application architecture are covered elsewhere.

## How to Use This Decision

Use this ADR when a team needs an AWS account, access model, guardrail change, or foundation-level exception:

| Question | Default answer |
| --- | --- |
| Does Organization need a new AWS account or account lifecycle change? | Use the central AVIV foundation process. |
| Should Organization fork or replace account-vending/access tooling now? | No. Use the inheritance period to learn and simplify. |
| Is this ordinary workload infrastructure inside an account? | Keep it with the workload team and normal IaC path. |
| Is this an OU, SCP, Control Tower, AFT, access, or account-baseline change? | Treat it as foundation-level and route through the central process. |
| Is the central process blocking a material need? | Request a documented, time-bound exception. |

The decision is a sequencing choice: inherit now, learn deliberately, and revisit with evidence. It is not a claim that central custom tooling is the desired permanent model.

## Why This Default

The current AWS estate already lives inside AVIV Group's foundation and commercial model. Moving away immediately would add high-blast-radius work at the moment Organization also needs workload momentum.

The default is conservative because foundation changes affect many accounts at once: security posture, auditability, billing, access, policy inheritance, account provisioning, and future delegation. A rushed local foundation would create more risk than value during the initial transition.

The expected direction is still simplification:

* more use of conventional AWS capabilities where they are sufficient;
* less reliance on custom access scripts;
* clearer country-level ownership for routine account and access work;
* retained group guardrails where security, audit, procurement, and FinOps need consistency;
* better self-service for common account and access requests.

## Boundary: Foundation vs Workload

| Topic | Foundation-owned | Workload-owned |
| --- | --- | --- |
| Account creation | Request process, account metadata, OU placement, lifecycle state. | Business purpose, owning team, environment, and workload justification. |
| Organization structure | OU hierarchy, inherited policies, Control Tower enrollment. | Understanding inherited constraints and requesting the right account type. |
| Guardrails and SCPs | Baseline controls, policy attachment, exception process. | Designing workloads inside guardrails and requesting justified exceptions. |
| Human access baseline | Central access model, permission sets or roles, federation, auditability. | Requesting least-privilege access and removing unused access. |
| Account customization | AFT workflow, baseline tags, default settings, account bootstrapping. | Workload-specific infrastructure deployed inside the account. |
| Cost allocation baseline | Account metadata, required tags, consolidated billing alignment. | Workload forecasts, budgets, rightsizing, and cost optimization. |
| Incident response | Foundation incidents, break-glass, account drift, policy failures. | Application incidents, service dashboards, and service runbooks. |

This boundary prevents two failure modes: product teams waiting on the foundation for normal workload choices, and product teams bypassing the foundation for high-blast-radius changes.

## Commitments During the Inheritance Period

The 6-12 month window should produce concrete learning, not passive waiting:

* Maintain a current inventory of Organization accounts, owners, environments, workloads, lifecycle states, and cost centers.
* Map OU placement, inherited SCPs, Control Tower controls, known policy failures, and common exception paths.
* Record lead time, access friction, customization pain points, drift incidents, and workarounds.
* Avoid new deep automation dependencies on custom access scripts.
* Standardize account, access, exception, and metadata request artifacts.
* Identify routine foundation tasks that could move to Organization ownership later.
* Prepare a future-state proposal for delegated ownership inside group guardrails.

These artifacts are the evidence for the revisit.

## Consequences

### Positive

* Organization gets a fast start without first building account-vending, landing-zone, or access tooling.
* Inherited accounts remain in their current organizational and guardrail context.
* The decision stays aligned with the AWS via AVIV Group contract in [[infra-tech-aws]].
* Engineering capacity can focus on workload-level migration, compute, messaging, and product delivery.
* The time-bounded horizon creates a natural revisit point.
* Future delegation can be based on observed friction and real account inventory rather than assumptions.

### Negative

* Organization depends on the central AVIV Group function for account creation, landing-zone changes, and access workflows during this period.
* Non-standard foundation requests may take longer than Organization prefers.
* Custom access scripts remain an external dependency and may change during the transition.
* Organization inherits central opinions about OU structure, SCPs, controls, tags, names, and access workflows.
* Organization will not build foundation-operating muscle unless learning is deliberate.
* If AVIV's expected simplification slips, Organization must explicitly re-evaluate instead of drifting indefinitely.

## What This Means for Teams

* Use the central account and access process for foundation-level changes.
* Keep application infrastructure inside workload-owned IaC and CI/CD paths; do not use AFT as an application deployment system.
* Make account purpose, owner, environment, cost center, data classification, and lifecycle explicit.
* Request least-privilege access and remove unused access.
* Escalate foundation gaps rather than normalizing bypasses or shadow tooling.
* Treat custom access scripts as transitional tooling, not stable Organization APIs.
* Capture friction so the revisit has evidence.

## Exception Criteria

An exception to the inheritance approach should explain:

* the specific foundation capability that central tooling cannot provide in time;
* affected accounts, OUs, workloads, users, and compliance or delivery impact;
* why a temporary central-process workaround is not sufficient;
* how auditability, least privilege, billing, tagging, logging, and support are preserved;
* how the exception will be retired or reconciled when the AVIV foundation changes;
* who owns the exception and when it will be reviewed.

Valid exceptions should be rare: security or compliance needs that cannot wait, production access blockers with material impact, mandatory migration support, or a short-lived migration bridge with an expiry date.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Inherit AVIV central foundation for 6-12 months** | Chosen. Maintains continuity while Organization learns, maps friction, and prepares for delegated ownership. |
| **Stand up Organization-specific foundation now** | More autonomy, but likely duplicates work that will change and distracts from workload momentum. |
| **Fork the central tooling** | Gives local control, but locks Organization into custom non-differentiating tooling. |
| **Adopt conventional AWS-native tooling immediately** | Aligns with the expected future direction, but creates avoidable near-term migration and central-operating friction. |
| **Freeze foundation changes** | Minimizes churn, but blocks legitimate migration, access, and lifecycle needs. |

## Scope and Revisiting

In scope:

* account creation and lifecycle management;
* AWS Organizations and OU placement;
* Control Tower landing-zone controls;
* AFT account provisioning and customization;
* human access management;
* foundation-level guardrails and exceptions.

Out of scope:

* public cloud provider selection, covered by [[infra-tech-aws]];
* AWS networking, covered by [[infra-tech-stack-networking]];
* workload compute, covered by [[infra-tech-stack-compute]];
* backend messaging, covered by [[infra-tech-stack-messaging]];
* application infrastructure inside accounts;
* workload-level FinOps beyond required account metadata and central reporting.

Revisit this decision in the planned **2026-11-01 to 2027-05-01** window, or earlier if:

* AVIV Group materially changes its AWS foundation toward a simpler, less centralized, more country-delegated model;
* central custom access tooling changes in a way that affects Organization workflows;
* a material foundation-level blocker affects inherited workload migration;
* the AWS strategy in [[infra-tech-aws]] changes.

At the revisit, Organization should decide which foundation capabilities it should own directly, which should remain central, and which custom-tool dependencies should be retired.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **AWS account** | A security, billing, quota, and operational boundary for workloads. |
| **Management account** | The AWS Organizations management account; not a normal workload account. |
| **Member account** | An account inside the AWS Organization where workloads or shared capabilities run. |
| **Organizational unit (OU)** | A group of AWS accounts inside AWS Organizations. |
| **Service control policy (SCP)** | An Organizations policy that sets maximum permissions for member accounts. |
| **Landing zone** | A governed multi-account AWS environment with baseline accounts, controls, and structure. |
| **Control / guardrail** | A rule that enforces, detects, or pre-checks governance requirements. |
| **AFT** | Account Factory for Terraform, used for account provisioning and account customization under Control Tower. |
| **IAM Identity Center** | AWS workforce access service for account and application access. |
| **Permission set** | An IAM Identity Center access template for roles in AWS accounts. |
| **Drift** | A difference between expected governed state and actual state. |
| **Delegation** | Moving selected foundation responsibilities to Organization while retaining agreed group guardrails. |
