> **Status**: DRAFT
>
> **Decision**: Organization will use Amazon MSK as the strategic Kafka backbone for backend messaging. Kafka topics on MSK become the default for durable event streams, asynchronous service-to-service integration, replay, and analytics-facing event flows. SQS, SNS, and EventBridge remain available for narrow use cases, but they are not the strategic messaging backbone. Building Kafka expertise is an explicit cost of this decision.

## Decision

Adopt **Amazon MSK** as Organization's strategic backend messaging platform.

Concretely:

* Durable business events and multi-consumer event streams default to Kafka topics on MSK.
* Organization consumes Kafka as an AWS managed service, not as self-managed Kafka and not through a third-party Kafka vendor by default.
* Kafka clients run from the standard backend compute model in [[infra-tech-stack-compute]].
* SQS, SNS, and EventBridge remain available for narrow patterns where Kafka would add unnecessary weight.
* Self-managed Kafka, Confluent Cloud, Kinesis, Pulsar, and other streaming platforms are not defaults.

This decision builds on the AWS cloud decision, [[infra-tech-aws]], and the compute decision, [[infra-tech-stack-compute]]. It defines the strategic messaging backbone; it does not define every topic convention, schema format, library, or migration sequence.

## How to Use This Decision

Use this ADR before introducing asynchronous integration:

| Question | Default answer |
| --- | --- |
| Is this a durable business event useful to multiple current or future consumers? | Use Kafka on MSK. |
| Does the flow need replay, late-joining consumers, analytics consumption, or independent consumer groups? | Use Kafka on MSK. |
| Is this a private work queue for one logical worker group with no event-history value? | SQS may fit. |
| Is this simple fan-out to AWS-native targets without replay needs? | SNS may fit. |
| Is this AWS-service event routing, SaaS event routing, or a schedule? | EventBridge may fit. |
| Does the caller need an immediate answer? | Use an API, not messaging by default. |
| Is this a long-running business workflow with state, timers, and compensation? | Use a workflow pattern; do not hide workflow state in Kafka consumers. |

Kafka is the backbone for event logs, not a universal replacement for every communication pattern.

## Why This Default

Organization needs durable, replayable, multi-consumer event streams for backend integration and analytics-facing flows. Kafka's model fits that shape better than assembling a backbone from SQS, SNS, and EventBridge.

MSK is chosen because it keeps Kafka inside the AVIV Group AWS contract and operating model while avoiding self-managed broker operations. AWS operates the managed service foundation. Organization still owns topic design, schemas, access, producers, consumers, lag, replay, and incident response.

The main risk is skill. Kafka is new to Organization and only lightly used elsewhere in AVIV Group. This ADR deliberately accepts a skills-investment cost instead of hiding it.

## What Kafka Means Here

Kafka should be used for events that read like business facts, not implementation commands. A good event has:

| Concern | Expectation |
| --- | --- |
| Meaning | The event describes something that already happened. |
| Ownership | A producer owns event meaning, schema compatibility, and support path. |
| Schema | Payload structure is versioned and compatibility is checked. |
| Partition key | Ordering and load distribution are deliberate design choices. |
| Retention | The replay window is explicit and tied to business, analytics, compliance, and cost needs. |
| Consumers | Consumer groups own idempotent processing, offset handling, lag monitoring, and replay readiness. |
| Failure path | Poison events, retries, dead-letter handling, and replay are designed before production use. |

Kafka mechanics become architecture decisions. Ordering is per partition. At-least-once processing is normal. Replay can duplicate side effects. Consumer lag is often a business signal, not only an infrastructure metric.

## Consequences

### Positive

* Organization gets one strategic messaging backbone for durable event streams.
* Event replay, independent consumers, and analytics-facing consumption become first-class patterns.
* MSK removes most broker-operations burden compared with self-managed Kafka.
* The platform stays inside the AVIV Group AWS contract and central AWS operating model.
* Kafka clients align with the ECS Fargate compute standard in [[infra-tech-stack-compute]].
* Messaging investment can concentrate on one platform, one set of conventions, and one education path.

### Negative

* Kafka has a steeper learning curve than SQS, SNS, or EventBridge.
* Organization has limited in-house Kafka expertise today; training, templates, reviews, and runbooks are part of the cost.
* MSK has a cluster cost floor and may be expensive for very low-volume tactical flows.
* Even on MSK, Organization owns topic governance, schemas, ACLs, client patterns, monitoring, and incident response.
* MSK is AWS-specific. Moving to another Kafka delivery model later is possible at the protocol level but not operationally free.
* "Kafka by default" can become overuse unless the SQS/SNS/EventBridge boundary is actively reviewed.

## What This Means for Teams

* Design durable business events as Kafka events by default.
* Do not create a topic until owner, event meaning, schema, partition key, retention, access control, monitoring, and failure handling are clear.
* Producers own event meaning and compatibility.
* Consumers own idempotency, offset commits, lag monitoring, retry behavior, and replay safety.
* Use recommended producer and consumer patterns rather than raw client usage everywhere.
* Use SQS, SNS, or EventBridge only when the flow is truly simpler outside Kafka, and document the reason.
* Treat Kafka literacy, runbooks, schema review, topic inventory, and incident drills as part of delivery, not optional enablement.

## Responsibility Boundary

| Area | Platform / enablement teams | Product / domain teams |
| --- | --- | --- |
| MSK foundation | Provide clusters, network access, access mechanisms, baseline monitoring, and upgrade guidance. | Consume the approved platform rather than creating parallel Kafka infrastructure. |
| Topic governance | Define naming, metadata, retention defaults, lifecycle rules, and review workflow. | Own event purpose, event meaning, consumers, and deprecation decisions. |
| Schemas | Provide schema tooling and compatibility checks. | Own schema evolution and compatibility with real consumers. |
| Client patterns | Provide recommended producer and consumer examples. | Use the patterns consistently and surface gaps. |
| Observability | Provide broker dashboards and standard alert templates. | Add service-level and business-level alerts for producer and consumer behavior. |
| Incident response | Own cluster-level runbooks. | Own domain incidents caused by bad events, consumer failures, lag, and replay effects. |

## Exception Criteria

An exception to Kafka on MSK should explain:

* the integration shape that Kafka does not serve well;
* the alternative service and why it is simpler or safer for this flow;
* who owns operations, monitoring, access control, retries, failure handling, and cost;
* whether the flow needs replay, late-joining consumers, analytics, or independent consumer evolution;
* the security and data classification model;
* when the exception will be revisited.

Examples of likely valid exceptions:

* SQS for a private work queue with one logical consumer group and no replay or fan-out requirement;
* SNS for simple pub/sub fan-out to AWS-native subscribers where durable replay is not needed;
* EventBridge for AWS service events, SaaS events, schedules, or simple integration rules;
* direct API calls for request/response interactions;
* workflow tooling for long-running business processes with explicit state, timers, retries, or compensation.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Amazon MSK as Kafka backbone** | Chosen. Best fit for durable event logs, replay, independent consumers, AWS alignment, and mainstream Kafka clients. |
| **SQS + SNS + EventBridge as backbone** | Excellent tactical tools, but not one coherent durable event-log backbone. |
| **Self-managed Kafka** | Maximum control, but too much broker-operations burden for Organization's desired model. |
| **Confluent Cloud / Platform** | Strong Kafka-native feature set, but introduces a separate vendor and commercial model outside the AVIV AWS path. |
| **Kinesis Data Streams** | AWS-native streaming, but narrower ecosystem and stronger AWS-specific API lock-in than Kafka. |

## Scope and Revisiting

In scope:

* durable business events;
* event streams with multiple independent consumers;
* asynchronous backend service integration;
* analytics-facing event flows;
* Kafka producer and consumer operation.

Out of scope:

* workflow orchestration;
* in-process pub/sub inside one service;
* real-time client-facing transport such as WebSockets or push notifications;
* database change-data-capture sources except where they feed Kafka topics;
* detailed topic naming, schema registry, serialization, and client-library standards.

Revisit this decision if:

* actual messaging patterns are dominated by low-volume flows that Kafka does not serve well;
* MSK cost or operations materially worsen relative to alternatives;
* AVIV Group changes its AWS operating model in a way that affects MSK;
* Kafka expertise does not materialize after a defined evaluation period;
* teams repeatedly need SQS, SNS, EventBridge, or workflow exceptions in ways that suggest the backbone boundary is wrong.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Event** | A record of a business or system fact that already happened. |
| **Command** | A request for another component to do something. |
| **Topic** | A named Kafka log that stores related events. |
| **Partition** | An ordered subset of a Kafka topic. |
| **Partition key** | The value used to choose the partition for an event. |
| **Consumer group** | A named group of consumers that share work and track offsets together. |
| **Offset** | A consumer group's position in a topic partition. |
| **Retention** | How long Kafka keeps records in a topic. |
| **Schema** | The versioned contract for event payload structure. |
| **Producer** | The service that publishes events. |
| **Consumer** | The service or job that reads events. |
| **Consumer lag** | The distance between latest available records and what a consumer group has processed. |
| **Idempotency** | The ability to process the same event more than once without incorrect side effects. |
| **Replay** | Re-reading older records from a topic. |
| **Dead-letter path** | The agreed handling path for events that cannot be processed. |
| **Transactional outbox** | A pattern that writes state changes and outbound events through one reliable path. |
