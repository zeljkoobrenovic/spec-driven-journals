> KEY POINTS:
> * Three concrete strategies for the three most common legacy landscapes, each implemented as a Claude Code workflow. 
> * Subagents for monolithic .NET on TFS, Java microservices on GitLab, and over-complex Node.js serverless on GitHub.

<hr>

The same Claude Code strategy does not mean the same playbook everywhere. Each landscape has a different modernization risk, so each needs a different workflow and a different guardrail.

## A. Monolithic .NET/C# + SQL Server + stored procedures + TFS

### Strategic stance

Treat this as the highest-value Claude Code modernization target — but not as a place to build new capability.

The main risks are hidden business rules, database coupling, stored procedure logic, weak automated tests, and old repository/workflow constraints. Claude Code earns its keep here by reading code and stored procs at a scale that no engineer would attempt by hand.

### Claude Code workflow

Use these subagents (see [MCP, Skills, and Subagents][mcps]):

* `legacy-explorer` — walks C# modules and produces a capability map
* `stored-proc-summarizer` — plain-language summary of stored procedures + tables touched
* `characterization-test-writer` — generates regression tests around critical flows
* `extraction-planner` — proposes service-extraction seams + ADR draft
* `wrapper-generator` — drafts API wrappers around stable legacy capabilities

MCP servers needed:

* DB schema MCP server (read-only against SQL Server) — lets subagents enumerate tables, dependencies, and stored proc metadata
* TFS MCP server (read-only) — enough to support the migration to Git, no more

A typical session: an engineer asks Claude Code "summarize the order-fulfillment module and identify a candidate for first extraction." The main agent spawns `legacy-explorer` to map the module, `stored-proc-summarizer` (in parallel) on each stored proc, then `extraction-planner` once both return. The result is a draft ADR plus a list of stored procs that need characterization tests before any change.

### Recommended approach

1. **Inventory and understand** — Use `legacy-explorer` and `stored-proc-summarizer` to produce a legacy capability map. Save it as living documentation in the repo.
2. **Add tests before changing** — `characterization-test-writer` generates regression and characterization tests around critical business flows. Prioritize stored procedures and high-change areas.
3. **Wrap first, extract second** — `wrapper-generator` proposes stable APIs or event adapters over selected legacy capabilities. Avoid direct database coupling from new services.
4. **Extract by business capability** — `extraction-planner` proposes seams; architects and domain experts decide final boundaries.
5. **Retire TFS gradually** — Migrate active code to Git-based repositories. Do not invest in further TFS MCP work beyond migration support.

### First pilot candidates

Pick one flow where business value is high but the first step can be read-only or test-only.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| Stored-procedure-heavy business flow | Tests Claude Code's ability to explain database-embedded logic | Stored procedure summary pack + dependency map |
| Critical user journey with weak tests | Builds confidence before any extraction | Characterization test pack |
| Stable capability suitable for wrapping | Proves wrap-first modernization | Wrapper API draft + extraction ADR |

### Claude Code guardrail

Run these subagents in `plan` permission mode by default. Do not allow Claude Code to rewrite stored procedures or business-critical logic without strong regression tests already in place and a domain-expert review on the PR. The `pr-review-helper` subagent enforces this — it flags any PR that modifies a stored proc without a paired test file.

## B. Java microservices + self-managed Kafka + GitLab

### Strategic stance

This estate is conceptually closer to the target but operationally misaligned if Kafka is self-managed and service standards vary.

The objective is not "rewrite everything," but **standardize, simplify, and migrate operational responsibility to AWS patterns** — and Claude Code makes the standardization step feasible at the per-service level instead of being a multi-quarter platform program.

### Claude Code workflow

Use these subagents:

* `service-scorecard-builder` — scores each service against the standard template
* `kafka-contract-reviewer` — reviews Kafka changes against org standards
* `kafka-topic-documenter` — generates AsyncAPI specs from existing producers/consumers
* `consumer-error-handling-fixer` — proposes idempotency and DLQ patterns where missing
* `template-aligner` — refactors a service toward the standard Java template
* `msk-migration-planner` — drafts the migration plan from self-managed Kafka to MSK

MCP servers needed:

* GitLab MCP server (official)
* Kafka / Schema Registry MCP server — lets subagents check actual registered schemas, not the ones in the diff
* Service catalog MCP server — service ownership and dependency lookup
* AWS MCP server — for MSK target state

A typical session: the platform team runs `service-scorecard-builder` across all services overnight (Claude Code in headless mode, results pushed to the service catalog). The next morning, every service has a scorecard for build health, test coverage, dependency age, observability maturity, Kafka usage, and operational risk. Teams below threshold get a Jira ticket auto-created with a `template-aligner` plan attached.

### Recommended approach

1. **Create a Kafka governance baseline** — encoded in the `kafka-contract-reviewer` subagent's system prompt. Topic naming, ownership, schema strategy, retention rules, compatibility rules, dead-letter handling, consumer group conventions, replay strategy.
2. **Generate service scorecards** — `service-scorecard-builder` runs continuously. The platform team's dashboard is built from its output.
3. **Standardize service templates** — `template-aligner` refactors one service at a time toward the Java base image, logging, tracing, metrics, health endpoints, Kafka client configuration, retry and idempotency patterns.
4. **Move Kafka operations toward Amazon MSK** — `msk-migration-planner` drafts service-by-service migration plans. Amazon MSK is a fully managed service for running applications that use Apache Kafka and provides control-plane operations for creating, updating, and deleting clusters while preserving Apache Kafka data-plane APIs. ([AWS Documentation][5])

### First pilot candidates

Pick services that represent common patterns rather than edge cases.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| High-volume Kafka consumer | Exposes error handling, retry, idempotency, and DLQ quality | Consumer review + remediation plan |
| Service with inconsistent template usage | Tests service-scorecard and template-alignment workflow | Service scorecard + template alignment PR |
| Important topic family | Tests event ownership and schema governance | AsyncAPI draft + ownership map |

### Claude Code guardrail

Do not let Claude Code invent event contracts. Event semantics are architecture decisions, not code-generation details. The `kafka-contract-reviewer` subagent's system prompt is explicit: it reviews proposed contracts against existing standards but does not author new ones without human-authored input.

## C. Node.js serverless stack + GitHub

### Strategic stance

This is modern but over-complex. The goal is **simplification**, not more automation.

Claude Code helps identify where serverless workflows, glue code, deployment pipelines, permissions, and operational handoffs have become too complex — and proposes consolidation toward ECS where appropriate.

### Claude Code workflow

Use these subagents:

* `serverless-complexity-reporter` — summarizes a serverless workflow's surface area and dependencies
* `event-flow-mapper` — produces a diagram of an event flow across Lambda functions, SQS, EventBridge, etc.
* `consolidation-recommender` — classifies functions into "keep serverless" / "consolidate to ECS" / "retire"
* `ci-simplifier` — proposes simplifications to GitHub Actions workflows
* `ecs-migration-planner` — for functions classified as "consolidate," produces a migration plan

MCP servers needed:

* GitHub MCP server (official)
* AWS MCP server — to inspect actual Lambda configs, IAM, EventBridge rules
* Service catalog MCP server

A typical session: an engineer asks "this Lambda is being paged on every deploy — should it stay serverless?" Claude Code spawns `serverless-complexity-reporter` and `event-flow-mapper` in parallel against the Lambda's repo + AWS state, then runs `consolidation-recommender` over both reports. The output is a recommendation with evidence, not a guess.

### Recommended approach

1. **Classify serverless components** — `consolidation-recommender` does the first pass. Keep as serverless where event-driven and simple. Migrate to ECS where workflows are long-running, operationally complex, or better modeled as services. Retire redundant glue code.
2. **Reduce workflow complexity** — `ci-simplifier` simplifies GitHub Actions, standardizes deployment environments, removes special-case pipelines. Align observability with target platform.
3. **Use Claude Code for decomposition and consolidation** — `event-flow-mapper` explains workflows; `ecs-migration-planner` proposes simplification options. Architecture review is required before migration decisions.

### First pilot candidates

Pick workflows that are painful enough to matter but bounded enough to map.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| Lambda chain with recurring operational incidents | Tests whether Claude Code can connect code, AWS state, and incident evidence | Event-flow map + complexity report |
| Overgrown GitHub Actions workflow | Proves CI simplification without changing product behavior | Simplified workflow PR |
| Long-running or stateful serverless process | Tests ECS consolidation decision quality | Keep / consolidate / retire decision record |

### Claude Code guardrail

Do not use Claude Code to generate **more** serverless orchestration unless the target state explicitly keeps that capability serverless. The `consolidation-recommender` subagent's system prompt biases toward consolidation precisely because the goal is simplification.
