> KEY POINTS:
> * Three concrete strategies for the three most common legacy landscapes, each implemented as a Claude Code workflow.
> * Subagents for monolithic .NET on TFS, Java microservices on GitLab, and overly complex Node.js serverless on GitHub.

<hr>

The use-case portfolio defines the **shared workflow library**. This chapter applies it to the three actual estates, because the same Claude Code strategy does not mean the same playbook everywhere. Each landscape has a different modernization risk, so each needs a different workflow and a different guardrail.

## Why one playbook is not enough

Agentic coding tools are general-purpose, but modernization is not. The right workflow depends on where the risk lives.

| Landscape | Main risk | What the agent should do first |
| --- | --- | --- |
| **.NET monolith + SQL Server + TFS** | Hidden business behavior, stored procedures, weak tests | Read, map, explain, and generate characterization tests before changing behavior |
| **Java microservices + Kafka + GitLab** | Inconsistent service standards and event governance | Score services, document topics, review contracts, and align templates |
| **Node serverless + GitHub** | Operational complexity across functions, queues, IAM, and workflows | Map event flows, count complexity, and recommend keep/consolidate/retire decisions |

For engineers used to conventional modernization programs, the agent does not change the strategy. It changes the **cost of doing the careful work**: reading old code, mapping dependencies, generating first-pass tests, producing dossiers, and checking standards repeatedly.

## A. Monolithic .NET/C# + SQL Server + stored procedures + TFS

### Strategic stance

Treat this as the **highest-value Claude Code modernization target** — but not as a place to build new capability.

The main risks are **hidden business rules**, **database coupling**, stored procedure logic, weak automated tests, and old repository/workflow constraints. Claude Code earns its keep here by reading code and stored procs at a scale that no engineer would attempt by hand.

### Claude Code workflow

Start with these subagents (see [Claude Code Enablement][enablement]):

* `legacy-explorer` — walks C# modules and produces a capability map
* `stored-proc-summarizer` — plain-language summary of stored procedures + tables touched
* `characterization-test-writer` — generates regression tests around critical flows
* `extraction-planner` — proposes service-extraction options + ADR draft
* `wrapper-generator` — drafts API wrappers around stable legacy capabilities

Required MCP servers:

* DB schema MCP server (read-only against SQL Server) — lets subagents enumerate tables, dependencies, and stored proc metadata
* TFS MCP server (read-only) — enough to support the migration to Git, no more

A typical session: an engineer asks Claude Code "summarize the order-fulfillment module and identify a candidate for first extraction." The main agent spawns `legacy-explorer` to map the module, `stored-proc-summarizer` (in parallel) on each stored proc, then `extraction-planner` once both return. The result is a draft ADR plus a list of stored procs that need characterization tests before any change.

### Recommended approach

1. **Inventory and understand** — Use `legacy-explorer` and `stored-proc-summarizer` to produce a legacy capability map. Save it as living documentation in the repo.
2. **Add tests before changing** — `characterization-test-writer` generates regression and characterization tests around critical business flows. Prioritize stored procedures and high-change areas.
3. **Wrap first, extract second** — `wrapper-generator` proposes stable APIs or event adapters over selected legacy capabilities. Avoid direct database coupling from new services.
4. **Extract by business capability** — `extraction-planner` proposes extraction options; architects and domain experts decide final boundaries.
5. **Retire TFS gradually** — Migrate active code to Git-based repositories. Do not invest in further TFS MCP work beyond migration support.

The first inventory must include more than C# source files. It should explicitly map **stored procedures, triggers, scheduled jobs, reports, cross-database dependencies, tables, callers, and operational runbooks**. These are often where the real business behavior lives.

For every proposed extraction, require a **modernization dossier**. This is the **canonical definition** referenced by the rest of the strategy; other chapters compose subsets of these fields depending on the workflow.

<a id="modernization-dossier"></a>

| Dossier item | Why it matters |
| --- | --- |
| **Business capability summary** | Confirms what behavior is actually being moved |
| **Source references** | Links the summary to C# files, stored procedures, tables, jobs, and callers |
| **Dependency map** | Inbound and outbound callers, downstream consumers, reports, and integrations |
| **Data ownership assessment** | Identifies tables and write paths that must not be split casually |
| **Transaction boundary** | Shows where consistency must be preserved |
| **Data migration plan** | Historical data, dual-write window, reconciliation strategy, reporting impact |
| **Characterization tests** | Captures current behavior before replacement |
| **Parallel-run or rollback plan** | Keeps extraction reversible |
| **Domain-owner review** | Prevents AI summaries from becoming unverified business truth |
| **ADR draft** | The decision being recommended, with context, alternatives, and review date |

### First pilot candidates

Pick one flow where business value is high but the first step can be read-only or test-only.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| Stored-procedure-heavy business flow | Tests Claude Code's ability to explain database-embedded logic | Stored procedure summary pack + dependency map |
| Critical user journey with weak tests | Builds confidence before any extraction | Characterization test pack |
| Stable capability suitable for wrapping | Proves wrap-first modernization | Wrapper API draft + extraction ADR |

### Claude Code guardrail

Run these subagents in **`plan` permission mode** by default. Do not allow Claude Code to rewrite stored procedures or business-critical logic without strong regression tests already in place and a domain-expert review on the PR. The `pr-review-helper` subagent enforces this — it flags any PR that modifies a stored proc without a paired test file.

### Data migration discipline

Most extractions fail not at the code-rewrite step but at the **data migration step**. The pattern that has worked elsewhere and should be the default here:

1. **Inventory the data first.** A `data-migration-planner` subagent (or the same `legacy-explorer` with a data-migration prompt) walks every table, column, foreign key, trigger, scheduled job, report, and external consumer touched by the candidate capability.
2. **Pick a migration mode per table.** Options: copy-on-cutover (small reference tables), dual-write window with reconciliation (the common case for transactional tables), CDC streaming (for high-volume tables), one-time migration with a freeze (only for tables with low write volume).
3. **Plan reconciliation up front.** Define the reconciliation query, run it against the existing data before cutover, and keep running it during the dual-write window. Reconciliation drift is the earliest signal that the migration is unsafe.
4. **Treat reports and external consumers as first-class.** Reporting queries and downstream integrations often pin business behaviour by accident; identify them in the dossier.
5. **Plan the rollback.** Every migration has a documented step to roll back the dual-write or revert the CDC stream, with a clear cut-off time after which rollback is no longer attempted.

The data-migration plan is a required field of the modernization dossier; the dossier is not complete without it.

### TFS-to-Git migration

TFS is a source to exit, but the exit needs to be planned, not improvised:

1. **Active vs archive.** Decide per repository whether it migrates or is archived in TFS read-only. Default: active repositories migrate; abandoned ones archive.
2. **History strategy.** Preserve history where it has audit or licensing value; squash where it does not. Document the choice per repository.
3. **Branch and tag mapping.** Pick the target Git host (GitHub or GitLab, aligned with the broader repository strategy) before migration so branch protections and CI bindings can move at the same time.
4. **CI cutover.** The TFS build definition is rebuilt as a GitHub Actions / GitLab CI workflow at the same time, not after.
5. **Reference rewriting.** Cross-repo links, ticket links, and embedded build URLs are rewritten by a one-time migration script.
6. **Read-only TFS for six months.** Keep TFS readable but not writable after migration to catch overlooked references.

AI investment in TFS is scoped to this migration: a read-only `tfs-inventory` subagent that lists changesets, work items, and build definitions, and a one-time migration helper. No further investment in TFS tooling.

## B. Java microservices + self-managed Kafka + GitLab

### Strategic stance

This estate is conceptually closer to the target but **operationally misaligned** if Kafka is self-managed and service standards vary.

The objective is not "rewrite everything," but **standardize, simplify, and migrate operational responsibility to AWS patterns** — and Claude Code makes the standardization step feasible at the per-service level instead of being a multi-quarter platform program.

### Claude Code workflow

Use these subagents:

* `service-scorecard-builder` — scores each service against the standard template
* `kafka-contract-reviewer` — reviews Kafka changes against org standards
* `kafka-topic-documenter` — generates AsyncAPI specs from existing producers/consumers
* `consumer-error-handling-fixer` — proposes idempotency and DLQ patterns where missing
* `template-aligner` — refactors a service toward the standard Java template
* `msk-migration-planner` — drafts the migration plan from self-managed Kafka to MSK

Required MCP servers:

* GitLab MCP server (official)
* Kafka / Schema Registry MCP server — lets subagents check actual registered schemas, not the ones in the diff
* Service catalog MCP server — service ownership and dependency lookup
* AWS MCP server — for MSK target state

A typical session: the platform team runs `service-scorecard-builder` across all services overnight (Claude Code in headless mode, results pushed to the service catalog). The next morning, every service has a **scorecard for build health, test coverage, dependency age, observability maturity, Kafka usage, and operational risk**. Teams below threshold get a Jira ticket auto-created with a `template-aligner` plan attached.

### Recommended approach

1. **Create a Kafka governance baseline** — encoded in the `kafka-contract-reviewer` subagent's system prompt. Topic naming, ownership, schema strategy, retention rules, compatibility rules, dead-letter handling, consumer group conventions, replay strategy, idempotency, and schema review rules.
2. **Generate service scorecards** — `service-scorecard-builder` runs continuously. The platform team's dashboard is built from its output.
3. **Standardize service templates** — `template-aligner` refactors one service at a time toward the Java base image, logging, tracing, metrics, health endpoints, Kafka client configuration, retry and idempotency patterns.
4. **Move Kafka operations toward Amazon MSK** — `msk-migration-planner` drafts service-by-service migration plans. Amazon MSK is a fully managed service for running applications that use Apache Kafka and provides control-plane operations for creating, updating, and deleting clusters while preserving Apache Kafka data-plane APIs. ([AWS Documentation][5])

The Kafka baseline should make these ownership questions explicit:

| Question | Required answer |
| --- | --- |
| **Who owns the topic?** | Owning domain and service team |
| **Who owns each schema?** | Schema owner and compatibility policy |
| **Who consumes the event?** | Consumer groups, service owners, and support contacts |
| **Can the event be replayed?** | Replay rules, retention, idempotency requirement, and operational runbook |
| **What happens on failure?** | Retry strategy, dead-letter topic, alert owner, and recovery procedure |
| **How is compatibility tested?** | Contract tests and Schema Registry checks in CI |

### First pilot candidates

Pick services that represent common patterns rather than edge cases.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| High-volume Kafka consumer | Exposes error handling, retry, idempotency, and DLQ quality | Consumer review + remediation plan |
| Service with inconsistent template usage | Tests service-scorecard and template-alignment workflow | Service scorecard + template alignment PR |
| Important topic family | Tests event ownership and schema governance | AsyncAPI draft + ownership map |

### Claude Code guardrail

Do not let Claude Code invent event contracts. **Event semantics are architecture decisions**, not code-generation details. The `kafka-contract-reviewer` subagent's system prompt is explicit: it reviews proposed contracts against existing standards but does not author new ones without human-authored input.

### Worked example: one event family migrated end-to-end

A representative sketch of what the Kafka workflow produces for a single event family migrating from self-managed Kafka to MSK:

1. **Inventory.** `kafka-topic-documenter` lists every producer and consumer of `orders.events.v1`, reads the registered schema, and notes the actual consumer-group offsets and lag.
2. **Schema bump (BACKWARD).** The owning team authors `orders.events.v2` adding a non-required field. `kafka-contract-reviewer` checks compatibility against `v1` consumers in the Schema Registry, not against the diff alone.
3. **Producer cutover.** Producers move to `v2`. The producer wrapper writes both topics during the dual-publish window if the schema change is breaking; otherwise, just `v2` with the registry handling compatibility.
4. **Consumer migration.** Each consumer team gets an auto-generated PR from `template-aligner` that pins the new schema version and updates the deserialiser. The PR includes a contract test against the new schema.
5. **Replay window.** Replay rules are documented in the topic README, with the retention horizon and any idempotency requirements named.
6. **DLQ wiring.** `<topic>.dlq` is created or verified, alerts are wired, and the runbook is updated.
7. **MSK cutover.** `msk-migration-planner` produces the service-by-service cutover order, dependencies, rollback plan, and reconciliation queries.

Each step produces an artefact in the repo (a PR, an ADR, an updated runbook) — none of it lives in chat.

## C. Node.js serverless stack + GitHub

### Strategic stance

This is modern but overly complex. The goal is **simplification**, not more automation.

Claude Code helps identify where serverless workflows, glue code, deployment pipelines, permissions, and operational handoffs have become too complex — and proposes consolidation toward ECS where appropriate.

### Claude Code workflow

Use these subagents:

* `serverless-complexity-reporter` — summarizes a serverless workflow's surface area and dependencies
* `event-flow-mapper` — produces a diagram of an event flow across Lambda functions, SQS, EventBridge, etc.
* `consolidation-recommender` — classifies functions into "keep serverless" / "consolidate to ECS" / "retire"
* `ci-simplifier` — proposes simplifications to GitHub Actions workflows
* `ecs-migration-planner` — for functions classified as "consolidate," produces a migration plan

Required MCP servers:

* GitHub MCP server (official)
* AWS MCP server — to inspect actual Lambda configs, IAM, EventBridge rules
* Service catalog MCP server

A typical session: an engineer asks "this Lambda is being paged on every deploy — should it stay serverless?" Claude Code spawns `serverless-complexity-reporter` and `event-flow-mapper` in parallel against the Lambda's repo + AWS state, then runs `consolidation-recommender` over both reports. The output is a **recommendation with evidence**, not a guess.

### Recommended approach

1. **Classify serverless components** — `consolidation-recommender` does the first pass. Keep as serverless where event-driven and simple. Migrate to ECS where workflows are long-running, operationally complex, or better modeled as services. Retire redundant glue code.
2. **Reduce workflow complexity** — `ci-simplifier` simplifies GitHub Actions, standardizes deployment environments, removes special-case pipelines. Align observability with target platform.
3. **Use Claude Code for decomposition and consolidation** — `event-flow-mapper` explains workflows; `ecs-migration-planner` proposes simplification options. Architecture review is required before migration decisions.

Use explicit criteria before moving a function to ECS. The goal is **simplification**, not containerization for its own sake.

| Keep serverless when | Move to ECS when |
| --- | --- |
| The function is small, isolated, and event-driven | The workflow has many steps, states, retries, or handoffs |
| Traffic is spiky or infrequent | The workload behaves like a long-running service |
| Managed integration removes code | Local debugging, deployment, and observability are painful |
| Operational ownership is clear | IAM, queues, state machines, and alerts obscure ownership |
| The workflow is easy to explain | The workflow needs a diagram before an engineer can reason about it |

The complexity report should count **functions, queues, EventBridge rules, state machines, IAM policies, deployment steps, alarms, retries, and runbooks**. If those numbers are high, the issue is likely operating-model complexity, not language or runtime choice.

### First pilot candidates

Pick workflows that are painful enough to matter but bounded enough to map.

| Candidate | Why it is useful | Expected artifact |
| --- | --- | --- |
| Lambda chain with recurring operational incidents | Tests whether Claude Code can connect code, AWS state, and incident evidence | Event-flow map + complexity report |
| Overgrown GitHub Actions workflow | Proves CI simplification without changing product behavior | Simplified workflow PR |
| Long-running or stateful serverless process | Tests ECS consolidation decision quality | Keep / consolidate / retire decision record |

### Claude Code guardrail

Do not use Claude Code to generate **more** serverless orchestration unless the target state explicitly keeps that capability serverless. The `consolidation-recommender` subagent's system prompt biases toward consolidation precisely because the **goal is simplification**.

### Worked example: CI simplification

The `ci-simplifier` subagent's expected kind of output, sketched in before/after form:

| Before | After |
| --- | --- |
| 6 GitHub Actions workflows per service (build / test / lint / deploy-dev / deploy-stage / deploy-prod) | 1 reusable workflow with matrix environments |
| Custom shell steps for SBOM, license check, and dependency scan | Standard composite actions from the org's shared action repository |
| Per-service `node_modules` cache keys | Standard cache key derived from `package-lock.json` hash |
| Environment-specific secrets duplicated in three places | OIDC federation to AWS, no long-lived secrets |
| Deployment via `npm run deploy:prod` (calls custom CDK wrapper) | Deployment via the standard `deploy` composite action |
| 230 lines of YAML per service | ~40 lines of YAML referencing the reusable workflow |

The win is not the line count; it is that the simplified pipeline is the same across services, so an engineer who has read one understands all of them.
