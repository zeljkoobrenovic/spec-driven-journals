This post turns the primitives into an operating model. The point is to show where Claude Code sits in the engineering system: not beside the platform, but as one surface through which teams use the platform's standards, templates, knowledge, and controls.

The recommended model has four layers. Claude Code is the consistent developer surface across all of them.

## Layer 1: Claude Code as the developer surface

Claude Code is the default AI developer tool, used inside the IDE, the terminal, and CI for day-to-day work:

* explain legacy code
* generate unit and characterization tests
* suggest refactorings
* write boilerplate
* create Dockerfiles and ECS task definitions
* assist with Kafka producer/consumer code
* generate migration scaffolding
* improve documentation
* support pull request reviews
* drive multi-step modernization tasks via subagents

The surface is uniform: every engineer's Claude Code reads the same `CLAUDE.md`, calls the same MCP servers, and has access to the same shared subagents. That uniformity — not the raw model capability — is the strategic property.

Other tools (Amazon Q Developer for AWS-console operations, GitHub Copilot for niche autocomplete cases) are allowed by exception. See [Tooling Decision][td] for the criteria.

## Layer 2: AI-assisted modernization factory

The modernization factory is the org's library of **subagents** that encode repeatable workflows, composing **MCP servers** (for stable third-party systems) and **Skills** (for our own code and conventions). See [MCP, Skills, and Subagents][mcps] for the implementation detail.

| Modernization task        | Claude Code subagent / MCP                                       | Human role                            |
| ------------------------- | ---------------------------------------------------------------- | ------------------------------------- |
| Legacy code understanding | `legacy-explorer` + repo MCP server                              | Validate business meaning             |
| Stored proc analysis      | `stored-proc-summarizer` + DB-schema MCP server                  | Confirm domain interpretation         |
| Test creation             | `characterization-test-writer` reads code, generates regression tests | Approve scenarios and edge cases  |
| Service extraction        | `extraction-planner` proposes seams; produces ADR draft          | Decide service boundaries             |
| Data migration            | Subagent + DB-schema MCP server walks dependencies                | Own data model and migration plan     |
| Kafka migration           | `kafka-contract-reviewer` + Schema Registry MCP server            | Approve event semantics and ownership |
| Containerization          | `ecs-template-generator` + AWS MCP server                         | Validate runtime, security, scaling   |
| CI/CD standardization     | Subagent generates pipeline from approved template                | Own release governance                |
| Documentation             | Subagent generates ADRs, API docs, runbooks                       | Review and publish                    |

This is what "modernization factory" means concretely: a versioned, reviewable, distributable set of subagents — not a slide deck of AI use cases.

## Layer 3: Platform engineering with Claude Code-ready golden paths

The internal developer platform gives Claude Code and engineers the same paved roads. The platform's job is to make the right thing the easy thing — for both.

The platform should include:

* standard ECS service template (referenced by `ecs-template-generator`)
* standard Java service template
* standard Node.js service template
* optional .NET modernization template
* Kafka producer/consumer libraries (enforced by `kafka-contract-reviewer`)
* shared observability package
* shared security and secrets handling
* CI/CD templates
* infrastructure-as-code modules
* service catalog (exposed via MCP server)
* architecture decision record templates
* runbook templates
* automated quality gates
* the org-level `CLAUDE.md` distribution mechanism
* the org's MCP server registry

Amazon ECS supports containerized workloads on AWS; Amazon MSK provides managed Apache Kafka, reducing the need for teams to become Kafka infrastructure specialists. ([Amazon Web Services, Inc.][3])

The platform team's deliverables now include subagents and MCP servers alongside Helm charts and Terraform modules. Treat them with the same product discipline.

## Layer 4: AI governance and quality controls

Governance for Claude Code is mostly *configuration*, not policy. The mechanisms:

* **Approved tool allowlist** in the org-level `settings.json` distributed to every engineer
* **Permission modes** — `default`, `acceptEdits`, `plan` — chosen per workflow class
* **Hooks** — pre/post tool execution scripts that run security scans, audit logging, secrets checks
* **MCP server registry** — only registered servers may appear in `settings.json`
* **Subagent registry** — only reviewed subagents may be distributed at the org level
* **Audit trail** — Claude Code session logs captured to a central store
* **Required tests for AI-generated code** (enforced in CI, not by trust)
* **Pull request checklist** — including a "Claude Code session id" field for traceability
* **Production approval rules** — unchanged: humans approve prod changes

The AWS Well-Architected Framework remains the backbone for what "good" looks like for a workload, organising quality around operational excellence, security, reliability, performance efficiency, cost optimization, and sustainability. ([AWS Documentation][4])

The shift Claude Code enables: governance is **executable**. A `settings.json` and a hook script either run or they don't. A policy PDF that says "engineers should not put secrets in prompts" is wishful. A pre-prompt hook that scans for secrets and refuses to send them is enforcement.
