> * A four-layer model with Claude Code as the developer surface.
> * The assistant layer, a modernization factory of subagents, Skills, and MCP servers, platform engineering with golden paths, and AI governance and quality controls.

<hr>

The previous chapters chose the tool and defined its building blocks. This chapter turns those pieces into an **engineering model**, showing where Claude Code sits in the system: not beside the platform, but as one surface through which teams use the platform's standards, templates, knowledge, and controls.

The recommended model has four layers. Claude Code is the **consistent developer surface** across all of them.

Think of this model as an internal developer platform with an agentic interface. Platform engineering still owns templates, CI/CD, observability, security, and service catalog. The difference is that engineers increasingly reach those platform capabilities through an agent that can read the repo, call approved tools, generate a PR, and run checks.

The important design rule: **the agent does not replace the platform**. It makes the platform easier to use.

## Layer 1: Claude Code as the developer surface

Claude Code is the default AI developer tool, used inside the IDE, the terminal, and CI for day-to-day work:

* explain legacy code
* generate unit and characterization tests
* suggest refactorings
* write boilerplate
* create Dockerfiles and ECS task definitions
* assist with Kafka producer/consumer code
* generate migration scaffolding
* improve documentation
* support pull request reviews
* drive multi-step modernization tasks via subagents

The surface is uniform: every engineer's Claude Code reads the same `CLAUDE.md`, can load the same Skills, calls the same MCP servers, and has access to the same shared subagents. That **uniformity** — not the raw model capability — is the strategic property.

Other tools (Amazon Q Developer for AWS-console operations, GitHub Copilot for niche autocomplete cases) are allowed by exception. See [Tooling Decision][td] for the criteria.

## Layer 2: AI-assisted modernization factory

The modernization factory is the org's library of **subagents** that encode repeatable workflows, composing **MCP servers** (for stable third-party systems) and **Skills** (for our own code and conventions). See [Claude Code Enablement][enablement] for the implementation detail.

| Modernization task        | Workflow assets                                                  | Human role                            |
| ------------------------- | ---------------------------------------------------------------- | ------------------------------------- |
| Legacy code understanding | `legacy-explorer` + repo MCP server                              | Validate business meaning             |
| Stored proc analysis      | `stored-proc-summarizer` + DB-schema MCP server                  | Confirm domain interpretation         |
| Test creation             | `characterization-test-writer` reads code, generates regression tests | Approve scenarios and edge cases  |
| Service extraction        | `extraction-planner` proposes extraction options; produces ADR draft | Decide service boundaries             |
| Data migration            | Subagent + DB-schema MCP server walks dependencies                | Own data model and migration plan     |
| Kafka migration           | `kafka-contract-reviewer` + Schema Registry MCP server            | Approve event semantics and ownership |
| Containerization          | `ecs-template-generator` + AWS MCP server                         | Validate runtime, security, scaling   |
| CI/CD standardization     | Subagent generates pipeline from approved template                | Own release governance                |
| Documentation             | Subagent generates ADRs, API docs, runbooks                       | Review and publish                    |

This is what "modernization factory" means concretely: a **versioned, reviewable, distributable set of subagents** — not a slide deck of AI use cases.

For an engineer, the factory should feel like a menu of known jobs:

* "map this legacy capability";
* "summarize these stored procedures";
* "generate characterization tests";
* "review this Kafka change";
* "create a new ECS service from the template";
* "explain why this serverless workflow is operationally complex."

Each job has a known output, owner, and review path. That is the difference between a modernization factory and an enthusiastic collection of prompts.

## Layer 3: Platform engineering with Claude Code-ready golden paths

The internal developer platform should give engineers and Claude Code the same **approved paths**. Its job is to make the right thing the easiest thing to do.

The platform should include:

* standard ECS service template (referenced by `ecs-template-generator`)
* standard Java service template
* standard Node.js service template
* optional .NET modernization template
* Kafka producer/consumer libraries (enforced by `kafka-contract-reviewer`)
* shared observability package
* shared security and secrets handling
* CI/CD templates
* infrastructure-as-code modules
* service catalog (exposed via MCP server)
* architecture decision record templates
* runbook templates
* automated quality gates
* the org-level `CLAUDE.md` distribution mechanism
* the org's MCP server, Skill, and subagent registries

Amazon ECS supports containerized workloads on AWS; Amazon MSK provides managed Apache Kafka, reducing the need for teams to become Kafka infrastructure specialists. ([Amazon Web Services, Inc.][3])

The platform team's deliverables now include **Skills, subagents, and MCP servers** alongside Helm charts and Terraform modules. Treat them with the same product discipline.

### Target architecture guardrails

The target architecture is a **default path**, not a rule that every workload must be forced into the same shape.

| Choice | Default | Exceptions |
| --- | --- | --- |
| **Runtime** | ECS for long-running services and standardized workloads | Lambda for small, isolated, event-driven tasks where serverless is simpler |
| **Messaging** | Kafka for durable asynchronous business events | Direct APIs for request/response interactions that do not need replay or independent consumer evolution |
| **Kafka operations** | Managed Kafka patterns, preferably Amazon MSK | Self-managed Kafka only with a time-bound exception and migration plan |
| **Service creation** | Approved service templates generated through Claude Code workflows | Custom templates require platform approval and an ADR |
| **Infrastructure** | Infrastructure as code and standard deployment pipelines | Manual console changes only for emergency response, followed by codified remediation |

Claude Code should reinforce these guardrails. It should not turn every local preference into a new pattern. When the right answer is "stay serverless," "use a direct API," or "do not extract yet," the workflow should say so and record the evidence.

### Platform readiness checklist

Agentic workflows will fail if the target platform is under-specified. Before scaling beyond pilots, the platform should have:

| Platform asset | Why the agent needs it |
| --- | --- |
| **ECS service template** | So generated services start from the same runtime, logging, health, and deployment pattern |
| **Kafka standards and libraries** | So event producers and consumers are reviewed against a consistent contract model |
| **Schema Registry access** | So contract review checks current registered schemas, not assumptions |
| **Service catalog** | So ownership, dependencies, and runtime metadata are queryable |
| **CI/CD templates** | So generated changes verify through the same commands everywhere |
| **Observability baseline** | So new services ship with logs, metrics, traces, alerts, and runbooks |
| **Security scanning and secret handling** | So generated code is subject to the same controls as human-written code |
| **ADR and runbook templates** | So modernization outputs become durable engineering artifacts |

## Layer 4: AI governance and quality controls

Governance for Claude Code is mostly **configuration, not policy**. The mechanisms:

* **Approved tool allowlist** in the org-level `settings.json` distributed to every engineer
* **Permission modes** — `default`, `acceptEdits`, `plan` — chosen per workflow class
* **Hooks** — pre- and post-tool execution scripts that run security scans, audit logging, secrets checks
* **MCP server registry** — only registered servers may appear in `settings.json`
* **Skill and subagent registries** — only reviewed shared Skills and subagents may be distributed at the org level
* **Audit trail** — Claude Code session logs captured to a central store
* **Required tests for AI-generated code** (enforced in CI, not by trust)
* **Pull request checklist** — including a "Claude Code session id" field for traceability
* **Production approval rules** — unchanged: humans approve prod changes

The AWS Well-Architected Framework remains the backbone for what "good" looks like for a workload, organizing quality around operational excellence, security, reliability, performance efficiency, cost optimization, and sustainability. ([AWS Documentation][4])

The shift Claude Code enables: governance is **executable**. A `settings.json` and a hook script either run or they do not. A policy PDF that says "engineers should not put secrets in prompts" is guidance. A pre-prompt hook that scans for secrets and refuses to send them is enforcement.
