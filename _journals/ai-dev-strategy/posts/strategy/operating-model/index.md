<hr>

After defining the landscape-specific work, the strategy needs **ownership**. Claude Code changes the engineering workflow, but it does not remove accountability. The operating model defines who owns the platform capabilities, who approves risk, and how service teams use AI without turning every repository into a separate experiment.

## What Changes When Agents Can Act

Many engineers already use AI as a **conversation partner**: ask a question, get an answer, decide what to do. Agentic coding changes the operating model because the tool can also **read repository context, call approved tools, edit files, run tests, and produce evidence**. That makes AI behavior part of the engineering system, not only part of an individual's workflow.

| Previous assumption | Agentic-coding implication |
| --- | --- |
| Prompt quality is a personal productivity habit. | Reusable instructions become **platform assets** and need owners, review, and versioning. |
| Documentation explains standards to humans. | `CLAUDE.md`, Skills, hooks, and templates also explain standards to **agents that act on the codebase**. |
| Tool access is mostly an IDE or CLI concern. | MCP servers, shell permissions, and hooks become **runtime control points**. |
| Review happens after code is written. | Standards should be encoded early enough that the agent **generates compliant work by default**. |
| A generated artifact is the developer's responsibility. | Responsibility stays with the service team, but the reusable workflow that produced it needs a **named owner**. |

The practical rule is simple: **if an artifact changes agent behavior, manage it like engineering infrastructure**. It needs a source of truth, an owner, a review path, and a retirement path.

## Core Principle

AI Engineering Enablement is **not a separate delivery team** that modernizes systems on behalf of everyone else.

It is an enablement function that builds **reusable assets**, sets guardrails, coaches teams, and measures whether modernization is getting safer and faster.

**Service teams still own their code.** Architects still own architecture decisions. Security still owns risk policy. Platform still owns the golden paths.

## AI Engineering Enablement Group

Throughout the strategy, this group is referred to by the short form **"AI Engineering Enablement"** in tables and RACI rows, and the full form **"AI Engineering Enablement Group"** in the operating-model chapter and forum descriptions. Both refer to the same body.

Create a small cross-functional group with representatives from:

* platform engineering
* architecture
* security
* developer experience
* QA and test automation
* cloud operations
* .NET legacy modernization
* Java/Kafka services
* Node.js/serverless services

This group should be **small enough to move quickly** and broad enough to avoid building a strategy that works only for one estate.

## Capacity and Funding

AI Engineering Enablement needs **dedicated capacity**. It should not be an informal side responsibility added to already-full platform and architecture backlogs.

Minimum standing capacity for the first year:

| Role | Focus |
| --- | --- |
| **Platform engineer** | `CLAUDE.md` baseline, templates, hooks, settings, registries, CI integration |
| **Security engineer** | data policy, tool approval, MCP risk review, audit and incident response |
| **Architecture lead** | service boundaries, Kafka governance, target-state exceptions |
| **QA/test automation lead** | characterization tests, workflow evaluations, regression strategy |
| **DevEx owner** | onboarding, documentation, training, developer feedback |
| **Landscape representatives** | .NET/TFS, Java/Kafka, and Node/serverless pilots |

The group should maintain a **visible backlog**. Requests for new Skills, MCP servers, subagents, hooks, and templates are prioritized by modernization value, risk reduction, reuse potential, and platform readiness.

### Backlog cadence

* **Source of truth.** The backlog lives in the same issue tracker used by the platform team (the strategy does not pick one — Jira, Linear, or a GitHub project all work). The board is public to every engineer.
* **Intake.** Any engineer or team can open a request through a standard template (problem, expected workflow, current workaround, affected estates).
* **Grooming.** The Enablement forum grooms the backlog weekly. Each item is scored on the four prioritisation criteria above and assigned a current quarter / next quarter / later band.
* **Promotion to delivery.** Items in the current quarter band move into normal sprint planning of the platform team; capacity is reserved (the four prioritisation criteria do not override capacity discipline).

### Funding model

The capacity sketch above is a placeholder until funded. The decision the executive sponsor is asked to make: **does AI enablement come out of the platform budget, a separate modernization budget, or a cross-charge to landscape teams?** The strategy recommends a **dedicated platform-budget line** for the first year so that prioritisation is centralised; landscape-specific work (pilots, scorecard rollout) can be cross-charged from the relevant estate.

## RACI for Core Artifacts

The strategy works only if each artifact has clear accountability.

| Artifact | Responsible | Accountable | Consulted | Informed |
| --- | --- | --- | --- | --- |
| Org AI policy | Security + Platform | Engineering leadership | Legal, Architecture | All engineers |
| Org `CLAUDE.md` baseline | Platform | Platform leader | Security, DevEx | Service teams |
| Project `CLAUDE.md` | Service team | Service owner | Platform | Reviewers |
| Shared Skills | Platform or domain owner | Named Skill owner | Teams using the Skill | AI Engineering Enablement |
| MCP servers | Platform or integration owner | Platform + Security | Data owner, Legal where needed | Users |
| Shared subagents | Workflow owner | Platform or domain lead | Security for risky tools | Service teams |
| Service templates | Platform | Platform leader | Architecture, Cloud Operations | All engineers |
| Modernization dossiers | Service team | Product/engineering owner | Domain experts, Architecture | Stakeholders |
| Tooling default | AI Engineering Enablement | Engineering leadership | Security, Procurement | All engineers |

## Governance Forums

Use **three forums**. Keep them separate so operational decisions do not get stuck in risk review, and risk decisions do not get buried in platform backlog grooming.

| Forum | Purpose | Frequency |
| --- | --- | --- |
| AI Engineering Enablement | Standards, roadmap, subagents, Skills, adoption, metrics | Weekly |
| Architecture Review | Service boundaries, extraction candidates, Kafka contracts, target-state exceptions | Weekly or biweekly |
| Risk and Compliance Review | Data handling, tool approval, vendor risk, licensing, audit, regulated use cases | Monthly or as needed |

## Decision Rights

Make **decision rights** explicit.

| Decision | Decision owner |
| --- | --- |
| Claude Code remains the default tool | AI Engineering Enablement, reviewed every six months |
| Alternative AI tool exception | AI Engineering Enablement + Security |
| New MCP server in org registry | Platform + Security |
| New shared Skill | Platform, with owning service team if domain-specific |
| New shared subagent | Platform, with Architecture or Security review if it affects standards or risk |
| New service boundary | Architecture Review |
| New Kafka event contract | Architecture Review + owning domain team |
| Production deployment | Existing production approval process |
| Modification of business-critical legacy logic | Owning service team + domain expert + required regression tests |

## How Work Flows

A normal AI-assisted modernization request should leave a trail that engineers, reviewers, architects, and auditors can inspect later.

| Step | Human decision | Agentic contribution | Durable artifact |
| --- | --- | --- | --- |
| **Frame the work** | What problem are we solving, and why now? | Summarize current repo context and likely dependencies | Issue, modernization candidate, or ADR stub |
| **Explore safely** | Which sources are authoritative? | Read code, schemas, tests, docs, service catalog, and approved MCP data | Source-linked current-state summary |
| **Choose target shape** | What architecture direction is acceptable? | Compare options against target standards | ADR and migration plan |
| **Make bounded changes** | Which files and behaviors are in scope? | Generate tests, docs, templates, or small code changes | PR with traceability |
| **Verify** | Is the result correct enough to proceed? | Run tests, summarize failures, list assumptions and gaps | Test report, review notes, checklist |
| **Promote learning** | Is this reusable? | Suggest Skill, subagent, template, hook, or library candidate | Backlog item for Platform |

### 1. A team identifies a modernization candidate

The **service team owns the candidate**. Claude Code may help produce a current-state summary, dependency map, test gaps, and first ADR draft.

Architecture Review decides whether the proposed target shape is acceptable.

### 2. The team uses shared workflows

The team invokes **approved subagents and Skills** instead of inventing a workflow from scratch. Typical examples:

* `legacy-explorer`
* `stored-proc-summarizer`
* `characterization-test-writer`
* `kafka-contract-reviewer`
* `ecs-template-generator`
* `pr-review-helper`

The output becomes **normal repository material**: tests, docs, ADRs, PR comments, scorecards, and migration plans.

### 3. Humans approve decisions

Claude Code can propose extraction options, event contract changes, service templates, and migration plans. **It does not approve them.**

Humans remain accountable for:

* architecture
* security
* licensing
* correctness
* domain semantics
* production readiness

### 4. Platform promotes reusable patterns

When a team repeats a workflow more than twice, Platform decides whether it belongs in:

* `CLAUDE.md`
* a Skill
* a subagent
* a template
* an MCP server
* a normal shared library

This is how one team's modernization learning becomes **a shared asset**.

## Team-Level Expectations

Each service team should maintain:

* a project `CLAUDE.md`
* repository ownership metadata
* standard build and test commands
* a runbook
* ADRs for major decisions
* a service catalog entry
* test strategy for risky changes
* explicit AI usage traceability on AI-assisted PRs

This is not documentation for its own sake. It is the **minimum context Claude Code needs** to produce useful, consistent work.

## Platform-Level Expectations

Platform should maintain:

* organization `CLAUDE.md`
* standard service templates
* approved Skills
* approved subagents
* MCP registry and setup guidance
* hook scripts
* CI quality gates
* policy-as-code examples
* scorecard definitions
* training and onboarding material

The platform product is no longer only infrastructure modules and CI templates. It also includes the **AI-facing instructions and workflows** that make those standards easy to follow.

## Operating Metrics

Each metric needs **baseline, owner, data source, target, and cadence**.

| Metric | Why it matters | Owner |
| --- | --- | --- |
| Repositories with valid `CLAUDE.md` | Measures context readiness | Platform |
| Shared Skills and subagents with owners and review dates | Measures reusable capability | Platform |
| Stale Skills, MCP servers, or subagents found in review | Measures maintenance risk | AI Engineering Enablement |
| AI-assisted PRs with traceability | Measures audit readiness | Security + Platform |
| Repeated standards comments in PRs | Measures whether standards are encoded early enough | DevEx |
| Modernization artifacts produced per candidate | Measures workflow completeness | Architecture |
| Exceptions to default tooling | Measures drift | AI Engineering Enablement |
| AI cost by team and workflow | Measures whether cost is visible and tied to value | Platform + Finance |
| Time to produce modernization dossier | Measures acceleration | AI Engineering Enablement |
| Characterization coverage on risky flows | Measures modernization safety | QA |

## Anti-Patterns

Each anti-pattern has a **detection signal** that the Enablement forum watches for. The pattern is not avoided by hoping it does not happen; it is avoided by acting on the signal.

| Anti-pattern | Detection signal |
| --- | --- |
| A central AI team generates code for service teams without ownership transfer | More than a small share of PRs on service-team repos authored by the enablement group, sustained across two months |
| Every team creates its own unreviewed subagents | More than a few private subagents discovered per repo scan, none registered |
| Broad tool autonomy before tests and standards are in place | `acceptEdits` mode usage rising without matching characterization coverage growth |
| Governance by wiki page without hooks, templates, or CI enforcement | Recurring PR comments about rules that are not enforced in CI |
| Treating `CLAUDE.md` as a long essay | Project `CLAUDE.md` files exceeding a few thousand tokens, or unchanged for more than a quarter while the code changes |
| Measuring "AI adoption" instead of modernization outcomes | Adoption dashboards rising while pilot outcome metrics stay flat |

The operating model is successful when service teams **move faster without losing ownership**, and platform standards become easier to follow than local variation.
