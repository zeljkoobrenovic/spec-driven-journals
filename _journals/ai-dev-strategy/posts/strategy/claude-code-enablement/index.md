> KEY POINTS:
> * Claude Code becomes useful at organizational scale only when it is surrounded by shared context, approved integrations, reusable task knowledge, repeatable workflows, and executable controls.
> * The durable assets are ordinary engineering artifacts: markdown instructions, scripts, templates, examples, settings, hooks, registries, and reviewed subagent definitions.
> * Enablement succeeds when a team can ask for a known workflow and get consistent, governed, reviewable behavior across repositories.

<hr>

This chapter answers one question: **what must Organization put around Claude Code so the default-tool decision becomes useful in daily engineering work?**

The answer is an enablement stack. Claude Code is the interface; the operating capability comes from the assets around it.

## The Enablement Stack

| Layer | Purpose | Durable artifact |
| --- | --- | --- |
| Repository instructions | Tell the agent how this repo works. | `CLAUDE.md`, `AGENTS.md`, README conventions. |
| Organizational knowledge | Teach repeatable internal patterns. | Skills, templates, examples, check scripts. |
| Approved system access | Read current state from trusted systems. | MCP servers and registry entries. |
| Repeatable workflows | Package common multi-step work. | Subagent definitions and workflow prompts. |
| Executable controls | Make policy enforceable. | Settings, hooks, CI checks, allowlists, audit logs. |
| Ownership | Keep the stack current. | Registry owners, review cadence, promotion rules. |

The stack should be small at first. A weak, outdated enablement library is worse than none because agents will confidently reuse bad instructions.

## Repository Instructions

Every serious repository should have a short instruction file that tells the agent:

* what the repository is;
* how to build, test, and verify;
* where important conventions live;
* what files or generated outputs require care;
* which commands are safe, expected, or prohibited;
* how to handle dirty worktrees and user changes.

Good repository instructions are operational, not motivational. They should help an agent avoid wasting time and avoid damaging work it did not create.

## Skills

Skills package Organization-specific knowledge that should travel across repositories: naming conventions, service templates, event design, ADR writing, modernization workflows, test generation, observability patterns, or security-review routines.

A Skill should include:

* a clear trigger: when should an agent use it;
* concise instructions;
* examples or templates;
* validation steps;
* ownership and review date.

Skills are promoted only when they have been used successfully in real work. Draft Skills stay local until there is evidence they improve consistency.

## MCP Servers

MCP servers connect the agent to approved systems of record. Use MCP for current state that should not be copied into prompts or static docs:

| System | Useful for |
| --- | --- |
| GitHub or GitLab | Repository state, pull requests, issues, and code search. |
| Jira | Work context, issue status, and delivery links. |
| AWS | Infrastructure inventory and approved operational queries. |
| Snyk or security tooling | Vulnerability and dependency context. |
| Schema Registry | Event contracts and compatibility checks. |
| Observability tools | Logs, traces, metrics, and incidents where approved. |

MCP access must be governed. Each server needs an owner, data classification, allowed operations, permission model, and audit trail.

## Subagents

Subagents package repeatable workflows. They are useful when the task has a stable sequence, such as:

* characterize a legacy module;
* prepare a modernization plan;
* review a pull request against internal standards;
* create an ADR from a discussion;
* validate an event-contract change;
* generate test scaffolding for an existing service.

A subagent should say what it owns, what evidence it must collect, what it must not change, and how its work is reviewed. Subagents should be treated like production automation: versioned, reviewed, and retired when they stop matching reality.

## Controls

The enablement stack needs controls because Claude Code can act:

| Control | Purpose |
| --- | --- |
| Permission modes | Limit what the agent can read, write, execute, or access. |
| Tool allowlists | Prevent uncontrolled shell, network, or system access. |
| Hooks | Block unsafe commands, missing tests, policy violations, or unreviewed generated files. |
| CI checks | Validate that generated outputs, specs, or templates remain consistent. |
| Audit trail | Preserve who asked for what, what changed, and what evidence was produced. |
| Registries | Make approved Skills, MCP servers, subagents, and templates discoverable. |

Controls should fail in a way engineers can understand. A hook that blocks work without a clear fix will be bypassed.

## Ownership

Enablement assets need product-like ownership:

| Asset | Owner |
| --- | --- |
| Repository instructions | Owning team. |
| Organization-level Skills | AI enablement or platform function, with domain reviewers. |
| MCP servers | System owner plus security review. |
| Subagents | Workflow owner plus engineering standards reviewer. |
| Hooks and CI controls | Platform, security, or standards owner. |
| Registries | Enablement function with curator support. |

Each asset should have a review cadence. Outdated instructions should be deleted or demoted before they become institutional folklore.

## Adoption Path

Start with a thin, proven stack:

1. Write repository instructions for pilot repos.
2. Create two or three Skills for the most repeated workflows.
3. Register only the MCP servers needed by pilots.
4. Add one or two hooks that protect real risks.
5. Review pilot sessions and promote only assets that demonstrably improved outcomes.
6. Expand by evidence, not enthusiasm.

The first goal is consistency in a small number of workflows. Breadth can come later.

## What Not To Do

* Do not put every policy into `CLAUDE.md`; long context files become unread.
* Do not create Skills without examples and validation.
* Do not expose MCP servers before access, data, and audit rules are clear.
* Do not use subagents to hide accountability.
* Do not rely on human memory for controls that can be executable.
* Do not keep stale enablement assets because they once worked.

## Success Measures

The stack is working when:

* engineers reuse the same workflow without re-explaining it each time;
* generated work is easier to review because evidence and structure are consistent;
* hooks catch real issues without excessive false positives;
* Skills and subagents are retired or improved based on pilot data;
* new repositories can adopt the default with limited local invention.
