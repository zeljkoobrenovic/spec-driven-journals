> KEY POINTS:
> * Claude Code becomes useful at organizational scale only when it has shared context, approved integrations, reusable task knowledge, repeatable workflows, and executable controls.
> * The durable assets are ordinary engineering artifacts that transfer well: Markdown instructions, scripts, templates, examples, settings, hooks, and reviewed subagent definitions.

<hr>

The previous chapters made two decisions. First, AI in software development should help us modernize a mixed legacy landscape instead of adding another layer of variation. Second, Claude Code should be the default tool because it gives us primitives we can build on.

This chapter answers the next question: **what do we need to put around Claude Code so that the decision becomes useful in daily engineering work?**

The answer is not two separate worlds called "context" and "capability." Engineers do not experience the tool that way. They experience **one working environment**: Claude Code reads the repository rules, understands the organization's patterns, reaches approved systems, runs repeatable workflows, and leaves an auditable trail.

That working environment is the **enablement stack**.

## The Enablement Stack as a System

For engineers new to agentic coding, the easiest way to understand the stack is to map each primitive to a familiar engineering concept.

| Agentic coding primitive | Similar engineering concept | What it does |
| --- | --- | --- |
| `CLAUDE.md` | Repository README plus build notes, but loaded automatically | Gives the agent the facts it must know every session |
| Skill | A small internal playbook with scripts and examples | Teaches the agent a task-specific process only when needed |
| MCP server | API client to a system of record | Lets the agent query current state instead of guessing |
| Subagent | A named CI job or runbook workflow | Encodes a repeated multi-step job with a clear output |
| Hook | Pre-commit hook or CI quality gate | Runs policy checks before or after agent actions |
| Registry | Internal developer-platform catalog | Defines what is approved, who owns it, and when it is reviewed |

The pieces are not valuable because they are AI-specific. They are valuable because they make agentic work **explicit, inspectable, and reusable**.

## The goal of enablement

The goal is to make Claude Code **useful, consistent, and governable** in this organization.

Useful means it can help with **the real work**: explaining legacy .NET and stored procedures, reviewing Kafka changes, aligning Java services to a target template, simplifying Node.js serverless workflows, and generating modernization scaffolding.

Consistent means the **same task is handled the same way across repositories**. A Kafka event review should not depend on which engineer remembered the best prompt. A stored procedure summary should follow the same structure in every legacy module. A new ECS service should use the same baseline every time.

Governable means the organization can decide **what Claude Code may read, what it may call, what it may edit, what must be reviewed**, and how shared instructions evolve.

The enablement stack answers five practical questions:

| Question | Enablement primitive | Purpose |
| --- | --- | --- |
| What standards and repository facts must Claude Code know every time? | `CLAUDE.md` and memory | Always-loaded context |
| How does it reuse our own task-specific conventions? | Skills | On-demand knowledge, scripts, templates, and examples |
| How does it reach current state in approved systems? | MCP servers | Grounding in external systems and live platform state |
| How do we make repeatable modernization workflows? | Subagents | Specialized workflows that compose context, Skills, and MCP |
| How do we enforce boundaries? | Settings, hooks, CI, and registries | Permissions, policy checks, audit, and quality gates |

The order matters less than the relationship. `CLAUDE.md` gives the baseline, Skills carry detailed organizational knowledge, MCP servers bring in current system state, subagents compose them into workflows, and **controls make the whole thing safe enough to scale**.

## How the pieces work together

Take a common modernization task: reviewing a Kafka producer change in a Java service.

Claude Code starts with the repository's `CLAUDE.md`, which says how the service is built, where Kafka code belongs, which producer wrapper is mandatory, and which tests must run before a PR is ready.

It then loads the `kafka-event-publishing` Skill because the task matches the Skill description. That Skill contains the organization's topic naming rules, schema compatibility expectations, contract-test pattern, and example payloads. The `kafka-contract-reviewer` subagent runs the review workflow, using the Schema Registry MCP server to fetch the actual registered schema instead of trusting the diff alone. Hooks and CI enforce the non-negotiables: no secrets, no unreviewed schema change, no merge without the required tests.

The same task as a step-by-step agent workflow:

| Step | What happens | Which primitive helps |
| --- | --- | --- |
| **1. Start with repository facts** | The agent learns the build command, test command, Kafka wrapper, and risky files | `CLAUDE.md` |
| **2. Load task knowledge** | The agent loads Kafka naming, schema, compatibility, and contract-test guidance | Skill |
| **3. Query current state** | The agent checks the actual registered schema and topic metadata | MCP server |
| **4. Run a repeatable review** | The agent follows the same review structure every time | Subagent |
| **5. Enforce boundaries** | Secret scanning, audit logging, CI checks, and permission rules run automatically | Hooks, settings, CI |
| **6. Produce evidence** | The output cites files, schemas, tests, assumptions, and required reviewers | Output contract |

No single primitive owns the whole task. The value comes from **composition**:

* `CLAUDE.md` prevents obvious repository mistakes.
* Skills package our conventions without loading every detail into every session.
* MCP servers keep the agent grounded in current state.
* Subagents turn recurring work into named workflows.
* Hooks and CI keep the workflow inside organizational boundaries.

That is the story to repeat across the modernization portfolio: not "prompt better," but **"turn recurring work into reusable, governed assets."**

## `CLAUDE.md`: the baseline every session needs

`CLAUDE.md` is the **first enablement file** most teams should create. It lives at the repository root and is loaded automatically when Claude Code works in that repository.

It should be **short and concrete**. A useful project `CLAUDE.md` answers the questions Claude Code would otherwise discover by trial and error:

* What is this repository responsible for?
* How do I build, test, and verify changes?
* Which local patterns are mandatory?
* Which files or areas require extra care?
* What looks wrong but is deliberate?

A practical structure:

```markdown
# Project: order-service

## What this is
Java 21 service that owns the orders capability. Publishes order events to
Kafka and runs on ECS.

## Build and verify
- `make build` compiles and runs unit tests.
- `make verify` adds integration and contract tests.
- Run `make verify` before declaring work done.

## Conventions
- Use `OrderEventProducer`; do not call the raw Kafka producer directly.
- Database migrations live in `db/migrations/`.
- Structured logging goes through `lib/logging`.

## Be careful
- Do not change event schemas without architecture review.
- The `legacy/` package is intentionally unrefactored during migration.
```

The organization also needs a shared `CLAUDE.md` baseline for cross-cutting rules: AI usage policy, secret-handling rules, required verification steps, standard make targets, pull-request expectations, and approved tool boundaries. User memory is a third layer and should remain personal: preferences, communication style, and working context for the individual engineer.

### Three layers of context

Claude Code has **three distinct places to put instructions**, and using each correctly is the difference between standards that work and standards engineers route around.

| Layer | File | What belongs here |
| --- | --- | --- |
| **Organization** | A shared `CLAUDE.md` distributed via template repo or sync | Cross-cutting rules: security policy, tool allowlist, definition of done |
| **Project** | `<repo-root>/CLAUDE.md` | Conventions specific to this codebase: framework choices, build commands, test invocations, where to put what |
| **User** | `~/.claude/CLAUDE.md` and the memory system | Personal preferences: response style, languages the engineer is fluent in, how they like reviews framed |

All three load automatically. None of them require the engineer to remember to attach context. See the [`CLAUDE.md` and memory appendix](claude-md-and-memory.html) for worked examples of each layer.

The trap is to turn `CLAUDE.md` into a wiki. It is not the place for long prose, aspirations, or every possible framework convention. If the guidance is **task-specific**, it belongs in a Skill. If it is a decision history, it belongs in an ADR. If it must be true on every run, `CLAUDE.md` is the right place.

## Skills: portable organizational knowledge

Skills carry **task-specific knowledge** that Claude Code loads only when relevant. A Skill is a small folder with a `SKILL.md` file and, optionally, scripts, templates, examples, and checklists.

This makes Skills the right home for conventions that matter deeply, but not in every session:

| Skill | What it carries |
| --- | --- |
| `kafka-event-publishing` | Topic naming, schema registration, contract-test examples |
| `new-ecs-service` | Service template, Dockerfile, CDK or Terraform starter files |
| `database-migration` | Migration naming, rollback examples, review checklist |
| `legacy-strangler-step` | Extraction planning, wrapper patterns, ADR template |
| `pr-readiness` | Definition of done, required tests, documentation checklist |

Skills are especially attractive because the **strategic asset is mostly plain text and ordinary files**. The instructions are readable. The scripts can be tested. The templates can be reviewed. The examples can live near the code they describe.

That makes Skills less brittle than many tool-specific investments. The loading mechanism is Claude Code-specific, but the knowledge inside a Skill is not. A `SKILL.md` about Kafka event publishing can be adapted to another agent system, including Codex-style agents, because it is fundamentally Markdown instructions plus repository assets. We should write Skills as clear operational knowledge, not as opaque prompt magic.

Plugins should be treated differently. A Claude-specific plugin may become a convenient way to distribute mature Skills, subagents, and settings across teams, but it should not be the core strategic unit. A plugin is packaging. The durable asset is the Skill content and the scripts, templates, and examples it carries.

### Skill quality standard

A shared Skill should meet a **minimum quality bar** before it is published beyond one repository.

| Field | Required content |
| --- | --- |
| **Purpose** | The specific task the Skill supports |
| **Trigger description** | When Claude Code should and should not load it |
| **Owner** | Team or person accountable for correctness and maintenance |
| **Inputs** | Files, commands, MCP data, templates, or examples it expects |
| **Output contract** | What the Skill should produce: PR, tests, ADR, report, checklist, migration plan |
| **Verification** | Commands, checks, tests, or review steps required before output is accepted |
| **Data restrictions** | Data classes the Skill may not read or include in prompts |
| **Review date** | The next date the Skill must be checked for drift |
| **Deprecation rule** | When it should be retired or replaced |

This keeps Skills from becoming **prompt folklore**. A Skill is a small platform product, not a clever paragraph.

### A good Skill reads like engineering documentation

Poor Skill:

> Be careful with Kafka and make sure the event looks good.

Good Skill:

```markdown
---
name: kafka-event-publishing
description: Use when adding or reviewing a Kafka producer, consumer, schema, or event contract.
---

# Kafka event publishing

## Required checks
- Use `<domain>.<entity>.events.v<n>` topic naming.
- Fetch the current schema from Schema Registry before reviewing compatibility.
- Add or update a contract test under `src/test/contract/`.
- Include idempotency key handling for consumers.

## Verification
- Run `make contract-test`.
- Link the Schema Registry compatibility result in the PR.

## Review
- Domain owner approves event semantics.
- Architecture approves new topic families.
```

The good version is not more "AI-like." It is better engineering documentation: specific, verifiable, and easy to review.

### Skill scope and promotion lifecycle

Skills live at three scopes, and a Skill should earn its way up:

| Scope | Location | Owner | Promotion bar |
| --- | --- | --- | --- |
| **Project** | `.claude/skills/<name>/` in the home repository | Repository owner | Free to create; reviewed in the home repo's PR process |
| **Domain** | Shared repo for the relevant domain (e.g. platform-kafka) | Domain platform team | Used in at least two repositories, meets the Skill quality bar, has an owner |
| **Org** | Distributed via the shared template or sync mechanism | Platform | Used across estates, passes the evaluation fixture, listed in the Skill registry, has a review date |

The point is to let useful project Skills emerge organically, then promote the ones that prove out. Promotion is a PR — adding the Skill to the shared repository and the registry — not a committee decision.

## MCP servers: current state from approved systems

MCP servers connect Claude Code to systems outside the local repository. They are most useful when the agent needs **current state** that should not be copied into prompts or stored in stale documentation.

Stand up the first MCP layer around systems that directly support modernization:

| MCP server | Why it matters |
| --- | --- |
| AWS | Inspect ECS, Lambda, IAM, CloudWatch, MSK, and deployment state |
| GitHub and GitLab | Read PRs, issues, CI status, repository metadata |
| Jira | Bring ticket and epic context into implementation and review |
| Snyk or equivalent | Ground dependency and vulnerability work in real findings |
| Schema Registry | Check Kafka schemas and compatibility from source of truth |
| Service catalog | Find ownership, dependencies, runtime, and operational metadata |

The guiding rule is simple: **MCP for stable systems whose interface is owned outside the repository; Skills for our own evolving conventions and code-backed processes.**

This distinction prevents expensive mistakes. A custom MCP server for a fast-changing internal convention will drift unless it is run like a product. A Skill that tries to wrap a third-party API will break when the vendor changes behavior. Use MCP where an external system already owns the interface. Use Skills where the capability evolves with our code.

### Decision criteria when it is not obvious

Some cases are ambiguous — an internal SaaS, a vendor product we have heavily customised, a service we own but expose to many teams. The decision criteria, in order:

1. **Who owns the interface?** If the interface changes when *we* change something, it is a Skill candidate. If it changes when a vendor decides, it is an MCP candidate.
2. **What is the change cadence?** Daily-changing capability → Skill (so it ships in the same PR). Quarterly-or-slower → MCP is fine.
3. **What is the audience?** A capability used by many engineers across many repos that need it loaded automatically → Skill (progressive disclosure handles this). A capability called occasionally and benefits from being a separately deployed service → MCP.
4. **Where does the source of truth live?** If "the README in this repo," it is a Skill. If "in another team's deployed system," it is MCP.

When still ambiguous, default to **Skill** — it ships with the code, staleness is harder to creep in, and it is cheaper to migrate to an MCP server later than the reverse.

Default MCP access should be **read-only**. Write access can exist for narrow workflows, but it should require a named owner, explicit permission mode, audit logging, and review.

### MCP registry requirements

Every approved MCP server must be listed in an **MCP registry** before it appears in org-level settings.

| Registry field | Why it matters |
| --- | --- |
| **Name and source** | Distinguishes official vendor server, internal wrapper, fork, or experimental server |
| **Owner** | Someone must patch, upgrade, monitor, and retire it |
| **Data classification** | The registry must state whether it can expose internal, confidential, regulated, or secret data |
| **Permission level** | Read-only, write-limited, or write-capable |
| **Authentication model** | Defines token scope, identity model, and rotation expectation |
| **Audit logging** | Required for investigation and compliance |
| **Allowed workflows** | Prevents a general integration from becoming an uncontrolled automation channel |
| **Runbook** | Explains what to do when the server fails or returns suspicious output |
| **Review date** | Keeps access aligned with the current platform and policy |

The default posture is **read-only and narrow**. Write-capable MCP servers require Security and Platform approval, workflow-specific justification, and a rollback plan.

## Subagents: repeatable workflows

Subagents are specialized Claude Code agents with their own instructions, tools, and scope. They are the **workflow layer**.

A good subagent should not be a second documentation store. It should describe a **repeatable job** and compose the right context, Skills, and MCP servers to perform that job.

Examples for this strategy:

| Subagent | Workflow |
| --- | --- |
| `legacy-explorer` | Map a legacy module and summarize responsibilities |
| `stored-proc-summarizer` | Explain stored procedures in business terms |
| `characterization-test-writer` | Generate regression tests before modernization |
| `extraction-planner` | Propose service extraction options and draft an ADR |
| `kafka-contract-reviewer` | Review producer and consumer changes against standards and registered schemas |
| `ecs-template-generator` | Create a service scaffold from the approved ECS template |
| `serverless-complexity-reporter` | Explain why a serverless workflow is operationally complex |
| `pr-review-helper` | Check a PR against the organization's definition of done |

This is where Claude Code becomes more than a smart editor. An engineer should be able to ask for a known workflow by name and get the same structure every time. The subagent provides the flow; Skills provide internal knowledge; MCP provides current state; `CLAUDE.md` provides repository rules.

Subagent definitions should live in version control, have named owners, and go through PR review. A subagent with edit or shell access needs stronger review than a read-only analysis subagent.

### Subagent versus Skill

This distinction is easy to blur.

| If you need... | Use... | Example |
| --- | --- | --- |
| Task knowledge the agent can apply inside many workflows | Skill | "How we publish Kafka events" |
| A repeatable end-to-end job with steps, tools, and output | Subagent | "Review this Kafka PR against standards and registered schema" |
| Current state from a system of record | MCP server | "Fetch the actual schema from Schema Registry" |
| A rule that must run regardless of agent judgment | Hook or CI | "Block secrets before a tool call" |

A subagent may use several Skills and MCP servers. A Skill should not try to become a workflow engine. An MCP server should not encode local conventions. Keeping those boundaries clear keeps the system maintainable.

### Subagent evaluation standard

A shared subagent should not become a standard because it sounds useful. It needs **evidence that it performs the workflow reliably**.

| Requirement | Purpose |
| --- | --- |
| **Representative examples** | Shows how the subagent behaves on real or realistic repositories |
| **Expected output contract** | Makes review possible and prevents loose prose from becoming a deliverable |
| **Allowed tools** | Limits blast radius and keeps risk visible |
| **Known failure modes** | Tells engineers when to distrust or escalate the output |
| **Evaluation fixture** | Gives the owner a repeatable way to test changes to the subagent |
| **Promotion criteria** | Defines when a pilot subagent becomes an org-level standard |
| **Deprecation path** | Prevents old workflows from staying active forever |

Subagents that can edit code, run shell commands, or call write-capable MCP servers require **stronger review** than read-only analysis subagents. The risk comes from what the workflow can do, not from the fact that it is called an AI workflow.

## Controls: settings, hooks, registries, and CI

Enablement is incomplete without controls. The more useful Claude Code becomes, the more important it is to make **boundaries executable**.

The minimum control set:

* an org-level settings baseline with approved tools and MCP servers
* permission modes aligned to workflow risk
* hooks for secret scanning, audit logging, and high-risk command checks
* a registry of approved MCP servers, Skills, and subagents
* CI gates for tests, security scans, schema checks, and policy checks
* PR templates that record AI-assisted work where traceability is required

The point is not to slow engineers down. The point is to move policy from memory and review comments into things that actually run. A hook that blocks a secret from leaving the machine is stronger than a policy page saying not to paste secrets into prompts. A CI gate that requires a contract test is stronger than a checklist item reviewers may miss.

Controls also need an **override path**. A false positive should not force engineers to route around the system. Each blocking hook or setting must define who can approve a temporary bypass, how the bypass is logged, and when the control is reviewed.

### Hook failure-mode rules

Hooks are policy-as-code, so they need the same operational discipline as any other code on the critical path:

* **Fail closed.** If a hook script crashes, times out, or returns an error code the runtime does not recognise, the tool call must not proceed. A silent-pass hook is worse than no hook.
* **Bounded latency.** Each hook has a documented timeout (suggested default: 5 seconds for pre-tool checks, 30 seconds for audit logging). If a hook regularly exceeds its budget, it is a defect, not a feature.
* **Reviewed bypass.** Temporary bypass requires a signed exception token issued by the platform team, time-limited (suggested default: 24 hours), and written to the audit store. Bypass-by-comment-out is not allowed.
* **Versioned and tested.** Hook scripts live in version control with unit tests covering the matching rules and a known-secret/known-violation fixture.

## Ownership and lifecycle

Every enablement asset needs **an owner**. Otherwise it becomes stale documentation with more authority than ordinary documentation.

| Asset | Primary owner | Lifecycle rule |
| --- | --- | --- |
| Project `CLAUDE.md` | Repository owner | Updated in the same PR as changed conventions |
| Org `CLAUDE.md` baseline | Platform engineering with security review | Versioned and distributed through templates or sync |
| Skills | Platform team for shared Skills; service teams for domain Skills | Registry entry, examples, review date, and verification path required |
| MCP servers | Platform or integration owner, with security review | Registered, versioned, monitored, access-scoped, and audited |
| Subagents | Workflow owner | Registry entry, output contract, allowed tools, and evaluation fixture required |
| Hooks and settings | Platform and security | Managed as policy-as-code |

The operating rule: if an instruction can cause Claude Code to change behavior, it is **product code for the engineering organization**. It deserves ownership, review, and deprecation.

## What not to do

Avoid these failure modes:

* Do not create a giant `CLAUDE.md` that tries to teach every workflow.
* Do not create custom MCP servers for conventions that change weekly in the codebase.
* Do not treat plugins as the strategy; use them only as packaging when they help distribution.
* Do not let private, unreviewed subagents become unofficial standards.
* Do not grant broad write access before tests, hooks, and audit trails exist.
* Do not let each repository invent a different enablement model.

The immediate path is modest: create a useful `CLAUDE.md` in pilot repositories, build a **small library of high-value Skills**, wire read-only MCP servers to the systems that matter most, and publish the first few subagents for legacy understanding, Kafka review, ECS scaffolding, and PR readiness.

With this enablement stack in place, the next chapter can move from primitives to the target engineering model: how Claude Code sits inside the modernization factory, platform golden paths, and governance layer.
