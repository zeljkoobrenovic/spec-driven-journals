> KEY POINTS:
> * A phased rollout plan for Claude Code-enabled modernization.
> * Establish policy and standards, build reusable workflows, scale adoption, and measure whether complexity is actually going down.

<hr>

With the operating model in place, the strategy needs **a sequence**. The roadmap below starts with the minimum foundation, then builds reusable modernization capabilities, then scales them across the three estates.

## Roadmap Principles

The roadmap follows **four rules**:

* start with understanding and tests before generation
* build shared workflows before broad autonomy
* pilot in all three landscapes, not only the easiest one
* measure reduction in complexity, not only AI usage

**AI adoption is not the deliverable.** A safer, simpler, more consistent engineering system is the deliverable.

## How to Read the Roadmap

The roadmap is an **adoption model**, not a list of AI experiments. Each phase answers a different engineering question.

| Phase | Question it answers | What engineers should experience |
| --- | --- | --- |
| **Phase 1: Foundation** | Can we use agents safely on real repositories? | Clear rules, working context files, representative pilots, and read-only or supervised workflows. |
| **Phase 2: Modernization Factory** | Can we turn repeated work into reusable capability? | Skills, subagents, templates, MCP integrations, scorecards, and repeatable review workflows. |
| **Phase 3: Scale and Governance** | Can this become normal engineering practice? | Standardized adoption across repositories, visible metrics, auditability, and regular tool review. |

For people new to Claude Code or Codex, the key mental model is this: **do not start by asking for more autonomy**. Start by making the engineering system understandable enough that an agent can operate inside it without guessing.

## Rollout Gates

Every AI workflow moves through **five gates**. Do not skip gates because a demo looks impressive.

| Gate | Allowed behavior | Required evidence to advance |
| --- | --- | --- |
| **1. Read-only exploration** | Explain code, summarize dependencies, map systems, inspect approved read-only MCP data | Source references, no sensitive data exposure, domain review for business interpretation |
| **2. Draft generation** | Generate docs, ADR drafts, test plans, modernization dossiers, and PR summaries | Human-reviewed outputs, assumptions listed, artifact stored in repo or service catalog |
| **3. Local edit assistance** | Edit code in a branch under engineer supervision | Tests pass, PR review, traceability, rollback path for risky changes |
| **4. Workflow automation** | Run a reviewed subagent or headless workflow for a bounded task | Approved Skill/subagent, owner, output contract, audit trail, failure mode documented |
| **5. Platform default** | Workflow becomes standard across teams | Metrics, support model, exception process, training, deprecation path |

For the first six months, high-risk modernization work should stay mostly in **Gates 1-3**. Gate 4 is appropriate for bounded review and generation workflows. Gate 5 is only for workflows with evidence, ownership, and support.

### Example: how a workflow matures

A stored-procedure modernization workflow should not jump directly to "rewrite this in a service." It should mature like this:

| Gate | Example behavior |
| --- | --- |
| **Gate 1** | Claude Code reads the procedure, calling code, schema references, and tests, then produces a source-linked behavior summary. |
| **Gate 2** | It drafts a characterization-test plan, ADR, and extraction candidate dossier for human review. |
| **Gate 3** | It helps add tests or documentation in a branch under engineer supervision. |
| **Gate 4** | A reviewed `stored-proc-summarizer` or `characterization-test-writer` subagent runs for similar procedures with a defined output contract. |
| **Gate 5** | The workflow becomes the standard path for legacy modernization candidates and is supported by Platform. |

This is the difference between **agentic assistance** and unsafe automation: the workflow earns more scope by producing evidence.

## Phase 1: Foundation, 0-3 Months

The first phase creates the **minimum operating base**: policy, standards, context, and three representative pilots.

| Workstream | Deliverable | Owner | Depends on |
| --- | --- | --- | --- |
| Tooling | Claude Code adopted as the default developer tool, with exception process for alternatives | AI Engineering Enablement | — |
| Policy | Initial Claude Code policy, permission-mode guidance, and data-handling rules | Security + Platform | Tooling |
| Context | Organization `CLAUDE.md` baseline and project `CLAUDE.md` template | Platform | Policy |
| Standards | Repository, service, Kafka, AWS, CI/CD, observability, and PR standards | Platform + Architecture | — (run in parallel with Context) |
| Governance | MCP server registry, Skill and subagent review process, hook baseline | Platform + Security | Policy, Standards |
| Baseline | Current-state inventory and measurement baseline for pilot domains | AI Engineering Enablement | Context (uses repo `CLAUDE.md` template) |
| Pilots | One pilot in .NET/TFS, one in Java/GitLab/Kafka, one in Node.js/GitHub/serverless | AI Engineering Enablement | Baseline, Governance |

### Pilot 1: .NET monolith and stored procedures

Goal: prove that Claude Code can help engineers **understand and test legacy behavior** before any extraction.

Expected artifacts:

* legacy capability map
* stored procedure summaries
* dependency map
* first characterization test pack
* first extraction candidate ADR

**Pilot exit criteria.** The pilot is complete when:

* a capability map exists for at least one module and cites source files for every claim;
* at least one stored procedure has a summary pack approved by the owning domain expert;
* at least one critical flow has a characterization test pack that runs in CI;
* one extraction-candidate ADR is in review with a complete modernization dossier (including the data-migration plan).

### Pilot 2: Java microservices and Kafka

Goal: prove that Claude Code can evaluate **service consistency and Kafka usage** against target standards.

Expected artifacts:

* service scorecard
* Kafka topic inventory
* AsyncAPI draft for one important event flow
* consumer error-handling findings
* MSK migration considerations

**Pilot exit criteria.** The pilot is complete when:

* service scorecards have been generated for the pilot service set and published to the service catalog;
* one event family has a complete AsyncAPI spec with ownership and schema compatibility recorded;
* at least one consumer's error-handling and DLQ wiring has been brought into line with the standard;
* an MSK migration plan has been drafted and reviewed by Architecture and Cloud Operations.

### Pilot 3: Node.js serverless workflows

Goal: prove that Claude Code can expose **operational complexity** and recommend simplification.

Expected artifacts:

* event-flow map
* serverless complexity report
* keep / consolidate / retire classification
* GitHub Actions simplification plan
* ECS migration candidate, if one exists

**Pilot exit criteria.** The pilot is complete when:

* an event-flow map exists for at least one painful Lambda chain;
* a serverless complexity report covers at least one workflow and cites concrete counts (functions, queues, EventBridge rules, state machines, IAM policies, deployment steps, alarms, retries, runbooks);
* a keep / consolidate / retire classification is in review for at least one workflow;
* the GitHub Actions simplification plan has been applied to one repository as a worked example.

## Phase 2: Modernization Factory, 3-9 Months

The second phase turns pilot workflows into **shared platform capabilities**. The goal is repeatability: the next team should not invent the prompt, checklist, or template from scratch.

| Workstream | Deliverable | Owner |
| --- | --- | --- |
| Subagents | `legacy-explorer`, `stored-proc-summarizer`, `characterization-test-writer`, `kafka-contract-reviewer`, `service-scorecard-builder`, `serverless-complexity-reporter` | Platform |
| Skills | Kafka event publishing, new ECS service, database migration, PR readiness, incident response | Platform + Service Teams |
| MCP | AWS, GitHub/GitLab, Jira, Snyk, Schema Registry, service catalog | Platform + Integration Teams |
| Templates | ECS service template, Java service template, Node.js service template, runbook and ADR templates | Platform |
| Scorecards | Repository and service maturity scorecards published to service catalog | Platform + DevEx |
| Review support | Headless Claude Code PR review pilot for selected repositories | Platform + Security |
| Cost controls | AI spend dashboard, model-routing guidance, monthly cost review | Platform + Finance |
| **Six-month tooling review (Nov 2026)** | Updated decision record for the default tool, refreshed Codex/Gemini/Claude Code capability comparison, pilot-evidence-based rating | AI Engineering Enablement |

## What Not to Do

Do not start by giving coding agents **broad autonomy**. That skips the foundation the strategy depends on.

Do not measure success by **number of prompts, number of AI users, or number of generated lines of code**. Those numbers can go up while the architecture gets worse.

Do not build custom integrations before deciding whether the capability belongs in MCP, a Skill, a subagent, a template, or a normal library.

## Roadmap Review

The roadmap should be **reviewed every quarter**. The overall strategy and tooling default should be reviewed every six months.

Each review asks:

* Are teams using Claude Code to move toward the target architecture?
* Are standards being applied earlier and more consistently?
* Are generated tests and documentation useful after the pilot ends?
* Are subagents and Skills staying current with the code?
* Are costs and model choices under control?
* Is complexity going down in the measured estates?

If the answer is no, the strategy needs adjustment. The point is not to protect the AI program. The point is to **modernize the engineering system**.
