The goal is to define how Claude Code **reaches real systems**, **applies internal conventions**, and **runs repeatable modernization workflows**. The answer is not one giant integration layer. It is a small set of primitives with different ownership and lifecycle rules.

## How the three primitives differ

`CLAUDE.md` tells Claude Code *what to do*. The other three primitives carry the **actual capability**:

* **MCP servers** — connect Claude Code to systems whose interface is owned by someone else (GitHub, Jira, Snyk, AWS, Schema Registry).
* **Skills** — package the org's *own* capabilities — code, scripts, conventions, examples — so Claude Code can apply them on demand.
* **Subagents** — encode repeatable *workflows* that combine MCP and Skills to do specific kinds of work end-to-end.

The strategy lives in the **combination**. Pick the wrong primitive for a given capability and we pay for it for years.

For readers new to agentic coding, it helps to map the terms back to familiar engineering concepts.

| Agentic primitive | Familiar engineering analogue | What it should contain |
| --- | --- | --- |
| **MCP server** | API client or integration adapter | Access to current state in a system of record: AWS, GitHub, GitLab, Jira, Snyk, Schema Registry, service catalog. |
| **Skill** | Internal playbook plus helper scripts | A task-specific process the organization owns: how to publish an event, create an ECS service, write a migration, prepare a PR. |
| **Subagent** | Named workflow or specialist job | An end-to-end task that composes context, Skills, MCP data, tools, and an output contract. |
| **Hook** | Pre-commit, CI, or runtime guard | A rule that should run automatically because relying on memory would be too weak. |

This distinction matters because **agentic tools amplify whatever workflow we give them**. A vague prompt produces a vague result. A maintained Skill, connected to a real system through MCP and wrapped in a governed subagent, produces repeatable engineering output.

## The principle: MCP for their systems, Skills for ours

This is the rule that prevents the most common drift:

> **MCP for stable third-party systems we do not own.** **Skills for capabilities backed by our own code, conventions, or evolving processes.** Subagents compose both into workflows.

Why this split exists: MCP servers and Skills look superficially similar — both expose capability to the agent — but the **lifecycle, ownership, and evolution model** are completely different.

| Dimension | MCP server | Skill |
| --- | --- | --- |
| **What it wraps** | An external system's API (GitHub, Jira, Snyk, AWS, a SaaS database) | Our own code, scripts, templates, internal conventions |
| **Owner of the interface** | The third-party vendor | Us |
| **Change cadence** | Slow and versioned (vendors care about backward compat) | Whatever our codebase changes at — could be daily |
| **Where it lives** | A separate process, often a separate repo or a vendor-published binary | In our codebase: `.claude/skills/<name>/SKILL.md` plus supporting files |
| **How it ships** | `settings.json` entry; `npx @vendor/server` or similar | Same PR as the code change it documents — lives with the invariants |
| **What "broken" looks like** | Vendor changed their API; bump the server version | Our code changed; the Skill is updated in the same PR |
| **Data sensitivity** | Network-bounded; auth handled by the vendor's protocol | In-repo; subject to the same access control as the code |

The mental model: **if the system is theirs, integrate with their interface via MCP. If the capability is ours, document it as code via Skills.**

The decision criteria for ambiguous cases (who owns the interface, change cadence, audience, source of truth) are listed in [Claude Code Enablement][enablement]. The rest of this appendix is implementation reference.

## Portability Across Agents

This strategy chooses Claude Code as the 2026 default, but it should not make operational knowledge Claude-only.

| Asset | Portability posture |
| --- | --- |
| **MCP server** | Portable where other agents support MCP or where the same integration can be wrapped behind another tool interface. |
| **Skill content** | Highly portable because the valuable content is Markdown, scripts, templates, examples, and checklists. |
| **Subagent definition** | Partly portable: the exact invocation model may be tool-specific, but the workflow, tool boundaries, and output contract should transfer. |
| **Plugin packaging** | Least portable; treat it as distribution convenience, not strategy. |
| **Standards, ADRs, templates, tests** | Fully strategic; keep them in organization-owned repositories and make them useful to humans and agents. |

The practical design rule: **write the knowledge in plain, reviewable files first; package it later**.

## Start with the MCP layer

These are the third-party integrations to wire first. Each removes a class of **"I don't know what's currently in production"** failure from Claude Code's answers.

### 1. AWS MCP server

Highest return per hour invested. Once Claude Code can **read our AWS state directly**, "what's the current ECS task definition for `orders-service`?" gets answered correctly instead of hallucinated.

```json
{
  "mcpServers": {
    "aws": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-aws"],
      "env": { "AWS_PROFILE": "engineering-readonly" }
    }
  }
}
```

Use a **read-only IAM role** for the default config; require explicit elevation (a separate profile) for any agent that needs to make changes. The permission model has to match the blast radius.

### 2. GitHub / GitLab MCP server

Both have official servers. Claude Code can then read PRs, search across repositories, check CI state, and post review comments without us copy-pasting URLs.

### 3. Jira MCP server

The official Atlassian server gives the agent ticket context. Useful for the surprisingly common "summarize the work in this epic" and "what tickets touch this service" queries — and for posting back from PR-review subagents.

### 4. Snyk MCP server

Vulnerability and dependency findings on demand, grounded in actual scan results rather than the agent guessing what might be vulnerable. Pairs with the security-focused subagents.

### 5. Schema Registry MCP server

We run Kafka — given the strategy, we will — and this lets Claude Code list topics, fetch schemas, check consumer-group state. Whether to use a vendor-published server or write a thin custom wrapper depends on our registry; in either case, the *interface* is owned by Confluent / the schema registry, not by us, so it stays MCP.

### 6. Observability and others (later)

Grafana, Datadog, an internal docs server — all are useful, but stand them up after the first five are stable. Do not try to expose everything through MCP in one quarter.

## The Skills layer: our own code, our own conventions

Skills are how the org packages **its own capabilities** for Claude Code. A Skill is a folder in the codebase containing a `SKILL.md` (the instructions), optionally bundled with **scripts, templates, and example files** the agent can reference.

### Where Skills live

```
.claude/skills/<name>/SKILL.md            # project-level skill
~/.claude/skills/<name>/SKILL.md          # user-level skill
<shared-repo>/skills/<name>/SKILL.md      # org-level, distributed via template, sync, or plugin later
```

Same loading hierarchy as `CLAUDE.md` and subagents. Critically, Skills load **on demand** — Claude Code reads the skill description and loads the full body only when the task matches. This is the **progressive disclosure** property: the org can have hundreds of Skills without bloating every session's context.

### What a Skill looks like

```markdown
---
name: kafka-event-publishing
description: Use when the task involves publishing a new event to Kafka, modifying an event schema, or wiring a new producer. Encodes our event-publishing conventions.
---

# Publishing Kafka events

Our standard publisher is `OrderEventProducer` (or the equivalent in your
service's language; see `lib/kafka/producer.{go,java,ts}`). Do not call the
raw KafkaProducer directly.

## Required steps for any new event

1. Register the schema in the Schema Registry first (BACKWARD compatibility
   mode). See `scripts/register-schema.sh` in this folder.
2. Add the event type to `events.proto` in this repo.
3. Use the producer wrapper, which handles correlation IDs, schema validation,
   and idempotency keys.
4. Add a contract test using the pattern in `examples/order-published_test.go`.

## Topic naming
`<domain>.<entity>.events.v<n>` — see ADR-0042 for the rationale.

## Files in this Skill
- `scripts/register-schema.sh` — schema registration helper
- `examples/order-published_test.go` — contract test pattern
- `examples/event-payload.json` — canonical event shape
```

The Skill carries instructions, scripts the agent can run, and examples it can pattern-match against. When Claude Code is asked to publish a new event, it loads this Skill, runs `register-schema.sh`, and follows the **contract-test pattern** — without the engineer having to remember any of it.

### Why Skills, not just CLAUDE.md

Two distinct properties:

* **`CLAUDE.md` is always loaded.** It is for invariants every session needs (the build command, the do-not-edit-this rules). Putting everything there bloats context.
* **Skills are loaded on demand.** They are for capabilities that only matter when a specific kind of task comes up. Most sessions do not need to know how to publish Kafka events — only the ones that are publishing Kafka events do.

The split keeps `CLAUDE.md` **short** and lets the org accumulate many Skills without paying token cost on every session.

### A starter library of org Skills

Build these in roughly this order — each one removes a recurring drain on senior engineering time:

| Skill | What it encodes | Bundled resources |
| --- | --- | --- |
| `kafka-event-publishing` | Our Kafka conventions, producer wrapper, registration | scripts, contract-test examples |
| `new-ecs-service` | Our service template + how to wire it in | template files, CDK stub |
| `database-migration` | Migration patterns, naming, review checklist | migration template, rollback example |
| `add-feature-flag` | Our feature-flag patterns, rollout convention | examples, config snippet |
| `pr-readiness` | What "done" means here: tests, docs, ADR, CI | checklist, ADR template |
| `secret-handling` | How secrets flow in our environment (Secrets Manager + IAM) | example IAM policy, retrieval snippet |
| `incident-response` | Runbook structure, on-call handoff template | runbook template |
| `legacy-strangler-step` | The next-extraction playbook for a strangler-fig migration | extraction-plan template, ADR template |

Notice the pattern: every one of these is **about our code, our conventions, or our process** — never about a third-party API. Those would be MCP.

### Packaged distribution (later)

When Skills mature, they can be packaged so a team can subscribe to "all platform Skills" with minimal setup and the platform team can own the lifecycle. A Claude-specific plugin may be one way to do that, but it should not be the strategic dependency. For Phase 1, in-repo Skills plus a shared template or sync mechanism is enough.

The durable asset is the **Skill content**: Markdown instructions, scripts, templates, examples, and checklists. That content is easy to read, review, and adapt to another agent system if the default tool changes. Packaging is replaceable; the operational knowledge should stay portable.

## Subagents: the workflow layer

A subagent is a specialized Claude Code agent with its own description, tools, and system prompt. It runs as a child of the main session, with its own context window. **Subagents are the workflow layer that composes MCP and Skills into end-to-end work.**

Three properties make subagents strategic, not just convenient:

1. **Reusable across the org.** Once `kafka-contract-reviewer` exists in a shared repo, every team gets the same review behavior.
2. **Composable.** A planning subagent can spawn an extraction subagent that spawns a test-generation subagent. The main session only sees the summarized result.
3. **Cheap to govern.** The subagent definition is a markdown file in version control. Reviewing it is reviewing code.

### Where subagents live

```
.claude/agents/<name>.md           # project-level
~/.claude/agents/<name>.md         # user-level
<shared-repo>/agents/<name>.md     # org-level, distributed via template
```

Same **loading hierarchy** as `CLAUDE.md` and Skills.

### Subagent definition shape

```markdown
---
description: Use proactively when reviewing a Kafka producer or consumer change.
tools: Read, Grep, Bash
model: sonnet
---

You are reviewing a Kafka producer/consumer change for compliance with the
org's event standards.

For the diff the user shows you:
1. Load the `kafka-event-publishing` Skill for our conventions.
2. Use the Schema Registry MCP server to fetch the actual registered schema
   (not the one in the diff — they may have drifted).
3. Check the diff against both: convention compliance from the Skill,
   schema-compatibility from the registry.

Report: per-rule pass/fail with file:line references. Do not edit code.
Suggest specific fixes for failures, but let the human apply them.
```

Notice how the subagent **explicitly composes a Skill (`kafka-event-publishing`) and an MCP server (Schema Registry)**. That's the typical shape: the subagent is the workflow, Skills carry our knowledge, MCP carries third-party state.

### A starter library of org subagents

Build these in roughly this order:

| Subagent | Composes | Tools |
| --- | --- | --- |
| `legacy-explorer` | Repo MCP + `legacy-strangler-step` Skill | Read, Grep, Bash |
| `stored-proc-summarizer` | DB-schema MCP + Skill for our proc-naming conventions | Read, Bash |
| `characterization-test-writer` | `pr-readiness` Skill | Read, Edit, Bash |
| `extraction-planner` | Repo MCP + `legacy-strangler-step` Skill | Read, Grep, Write |
| `kafka-contract-reviewer` | Schema Registry MCP + `kafka-event-publishing` Skill | Read, Grep, Bash |
| `ecs-template-generator` | AWS MCP + `new-ecs-service` Skill | Read, Write, Bash |
| `pr-review-helper` | Repo MCP + `pr-readiness` Skill | Read, Grep, Bash |
| `dependency-upgrade-runner` | Snyk MCP + Repo MCP | Read, Edit, Bash |

Each subagent is the **product**: it's what an engineer invokes ("review this PR"). The Skills and MCP servers are the **components** it composes. That separation is what lets Skills and MCP servers be **reused across many subagents without duplication**.

## Governance for all three

A short policy:

* **MCP servers** are reviewed like any production dependency. Custom servers go through code review, security review, and have a named owner. Third-party servers require a license and security check before being added to the org default `settings.json`.
* **Skills** are reviewed like any shared library — because they *are* shared library code. They live in version control alongside the code they document. A Skill that grants access to scripts requires platform-team review of those scripts. Stale Skills (where the underlying code has changed but the Skill hasn't) are a higher-priority defect than stale documentation, because the agent will follow them.
* **Subagents** are reviewed like any shared library. Definitions live in version control. Changes go through PR. Each subagent has an owner. A subagent that grants tools beyond `Read`/`Grep` requires platform-team review.
* **Distribution** of org-level MCP configs, Skills, and subagents goes through the same template-repo, sync, or packaged-distribution mechanism.

This is what "AI-assisted modernization factory" actually means in implementation terms: a curated set of Skills (our knowledge), grounded by a curated set of MCP servers (their systems), composed into subagents (our workflows), and **governed like any other piece of platform engineering**.
