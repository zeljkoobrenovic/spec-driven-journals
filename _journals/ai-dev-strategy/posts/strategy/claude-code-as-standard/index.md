<hr>

The tooling decision says what to standardize on. This chapter explains why that decision is useful beyond individual coding help: Claude Code gives the organization a **coherent operating surface** it can build, version, govern, and reuse.

Some comparable primitives now exist in other tools. Codex has documented Skills, subagents, and MCP. Gemini CLI has `GEMINI.md` context files and MCP. That does not invalidate the Claude Code decision. It changes the argument: this chapter is a **capability rationale for the chosen default**, not a claim that alternatives cannot support agentic engineering work.

## The case in one sentence

Claude Code is the recommended default because it combines **codebase-aware agentic work, shared instructions, reusable Skills, MCP grounding, subagents, and executable controls** in a form this organization can standardize around.

The strategic value comes from **four capabilities**. Each one addresses a real failure mode of the multi-vendor approach.

## What an agentic loop actually does

A coding agent is not just a model generating text. It is a model wrapped in a loop with tools.

For an engineer, the difference looks like this:

| Assistant type | Typical interaction | What the engineer must do |
| --- | --- | --- |
| **Chat** | "Explain this code" | Copy context in, judge the answer, apply changes manually |
| **Autocomplete** | Suggests the next line or block | Keep the whole design and verification loop in your head |
| **Agentic coding** | Reads code, plans work, edits files, runs checks, revises, summarizes | Set the goal, constrain the scope, review evidence, approve changes |

A safe agentic loop is usually:

1. **Explore**: read the relevant files, tests, schemas, and docs.
2. **Plan**: propose a small set of changes and assumptions.
3. **Act**: edit only inside the approved scope.
4. **Verify**: run build, tests, linters, contract checks, or policy checks.
5. **Explain**: summarize what changed, what failed, what remains uncertain.

The strategy is built around making that loop reliable. `CLAUDE.md` tells the agent what world it is in. Skills teach specific tasks. MCP gives current state. Subagents make repeated workflows consistent. Hooks and permissions keep the loop inside safe boundaries.

## 1. Agentic loop with subagents

Claude Code runs a **tool-using loop**: read files, run commands, edit code, run tests, decide what to do next. For multi-step work, the loop is the unit of work, not the line of code. A request such as "extract this stored procedure into a service, generate characterization tests first, then write the wrapper, then propose the Terraform" needs an **agentic workflow**, not autocomplete.

**Subagents** let us delegate parts of the loop to specialized agents:

* an `Explore` subagent for read-only codebase search (used heavily for legacy understanding)
* a `Plan` subagent that produces an implementation plan but does not write code
* domain-specific subagents we define ourselves: `kafka-contract-reviewer`, `stored-proc-summarizer`, `ecs-template-generator`

A minimal custom subagent definition lives at `.claude/agents/<name>.md`:

```markdown
---
description: Summarizes a SQL Server stored procedure in business terms.
tools: Read, Bash
---

You are reviewing a stored procedure for a modernization team. For the file
the user names, produce: (1) a one-paragraph plain-language summary of what
the procedure does for the business, (2) the inputs, outputs, and side
effects, (3) the tables it reads from and writes to, (4) any obviously
duplicated logic that probably also lives in application code. Do not
propose changes.
```

The strategic point: **subagents make the agentic loop reusable across teams**. Instead of every developer prompting from scratch, the organization publishes a library of subagents that encode how modernization work is done here.

For readers familiar with CI/CD, a subagent is closer to a **named job** than to a saved prompt. A good subagent has inputs, allowed tools, an output contract, known failure modes, and an owner. That is what makes it governable.

## 2. MCP for connecting systems

The Model Context Protocol is how Claude Code talks to external systems — not as bespoke per-workflow integrations, but as **standard servers that any agent can call**.

Practical MCP servers worth standing up early:

| Server | What it gives Claude Code |
| --- | --- |
| **AWS** (official) | Read account state, query CloudWatch, inspect ECS services, draft CDK/Terraform |
| **GitHub / GitLab** (official) | Open and review PRs, read issues, search across repos |
| **Kafka / Schema Registry** (custom) | List topics, read schemas, validate consumer/producer contracts |
| **Service catalog** (custom) | Look up service ownership, dependencies, deployment state |
| **ADR / docs** (custom) | Search architecture decisions, runbooks, internal standards |

A typical entry in `~/.claude/settings.json`:

```json
{
  "mcpServers": {
    "aws": { "command": "npx", "args": ["-y", "@modelcontextprotocol/server-aws"] },
    "service-catalog": { "command": "node", "args": ["./mcp/service-catalog/index.js"] }
  }
}
```

Once these are in place, an engineer asking "what consumes the `orders.events` topic?" gets an answer that is **grounded in current production state**, not in stale wiki text. That is the unlock that makes legacy understanding workflows trustworthy.

The important distinction: MCP is not "more context pasted into the prompt." It is an **approved way to query a system of record**. That difference matters for security, audit, freshness, and ownership.

## 3. `CLAUDE.md` and memory

Standards documents are notoriously ignored. `CLAUDE.md` is different: Claude Code loads it **automatically into every session** in that repository.

A project `CLAUDE.md` at the repository root is the right home for:

* "This service uses pattern X for error handling — match it."
* "All Kafka producers must include a correlation ID — see `lib/kafka/producer.go`."
* "Database migrations go in `db/migrations/`, never hand-edited SQL."
* "Run `make verify` before declaring work done."

A user-level `~/.claude/CLAUDE.md` (or memory directory) carries individual preferences. An organization-wide `CLAUDE.md` distributed via a shared template repo carries cross-cutting rules.

The result: a new engineer's first AI-assisted PR follows the same conventions as a ten-year veteran's, because the conventions live in a file Claude Code always reads.

In Codex-style environments, the equivalent idea is usually an `AGENTS.md` file or agent-specific instructions. The file name matters less than the engineering practice: repository rules should be **visible, versioned, and short enough to be followed**.

## 4. Skills for portable organizational knowledge

`CLAUDE.md` is always loaded, so it should stay short. Skills carry the next layer: **task-specific instructions, examples, scripts, templates, and checklists** that Claude Code loads only when a relevant task appears.

That distinction matters for modernization. A repository may not need Kafka event-publishing guidance in every session, but when a developer asks to add or review a Kafka event, Claude Code should load a shared `kafka-event-publishing` Skill that explains the naming convention, schema-registration steps, contract-test pattern, and example payloads.

Skills are strategically attractive because they are mostly text and ordinary files:

* `SKILL.md` instructions are readable by humans
* bundled scripts and templates live in version control
* examples can be reviewed like test fixtures
* changes can ship in the same PR as the code or standard they describe
* the underlying knowledge can be moved to another agent even if the packaging changes

That last point is important. A Skill folder is Claude Code-specific as a loading mechanism, but its substance is not deeply proprietary: Markdown instructions, scripts, examples, and templates can be adapted to other agent systems, including Codex-style agents, with much less effort than rewriting a custom integration or a closed workflow.

Plugins should be treated differently. They may become a convenient distribution mechanism for mature org-level Skills and subagents, but they should not be the core strategic unit. **A plugin is packaging**. The durable asset is the text-first operational knowledge inside the Skill, plus the tests, templates, and scripts it carries.

If a Skill cannot be understood by an engineer reviewing the repository, it is not a good Skill. We want operational knowledge, not hidden prompt craft.

## Why this combination matters

Each capability on its own is incremental. Together they create a different kind of tool:

* **Agentic loop + subagents** = repeatable workflows, not one-shot completions
* **MCP** = grounded in real state, not in training data
* **CLAUDE.md** = standards enforced by default, not by review
* **Skills** = task-specific organizational knowledge without bloating every session

Multi-tool strategies struggle because investments in subagents, hooks, settings, and automatic context loading do not transfer one-for-one. But the strategy should avoid unnecessary lock-in: keep Skills text-first, keep scripts and templates in normal repos, use MCP for external systems, and treat plugins as optional packaging. The case for picking Claude Code as the default is not that it has the highest benchmark scores this quarter — it is that **the organization can build on it without making every asset opaque**.

## What this is not

Claude Code is not a replacement for engineers, architects, or platform teams. It is also not a magic legacy-modernization machine — it works only as well as the standards, Skills, MCP integrations, and subagents we invest in. **That investment is the actual strategy**. The tool choice just makes it possible.
