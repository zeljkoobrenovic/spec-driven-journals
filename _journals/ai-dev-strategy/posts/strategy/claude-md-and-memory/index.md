> KEY POINTS:
> * How to make engineering standards stick: project, user, and organization-level CLAUDE.md files plus the memory system. 
> * Standards become defaults, not aspirations.

<hr>

After choosing Claude Code as the default, the first practical question is context. `CLAUDE.md` is where the organization turns repeated review comments, repository traps, and build rules into instructions the tool reads every time.

## Three layers of context

Claude Code has three distinct places to put instructions, and using each correctly is the difference between standards that work and standards engineers route around.

| Layer | File | What belongs here |
| --- | --- | --- |
| **Organization** | A shared `CLAUDE.md` distributed via template repo or symlink | Cross-cutting rules: security policy, tool allowlist, definition of done |
| **Project** | `<repo-root>/CLAUDE.md` | Conventions specific to this codebase: framework choices, build commands, test invocations, where to put what |
| **User** | `~/.claude/CLAUDE.md` and the memory system | Personal preferences: response style, languages the engineer is fluent in, how they like reviews framed |

All three load automatically. None of them require the engineer to remember to attach context.

## Project `CLAUDE.md`: the highest-leverage file

This is the file that pays for itself fastest. It lives at the repository root and answers the questions Claude Code would otherwise have to discover by trial and error.

A workable starter template:

```markdown
# Project: order-service

## What this is
Java 21 service that owns the `orders` business capability. Publishes
`orders.events.v1` to MSK; reads from PostgreSQL `orders` schema; deployed to
ECS via CDK in `infra/`.

## Build and verify
- `make build` — compile + unit tests
- `make verify` — adds integration tests against testcontainers
- Always run `make verify` before declaring work done.

## Conventions
- Logging: structured JSON via `lib/logging` — never use `System.out`.
- Errors: domain errors extend `OrderDomainError`; transport errors stay in
  the controller layer.
- Kafka: use `OrderEventProducer`, which adds correlation IDs and schema
  validation. Do not call the raw KafkaProducer.
- Database: migrations live in `db/migrations/` and run automatically. Never
  hand-edit production SQL.

## Things that look wrong but aren't
- The `legacy/` package is intentionally unrefactored; it is being strangled
  out and tracked in ADR-0042. Don't tidy it.
- `OrderRepository.findByLegacyId` returns `null` on miss, not Optional —
  matches the legacy ID format from the .NET system. Do not "fix" this.

## When in doubt
Check `docs/adr/` for architecture decisions. Ask before changing event
schemas, IAM policies, or anything in `infra/prod/`.
```

Two things make this work:

1. **It encodes invariants, not preferences.** "Always use `OrderEventProducer`" is an invariant — breaking it produces wrong data. "Prefer functional style" is a preference that bloats the file and gets ignored.
2. **It calls out traps.** The "things that look wrong but aren't" section saves more time than the rest of the file combined, because it stops Claude Code (and new engineers) from "fixing" things that are deliberate.

## Organization `CLAUDE.md`: cross-cutting rules

This carries rules that apply to every repository. The mechanism varies by org — common options:

* **Template repo**: every new repo is created from `org/repo-template`, which includes a baseline `CLAUDE.md` that projects extend.
* **Distribution via CI**: a workflow drops the latest org-level `CLAUDE.md` into each repo on push.
* **Symlink to a shared dir**: less common but works for monorepo-like setups.

Content that belongs at the org level:

* approved AI tool list and how to request exceptions
* secrets and customer-data prompting rules
* PR review and merge requirements
* security scanning, license scanning expectations
* the standard release tagging convention
* the standard `make` targets every repo should provide

Content that does **not** belong at the org level: framework-specific advice, language-specific style. Those go in project files where they belong.

## User-level memory: preferences and history

The user-level memory system at `~/.claude/.../memory/` carries information about the individual engineer:

* role and current focus area (so Claude Code frames explanations appropriately)
* tools they prefer
* feedback they've given in past sessions ("don't summarize at the end of every response")
* references to internal systems they use

This is the layer that makes Claude Code feel like a colleague who knows the engineer, rather than a fresh hire every session. The org should not write user-level memory — it accrues organically as engineers work.

## What to keep out of CLAUDE.md

Three categories that bloat the file without helping:

* **Things derivable from the code itself.** Don't restate what `package.json` already says.
* **Aspirations.** "We want to migrate to ECS eventually" is not actionable. Either it's true today (then say what to do today) or it goes in an ADR.
* **Long prose.** Claude Code reads `CLAUDE.md` every session. Every paragraph costs context. Bullet lists outperform essays.

## Distribution and lifecycle

Treat `CLAUDE.md` like any other piece of code:

* it lives in version control
* changes go through PR review
* the project owner is also the `CLAUDE.md` owner
* when an invariant changes, the file changes in the same PR

A `CLAUDE.md` that drifts from the codebase becomes worse than no file at all — it confidently sends Claude Code in the wrong direction. The "trust but verify" rule applies: a stale instruction in `CLAUDE.md` will get followed before a fresh observation of the code.

## Measuring whether it works

Two cheap signals tell us if our `CLAUDE.md` strategy is paying off:

1. **PR comment patterns**: are reviewers writing "please use the standard logger" repeatedly? That convention belongs in `CLAUDE.md`.
2. **New-engineer onboarding time**: how long before a new hire's first Claude Code-assisted PR is review-clean? A good `CLAUDE.md` cuts this from weeks to days.

The goal is not a comprehensive document. The goal is that engineers stop having to re-explain the same things — to each other and to Claude Code.
