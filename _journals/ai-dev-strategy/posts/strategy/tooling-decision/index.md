> KEY POINTS:
> * Pick a default AI developer tool. The recommendation is Claude Code. 
> * Other assistants are allowed only with a documented case. 
> * Repository, architecture, and MCP-vs-Skills decisions all flow from this one choice.

<hr>

This is the first implementation decision after the principles. Without a default tool, the rest of the strategy stays abstract: every team chooses a different assistant, governance fragments, and shared workflows never compound.

## The decision

**Adopt Claude Code as the default AI developer tool across the organization.** Other assistants may be used case-by-case, but only with a one-page justification reviewed by the AI Engineering Enablement function.

This is the single most consequential decision in the AI strategy. Without a default:

* every team picks differently
* no shared `CLAUDE.md`, Skill, subagent, or MCP investment compounds
* governance has to chase a moving target
* training, secure-prompting guidelines, and enterprise contracts fragment

With a default, all of those become single, investable lines of work.

## Why Claude Code, in one paragraph

Claude Code wins the *default* slot on **platform maturity**, not on benchmark scores. Subagents, hooks, Skills, and `CLAUDE.md` give the org things it can build, version, govern, and reuse across teams. Codex CLI and Gemini CLI run a similar agentic loop and will close gaps over time — and the strategy is [reviewed every six months](strategic-principles.html) to catch the moment they do — but today the surface for org-level investment is largest on Claude Code.

## How Claude Code compares to the realistic alternatives

The serious agentic-CLI field has narrowed to three: **Claude Code**, **OpenAI's Codex CLI**, and **Google's Gemini CLI**. All three run a tool-using loop in the terminal, can read our codebase, and speak MCP. The differences that matter for a default are not benchmark scores — they are the seams we will build on.

| Property | Claude Code | Codex CLI (OpenAI) | Gemini CLI (Google) |
| --- | --- | --- | --- |
| **Agentic loop maturity** | Mature; stable subagent model, parallel sub-tasks, plan/edit/default permission modes | Working; subagent equivalent less developed | Working; harness still maturing |
| **Project instructions file** | `CLAUDE.md`, loaded automatically at org/project/user levels | `AGENTS.md`, growing convention; smaller ecosystem | `GEMINI.md`, newer; smaller ecosystem |
| **Subagents as a primitive** | First-class; versionable markdown in `.claude/agents/` | Limited; planned | Limited |
| **Skills as a primitive** | First-class; on-demand loading via `.claude/skills/` | Not yet | Not yet |
| **MCP support** | Mature; large third-party catalog | Yes | Yes; newer surface |
| **Hooks / runtime governance** | Pre/post tool hooks, executable allowlists, permission modes | Sandboxing; less granular | Less developed |
| **Strongest model class** | Claude (Opus/Sonnet/Haiku) — long-horizon agentic work | GPT-class — strong general reasoning, isolated coding | Gemini — very large contexts, multimodal |
| **Enterprise contract surface** | Anthropic direct, AWS Bedrock, GCP Vertex | OpenAI direct, Azure | Google Cloud direct |

A coding assistant that only completes the next token cannot run a strangler-fig extraction or a migration sweep. All three of these can — for individual tasks. The difference is what *compounds* across teams, and that gap currently favors Claude Code.

## Where Claude Code is weak — be honest about this

Picking a default is not pretending it has no downsides. The cons that matter, with the mitigation budgeted in:

| Weakness | Why it matters | Mitigation |
| --- | --- | --- |
| **Cost at scale** | Heavy agentic Opus use is expensive; costs can scale faster than developer count | Tiered model routing — Sonnet/Haiku for the bulk of work, Opus reserved for genuinely hard tasks |
| **Vendor concentration on Anthropic** | One tool, one model vendor | Deploy via AWS Bedrock or GCP Vertex so the contract sits with a hyperscaler the org already pays; keep MCP servers, Skills, and `CLAUDE.md` portable |
| **Context-window economics** | Long sessions hit cache misses; large `CLAUDE.md` costs tokens every turn | Discipline on context size; use Skills (on-demand) instead of bloating `CLAUDE.md` (always-loaded) |
| **Closed-source agent harness** | Self-hosting the agent loop is not an option | Accept it, or — for hard requirements — use Aider / OpenCode and accept reduced capability. Codex CLI / Gemini CLI do not solve this either |
| **IDE-integration corners** | Some JetBrains / niche-editor flows less polished than GitHub Copilot | Pure-autocomplete users may keep Copilot alongside; do not block |
| **Tool-specific investment** | Subagents, Skills, MCP wiring don't port one-for-one to Codex/Gemini CLI | The cost of picking *any* default; the six-month review is when this is re-evaluated |

None of these are reasons to stay vendor-neutral. Vendor-neutrality has its own, larger costs: no compounding investment, no shared governance, no executable standards. They are reasons to **pick the default with eyes open** and to budget the mitigations.

## Policy on alternative tools

Other tools are not banned. They require a documented case. The default position for each:

| Tool | Default stance | Allowed when |
| --- | --- | --- |
| **OpenAI Codex CLI** | Not standard; re-evaluate at six-month review | A team has a workload where its model genuinely outperforms and the loss of shared `CLAUDE.md` / Skills / subagents is acceptable for that team |
| **Google Gemini CLI** | Not standard; re-evaluate at six-month review | Workloads dominated by very-large-context analysis or multimodal inputs (screenshot + spec + code) where Gemini's strengths are decisive |
| **GitHub Copilot** | Not standard for agentic work | Pure-autocomplete use; team locked into GitHub Enterprise features that require it. May coexist with Claude Code |
| **Amazon Q Developer** | Not standard for IDE work | Useful for AWS-console / operational queries; AWS support contexts |
| **GitLab Duo** | Not standard | GitLab CI workflows where Duo provides specific integration value during transition |
| **Cursor / Windsurf / other agentic IDEs** | Not standard | Pilot only, with security review. IDE polish does not outweigh losing shared subagent / hook investment |
| **TFS-integrated AI** | Avoid | TFS is a source to exit, not a platform to enrich |
| **Self-hosted open-source agents (Aider, OpenCode, etc.)** | Not standard | Hard data-residency or air-gap requirements that no hosted vendor can satisfy. Accept reduced capability |

The bar for an exception: a one-page case stating (1) what gap exists, (2) why Claude Code cannot close it, (3) what the migration-back path looks like. This is not a hostile bar — it just stops drift.

## Consequences of the choice

Three things follow directly from picking Claude Code as the default. Each is short because the substance lives in its own post.

### 1. MCP for their systems, Skills for ours

> **MCP for stable third-party systems we don't own** (GitHub, Jira, Snyk, AWS, Schema Registry). **Skills for capabilities backed by our own code, conventions, or evolving processes.**

Third-party APIs change on someone else's schedule and live behind a vendor's interface — MCP is the right seam. Our own conventions change daily, in our PRs — Skills ship with the code they document. The wrong choice leaves us with a custom MCP server that goes stale every sprint, or a Skill that wraps a vendor API and breaks every release.

See [MCP, Skills, and Subagents][mcps] for the full decision criteria and starter libraries.

### 2. Repository strategy is unblocked, not forced

Claude Code works across GitHub, GitLab, and other Git hosts, so the tool decision does not pick the repository host. But it *enables* that decision: once the AI tool is constant, consolidating to one host is no longer tangled with retraining engineers on a host-specific assistant.

Move toward one strategic repository ecosystem over time — GitHub or GitLab. In the interim, the same standards (repository metadata, pipeline standards, code review rules, security scanning, service templates, documentation requirements) work across both because Claude Code reads them either way.

### 3. Standardized architecture becomes cheaper to maintain

Use Claude Code to accelerate migration to:

* simpler microservices
* ECS-hosted containers
* Kafka/MSK messaging
* standardized CI/CD
* standardized observability
* clear ownership

Do **not** use Claude Code (or any AI) to create more architectural variation. The point of standardizing on one tool is to make standardized architecture cheaper to maintain — not to give every team a more powerful way to invent their own platform.
