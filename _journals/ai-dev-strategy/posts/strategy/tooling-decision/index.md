> KEY POINTS:
> * Pick a default AI developer tool. The recommendation is **Claude Code for 2026**.
> * The decision is based on **workflow fit, governance maturity, enterprise readiness, and compounding enablement investment**, not on a claim that competitors lack all comparable primitives.
> * Other assistants are allowed only with a documented case, and the default is reviewed every six months.

<hr>

The previous chapter defined the bar: the tool must support incremental modernization, executable governance, reusable workflows, and standards that work across repositories. This chapter applies that bar and makes the first implementation decision.

For readers new to agentic coding, read this chapter as a **platform decision**, not as a model leaderboard. We are not choosing the model that won the latest coding benchmark. We are choosing the tool surface where the organization will place shared instructions, Skills, MCP servers, subagents, hooks, settings, templates, and audit controls.

## The decision

**Adopt Claude Code as the default AI developer tool across the organization.** Other assistants may be used case-by-case, but only with a one-page justification reviewed by the AI Engineering Enablement function.

This is the **single most consequential decision** in the AI strategy. Without a default:

* every team picks differently
* no shared `CLAUDE.md`, Skill, subagent, or MCP investment compounds
* governance has to chase a moving target
* training, secure-prompting guidelines, and enterprise contracts fragment

With a default, all of those become **shared workstreams** the organization can fund, improve, and measure.

## Why Claude Code, in one paragraph

Claude Code wins the *default* slot on **organizational fit**, not on benchmark scores and not on a claim that no other tool has comparable primitives. As of this review, Codex documents Skills, subagents, and MCP; Gemini CLI documents hierarchical `GEMINI.md` context and MCP. The reason to standardize on Claude Code is that its current combination of `CLAUDE.md`, Skills, MCP, subagents, hooks, permission modes, multi-surface usage, and enterprise deployment options gives this organization the strongest surface for **shared modernization workflows and executable governance**.

## What choosing a default means

A default tool is not the same as a ban on every other tool.

| Misreading | Better reading |
| --- | --- |
| "Only Claude Code is allowed anywhere." | Claude Code is the standard surface for governed modernization workflows; exceptions are allowed with a documented case. |
| "The model is always better." | The decision is about workflow fit, controls, and compounding organizational investment. |
| "Prompting is now standardized." | Reusable assets are standardized: context files, Skills, MCP servers, subagents, hooks, templates, and review rules. |
| "Developers lose choice." | Teams still choose implementation details, but shared modernization workflows should not fragment by tool. |
| "This is permanent." | The default is reviewed every six months because the tool market changes quickly. |

This distinction matters. If every team picks its own agentic coding stack, the organization gets local speed and global disorder. If the organization standardizes the surface while keeping durable knowledge portable, it gets local speed **and** compounding shared capability.

## Tool concepts across Claude Code and Codex

The names differ across tools, but the core ideas are familiar if you think in platform terms.

| Capability | Claude Code language | Codex-style language | Engineering idea |
| --- | --- | --- | --- |
| Always-loaded project context | `CLAUDE.md` | `AGENTS.md` and agent instructions | Repository-specific rules and build commands |
| Task-specific knowledge | Skills | Skills | Reusable task packs with instructions, scripts, templates, and examples |
| External system access | MCP servers | MCP servers | Controlled integrations to current state |
| Repeated specialist workflow | Subagents / custom agents | Subagents / custom agents | Named workflow with its own instructions and tools |
| Runtime enforcement | Settings, hooks, permission modes | Config, approvals, hooks/sandbox controls | Policy-as-code around what the agent may do |
| Distribution | Templates, sync, plugins | Skills/config distribution mechanisms | Platform packaging for teams |

The strategy should therefore avoid "prompt magic." Durable knowledge should live in plain text, scripts, templates, examples, tests, and registries that engineers can inspect and move if the default tool changes.

## How Claude Code compares to the realistic alternatives

The serious agentic-CLI field has narrowed to three for this strategy: **Claude Code**, **OpenAI Codex**, and **Google Gemini CLI**. All three can support meaningful engineering work. The decision criteria therefore need to be weighted, dated, and reviewed.

| Criterion | Weight | Why it matters |
| --- | --- | --- |
| **Modernization workflow fit** | High | The tool must support multi-step legacy understanding, test generation, service scaffolding, and PR review |
| **Governance controls** | High | The tool must support permission modes, allowlists, hooks, audit, and enterprise controls |
| **Context and standards model** | High | Repository and organization standards must load reliably without prompt ceremony |
| **Reusable capability model** | High | Skills, subagents, templates, and MCP configurations must be versioned and shared |
| **Portability of durable assets** | High | The organization must avoid locking its knowledge into opaque tool-specific artifacts |
| **Enterprise fit** | Medium | Procurement, security, identity, deployment options, and support matter at scale |
| **Model strengths** | Medium | Model quality matters, but it changes quickly and should not be the only basis for strategy |
| **Developer experience** | Medium | The default must be good enough that teams do not route around it |
| **Cost management** | Medium | The tool must support budget visibility and sensible model routing |

Current capability view, checked against primary docs on **May 14, 2026**:

| Property | Claude Code | OpenAI Codex | Gemini CLI |
| --- | --- | --- | --- |
| **Agentic coding loop** | Strong; reads code, edits files, runs commands, integrates with development tools | Strong; CLI, IDE, app, subagent workflows | Strong CLI with file, shell, web, and tool support |
| **Project instructions** | `CLAUDE.md` and memory; loaded as part of the Claude Code workflow | `AGENTS.md` and config model | `GEMINI.md` hierarchical context files and memory commands |
| **Skills / task packs** | Skills for repeatable team workflows | Skills are documented, use progressive disclosure, and can package instructions, resources, and scripts | No equivalent organization-wide Skill primitive in the same form; context files, extensions, commands, and MCP can cover parts of the need |
| **Subagents / custom agents** | Custom subagents and agent teams are first-class workflow concepts | Subagent workflows and custom agents are documented and enabled by default | No comparable documented subagent primitive as a standard workflow layer |
| **MCP support** | Strong MCP integration, including managed configuration and server restrictions | MCP supported in CLI and IDE extension | MCP servers configured through `settings.json`, with allow/exclude controls |
| **Hooks and executable controls** | Hooks, settings, permission modes, and managed MCP controls are explicit | Hooks, sandboxing, approvals, and managed configuration exist | Settings and MCP trust controls exist; governance model differs |
| **Enterprise deployment surface** | Anthropic direct; also usable through supported third-party providers where appropriate | OpenAI enterprise surface and Azure-aligned options | Google Cloud-aligned enterprise surface |
| **Strategic fit for this plan** | Best default for the 2026 enablement model described in this strategy | Credible alternative to re-evaluate, especially where OpenAI stack fit is decisive | Credible alternative for large-context or Google ecosystem workloads |

The conclusion is deliberately narrow: **Claude Code is the best default for this organization now**. It is not a permanent claim about the market. It is a dated operating decision that lets the organization build one coherent enablement stack.

## Where Claude Code is weak — be honest about this

Picking a default is not pretending it has no downsides. The cons that matter, with the mitigation budgeted in:

| Weakness | Why it matters | Mitigation |
| --- | --- | --- |
| **Cost at scale** | Heavy agentic Opus use is expensive; costs can scale faster than developer count | Tiered model routing (see table below) — Haiku for routine review, Sonnet for the bulk of generation, Opus reserved for genuinely hard planning |
| **Vendor concentration on Anthropic** | One tool, one model vendor | Deploy via AWS Bedrock or GCP Vertex so the contract sits with a hyperscaler the org already pays; keep MCP servers, Skills, and `CLAUDE.md` portable |
| **Context-window economics** | Long sessions hit cache misses; large `CLAUDE.md` costs tokens every turn | Discipline on context size; use Skills (on-demand) instead of bloating `CLAUDE.md` (always-loaded) |
| **Vendor-managed agent runtime** | The organization depends on a vendor's agent behavior, release cadence, and product direction | Keep durable knowledge outside the tool; use dated evaluations; for hard self-hosting or air-gap requirements, evaluate open-source agents separately and accept capability trade-offs |
| **IDE-integration corners** | Some JetBrains / niche-editor flows less polished than GitHub Copilot | Pure-autocomplete users may keep Copilot alongside; do not block |
| **Tool-specific investment** | Subagents, hooks, and loading conventions do not port one-for-one to Codex/Gemini CLI | Keep Skills text-first, keep scripts/templates in normal repos, and treat plugins as optional packaging |

None of these are reasons to stay vendor-neutral. Vendor-neutrality has its own, larger costs: **no compounding investment**, **no shared governance**, **no executable standards**. They are reasons to **pick the default with eyes open** and to budget the mitigations.

## Model routing policy

The cost mitigation only works if it is concrete. The default routing rules below apply to org-level subagents; teams may downgrade to a smaller model but should not silently upgrade without a workflow-level reason.

| Workflow class | Default model tier | Reason |
| --- | --- | --- |
| PR review assistance (`pr-review-helper`, `kafka-contract-reviewer`) | Haiku | High volume, short context, evidence-citing — does not need the strongest model |
| Standards lookup and self-service knowledge | Haiku | Mostly retrieval over the curated knowledge base |
| Code generation from approved templates (`ecs-template-generator`) | Sonnet | Predictable structure, but template wiring needs reliable reasoning |
| Characterization test generation (`characterization-test-writer`) | Sonnet | Needs reliable reading of existing code; volume is moderate |
| Legacy understanding and mapping (`legacy-explorer`, `event-flow-mapper`) | Sonnet | Multi-step exploration; cost-sensitive at fleet scale |
| Stored-procedure summarisation (`stored-proc-summarizer`) | Sonnet by default, Opus by exception | Domain-heavy; Opus only when business semantics are unusually complex |
| Modernization dossier composition (`modernization-doc-builder`) | Opus | Composes outputs from multiple subagents; quality matters; volume is low |
| Extraction planning (`extraction-planner`) | Opus | Architecture-shaping output; low volume; high impact |

Routing is a default, not a hard cap. A subagent may override its tier in its frontmatter, but the override is reviewed in the subagent's PR and logged in the subagent registry.

## Migration-back path

The reversibility rule is operationalised by keeping durable assets in formats this organisation controls. If a future six-month review picks a different default, the migration-back path is:

1. **Standards, ADRs, and runbooks** are already Markdown in our repositories — no migration needed.
2. **Skills** are Markdown plus scripts, templates, and examples. The body of each Skill is portable; only the loading mechanism is Claude Code-specific. Re-targeting to Codex Skills or Gemini context files is a translation of the discovery layer, not a rewrite.
3. **MCP servers** are vendor-neutral by protocol. The same MCP servers continue to work for any agent that speaks MCP.
4. **Subagent definitions** are the most tool-specific asset. Mitigation: keep them text-first in Markdown with explicit input/output contracts, so they can be ported to whatever workflow model the new tool uses.
5. **Hooks and settings** are the most tool-specific asset. Mitigation: keep the *policy* (what is blocked, what is audited) in a tool-neutral spec; treat the Claude Code-specific hook scripts and `settings.json` as the *implementation* of that spec.
6. **Plugins** are packaging only. Do not encode operating knowledge that only exists inside a plugin.

Migration would be **work, not a crisis** — typically two to three months of platform-team time to translate the subagent layer and the hook scripts, with the rest of the assets reusable as-is.

## Policy on alternative tools

Other tools are not banned. They require a documented case. The default position for each:

| Tool | Default stance | Allowed when |
| --- | --- | --- |
| **OpenAI Codex** | Not standard; re-evaluate at six-month review | A team has a workload where Codex model behavior, Skills, subagents, enterprise controls, or OpenAI ecosystem fit clearly outperform the Claude Code default; the team must map shared Skills, MCP, and audit expectations back to the org standards |
| **Google Gemini CLI** | Not standard; re-evaluate at six-month review | Workloads dominated by very-large-context analysis or multimodal inputs (screenshot + spec + code) where Gemini's strengths are decisive |
| **GitHub Copilot** | Not standard for agentic work | Pure-autocomplete use; team locked into GitHub Enterprise features that require it. May coexist with Claude Code |
| **Amazon Q Developer** | Not standard for IDE work | Useful for AWS-console / operational queries; AWS support contexts |
| **GitLab Duo** | Not standard | GitLab CI workflows where Duo provides specific integration value during transition |
| **Cursor / Windsurf / other agentic IDEs** | Not standard | Pilot only, with security review. IDE polish does not outweigh losing shared subagent / hook investment |
| **TFS-integrated AI** | Avoid | TFS is a source to exit, not a platform to enrich |
| **Self-hosted open-source agents (Aider, OpenCode, etc.)** | Not standard | Hard data-residency or air-gap requirements that no hosted vendor can satisfy. Accept reduced capability |

The bar for an exception is a **one-page case** stating (1) what gap exists, (2) why Claude Code cannot close it, and (3) what the migration-back path looks like. The bar is not meant to block exceptions; it is meant to stop **unmanaged drift**.

## Consequences of the choice

Three things follow directly from picking Claude Code as the default. Each is summarized here; the following chapters expand the details.

### 1. MCP for their systems, Skills for ours

> **MCP for stable third-party systems we do not own** (GitHub, Jira, Snyk, AWS, Schema Registry). **Skills for capabilities backed by our own code, conventions, or evolving processes.**

Third-party APIs change on someone else's schedule and live behind a vendor's interface — **MCP is the right fit**. Our own conventions change daily, in our PRs — **Skills ship with the code they document**. The wrong choice leaves us with a custom MCP server that goes stale every sprint, or a Skill that wraps a vendor API and breaks every release.

See [Claude Code Enablement](claude-code-enablement.html) for the full decision criteria and starter libraries.

### 2. Repository strategy is unblocked, not forced

Claude Code works across GitHub, GitLab, and other Git hosts, so the tool decision does not pick the repository host. But it *enables* that decision: once the AI tool is constant, consolidating to one host is no longer tangled with retraining engineers on a host-specific assistant.

Move toward one strategic repository ecosystem over time — GitHub or GitLab. In the interim, the same standards (repository metadata, pipeline standards, code review rules, security scanning, service templates, documentation requirements) work across both because Claude Code reads them either way.

### 3. Standardized architecture becomes cheaper to maintain

Use Claude Code to accelerate migration to:

* simpler microservices
* ECS-hosted containers
* Kafka/MSK messaging
* standardized CI/CD
* standardized observability
* clear ownership

Do **not** use Claude Code (or any AI) to create more architectural variation. The point of standardizing on one tool is to make standardized architecture cheaper to maintain — not to give every team a more powerful way to invent their own platform.

The next chapter explains why this choice is strong enough to build on: the Claude Code capabilities that turn a coding assistant into a modernization platform.

## Strategic Decision Records

Five decisions in this strategy carry enough weight to deserve a decision-log entry. Each follows the same shape — Context, Decision, Rationale, Implications, Review date — so they can be reviewed independently of the surrounding narrative.

### ADR-AI-001: Claude Code as the 2026 default AI developer tool

**Context.** The organisation is starting a modernization program across three estates and needs a shared developer-facing AI surface to avoid every team building its own enablement stack.

**Decision.** Adopt Claude Code as the default AI developer tool across the organisation for 2026. Other assistants are allowed by exception only, with a one-page case reviewed by AI Engineering Enablement.

**Rationale.** Workflow fit, governance maturity, the `CLAUDE.md`/Skills/MCP/subagents/hooks combination, and enterprise deployment options give the strongest surface for shared modernization workflows and executable governance as of 2026-05-14. Codex and Gemini CLI offer comparable primitives in places; the call is about organisational fit, not market superiority.

**Implications.** Enablement investment concentrates on one tool surface. Skills, subagents, and hook scripts are written tool-specifically; the substance (instructions, scripts, templates) stays portable. Other tools require an exception with migration-back plan.

**Review date.** November 2026 (six months), then every six months thereafter.

### ADR-AI-002: AWS / ECS / Kafka as the target engineering platform

**Context.** The current estate spans .NET on TFS with SQL Server, Java microservices with self-managed Kafka on GitLab, and Node.js serverless on GitHub. Modernization needs a target to converge on.

**Decision.** Target an AWS-based engineering platform with ECS as the default service runtime, managed Kafka (preferably Amazon MSK) as the durable event backbone, and direct synchronous APIs as the alternative for request/response interactions.

**Rationale.** ECS removes the operational variation produced by self-managed runtimes. MSK removes Kafka operations from product teams. The combination is well-supported, well-documented, and reduces undifferentiated operational burden.

**Implications.** New services come from approved ECS templates. Lambda is the documented exception for spiky/event-driven workloads. Self-managed Kafka migrates to MSK on a per-service schedule. Synchronous APIs remain valid where event replay or independent consumer evolution is not needed.

**Review date.** Annually, with the strategic principles.

### ADR-AI-003: Skills as durable knowledge, plugins as packaging

**Context.** Claude Code supports both Skills (Markdown + scripts + templates) and plugins (a packaging mechanism). The organisation must decide which is the strategic asset.

**Decision.** Treat Skills as the **durable** strategic asset. Treat plugins as **packaging only**.

**Rationale.** A Skill's substance — Markdown instructions, scripts, examples, templates — is portable across agent systems with manageable effort. A plugin is a Claude Code-specific distribution mechanism. Concentrating organisational knowledge in plugins would create lock-in without proportional benefit.

**Implications.** Skills are written tool-neutrally where possible. Plugins may be adopted later for distribution convenience but never as the primary store of operational knowledge. Skill registry fields focus on content (purpose, trigger, owner, inputs, output contract, verification, data restrictions, review date), not on packaging.

**Review date.** November 2026, with the tooling review.

### ADR-AI-004: MCP for their systems, Skills for ours

**Context.** Both MCP servers and Skills can expose capability to the agent. Picking the wrong primitive for a given capability is paid for over years (stale wrappers, brittle custom servers, custom Skills that break on vendor upgrade).

**Decision.** Use **MCP** for stable third-party systems whose interface is owned outside the repository (GitHub, Jira, Snyk, AWS, Schema Registry). Use **Skills** for capabilities backed by our own code, conventions, or evolving processes. Subagents compose both.

**Rationale.** Lifecycle differs. MCP servers' interfaces change on the vendor's schedule and are best treated as integrations. Skills evolve daily with our code and are best treated as repository assets that ship in the same PR as the convention they encode.

**Implications.** Custom MCP servers for internal conventions are avoided. Skills that wrap third-party APIs are avoided. When ambiguous, default to Skill (cheaper to migrate to an MCP server later than the reverse).

**Review date.** Quarterly, by the Enablement forum.

### ADR-AI-005: Humans remain accountable for production decisions

**Context.** AI agents can propose, generate, summarise, and review. The organisation must decide what humans must still own.

**Decision.** Humans remain accountable for architecture, business semantics, security, licensing, correctness, and production readiness. No AI-assisted change merges to production without a named human approver.

**Rationale.** AI accelerates analysis and change; it does not absorb consequence. Domain rules, transaction boundaries, event semantics, security posture, and production state require accountability that cannot be delegated to an agent.

**Implications.** Permission modes map to risk classes. PR review is human-final even when an agent posts a first-pass review. Production deployment uses the existing change-management process. AI-assisted PRs include a session-id trailer for traceability.

**Review date.** Annually, with the AI policy.
