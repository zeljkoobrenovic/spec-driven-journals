> KEY POINTS:
> * Claude Code is the recommended default AI developer tool for 2026 pilots and enablement investment.
> * The choice is based on workflow fit, governability, repository-level operation, and reusable organizational assets.
> * This is a dated default, not a claim that every competitor is inferior for every use case.
> * Codex, Gemini, and other tools remain available through documented exceptions and review.

<hr>

This chapter makes the first tooling decision in the AI development strategy: **standardize enablement around Claude Code for the 2026 pilot phase**.

The detailed ADR lives in [[ai-tooling]]. This strategy chapter explains the operating implication: where Organization should invest shared instructions, Skills, MCP integrations, subagents, hooks, templates, and training.

## The Decision

Use **Claude Code as the default agentic coding tool** for the 2026 AI development strategy.

Default means:

* shared enablement assets are written for Claude Code first;
* pilot workflows use Claude Code unless an exception is documented;
* training, examples, and support material target Claude Code first;
* governance controls are validated against Claude Code first;
* the default is reviewed after real pilot evidence.

Default does **not** mean:

* every engineer must use only Claude Code forever;
* other tools are banned;
* model benchmarks alone decide tool choice;
* the organization should ignore stronger alternatives as they mature.

## Why Claude Code

Claude Code is the right default for the first operating cycle because it fits the strategy's main need: turning AI from chat into a governed engineering workflow.

| Criterion | Why it matters |
| --- | --- |
| Repository operation | The tool must read, edit, run commands, and work against real codebases. |
| Instruction hierarchy | Shared rules must travel from organization to repository to session. |
| Reusable knowledge | Skills and templates should compound across teams. |
| System integration | MCP can connect agents to approved current-state sources. |
| Workflow packaging | Subagents can make repeated work more consistent. |
| Governability | Settings, hooks, permissions, and audit trails can make policy executable. |

The decision is about organizational leverage, not only assistant quality in isolated prompts.

## Alternatives

| Alternative | Position |
| --- | --- |
| **Codex** | Strong coding assistant and useful comparison point. Keep in review; use where it clearly outperforms or integrates better. |
| **Gemini** | Useful for large-context reasoning and synthesis. Treat multi-model orchestration as a pilot hypothesis, not the default. |
| **IDE assistants** | Useful for local completion and small edits, but not enough as the primary governed workflow surface. |
| **No default** | Preserves local choice but fragments instructions, controls, training, and evidence. |

The default should be reversible. A six-month review should compare evidence, not preference.

## Exception Policy

Teams may use another tool when they document:

* the workflow that Claude Code does not handle well;
* the tool they propose instead;
* data and access implications;
* review and audit expectations;
* how outputs remain compatible with engineering standards;
* when the exception will be revisited.

Exceptions are not failures. They are evidence for future default-tool decisions.

## Model Routing

The default tool does not eliminate model routing. The strategy should treat model choice as part of workflow design:

| Work | Default posture |
| --- | --- |
| Repository edits and governed workflows | Claude Code first. |
| Broad synthesis across large document sets | Consider Gemini or other large-context tools as controlled support. |
| Alternative implementation comparison | Codex can be used as a review or second-opinion tool. |
| Sensitive data or production-impacting decisions | Follow policy; human accountability remains mandatory. |

Routing should be recorded when it affects the result.

## Migration-Back Path

The organization must be able to move away from Claude Code without losing the whole enablement investment. Keep assets portable:

* write instructions in markdown;
* keep templates and scripts outside vendor-specific UI;
* document Skills as reusable knowledge, not magic prompts;
* keep MCP integration ownership explicit;
* version subagents and hooks in repositories;
* store evidence in normal engineering systems.

If the default changes, the organization should migrate assets, not restart its AI strategy.

## Consequences

### Positive

* Shared enablement effort compounds instead of fragmenting across tools.
* Governance can be tested against one primary workflow first.
* Training becomes simpler.
* Pilot evidence is easier to compare.
* The organization can build reusable Skills, hooks, and templates around real usage.

### Negative

* The default may bias teams toward Claude Code even when another tool fits a niche better.
* Enablement work may need rework if the market changes quickly.
* Tool-specific features can creep into standards unless portability is actively protected.
* Teams need a clear exception path to avoid shadow tooling.

## Review Criteria

Review the default after the pilot cycle using evidence:

| Criterion | Evidence |
| --- | --- |
| Delivery value | Pilot outcomes, cycle time, modernization throughput, review burden. |
| Quality | Defect escape, test coverage, review findings, incidents. |
| Governability | Hook results, audit trails, permission incidents, policy exceptions. |
| Cost | License, token, support, enablement, and review cost per workflow. |
| Portability | How much knowledge survives outside the tool. |
| Developer experience | Adoption, friction, training needs, and trust. |

The review should decide whether to keep, narrow, expand, or replace the default.
