The introduction explains the strategic framing. This chapter makes the current-state problem explicit, because the AI strategy only makes sense when it is tied to the actual modernization challenge.

## Current Context

We operate **three architecture paradigms side by side**.

### 1. Monolithic web application

The oldest part of the landscape is a .NET/C# monolithic web application backed by Microsoft SQL Server. A significant amount of business logic lives in stored procedures, not only in application code. The code is hosted in a self-managed TFS environment.

This estate carries **high modernization value** because it likely contains core business rules. It also carries **high risk**: hidden behavior, weak test coverage, database coupling, and old delivery workflows make large rewrites unsafe.

### 2. Legacy microservices architecture

The second part is a Java-based microservices landscape using self-managed Kafka. The code is hosted in GitLab.

This estate is closer to the **target architecture in shape**, but not necessarily in operating model. Service patterns may vary, Kafka ownership may be inconsistent, and running Kafka ourselves creates operational responsibility that should not be spread across product teams.

### 3. Newer serverless architecture

The newest part is a Node.js-based serverless stack hosted in GitHub.

This estate is more modern and generic, but it has become **overly complex in workflow and operations**. Serverless functions, event flows, deployment pipelines, permissions, and handoffs can become difficult to understand and expensive to operate when the architecture grows without enough standardization.

## The Problem

The technology problem is not a lack of AI tools. It is that we are modernizing from a **fragmented baseline**:

* different architecture paradigms
* different programming languages
* different repository platforms
* different delivery workflows
* different operational models
* inconsistent standards across services and teams
* hidden business rules in legacy code and stored procedures
* limited shared understanding of dependencies and migration risk

If we apply AI casually to this environment, we will **accelerate inconsistency**. Teams will generate more code inside already-complex systems, invent new patterns, and make modernization harder to govern.

The AI strategy must do the opposite. It must make the **target direction easier to follow** than the legacy habits.

## Strategic Direction

The strategic direction is to gradually rebuild and modernize the landscape on AWS. The target architecture should be **simpler than the current mix**:

* microservices organized around clear business capabilities
* containerized workloads running on Amazon ECS
* Kafka as the main messaging infrastructure
* managed Kafka operations through AWS patterns, preferably Amazon MSK
* standardized CI/CD
* standardized observability
* consistent security and secrets handling
* clear repository ownership and service metadata
* engineering practices that work across GitHub, GitLab, and migration paths out of TFS

The modernization approach should be **incremental**. The legacy monolith should be understood, tested, wrapped, extracted, and retired progressively. Existing Java services should be standardized and moved toward the target operating model. The Node.js serverless estate should be simplified, with some capabilities staying serverless and others moving to ECS where a service model is clearer.

## Strategic Assumptions

The strategy depends on these assumptions. If one changes materially, the strategy should be reviewed.

| Assumption | Meaning |
| --- | --- |
| **AWS remains the modernization platform** | New platform investment should converge on AWS patterns, not reopen the cloud decision in every modernization effort |
| **ECS is the default service runtime** | Long-running services should prefer containerized ECS workloads unless serverless is clearly simpler |
| **Kafka remains strategic messaging infrastructure** | Event-driven integration should converge on governed Kafka patterns, preferably managed through Amazon MSK where appropriate |
| **TFS is a source to exit** | AI investment in TFS should support understanding and migration, not make TFS a renewed platform |
| **GitHub and GitLab coexist during transition** | AI standards must work across both while repository strategy is rationalized |
| **Humans remain accountable** | AI may propose, generate, summarize, and review, but humans own architecture, business semantics, security, and production readiness |

## Current-State Baseline to Collect

The first version of the strategy can proceed without perfect inventory, but rollout must collect a **measurable baseline**. Otherwise the organization cannot prove that modernization is getting faster or complexity is going down.

| Baseline area | Measures to collect |
| --- | --- |
| **Repository landscape** | Number of active TFS, GitLab, and GitHub repositories; owner coverage; archived vs active repositories |
| **Architecture landscape** | Number of monolith modules, services, Lambda functions, ECS services, Kafka topics, and shared libraries |
| **Database complexity** | Number of stored procedures, triggers, scheduled jobs, cross-database dependencies, and tables without clear ownership |
| **Delivery performance** | Lead time, deployment frequency, failed deployments, rollback rate, PR cycle time, and manual release steps |
| **Operational complexity** | Incident count, noisy alarms, missing runbooks, workflow engines, special-case deployment pipelines |
| **AI readiness** | Repositories with valid `CLAUDE.md`, standard test command, service catalog entry, approved templates, and traceability rules |
| **Cost baseline** | Current tool spend, expected AI budget, and future AI cost by team and workflow |

## Why This Baseline Matters

Without this baseline, the organization will be left with anecdotes: teams feel faster, developers like the tool, or more code was generated. None of that proves that modernization risk went down.

The baseline turns the strategy into a measurable change program. It should show whether teams are understanding legacy systems faster, adding tests before risky extraction, reducing deployment variation, improving ownership metadata, and making reusable modernization workflows available across the estate.
