> KEY POINTS:
> * A Claude Code policy starts with **human accountability**, data classification, approved tools, and explicit review.
> * The rules are paired with executable mechanisms: permission modes, hooks, allowlists, registries, audit trails, and incident handling.

<hr>

The roadmap assumes teams can move quickly **without bypassing controls**. This chapter makes that assumption explicit: once Claude Code becomes the default developer surface, policy cannot stay as prose on a wiki. The rules below are written so they can become configuration, hooks, allowlists, CI gates, and review requirements.

## Why Agentic Coding Needs Specific Policy

Generic "use AI responsibly" guidance is not enough for agentic coding. A chat answer is advisory; an agentic coding session can **inspect repositories, call tools, edit files, run commands, and produce artifacts that enter the delivery flow**. That means the policy must cover not only prompts and outputs, but also **tool access, repository context, automation boundaries, and evidence**.

| Policy concern | Why it matters for agentic coding |
| --- | --- |
| **Data classification** | The agent may read many files quickly; repositories must not contain secrets or production data that should never enter context. |
| **Permission mode** | The difference between read-only analysis and file-editing assistance is a real risk boundary. |
| **Tool allowlists** | Shell commands, MCP servers, and integrations can expand what the agent can see or change. |
| **Hooks** | Runtime checks catch secrets, license problems, unsafe commands, or missing audit records before work proceeds. |
| **Traceability** | Reviewers need to know when AI was used, what changed, which tests ran, and which assumptions remain. |
| **Human accountability** | The agent can recommend and generate; accountable engineers still approve architecture, semantics, security, and production readiness. |

## The policy

> Claude Code is the default AI developer tool for code understanding, code generation, testing, refactoring, documentation, modernization planning, and operational support.
>
> Claude Code-generated output must be reviewed, tested, and approved by accountable engineers.
>
> Claude Code must follow approved architecture patterns, security standards, coding standards, and data-handling rules. These rules live in `CLAUDE.md`, approved Skills, approved subagents, and the org's hook scripts.
>
> Claude Code must not be used to bypass peer review, security review, change management, licensing obligations, or production-readiness criteria. Other AI developer tools may be used only with documented governance approval.

## Specific rules

### Data and prompts

* **No secrets, credentials, customer-sensitive data, or production data** in prompts.
* No production data files in repositories that Claude Code reads.
* Pre-prompt secret-scanning hook runs on every session; sessions abort if a secret is detected in context.

Data handling follows a simple classification model:

| Data class | Default AI handling |
| --- | --- |
| **Public** | Allowed |
| **Internal source code and docs** | Allowed only in approved tools under enterprise terms |
| **Confidential business data** | Redact unless explicitly approved for the workflow |
| **Customer, employee, financial, or regulated data** | Not allowed unless Legal, Security, and the data owner approve a specific redacted workflow |
| **Production logs** | Redacted and scoped; sensitive fields removed before use |
| **Security findings** | Approved tools only; access limited by role and audit logging |
| **Secrets and credentials** | Never allowed; blocked by hooks, scanning, and repository hygiene |

The default rule is **minimum necessary context**. Do not include a database dump, production log bundle, or ticket export when file references, schema metadata, or a small redacted sample are enough.

### Code and review

* **No code generated by Claude Code merged without human review.**
* Every PR includes a `Claude-Code-Session: <id>` trailer in the commit message for traceability.
* No Claude Code-generated migration of business-critical logic without regression tests in the same PR.
* No Claude Code-created Kafka event contracts without architecture/domain approval (enforced by `kafka-contract-reviewer` subagent).
* No new service generated outside approved platform templates (enforced by `ecs-template-generator` being the only sanctioned path).

### Tools and exceptions

* **Only MCP servers in the org registry** may appear in `settings.json`.
* Only reviewed shared Skills and subagents may be distributed at the org level.
* Other AI tools (Copilot, Q Developer for IDE work, Cursor, etc.) require a one-page case approved by AI Engineering Enablement.
* All AI-assisted modernization work must produce tests and documentation as part of the same PR.
* Write-capable MCP servers require a named owner, Security review, audit logging, rollback plan, and workflow-specific approval.
* Generated code must follow the organization's license and IP policy. Claude Code must not copy code from unknown sources into the repository.

### Vendor, retention, and IP

Approved AI tools must have **documented enterprise terms** before organization-wide rollout.

The approval record should state:

* whether prompts and outputs are retained
* whether customer data is allowed at all
* whether submitted data is used for training
* where audit logs are stored
* which identity and access controls apply
* which contractual entity owns the relationship
* how generated-code licensing concerns are handled

If those answers are missing, the tool may be used only in a controlled pilot with non-sensitive code and explicit Security approval.

## How the policy is enforced

The shift from previous policy frameworks: **enforcement is configuration, not trust**. A `settings.json` and a hook script either run or they do not.

### Permission modes

Claude Code's permission modes map to risk classes:

| Mode | Use it for |
| --- | --- |
| `plan` | Architecture work, modernization analysis, anything that should not edit code |
| `default` | Day-to-day development; engineer approves each tool call |
| `acceptEdits` | Bounded scope (e.g. generating tests in `tests/`, scaffolding a new service) where every change is reviewable as code |

The org-level `settings.json` sets **`default` as the floor**. Engineers may downgrade to `plan` for sensitive work; `acceptEdits` requires explicit per-session opt-in.

### Hooks

Pre- and post-tool execution hooks are how policy becomes runtime enforcement. Standard hooks distributed at the org level:

```json
{
  "hooks": {
    "PreToolUse": [
      { "matcher": "Bash", "hooks": [{ "type": "command", "command": ".claude/hooks/secret-scan.sh" }] },
      { "matcher": "Edit|Write", "hooks": [{ "type": "command", "command": ".claude/hooks/license-check.sh" }] }
    ],
    "PostToolUse": [
      { "matcher": ".*", "hooks": [{ "type": "command", "command": ".claude/hooks/audit-log.sh" }] }
    ]
  }
}
```

The hooks themselves are short scripts kept in version control. They get reviewed like any other code, and they **fail closed** — if `secret-scan.sh` exits non-zero, the tool call does not run.

### Tool allowlists

The org-level `settings.json` allowlist specifies which tools and which Bash commands can run without approval. Anything not on the allowlist falls back to `default` mode (engineer approves each call). This is what keeps the audit story manageable: most operations are pre-approved at the org level, so the only prompts engineers see are the ones that genuinely matter.

### Audit trail

Every Claude Code session writes **a session log**. The audit hook ships those logs to a central store (S3 with object lock, typically). The retention policy matches the org's general code-change audit policy. Investigations after the fact — "why did this change happen?" — can replay the session.

### Incident handling

AI-related incidents follow the normal security and production incident process, with an AI-specific evidence pack:

| Incident type | Required response |
| --- | --- |
| **Secret or sensitive data exposure** | Stop the session, revoke affected credentials, preserve logs, notify Security, rotate secrets |
| **Unsafe generated change** | Revert or contain through normal change-management process, attach session log to incident |
| **Unapproved tool or MCP server** | Disable access, review local settings, identify data exposed, decide whether exception is possible |
| **Prompt or tool injection** | Preserve source content and MCP output, block the source if needed, update hooks or instructions |
| **Cost runaway** | Pause workflow, inspect model and context usage, add budget guardrail or routing rule |

Every incident should produce an update to policy, hooks, Skills, subagents, or training. Otherwise the organization learns the same lesson twice.

## What this policy is not

It is not a substitute for security review, threat modeling, or code review. Those processes still exist. The policy ensures Claude Code is **subject** to them, not exempt from them.

It is also not static. The policy lives in version control. When a new attack pattern emerges, when a new MCP server class needs review criteria, or when a workflow class needs a new permission mode — the policy changes through the same PR process as any other piece of platform code. **Treat policy as code**; it ages much better that way.
