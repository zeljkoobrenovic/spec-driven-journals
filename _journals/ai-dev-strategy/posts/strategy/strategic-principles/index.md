> KEY POINTS:
> * Our AI strategy should be positioned as an engineering modernization accelerator, not as a standalone AI adoption initiative. 
> * Six principles keep AI aligned with the target architecture instead of reinforcing legacy complexity.

<hr>

These principles are the guardrails for the rest of the strategy. They explain what Claude Code is allowed to accelerate, what it must not reinforce, and how the organization keeps the tool choice reversible while still investing with intent.

Our AI strategy should be positioned as an **engineering modernization accelerator**, not as a standalone "AI adoption" initiative.

The core goal:

> Use AI — with **Claude Code as the default developer tool** — to accelerate the controlled transition from fragmented legacy delivery models toward a consistent AWS-based microservices engineering platform, while reducing cognitive load, improving quality, and keeping humans accountable for architecture, design, security, and production decisions.

The "with Claude Code as the default" qualifier matters. A vendor-neutral AI strategy sounds prudent on paper but in practice produces a long evaluation phase, three pilot programs that never converge, and engineers picking tools by personal preference. Picking a default unlocks investment in shared standards, reusable subagents, and MCP integrations that compound over time. Other tools are not banned — they need a governance case.

## 1. AI supports the target architecture, not the old complexity

AI should not be used mainly to make the current fragmented landscape easier to tolerate. It should help teams move toward the future state:

* AWS-based platform
* containerized services on ECS
* Kafka-based asynchronous messaging
* simpler microservices
* standardized CI/CD
* standardized observability
* consistent engineering practices across repositories

This avoids the trap of using AI to produce more code inside already-complex systems. Claude Code's ability to read large legacy codebases and propose extractions, rather than just autocomplete inside them, is a direct lever for this principle.

## 2. Modernization should be incremental

The recommended modernization model is a **strangler-style transition**: wrap, extract, replace, and retire legacy capabilities progressively. AWS describes the strangler fig pattern as a way to migrate monolithic applications to microservices incrementally, reducing transformation risk and business disruption. ([AWS Documentation][1])

Claude Code supports this incremental model directly: a planning subagent can map a legacy module's seams; an extraction subagent can produce a wrapper; a test-generation subagent can build characterization tests before any change. Each step is small, reviewable, and reversible.

## 3. AI output must be governed by engineering standards

AI-generated code should never bypass normal engineering controls. The policy should be:

> AI may propose, generate, explain, test, and refactor; humans remain accountable for correctness, security, architecture, licensing, and production readiness.

This is especially important across C#, SQL stored procedures, Java services, Node.js serverless code, Terraform/CDK, Docker, Kafka, and CI/CD definitions.

Claude Code's permission modes, hooks, and tool allowlists are the mechanism for enforcing this. Governance moves from a policy PDF that no one reads into a `settings.json` that runs every session.

## 4. Standardization comes before large-scale automation

Before giving AI agents broad autonomy, define standard patterns:

* service template
* API conventions
* Kafka topic conventions
* container standards
* CI/CD pipeline standards
* observability baseline
* security controls
* testing expectations
* repository structure
* definition of done

Without this, AI will accelerate inconsistency. With Claude Code, those standards live in a project `CLAUDE.md` and in shared subagents — so a developer asking "create a new service" gets the same answer everywhere.

## 5. The strategy is revisited every six months

This document is not a one-time bet. The AI developer-tooling landscape moves faster than any other layer of the stack — model capabilities, agentic harnesses, and primitives like Skills are evolving on a quarterly cadence. A strategy that doesn't expire goes stale.

> **This strategy is reviewed every six months.** Each review re-tests the tooling default (is Claude Code still the right primary?), the MCP/Skills/subagent inventory (what should be deprecated?), the cost/model-routing assumptions, and the modernization progress. Outputs go to the same governance forum that approves the strategy itself.

This cadence is part of what makes a default safe to pick. If the choice is reversible by design — every six months we can swap defaults, retire stale Skills, switch from one MCP server to another — then committing now is much less expensive than waiting indefinitely for the "right" tool to emerge. The committed direction compounds; the periodic review keeps it honest.

What the review specifically checks:

* **Tooling default** — has a serious challenger (Codex CLI, Gemini CLI, something new) moved ahead on the dimensions the org actually decides on?
* **Cost and model routing** — are we paying for Opus on work that Sonnet or Haiku handles? Is the routing policy still right?
* **Vendor-concentration mitigations** — Bedrock/Vertex deployment, MCP/Skill portability — are they still in place?
* **Skill and subagent inventory** — what is stale? What new patterns should be promoted into shared Skills?
* **MCP server inventory** — are any third-party integrations broken, deprecated, or worth replacing?
* **Governance fit** — are hook scripts, permission modes, allowlists still matching how teams actually work?

A six-month cycle is short enough that no investment is locked in for years, long enough that compound returns on shared Skills and subagents are visible. Anything shorter becomes thrash; anything longer becomes legacy.

## 6. MCP for their systems, Skills for ours

When extending Claude Code with new capabilities, follow this split:

> **MCP for stable third-party systems we don't own (GitHub, Jira, Snyk, AWS, Schema Registry). Skills for capabilities backed by our own code, conventions, or evolving processes.**

Third-party APIs change on someone else's schedule and live behind a vendor's interface — MCP is the right seam. Our own conventions and capabilities change daily, in our PRs, by our engineers — Skills are how those capabilities ship with the code they document, in the same PR.

This split prevents the two most common mistakes: building custom MCP servers that wrap our own evolving code (and going stale every sprint), or building Skills that wrap vendor APIs (and breaking on every vendor release). It also keeps the platform investment in the right place — MCP servers are owned by platform/integration teams, Skills are owned by the teams whose code they describe.
