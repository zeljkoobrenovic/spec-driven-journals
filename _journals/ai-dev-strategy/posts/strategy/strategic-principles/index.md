> KEY POINTS:
> * Our AI strategy should be positioned as an engineering modernization accelerator, not as a standalone AI adoption initiative.
> * Six principles keep AI aligned with the target architecture instead of reinforcing legacy complexity.
> * For engineers new to agentic coding, the key mental shift is this: **the agent should follow the engineering system**, not invent a new one.

<hr>

These principles are the **guardrails for the rest of the strategy**. They come before the tooling decision on purpose: first define what the organization needs AI to do, then choose the tool that best supports that work.

The core goal is:

> Use AI to accelerate the controlled transition from fragmented legacy delivery models toward a consistent AWS-based microservices engineering platform, while reducing cognitive load, improving quality, and keeping humans accountable for architecture, design, security, and production decisions.

The strategy does not avoid vendor decisions forever. It is **principle-led**: define the modernization job first, then choose the tool that best supports that job.

## From Prompt to Workflow

For a single developer, an AI assistant may feel like a better chat window. For an organization, that is the wrong unit of design.

The educational distinction:

| Level | What it looks like | Risk |
| --- | --- | --- |
| **Prompt** | "Explain this stored procedure" | Useful once, but quality depends on the engineer asking |
| **Repeatable prompt** | A saved prompt pasted into chat | Better, but still easy to drift and hard to govern |
| **Agentic workflow** | Context file + Skill + MCP data + subagent + tests + review rule | Repeatable, reviewable, measurable, and governable |

This strategy is about the third level. It is not trying to make everyone a better prompt writer. It is trying to make modernization work **repeatable by design**.

That is why the principles below sound like platform-engineering principles. Agentic coding is useful only when it plugs into existing engineering controls: source control, tests, CI, security review, architecture decisions, service ownership, and production approval.

## 1. AI supports the target architecture, not the old complexity

AI should not be used mainly to make the current fragmented landscape easier to tolerate. It should help teams move toward the **future state**:

* AWS-based platform
* containerized services on ECS
* Kafka-based asynchronous messaging
* simpler microservices
* standardized CI/CD
* standardized observability
* consistent engineering practices across repositories

This avoids the trap of using AI to produce more code inside already-complex systems. AI-generated work should be judged by whether it moves the system toward the **target architecture**, not by whether it makes local delivery faster in the short term.

## 2. Modernization should be incremental

The recommended modernization model is a **strangler-style transition**: wrap, extract, replace, and retire legacy capabilities progressively. AWS describes the strangler fig pattern as a way to migrate monolithic applications to microservices incrementally, reducing transformation risk and business disruption. ([AWS Documentation][1])

AI should support that incremental model by helping engineers:

* understand legacy behavior
* summarize dependencies
* identify extraction candidates
* generate characterization tests
* draft wrapper and adapter options
* document migration assumptions
* prepare ADRs and rollback plans

Each step should be **small, reviewable, and reversible**. AI is most valuable when it reduces the cost of safe increments, not when it encourages large rewrites.

## 3. AI output must be governed by engineering standards

AI-generated code should never bypass normal engineering controls. The policy should be:

> AI may propose, generate, explain, test, and refactor; humans remain accountable for correctness, security, architecture, licensing, and production readiness.

This is especially important across C#, SQL stored procedures, Java services, Node.js serverless code, Terraform/CDK, Docker, Kafka, and CI/CD definitions.

Governance must be **executable where possible**:

* approved tools and access paths
* permission modes or equivalent risk controls
* audit trail for AI-assisted work
* security and license scanning
* CI quality gates
* required tests for generated code
* architecture review for service boundaries and event contracts
* production approval by humans

Policy that lives only in prose will not be enough. The selected tooling must support **governance that can run in the workflow**.

## 4. Standardization comes before large-scale automation

Before giving AI agents broad autonomy, define **standard patterns**:

* service template
* API conventions
* Kafka topic conventions
* container standards
* CI/CD pipeline standards
* observability baseline
* security controls
* testing expectations
* repository structure
* definition of done

Without this, AI will **accelerate inconsistency**. With standards in place, AI can help apply the same patterns across TFS migration work, GitLab Java services, and GitHub Node.js repositories.

The important sequence is:

1. define the standard
2. encode it in templates, instructions, checks, and review workflows
3. use AI to apply it repeatedly
4. measure drift and improve the standard

## 5. The tooling decision should be explicit and reversible

A purely "let every team choose" approach sounds flexible, but it makes shared investment hard. Each team trains differently, configures differently, prompts differently, and governs differently.

The strategy therefore needs a **default AI developer tool**. A default allows investment in:

* shared instructions
* reusable workflows
* common security controls
* common audit practices
* shared training
* shared templates
* shared integration patterns

The default is not permanent. It should be **reviewed every six months** against the organization's actual needs:

* Is it still the best fit for modernization workflows?
* Does it still support the required governance model?
* Are costs and model-routing assumptions still acceptable?
* Are there serious challengers that now meet the same bar better?
* Can teams still move work and standards if the default changes?

This cadence makes it possible to choose a tool without pretending the choice is forever. The next chapter applies this principle and makes the first decision under it.

## 6. Capabilities should follow ownership and lifecycle

AI tools need context and integrations. The strategy should place each capability where it can be **owned and maintained correctly**.

The principle:

> Stable external systems should be integrated through stable interfaces. Internal conventions should live close to the code and teams that own them. Repeatable modernization workflows should be versioned, reviewed, and shared.

This prevents two common mistakes:

* wrapping fast-changing internal code in heavyweight integration layers that go stale every sprint
* encoding third-party system behavior in local instructions that break when the vendor changes

The selected tool must support this separation. We need a way to **connect to external systems**, a way to **package internal conventions**, and a way to **compose both into repeatable workflows**.

That requirement is one of the main criteria for the tooling decision in the next chapter. With the problem and principles clear, the strategy can make a concrete choice instead of running an open-ended tool evaluation.

## Decision Tests

Use these tests when reviewing AI-assisted work:

| Test | Reject or rework when |
| --- | --- |
| **Target architecture test** | The output adds another local pattern instead of moving toward ECS, Kafka, standard CI/CD, and standard observability |
| **Evidence test** | The output cannot cite source files, stored procedures, schemas, tickets, runtime state, or tests |
| **Incremental safety test** | The proposal rewrites business-critical behavior without characterization tests, rollback, or domain review |
| **Governance test** | The workflow depends on unapproved tools, MCP servers, Skills, subagents, or data handling |
| **Ownership test** | The generated artifact has no human owner, review path, or maintenance model |
| **Reversibility test** | The work creates lock-in through opaque prompts, plugin-only logic, or undocumented tool behavior |

If a proposal fails one of these tests, the answer is not automatically "no." The answer is **make the risk explicit**, add the missing evidence or control, and then decide.

## How an Engineer Should Use These Principles

When using Claude Code, Codex, or another coding agent, do not start by asking "can the agent do this?" Start with four engineering questions:

1. **What is the target state?**
   If the target pattern is not clear, the agent will optimize for local completion.

2. **What evidence should the agent cite?**
   Ask for file paths, stored procedures, schemas, tickets, AWS resources, tests, and assumptions.

3. **What is the allowed action level?**
   Exploration, draft generation, branch edits, or automated workflow are different risk classes.

4. **Who reviews the result?**
   AI output is not approval. A domain owner, architect, security reviewer, or service owner still makes the decision.

This is how the strategy becomes a daily engineering habit rather than a document people read once.
