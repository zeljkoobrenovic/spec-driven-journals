> KEY POINTS:
> * The repository, microservice, Kafka, and AWS standards that Claude Code should help teams follow.
> * Encoded in `CLAUDE.md`, Skills, subagents, and hooks so they are enforced by default rather than by review.

<hr>

The policy sets the boundaries. These standards define the **target shape of the engineering system** Claude Code should guide teams toward: consistent repositories, services, event contracts, and AWS deployment patterns.

## Standards as code, not as documents

Each standard below is paired with **the Claude Code mechanism that enforces it**. A standard with no enforcement mechanism is aspiration, not policy. The point of having Claude Code as the default tool is that standards can ship as **machine-readable artifacts**: `CLAUDE.md` lines, Skills, subagent system prompts, and hook scripts.

For an engineer new to agentic coding, the important shift is this: a standard should not only tell people **what good looks like**. It should help the agent **generate the right thing**, help reviewers **detect drift**, and help teams **explain exceptions**.

| Form of the standard | Example | Why it matters |
| --- | --- | --- |
| **Human-readable rule** | "Services should use structured JSON logging." | Engineers, architects, and auditors can understand the intent. |
| **Generation path** | Service template includes the logging library and default configuration. | New work starts compliant instead of relying on review comments. |
| **Agent instruction** | `CLAUDE.md` and the relevant Skill say which logger to use and what to avoid. | Claude Code or Codex can follow the rule during implementation. |
| **Verification path** | Hook, CI check, or review subagent flags `console.log` / `System.out` usage. | Drift is caught before merge. |
| **Exception path** | ADR explains why a service deviates and when the exception expires. | Standards remain strong without becoming unrealistic. |

This is **standards-as-code** in practice: documentation, generation, verification, and exception handling move together.

## Standards maturity levels

Not every standard is equally ready. Classify each one so teams know how strongly it applies.

| Level | Meaning | Enforcement |
| --- | --- | --- |
| **Draft** | Proposed direction; useful for discussion and pilots | No enforcement; collect feedback |
| **Recommended** | Preferred pattern for new work | Claude Code suggests it; reviewers may request it |
| **Default** | Expected path for strategic repositories | Templates, Skills, and subagents generate it by default |
| **Required** | Non-negotiable for covered systems | Hooks, CI, or policy checks block drift |
| **Deprecated** | Old pattern being removed | Claude Code flags it and proposes migration |

Every standard should have **owner, maturity level, enforcement mechanism, exception path, and review date**.

## Evidence rule

AI-generated standards feedback must cite evidence. A PR review comment, modernization dossier, or service scorecard should point to **files, schemas, service catalog entries, CI results, AWS resources, or runbooks**. If the agent cannot cite evidence, it should mark the finding as an assumption or question.

## Repository standards

| Standard | Enforcement mechanism |
| --- | --- |
| One clear owner per repository | Required field in `CLAUDE.md` header; `pr-review-helper` checks it exists |
| Standard README | Template repo seeds it; `repo-audit` subagent flags drift |
| Standard build file | Template; `make verify` is the canonical entry point |
| Standard test command | `make verify` — encoded in `CLAUDE.md` so Claude Code uses it |
| Standard Dockerfile | Generated by `ecs-template-generator`; deviation flagged in PR review |
| Standard CI/CD workflow | Template; `ci-simplifier` subagent proposes fixes for drift |
| Standard dependency scanning | CI gate; pre-commit hook for local runs |
| Standard security scanning | CI gate; `pr-review-helper` checks results |
| Standard release tagging | Org-level `CLAUDE.md` rule; release subagent enforces |
| Standard ownership metadata | Service catalog MCP server is the source of truth |

## Microservice standards

Each new service generated by `ecs-template-generator` includes these by default. Existing services get a **scorecard** from `service-scorecard-builder` and a **remediation plan** from `template-aligner`.

| Standard | Enforcement mechanism |
| --- | --- |
| Business capability ownership | Required in service catalog entry |
| API contract | OpenAPI spec in `api/`; `pr-review-helper` checks for breaking changes |
| Event contract where applicable | AsyncAPI spec; `kafka-contract-reviewer` enforces |
| Health endpoint | Generated by template; deviation flagged |
| Metrics | Shared observability library; CI checks for instrumentation |
| Structured logs | Shared logging library; `CLAUDE.md` forbids `System.out`/`console.log` |
| Distributed tracing | Shared tracing library; auto-included by template |
| Idempotent message handling | `kafka-contract-reviewer` checks for idempotency key |
| Retry and dead-letter strategy | `kafka-contract-reviewer` checks for DLQ wiring |
| Clear database ownership | Service catalog field; one schema, one service |
| Runbook | Required path: `docs/runbook.md`; `pr-review-helper` checks it exists |
| ADRs for major decisions | `extraction-planner` and other planning subagents produce ADR drafts by default |

## Kafka standards

The `kafka-contract-reviewer` subagent's system prompt is the **executable form of these standards**. When the standards change, the subagent's prompt changes — in the same PR, so the docs and the enforcement never drift apart.

| Standard | Enforcement mechanism |
| --- | --- |
| Topic naming convention `<domain>.<entity>.events.v<n>` | `kafka-contract-reviewer` |
| Schema ownership | Schema Registry MCP server; ownership tag required |
| Schema compatibility policy (BACKWARD by default) | Schema Registry config + reviewer |
| Producer ownership | Service catalog + reviewer |
| Consumer ownership | Service catalog + reviewer |
| Retention policy | Topic creation template |
| Replay policy | Documented in topic README; reviewer checks for ADR |
| Dead-letter topic convention `<topic>.dlq` | Reviewer enforces wiring |
| Monitoring baseline | Shared library + CI check |
| Partitioning rules | Reviewer flags non-default partition count |
| Event versioning rules | `<n>` in topic name + schema version field |

## AWS standards

The `ecs-template-generator` and AWS MCP server pair makes these the **default path**. Deviating requires going off-template, which `pr-review-helper` flags.

| Standard | Enforcement mechanism |
| --- | --- |
| ECS as default compute target | `ecs-template-generator` is the only sanctioned generator; Lambdas require an exception ADR |
| Amazon MSK as default Kafka target | `msk-migration-planner` is the canonical path; self-managed Kafka requires exception ADR |
| Infrastructure as code | No manual console changes as the source of truth; `pr-review-helper` flags PRs that do not include IaC for infra changes |
| Least-privilege IAM | Template generates least-privilege roles; reviewer checks for `*` actions |
| Centralized logging and metrics | Generated by template via shared library |
| Standard deployment pipeline | CI template; deviation flagged |
| Cost tagging | IaC modules require tags; CI gate enforces |
| Environment separation | Template generates per-env stacks; reviewer flags shared resources |
| Well-Architected review for critical workloads | Subagent walks the checklist; produces a draft review |

## How to evolve the standards

Three rules keep the **standards-as-code** approach from becoming its own legacy problem:

1. **Standards live with their enforcement.** When we change a standard, we change the subagent, hook, or template that enforces it in the same PR. If we cannot change the enforcement, we do not actually have a standard — we have a wish.
2. **Deprecation is a first-class operation.** When a standard is being phased out, the subagent that enforced it gets a deprecation date. After that date, it stops running. Do not keep obsolete standards alive.
3. **The standards owner is the platform team, not a committee.** A small platform team that owns the templates, Skills, subagents, and hooks can ship changes weekly. A committee that owns "the standards" usually ships changes quarterly. The first model works better for executable standards.

Exceptions are allowed, but they must be **explicit**. An exception ADR should state the standard being bypassed, why the default is not appropriate, what compensating controls exist, who approved the exception, and when it expires.

### Worked exception ADR

```markdown
# ADR-0118: Lambda runtime for the `inventory-sync` job (exception to ECS default)

## Status
Approved exception. Expires 2027-03-31; re-review at that date or sooner if traffic changes.

## Context
The `inventory-sync` job runs four times an hour, processes ~2,000 events
per run from EventBridge, and finishes in under 90 seconds. Steady-state
load does not justify a long-running ECS service: average utilisation
would be < 3%.

## Decision
Run `inventory-sync` as an AWS Lambda function on a scheduled EventBridge
rule, deviating from the "ECS by default" standard.

## Compensating controls
- IaC: Terraform module owned by the platform team; identical
  observability and tagging to ECS services
- Standard alarms (errors, duration, throttles) wired to the platform
  alerting channel
- DLQ on the EventBridge target with the same runbook pattern as MSK DLQs
- Code reviewed by the platform team for cold-start and timeout behaviour

## Approver
Architecture Review (2026-04-19); Platform team consulted.

## Review date
2027-03-31. If average utilisation rises above ~30% or run duration
exceeds 60% of the timeout, escalate to Architecture before the review
date.
```

The ADR is checked into the service repository and indexed by the standards owner. Exception ADRs are reviewed in aggregate at each quarterly Architecture forum to catch patterns that suggest the standard itself should change.
