<hr>

The engineering model is useful only if it turns into a **build order**. This chapter identifies the first workflows worth building, who should own them, and what human decision point keeps each one safe.

## How to read this list

Each use case below is **a specific Claude Code workflow** — a subagent (or small set of subagents), the Skills it uses, the MCP servers it depends on, and the **human accountability point**. This is what to build and in what order. The workflows compose: the legacy-understanding subagents feed the modernization-documentation workflow, which feeds the test-generation workflow, and so on.

For readers new to agentic coding, the important pattern is:

> A use case is not "ask the agent a question." A use case is **a workflow with inputs, tools, outputs, controls, and a human decision point**.

That framing keeps the portfolio close to normal engineering practice. A workflow should be reviewed like a build job, a deployment template, or a shared library.

| Use case | Owner | When to build |
| --- | --- | --- |
| 1. Legacy understanding assistant | Platform | First quarter |
| 2. Modernization documentation | Platform | First quarter |
| 3. Test generation | Platform + QA | First quarter |
| 4. Service template generation | Platform | Second quarter |
| 5. Pull request review assistance | Platform | Second quarter |
| 6. Developer self-service knowledge | Platform + DevEx | Continuous |

## Evidence and Output Contract

Every workflow must produce **reviewable engineering evidence**, not just a conversational answer.

| Requirement | Meaning |
| --- | --- |
| **Source references** | Cite files, functions, stored procedures, schemas, tickets, cloud resources, or service catalog entries used as evidence |
| **Assumptions** | State what the workflow inferred and what it could not verify |
| **Risk note** | Identify behavioral, data, security, operational, or contract risks |
| **Verification path** | Name the tests, commands, reviewers, or checks needed before acting on the output |
| **Human owner** | Name the role accountable for accepting, rejecting, or refining the output |
| **Artifact location** | Store results in the repository, service catalog, ADR folder, PR, or runbook; avoid one-off chat-only outputs |

This contract matters most for stored procedures, Kafka contracts, and serverless simplification. Those areas often fail through **plausible but incomplete understanding**.

Example of weak output:

> This procedure appears to calculate order status and could probably be extracted.

Example of useful output:

> `dbo.UpdateOrderStatus` reads `Orders`, `OrderItems`, and `PaymentEvents`; writes `Orders.Status`; is called from `OrderController.Submit`, nightly job `ReconcileOrders`, and report `FulfillmentAging`. Extraction requires characterization tests for submitted, cancelled, partially paid, and failed-payment states. Domain owner must confirm that `PaymentEvents.EventType = 17` means "manual refund."

The second output is longer, but it is reviewable. It names evidence, assumptions, and the human validation point.

## Workflow metrics

Each workflow needs at least one outcome metric so it is judged by what it changes, not by how often it runs.

| Workflow | Primary outcome metric | Secondary signal |
| --- | --- | --- |
| 1. Legacy understanding | Median time to produce a capability map for one module | % of capability maps with cited source files for every claim |
| 2. Modernization documentation | Median time from candidate identification to reviewed dossier | % of dossiers that lead to an approved ADR within one quarter |
| 3. Test generation | Characterization coverage on top-N business-critical flows | Mutation-test or behavioural-change detection rate on generated tests |
| 4. Service template generation | % of new services generated from approved templates | Drift between generated services and current template version |
| 5. PR review assistance | Median time to first useful review comment | Rate of standards-drift comments resurfacing in subsequent PRs |
| 6. Developer self-service | Repeat-question rate in support channels | `CLAUDE.md` / Skill freshness (median age since last update) |

Vanity measures — number of prompts, generated lines of code, number of AI users — should not appear on these dashboards unless paired with one of the metrics above.

## What is not yet allowed

Some workflow behaviours are gated even when the underlying capability exists. For Phase 1:

* No auto-merge of any PR, even when `pr-review-helper` approves.
* No edits to stored procedures, triggers, or scheduled jobs without characterization tests in the same PR and a domain-owner reviewer named.
* No new Kafka event contracts authored end-to-end by an agent. The `kafka-contract-reviewer` reviews proposed contracts; humans author them.
* No schema migrations in `acceptEdits` mode. Schema migrations stay in `default` mode where every change is approved.
* No write-capable MCP server may be called by a Phase 1 subagent unless its registry entry explicitly lists the subagent under "Allowed workflows."
* No edits outside the new-service directory by `ecs-template-generator` in `acceptEdits` mode.

These boundaries are reviewed at each phase gate. They exist to keep the agentic surface area calibrated to the maturity of the controls behind it.

## 1. Legacy understanding assistant

### What it does

Engineers ask questions about the legacy estate and get answers **grounded in current code, schemas, and runbooks** — not in stale wiki pages.

* "Where is customer pricing calculated?"
* "Which stored procedures update order status?"
* "Which services consume this Kafka topic?"
* "What workflows call this Lambda?"
* "Which repositories implement this business capability?"

### Claude Code implementation

* **Subagents**: `legacy-explorer`, `stored-proc-summarizer`, `event-flow-mapper`
* **Skills**: `legacy-strangler-step`, `database-migration`, `pr-readiness`
* **MCP servers**: repo (GitHub/GitLab), DB schema, Kafka/Schema Registry, AWS, service catalog
* **Output contract**: capability map with file, stored procedure, table, event, and runtime references
* **Human accountability**: domain experts validate business interpretation; platform team owns the subagents

### Why it goes first

Without this, **every other workflow runs blind**. The investment in the underlying MCP servers — repo, schema, Kafka — is what enables every later use case.

## 2. Modernization documentation

### What it does

For every modernization candidate, Claude Code produces a complete **modernization dossier** (canonical definition in [Strategy by Landscape](strategy-by-landscape.html#modernization-dossier)). The dossier composes:

* business capability summary and current-state description
* source references (files, stored procedures, tables, callers, jobs)
* data ownership and transaction-boundary assessment
* dependency map and test gaps
* proposed target design and migration steps
* characterization tests and parallel-run / rollback plan
* domain-owner review note and an ADR draft

### Claude Code implementation

* **Subagent**: `modernization-doc-builder` (composes outputs from legacy-understanding subagents)
* **Skills**: `legacy-strangler-step`, ADR template, migration-plan template
* **MCP servers**: same as use case 1 + ADR/docs MCP server
* **Output**: a markdown dossier committed to the repo as a PR; ADR draft committed to `docs/adr/`
* **Human accountability**: architect approves the ADR; platform team approves the migration plan

### Why this matters

Modernization without documentation is what produced the current legacy estate. Making documentation a **byproduct of the workflow** — not a separate manual step — is how we avoid repeating that.

## 3. Test generation and characterization

### What it does

This should be one of the first production-grade Claude Code uses. Focus areas:

* stored procedure tests
* Java service contract tests
* Kafka consumer/producer tests
* Node.js workflow tests
* regression tests before refactoring
* golden-master tests for legacy behavior

### Claude Code implementation

* **Subagent**: `characterization-test-writer` — reads code (and DB schema, for stored procs), generates test files, runs them to confirm they pass against current behavior
* **Skills**: `pr-readiness`, language-specific test-pattern Skills, `database-migration` where stored procedures are involved
* **MCP servers**: repo, DB schema; AWS for any test-double setup
* **Output contract**: tests committed in a PR, with generated assumptions documented and edge cases reviewed by QA
* **Permission mode**: `acceptEdits` is acceptable here because the test files are new code, not modifications to behavior. Generated tests still go through PR review.
* **Human accountability**: QA approves the scenarios and edge cases; the test author owns ongoing maintenance

### Why this is high-leverage

**Tests are the unblocker** for every other modernization workflow. Without them, "wrap first, extract second" is too risky to start.

## 4. Service template generation

### What it does

When an engineer needs a new service, Claude Code generates it from the **approved template** — never from scratch. Templates include:

* Dockerfile
* ECS task definition
* health checks
* logging
* metrics
* tracing
* security defaults
* Kafka integration
* CI/CD pipeline
* README
* runbook
* ADR

### Claude Code implementation

* **Subagent**: `ecs-template-generator` (and language-specific variants: `java-service-generator`, `node-service-generator`)
* **Skills**: `new-ecs-service`, language template Skills, `secret-handling`
* **MCP servers**: AWS (for VPC/SG lookup), service catalog (to register the new service)
* **Output contract**: generated service scaffold, runbook, ADR, service catalog entry, verification commands, and traceable template version
* **Permission mode**: `acceptEdits` within a new directory; `default` (i.e. ask for approval) for any modification outside it
* **Human accountability**: platform team owns the templates; service owner approves the generated PR

### The strategic property

Once `ecs-template-generator` exists, "create a new service" is no longer a place where teams diverge. **The template is the standard**, and the subagent is how the standard gets applied. Drift becomes much harder.

## 5. Pull request review assistance

### What it does

When a PR is opened, a Claude Code subagent posts a first-pass review:

* code summary
* risk areas
* missing tests
* security concerns
* inconsistent patterns
* migration checklist compliance

**Final approval remains human.**

### Claude Code implementation

* **Subagent**: `pr-review-helper` (or domain-specific variants like `kafka-contract-reviewer`, invoked when the diff touches Kafka code)
* **Skills**: `pr-readiness`, plus domain Skills such as `kafka-event-publishing` when relevant
* **MCP servers**: repo, service catalog (to identify owners), DB schema (to check migration impact)
* **Trigger**: PR-opened webhook → headless Claude Code session → review comment posted
* **Output contract**: review comment grouped by correctness, security, tests, standards drift, and unresolved questions
* **Human accountability**: human reviewers own approval; the subagent is a checklist-runner, not a gatekeeper

### What it does not do

It does not auto-approve. It does not auto-merge. It does not block PRs; **CI owns blocking**. It saves human reviewers from spending the first thirty minutes of a review re-discovering things the subagent could have flagged immediately.

## 6. Developer self-service knowledge

### What it does

Claude Code answers questions about internal standards from a **curated knowledge base** — so engineers stop asking the same questions in Slack:

* "How do I create a new ECS service?"
* "How do I publish a Kafka event?"
* "What is the standard logging format?"
* "How do I migrate a GitLab service to the target platform?"
* "What is the approved pattern for retries?"

### Claude Code implementation

* **Mechanism**: org-level `CLAUDE.md` distributed to every repo + curated Skills + an internal docs MCP server that indexes runbooks, ADRs, and standards docs
* **Skills**: `new-ecs-service`, `kafka-event-publishing`, `database-migration`, `incident-response`
* **Subagent (optional)**: `internal-docs-search` — for queries that span multiple sources, this subagent can search across all of them in parallel
* **Human accountability**: DevEx owns the docs MCP server's freshness; platform team owns `CLAUDE.md` content

### Why this is "continuous"

Unlike the other use cases, this one does not have a launch date. It improves whenever someone updates a runbook, adds an ADR, or refines `CLAUDE.md`. The investment is **curating the knowledge base** — Claude Code is just the interface to it.
