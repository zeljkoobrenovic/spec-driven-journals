> KEY POINTS:
> * The AI strategy has real risks: vendor concentration, hallucinated changes, stale instructions, secret exposure, over-automation, cost growth, and architectural drift. 
> * Each risk needs an explicit mitigation.

<hr>

Choosing Claude Code as the default is a pragmatic strategy, not a risk-free one. The right response is not to avoid commitment. The right response is to name the risks and design mitigations into the operating model.

## Risk Register

| Risk | Why it matters | Mitigation |
| --- | --- | --- |
| Vendor concentration | One default tool can create dependency on one vendor and one agent harness | Six-month tooling review; keep standards portable; keep MCP, Skills, templates, and ADRs in our repos |
| Cost growth | Agentic workflows can consume more tokens and more expensive models than expected | Model routing policy; budgets by team; monthly cost review; use smaller models for routine tasks |
| Hallucinated or incorrect output | AI may invent APIs, misunderstand code, or produce plausible but wrong explanations | Require grounding in repo, MCP data, tests, and file references; human review remains mandatory |
| Secret or data exposure | Prompts and context can accidentally include credentials or sensitive data | Pre-prompt secret scanning, repository hygiene, data-handling policy, blocked production data files |
| Stale `CLAUDE.md` | Claude Code will follow outdated instructions confidently | Project owner owns `CLAUDE.md`; changes to invariants update code and `CLAUDE.md` in the same PR |
| Stale Skills | A Skill can encode an old process and keep applying it | Skills have owners, review dates, and deprecation dates; stale Skills are treated as platform defects |
| Stale subagents | A subagent can enforce a rule that no longer matches the target platform | Subagents live in version control, have owners, and are reviewed with the standards they enforce |
| Architectural drift | AI can make it easier for teams to create more variation | Generation only from approved templates; `pr-review-helper` flags drift; exceptions require ADRs |
| Over-automation | Teams may let AI make decisions that require human accountability | Permission modes by risk class; architecture and production decisions remain human |
| Weak tests | Generated code without regression tests increases modernization risk | Characterization tests required before changing business-critical logic |
| Event-contract mistakes | AI can generate syntactically valid but semantically wrong Kafka events | Domain owner and architecture approval required for event semantics |
| Tool sprawl | Teams may adopt multiple AI tools and lose shared governance | Claude Code default; one-page exception process; six-month re-evaluation |
| False productivity metrics | Lines generated and prompts sent can rise while complexity worsens | Measure modernization outcomes, standard alignment, test coverage, cycle time, and complexity reduction |

## Risk 1: Vendor Concentration

Standardizing on Claude Code creates concentration risk. Subagents, Skills, hooks, and `CLAUDE.md` are not fully portable to other tools.

The mitigation is not pretending the investment is portable. The mitigation is to keep the important assets in formats the organization controls:

* standards in Markdown and templates
* architecture decisions in ADRs
* workflows in version-controlled subagent definitions
* reusable domain guidance in Skills
* integrations behind MCP where possible
* six-month review of the default tool

If a future tool becomes the better default, migration should be work, not a crisis.

## Risk 2: AI Reinforces Legacy Complexity

The biggest strategic risk is not a bad code suggestion. It is using AI to make the current fragmented landscape easier to tolerate.

That happens when teams ask Claude Code to add more features inside the monolith, create more serverless orchestration, or invent new service patterns because generation is cheap.

Mitigation:

* the target architecture is explicit
* new services come from approved templates
* serverless expansion requires target-state justification
* legacy work starts with understanding and tests
* architecture review approves service boundaries and event semantics

AI should make the target path cheaper, not make every local variation easier.

## Risk 3: Unsafe Legacy Changes

The .NET/C# and stored procedure estate carries hidden behavior. AI may summarize code well and still miss a domain rule buried in a stored procedure, batch job, trigger, or operational convention.

Mitigation:

* run legacy analysis in `plan` mode by default
* generate characterization tests before modifying critical logic
* require domain expert review for stored procedure changes
* document assumptions in ADRs and migration dossiers
* prefer wrap-first extraction over rewrite-first replacement

The rule is simple: no business-critical rewrite without evidence.

## Risk 4: Misgoverned Kafka Changes

Kafka event contracts are architecture decisions. AI can generate event names, payloads, schemas, and consumers, but it cannot decide domain semantics.

Mitigation:

* `kafka-contract-reviewer` checks standards, not business meaning
* domain owners approve event semantics
* Schema Registry is the source of truth for registered schemas
* AsyncAPI specs are required for strategic event flows
* compatibility rules are enforced in CI

Claude Code can help make Kafka governance cheaper. It must not replace Kafka governance.

## Risk 5: Stale Context

Claude Code is only as good as the context it loads. A stale `CLAUDE.md`, stale Skill, or stale subagent is worse than missing documentation because the tool will act on it.

Mitigation:

* each context artifact has an owner
* each shared Skill and subagent has a review date
* standards and enforcement change in the same PR
* stale instructions are tracked as defects
* long prose is kept out of always-loaded context

Context is product surface. It needs lifecycle management.

## Risk 6: Cost and Model Misuse

Agentic work can be expensive when every task uses the most capable model, large context, and long-running sessions.

Mitigation:

* route routine work to cheaper models
* reserve the strongest model for complex planning and high-risk modernization work
* keep `CLAUDE.md` short
* use Skills for on-demand context
* report spend by team and workflow
* review cost and value monthly during rollout

The goal is not lowest cost. The goal is visible cost with a clear modernization return.

## Risk 7: Governance Theater

The organization can write a good AI policy and still have weak governance if nothing enforces it.

Mitigation:

* permission modes map to risk classes
* hooks scan for secrets and write audit logs
* tool allowlists are distributed through settings
* CI enforces tests, scans, and compatibility checks
* AI-assisted PRs include traceability
* exceptions require a documented case

Policy should become configuration, not a PDF people remember during audits.

## Early Warning Signals

Watch for these signals:

* more generated code but no increase in tests
* more service patterns instead of fewer
* teams creating private subagents or Skills with no review
* repeated PR comments that should have been encoded in `CLAUDE.md`
* rising model spend without modernization artifacts
* AI-generated Kafka events with unclear ownership
* project `CLAUDE.md` files that have not changed while the codebase changed
* teams treating Claude Code output as approval rather than input

These signals mean the strategy is drifting from modernization into unmanaged tool adoption.

## Residual Risk

Even with mitigations, AI-assisted development will still produce mistakes. The strategy does not remove engineering risk. It changes where risk is caught.

The desired shift is earlier detection:

* secrets before prompts
* standards drift before review
* missing tests before merge
* architecture mismatch before implementation
* stale context before repeated misuse

That is the governance target: not zero risk, but risk surfaced early enough that teams can act on it.
