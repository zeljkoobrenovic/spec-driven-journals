> KEY POINTS:
> * The AI strategy has real risks: vendor concentration, hallucinated changes, stale instructions, secret exposure, prompt and tool injection, over-automation, cost growth, and architectural drift.
> * Each risk needs an explicit mitigation.

<hr>

After the policy and standards, the last step is to **stress-test the strategy**. Choosing Claude Code as the default is pragmatic, not risk-free. The right response is not to avoid commitment. The right response is to name the risks and design mitigations into the operating model.

## How Agentic Risk Differs from Chatbot Risk

Most software engineers already understand the basic risks of generative AI: it can be wrong, plausible, outdated, or overconfident. Agentic coding keeps those risks and adds a new class: the tool may be able to **act on bad assumptions**.

| Familiar AI risk | Agentic-coding version |
| --- | --- |
| The model gives a wrong answer. | The agent edits files, writes tests, or posts review comments based on the wrong answer. |
| The prompt contains sensitive data. | The agent reads a repository, log, ticket, or MCP response that contains sensitive data. |
| The model hallucinates an API. | The agent scaffolds code around a non-existent API and makes the PR look complete. |
| The model follows misleading text. | A ticket, log line, README, or dependency output contains instructions that steer the agent away from policy. |
| The output is hard to audit. | The agent makes many small decisions across tools, files, and commands without a clear session trace. |
| A developer over-trusts the answer. | A team lets generated code, generated tests, or generated architecture notes stand in for accountable review. |

The mitigation pattern is consistent: **ground the agent, restrict its tools, require evidence, and keep humans accountable for irreversible decisions**.

Thresholds are deliberately left as words ("≥ threshold," "small share") rather than numbers. Each owner sets the concrete threshold for their risk at the first quarterly review based on pilot data.

## Risk 1: Vendor Concentration

Standardizing on Claude Code creates **concentration risk**. Subagents, hooks, plugin packaging, and automatic `CLAUDE.md` loading are not fully portable to other tools.

The mitigation is not pretending the investment is portable. The mitigation is to keep the important assets in **formats the organization controls**:

* standards in Markdown and templates
* architecture decisions in ADRs
* workflows in version-controlled subagent definitions
* reusable domain guidance in text-first Skills
* integrations behind MCP where possible
* six-month review of the default tool

Skills deserve a nuance: the Claude Code loading mechanism is tool-specific, but the useful substance — **Markdown instructions, scripts, examples, templates, and checklists** — is relatively easy to transfer to another agent such as Codex. Plugins should be treated as convenience packaging, not as the strategic asset.

If a future tool becomes the better default, migration should be **work, not a crisis**.

## Risk 2: AI Reinforces Legacy Complexity

The biggest strategic risk is not a bad code suggestion. It is using AI to make the **current fragmented landscape easier to tolerate**.

That happens when teams ask Claude Code to add more features inside the monolith, create more serverless orchestration, or invent new service patterns because generation is cheap.

Mitigation:

* the target architecture is explicit
* new services come from approved templates
* serverless expansion requires target-state justification
* legacy work starts with understanding and tests
* architecture review approves service boundaries and event semantics

AI should make the **target path cheaper**, not make every local variation easier.

## Risk 3: Unsafe Legacy Changes

The .NET/C# and stored procedure estate carries hidden behavior. AI may summarize code well and still miss a domain rule buried in a stored procedure, batch job, trigger, or operational convention.

Mitigation:

* run legacy analysis in `plan` mode by default
* generate characterization tests before modifying critical logic
* require domain expert review for stored procedure changes
* document assumptions in ADRs and migration dossiers
* prefer wrap-first extraction over rewrite-first replacement

The rule is simple: **no business-critical rewrite without evidence**.

## Risk 4: Misgoverned Kafka Changes

Kafka event contracts are architecture decisions. AI can generate event names, payloads, schemas, and consumers, but it cannot decide domain semantics.

Mitigation:

* `kafka-contract-reviewer` checks standards, not business meaning
* domain owners approve event semantics
* Schema Registry is the source of truth for registered schemas
* AsyncAPI specs are required for strategic event flows
* compatibility rules are enforced in CI

Claude Code can help make Kafka governance cheaper. It must **not replace Kafka governance**.

## Risk 5: Stale Context

Claude Code is only as good as the context it loads. A stale `CLAUDE.md`, stale Skill, or stale subagent is worse than missing documentation because the tool will act on it.

Mitigation:

* each context artifact has an owner
* each shared Skill and subagent has a review date
* standards and enforcement change in the same PR
* stale instructions are tracked as defects
* long prose is kept out of always-loaded context

**Context is product surface.** It needs lifecycle management.

## Risk 6: Cost and Model Misuse

Agentic work can be expensive when every task uses the most capable model, large context, and long-running sessions.

Mitigation:

* route routine work to cheaper models
* reserve the strongest model for complex planning and high-risk modernization work
* keep `CLAUDE.md` short
* use Skills for on-demand context
* report spend by team and workflow
* review cost and value monthly during rollout

The goal is not lowest cost. The goal is **visible cost with a clear modernization return**.

## Risk 7: Prompt and Tool Injection

Claude Code reads repository files, tickets, logs, MCP output, docs, and sometimes generated artifacts. Those inputs can contain hostile or misleading instructions. A comment in a ticket or a string in a log can tell the agent to ignore policy, exfiltrate data, or run unsafe commands.

Mitigation:

* treat external text as **untrusted input**
* keep risky sessions in `plan` or `default` mode
* require tool approvals for high-risk commands
* restrict MCP servers to approved workflows and scopes
* keep write-capable MCP servers rare and audited
* train reviewers to look for prompt-injection paths in agent outputs
* add evaluation cases that include malicious instructions in files, tickets, and tool output

This is not hypothetical. As soon as agents can call tools, the boundary between "content" and "instruction" becomes a security concern.

## Risk 8: Governance Theater

The organization can write a good AI policy and still have weak governance if nothing enforces it.

Mitigation:

* permission modes map to risk classes
* hooks scan for secrets and write audit logs
* tool allowlists are distributed through settings
* CI enforces tests, scans, and compatibility checks
* AI-assisted PRs include traceability
* exceptions require a documented case

Policy should become **configuration**, not a PDF people remember during audits.

## Risk 9: Model Drift and Workflow Drift

Agent behavior can change even when our repository does not. Vendors update models, tools change, MCP servers upgrade, and prompts that worked last quarter may degrade.

Mitigation:

* keep evaluation fixtures for critical Skills and subagents
* run regression checks before promoting new workflow versions
* record model and tool versions where practical for high-risk workflows
* review the default tool every six months
* treat stale or failing AI workflows as platform defects

The strategy should assume **continuous maintenance**, not one-time rollout.

## Early Warning Signals

Watch for these signals:

* more generated code but no increase in tests
* more service patterns instead of fewer
* teams creating private subagents or Skills with no review
* unapproved MCP servers appearing in local settings
* generated tests that do not fail when behavior is intentionally changed
* model spend rising faster than modernization output
* recurring agent mistakes after model or tool upgrades
* repeated PR comments that should have been encoded in `CLAUDE.md`
* rising model spend without modernization artifacts
* AI-generated Kafka events with unclear ownership
* project `CLAUDE.md` files that have not changed while the codebase changed
* teams treating Claude Code output as approval rather than input

These signals mean the strategy is drifting from **modernization into unmanaged tool adoption**.

## Residual Risk

Even with mitigations, AI-assisted development will still produce mistakes. The strategy does not remove engineering risk. It changes where risk is caught.

The desired shift is **earlier detection**:

* secrets before prompts
* standards drift before review
* missing tests before merge
* architecture mismatch before implementation
* stale context before repeated misuse

That is the governance target: **not zero risk, but risk surfaced early enough** that teams can act on it.

**Residual risk is accepted by Engineering leadership**, with Security and Architecture consulted, at each six-month strategy review. The acceptance is recorded in the same place as the strategy version (the introduction's status block). If a material new risk emerges between reviews, the AI Engineering Enablement forum escalates it through the Risk and Compliance Review forum.

Read end to end, the strategy now forms a loop: context defines the problem, principles define the bar, the tooling decision creates a shared surface, the enablement chapter makes it usable, the workflow chapters apply it, the execution chapters assign ownership and timing, and the governance chapters keep the modernization work safe.
