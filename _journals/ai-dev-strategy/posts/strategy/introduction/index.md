> KEY POINTS:
> * The AI strategy starts from a difficult reality: three architecture paradigms, three repository ecosystems, and too much operational variation. 
> * The goal is to use Claude Code to accelerate modernization toward a simpler AWS-based engineering platform.

<hr>

This strategy defines how we use AI in software development. It is not an "AI adoption" plan and it is not mainly about helping individual developers type code faster.

The strategy exists because the engineering landscape is already complex, and modernization will add even more work before it makes things simpler. AI should help us move through that transition faster, with better understanding, stronger tests, more consistent standards, and clearer governance.

## Current Context

We operate three different architecture paradigms at the same time.

### 1. Monolithic web application

The oldest part of the landscape is a .NET/C# monolithic web application backed by MS SQL Server. A significant amount of business logic lives in stored procedures, not only in application code. The code is hosted in a self-managed TFS environment.

This estate carries high modernization value because it likely contains core business rules. It also carries high risk: hidden behavior, weak test coverage, database coupling, and old delivery workflows make large rewrites unsafe.

### 2. Legacy microservices architecture

The second part is a Java-based microservices landscape using self-managed Kafka. The code is hosted in GitLab.

This estate is closer to the target architecture in shape, but not necessarily in operating model. Service patterns may vary, Kafka ownership may be inconsistent, and running Kafka ourselves creates operational responsibility that should not be spread across product teams.

### 3. Newer serverless architecture

The newest part is a Node.js-based serverless stack hosted in GitHub.

This estate is more modern and generic, but it has become overly complex in workflow and operations. Serverless functions, event flows, deployment pipelines, permissions, and handoffs can become difficult to understand and expensive to operate when the architecture grows without enough standardization.

## The Problem

The technology problem is not that we lack AI tools. The problem is that we are modernizing from a fragmented baseline:

* different architecture paradigms
* different programming languages
* different repository platforms
* different delivery workflows
* different operational models
* inconsistent standards across services and teams
* hidden business rules in legacy code and stored procedures
* limited shared understanding of dependencies and migration risk

If we apply AI casually to this environment, we will accelerate inconsistency. Teams will generate more code inside already-complex systems, invent new patterns, and make modernization harder to govern.

The AI strategy must do the opposite. It must make the target direction easier to follow than the legacy habits.

## Strategic Direction

The strategic direction is to gradually rebuild and modernize the landscape on AWS.

The target architecture should be simpler than the current mix:

* microservices organized around clear business capabilities
* containerized workloads running on Amazon ECS
* Kafka as the main messaging infrastructure
* managed Kafka operations through AWS patterns, preferably Amazon MSK
* standardized CI/CD
* standardized observability
* consistent security and secrets handling
* clear repository ownership and service metadata
* engineering practices that work across GitHub, GitLab, and migration paths out of TFS

The modernization approach should be incremental. The legacy monolith should be understood, tested, wrapped, extracted, and retired progressively. Existing Java services should be standardized and moved toward the target operating model. The Node.js serverless estate should be simplified, with some capabilities staying serverless and others moving to ECS where a service model is clearer.

## Role of AI

AI should be treated as a modernization accelerator, not as a separate technology program.

Its role is to help engineers:

* understand legacy modules, stored procedures, services, events, and workflows
* generate characterization tests before risky changes
* document current state and migration options
* identify extraction candidates and service boundaries
* generate scaffolding from approved templates
* improve consistency in CI/CD, observability, and repository structure
* review pull requests against target standards
* answer questions from a curated engineering knowledge base

Claude Code is the recommended default developer tool because it can operate across repositories, read code, run commands, use MCP servers, apply Skills, follow `CLAUDE.md`, and run repeatable subagent workflows. That combination matters more than generic code completion.

## Goals

The AI strategy has six goals.

### 1. Accelerate modernization safely

Use AI to reduce the time needed to understand, test, document, wrap, extract, and migrate legacy capabilities. Speed matters only when changes remain reviewable, tested, and reversible.

### 2. Reduce complexity

Use AI to expose unnecessary complexity, not hide it. The strategy should help teams simplify serverless workflows, standardize service patterns, reduce custom pipelines, and avoid adding more architectural variation.

### 3. Improve developer productivity

Use AI to remove repetitive work from engineers: boilerplate, first-pass documentation, dependency exploration, test scaffolding, PR summaries, and standards lookup. The productivity gain should come from better flow and less cognitive load, not from bypassing engineering discipline.

### 4. Ensure consistency across platforms

Use a shared Claude Code setup, common `CLAUDE.md` guidance, approved Skills, subagents, MCP servers, templates, and hooks so teams get the same modernization guidance across TFS migration work, GitLab Java services, and GitHub Node.js repositories.

### 5. Govern AI-generated work

AI-generated output must go through normal engineering controls: human review, tests, security checks, license checks, architecture review where needed, and production approval. AI may propose and generate; humans remain accountable.

### 6. Build reusable modernization capabilities

Invest in reusable workflows rather than one-off prompts. A stored procedure summarizer, Kafka contract reviewer, service scorecard builder, ECS template generator, and PR review helper should become shared platform assets that improve over time.

## Success Criteria

The strategy is working when:

* engineers understand legacy components faster
* critical legacy flows gain characterization tests before change
* modernization candidates have clear documentation and ADRs
* new services start from approved ECS and observability templates
* Kafka contracts become easier to review and govern
* serverless workflows become simpler, not more elaborate
* repository and service standards are applied consistently
* PR review catches standards drift earlier
* AI usage has traceability, governance, and human accountability

## The Framing

The AI strategy should not be:

> ~~Let developers use AI tools to code faster.~~

It should be:

> Use Claude Code as the default developer environment to make modernization repeatable, safe, and faster by combining legacy understanding, automated testing, platform standards, reusable workflows, and governed engineering controls.

That framing connects AI directly to the real technology challenge: moving from a fragmented legacy landscape to a simpler, consistent, AWS-based engineering platform.
