> KEY POINTS:
> * This is a modernization strategy first and an AI tooling strategy second.
> * The goal is to make modernization **repeatable, safe, and faster**, not to maximize AI activity or generated code.
> * Agentic coding matters because it can turn legacy understanding, tests, standards, and governance into repeatable engineering workflows.

<hr>

## Why This Strategy Exists

The AI strategy should not be:

> ~~Let developers use AI tools to code faster.~~

It should be:

> Use AI to make modernization **repeatable, safe, and faster** by combining legacy understanding, automated testing, platform standards, reusable workflows, and governed engineering controls.

That distinction is the whole point. This is not an "AI adoption" plan, and it is not mainly about helping individual developers type code faster. It defines how to use AI in software development during modernization, where the real challenge is architectural fragmentation, hidden business logic, inconsistent delivery models, and the need to move safely toward a simpler target engineering model.

Modernization will add work before it removes work. Teams need to understand old behavior, create tests where few exist, standardize service patterns, migrate repositories and pipelines, simplify operations, and keep delivery moving. AI should help with that transition by improving understanding, increasing test coverage, applying standards consistently, and making governance easier to execute.

If AI is applied casually to a fragmented estate, it will accelerate inconsistency. Teams will generate more code inside already-complex systems, invent new local patterns, and make modernization harder to govern. The strategy must do the opposite: make the target direction easier to follow than the legacy habits.

## What This Strategy Decides

This strategy makes one central decision: **use AI-assisted development to accelerate modernization toward a simpler target engineering model**.

The target is an AWS-based landscape built around clear microservices, ECS-hosted container workloads, Kafka as the main messaging backbone, standard CI/CD, standard observability, and consistent engineering controls.

The strategy also makes one implementation decision: **Claude Code is the default AI developer tool for 2026**. That default is not a statement that no other tool can do useful work. It is a decision to concentrate enablement investment in one governed surface: repository instructions, Skills, MCP servers, subagents, hooks, settings, templates, and audit controls. The tool default is reviewed every six months.

The expected outcome is not "more AI usage." The expected outcome is **faster, safer modernization with less architectural variation**.

## Current Reality in Brief

The current estate has three architecture paradigms side by side:

| Estate | Main issue |
| --- | --- |
| **.NET/C# monolith with SQL Server and TFS** | High modernization value, high risk, hidden behavior in code and stored procedures |
| **Java microservices with self-managed Kafka and GitLab** | Closer to the target shape, but inconsistent in operating model and ownership |
| **Node.js serverless stack with GitHub** | Modern but operationally complex when workflows, permissions, and event flows grow without enough standardization |

The detailed current-state framing is in [Current State and Problem](current-state-and-problem.html). The important point for the introduction is this: the problem is not a lack of AI tools. The problem is modernization from a fragmented baseline.

## Strategic Response

The strategic direction is to gradually rebuild and modernize the landscape on AWS. The target architecture should be simpler than the current mix:

* microservices organized around clear business capabilities
* containerized workloads running on Amazon ECS
* Kafka as the main messaging infrastructure
* managed Kafka operations through AWS patterns, preferably Amazon MSK
* standardized CI/CD
* standardized observability
* consistent security and secrets handling
* clear repository ownership and service metadata
* engineering practices that work across GitHub, GitLab, and migration paths out of TFS

The modernization approach should be incremental. The legacy monolith should be understood, tested, wrapped, extracted, and retired progressively. Existing Java services should be standardized and moved toward the target operating model. The Node.js serverless estate should be simplified, with some capabilities staying serverless and others moving to ECS where a service model is clearer.

AI should be treated as a **modernization accelerator**, not as a separate technology program or a local productivity experiment.

Its role is to help engineers:

* understand legacy modules, stored procedures, services, events, and workflows
* generate characterization tests before risky changes
* document current state and produce modernization dossiers
* generate scaffolding from approved templates and review PRs against target standards
* answer questions from a curated engineering knowledge base

## What AI Changes

Most engineers have seen generative AI used in two familiar ways:

* **chat**: ask a question, get an explanation or draft;
* **autocomplete**: type code, get inline suggestions.

Agentic coding is different. An agentic coding tool such as Claude Code or Codex can run a **multi-step engineering loop**:

1. read files in a repository;
2. inspect tests, build files, logs, schemas, or tickets;
3. decide what information is missing;
4. call approved tools or MCP servers to get current state;
5. propose a plan;
6. edit files in a branch;
7. run commands or tests;
8. revise its work based on failures;
9. produce a reviewable artifact such as a PR, ADR, test pack, or modernization dossier.

That loop is powerful, but it is not magic. It behaves well only when the organization gives it **good context**, **safe tools**, **clear boundaries**, and **human review**.

This strategy therefore treats agentic coding as an **engineering system**, not as a clever prompt. The useful unit is not "a prompt that works once." The useful unit is a reusable workflow: repository context, a Skill, an MCP server, a subagent, a template, a hook, and a review rule that together make the same work repeatable across teams.

If you understand CI pipelines, service templates, API contracts, and platform golden paths, these concepts should feel familiar. Agentic coding adds a new developer surface, but the governance model is still ordinary engineering discipline.

## Scope

The strategy covers **AI in software development for modernization**. It is explicitly **not**:

* an AI adoption plan for non-development functions such as support, marketing, or analytics
* a model procurement strategy or commercial contract framework
* a plan for autonomous production operations
* a substitute for normal architecture, security, or change management

## Goals and Success Criteria

The AI strategy has six goals.

| Goal | Success measure |
| --- | --- |
| **Accelerate modernization safely** | Time from candidate identification to reviewed modernization dossier drops by 30%; risky flows gain characterization tests before extraction |
| **Reduce complexity** | Distinct deployment patterns, custom CI workflows, and special-case serverless orchestrations decrease quarter over quarter |
| **Improve developer productivity** | Time to understand a legacy module, produce first tests, or create an approved service scaffold drops in measured pilots |
| **Ensure consistency across platforms** | Strategic repositories have current `CLAUDE.md`, standard build/test commands, ownership metadata, and service catalog entries |
| **Govern AI-generated work** | AI-assisted PRs include traceability, reviewed outputs, tests, and evidence references |
| **Build reusable modernization capability** | Shared Skills, MCP servers, subagents, templates, and hooks have owners, review dates, usage metrics, and deprecation paths |

Avoid vanity measures such as prompt count, generated lines of code, or number of AI users unless they are paired with modernization outcomes.
