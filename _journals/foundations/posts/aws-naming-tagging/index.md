> **Status**: DRAFT
>
> **Decision**: Organization standardizes naming and tagging across AWS. Accounts use `org-<domain>-<env>`. Resources use `org-<domain>-<system>-<component>-<env>[-<qualifier>]` in lowercase kebab-case. Every taggable resource carries required ownership, cost, environment, data, compliance, management, and repository tags. Tags are authoritative for tooling; names are for human recognition.

## Decision

Adopt one AWS naming and tagging scheme across Organization.

| Area | Standard |
| --- | --- |
| AWS Organization | One organization for Organization-managed accounts. |
| Account name | `org-<domain>-<env>` |
| Resource name | `org-<domain>-<system>-<component>-<env>[-<qualifier>]` |
| Case | Lowercase kebab-case for names; PascalCase for tag keys. |
| Tags | Required on every taggable production resource. |
| Authority | Tags are authoritative for tooling; names are supporting context. |
| Enforcement | AWS Organizations, tag policies, SCPs, AWS Config, IaC checks. |

The convention should let a human recognize a resource and let automation answer: who owns it, what system it belongs to, what environment it runs in, what it costs, and which compliance or data constraints apply.

## How to Read This Decision

Names and tags serve different purposes:

| Mechanism | Optimized for | Failure mode |
| --- | --- | --- |
| **Name** | Human recognition in consoles, logs, dashboards, and tickets. | Names are often immutable and cannot carry all metadata. |
| **Tag** | Queryable ownership, cost, compliance, lifecycle, and automation data. | Free-form tags drift unless keys and values are governed. |

When names and tags disagree, fix the tags first. Tooling should trust tags.

## Account Naming

AWS accounts use:

```text
org-<domain>-<env>
```

Examples:

| Account | Meaning |
| --- | --- |
| `org-listings-prod` | Production account for the listings domain. |
| `org-search-dev` | Development account for search. |
| `org-platform-shared` | Shared platform account. |
| `org-security-prod` | Security-owned production account. |

Use one primary purpose per account. Do not combine unrelated systems or ownership models inside one account just to reduce account count.

## Resource Naming

Resources use:

```text
org-<domain>-<system>-<component>-<env>[-<qualifier>]
```

Examples:

| Resource | Meaning |
| --- | --- |
| `org-listings-search-api-prod` | Production search API in the listings domain. |
| `org-platform-deploy-runner-dev` | Development deployment runner owned by platform. |
| `org-finance-billing-events-prod-dlq` | Dead-letter queue for billing events. |

Rules:

* Use lowercase kebab-case.
* Use the same `domain`, `system`, and `component` vocabulary as GitHub and package names.
* Add a `qualifier` only when the base name is not specific enough.
* Do not encode secrets, personal names, team jokes, or temporary project names.
* Prefer stable system names over current implementation details.

## Required Tags

Every taggable resource must carry:

| Tag | Meaning |
| --- | --- |
| `Owner` | Owning team or accountable group. |
| `Domain` | Business or platform domain. |
| `System` | System or product capability. |
| `Component` | Deployable or logical component. |
| `Environment` | `prod`, `stage`, `dev`, `test`, `sandbox`, or approved equivalent. |
| `CostCenter` | Finance allocation code. |
| `DataClass` | Highest data classification processed or stored. |
| `Compliance` | Relevant compliance scope, or `none`. |
| `ManagedBy` | `terraform`, `cloudformation`, `cdk`, `manual-exception`, or approved tool. |
| `RepoUrl` | Canonical source repository URL. |

Recommended tags such as `ServiceTier`, `Lifecycle`, `Criticality`, and `RunbookUrl` may be added when useful, but they do not replace the required set.

## Controlled Vocabularies

These values must come from governed lists:

| Field | Why controlled |
| --- | --- |
| `Domain` | Enables ownership, cost, and architecture grouping. |
| `System` | Connects AWS resources to repositories and packages. |
| `Environment` | Enables environment-scoped controls and dashboards. |
| `CostCenter` | Enables finance reporting. |
| `DataClass` | Enables security and privacy controls. |
| `Compliance` | Enables audit and regulatory reporting. |
| `ManagedBy` | Enables drift detection and manual-exception review. |

If a needed value is missing, raise a vocabulary change. Do not invent a local spelling.

## Enforcement

Enforcement should be automated:

| Layer | Check |
| --- | --- |
| IaC review | Names and tags exist before merge. |
| Tag policies | Allowed keys and values are constrained. |
| SCPs | Non-compliant resource creation can be blocked in production. |
| AWS Config | Existing resources are monitored for drift. |
| Inventory export | Ownership, cost, compliance, and repo links are queryable. |

Manual review is useful during migration, but it cannot be the long-term control.

## What This Means for Teams

* Choose `domain`, `system`, and `component` before provisioning.
* Keep AWS, GitHub, and package names aligned.
* Treat missing required tags as a release blocker for production resources.
* Use `manual-exception` only with a tracked reason and expiry.
* Fix tags when ownership, data classification, or cost allocation changes.

## Anti-Patterns

* **Name-only governance.** Automation tries to parse everything from resource names.
* **Free-form tag values.** Dashboards fill with spelling variants.
* **Multi-purpose accounts.** Isolation and cost signals become unclear.
* **Owner as person.** A resource is tagged to an individual instead of an accountable team.
* **Manual resources without expiry.** Emergency work becomes permanent drift.
* **Different vocabularies per system.** AWS, GitHub, and packages describe the same thing differently.

## Scope and Revisiting

This applies to Organization-managed AWS accounts and taggable resources. It does not define non-AWS cloud naming, IAM policy design, account-vending workflow, or migration steps for every existing resource.

Revisit when AWS introduces materially better governance mechanisms, when the domain vocabulary changes, or when repeated exceptions show that the standard is missing a real pattern.
