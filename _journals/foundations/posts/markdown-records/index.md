> **Status**: DRAFT
>
> **Decision**: High-quality Organization technical writing - decisions, strategies, principles, foundations, and discussion records - will live as markdown records in git and be published as a static site. The markdown source is canonical. Pull requests, linked Slack threads, meetings, and temporary Google Docs may support discussion, but they do not replace the record.

## Decision

Use **markdown records in git, published as a static site**, as the default format for durable technical writing.

Concretely:

* Source files live under `_journals/<journal>/posts/<slug>/index.md`.
* Each post has stable front matter: `title`, `date`, `author`, `permalink`, `excerpt`, and where relevant `id` and `status`.
* Published pages are generated by the repository's small static-site build.
* Durable decisions use stable `permalink` and `id` values.
* Cross-record links use `[[slug]]` so titles can change without breaking references.
* Substantive new or rewritten posts should have a sibling `spec.md` as described in [[spec-driven-authoring]].

The goal is not to make markdown sacred. The goal is to keep important reasoning reviewable, versioned, searchable, linkable, and easy for humans and AI tools to use.

## How to Read This Decision

This record decides where durable reasoning lives. It is not a full writing manual.

| Use markdown records for | Do not use them for |
| --- | --- |
| Architecture decisions and ADRs. | Ephemeral meeting notes. |
| Strategy documents that should remain citable. | Draft-only collaboration where the final artifact will be elsewhere. |
| Engineering principles and foundations. | Jira ticket status updates. |
| Discussion records that need a stable URL. | Private people, legal, or vendor-sensitive material. |
| Standards and conventions that automation or AI agents should read. | One-off chat answers. |

If a document is expected to guide future teams, survive beyond one project, or be cited by another record, it belongs here.

## Record Types

| Type | Purpose | Typical status |
| --- | --- | --- |
| **Decision** | Records a choice and its consequences. | `proposed`, `accepted`, `superseded`, `deprecated` |
| **Foundation** | Establishes an enduring operating convention or ritual. | `draft`, `accepted` |
| **Principle** | States a durable engineering belief and its implications. | `accepted` |
| **Strategy** | Explains direction, sequencing, and trade-offs over time. | `draft`, `accepted`, `superseded` |
| **Discussion** | Captures a structured argument before a decision exists. | `draft`, `closed` |

Use the smallest type that honestly fits. Do not call a guide a decision unless it decides something.

## Minimum Structure

Most durable records should be readable in this order:

1. Front matter.
2. Opening highlight: `Status` plus `Decision` or `Principle`.
3. The decision or statement in direct language.
4. How to read or apply the record.
5. Rationale and consequences.
6. What this means for teams.
7. Anti-patterns.
8. Related records.
9. Scope and revisiting.
10. Authoritative references.
11. Appendices for vocabulary, templates, examples, or playbook material.

This order is a convention, not a prison. The main test is whether a busy reader can understand the decision before reaching supporting detail.

## Metadata Contract

Keep these fields stable:

| Field | Rule |
| --- | --- |
| `id` | Stable record identifier. Do not change when the title changes. |
| `permalink` | Stable URL slug. Do not change after publication. |
| `status` | Visible decision state. Keep it aligned with the opening highlight. |
| `title` | Human-readable title. May be improved when the meaning stays the same. |
| `excerpt` | Short index-card summary. Keep it decision-focused. |
| `tags` | Search and grouping hints. Avoid turning tags into prose. |

Changing `id` or `permalink` is a migration, not an edit.

## Authoring Workflow

The default workflow is:

1. Write or update the sibling `spec.md` for substantive work.
2. Draft the record in markdown.
3. Link related records with `[[slug]]`.
4. Keep examples and templates after the decision path, preferably in appendices.
5. Run `python3 _wiring/build.py`.
6. Review the diff in source and generated docs.
7. Merge when the record is clear enough to cite.

Temporary collaboration in Slack, Google Docs, Miro, or meetings is fine. The durable outcome must land back in markdown.

## Discussion and Publication

Markdown records work because they separate discussion from the source of truth:

| Place | Role |
| --- | --- |
| Pull request | Review the wording, links, status, and implications. |
| Slack thread | Collect fast feedback or advertise a proposal. |
| Architecture Advisory Forum | Discuss cross-team trade-offs and exceptions. |
| Google Docs | Optional temporary drafting space for people who need comments or suggestions. |
| Markdown file | Canonical decision, strategy, principle, or foundation. |
| Generated HTML | Published reading surface. |

Do not leave the final decision in a chat thread or slide deck.

## Quality Bar

A good record:

* says the decision before explaining the background;
* names the accountable owner or decision context where relevant;
* distinguishes what is decided from what remains team-owned;
* links related records instead of repeating them;
* includes consequences, not only benefits;
* states how it will be revisited;
* keeps vocabulary, templates, and long examples in appendices.

If a record grows into a manual, split it. The decision should stay short enough to review.

## AI Use

AI assistants may draft, restructure, summarize, and cross-link records, but the repository remains an engineering artifact:

* The human author owns meaning and approval.
* The agent should preserve `id` and `permalink`.
* The agent should build and validate after substantive changes.
* The agent should not invent sources or silently change decision semantics.
* Session logs may explain how a change was produced, but the record itself must stand on its own.

This is why records should be explicit and structured: they are read by people and by tools.

## Anti-Patterns

* **Decision in Jira only.** A ticket may track work; it is not a durable decision record.
* **Wiki drift.** A page changes over time without review, status, or stable links.
* **Slide-deck strategy.** Direction exists only in a presentation that cannot be versioned or linked reliably.
* **Record as manual.** The decision is buried under implementation instructions.
* **Unstable slugs.** A published URL changes because the title was improved.
* **Copy-paste references.** Related decisions are repeated instead of linked.

## Scope and Revisiting

This decision applies to durable technical writing owned by Organization engineering.

Out of scope:

* confidential people, legal, vendor, or security material;
* team-local runbooks whose value is only operational;
* generated API docs or code reference docs;
* transient planning notes.

Revisit this decision if the repository can no longer scale, access-control needs exceed the static-site model, or another tool provides a better combination of versioning, review, publication, and AI readability.
