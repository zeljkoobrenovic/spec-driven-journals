> **Decision**: Organization will use Jira as the standard engineering issue tracker and adopt four shared workflows across teams: Epic, Spike, Story & Bug, and Task. The shared element is the state machine and the meaning of each transition. Teams still own their boards, sprint rituals, estimation, priorities, filters, and day-to-day operating model.

## Decision

Adopt Jira as the canonical issue tracker for engineering delivery work and standardize four workflows:

| Workflow | Used for | Main flow | QA gate |
| --- | --- | --- | --- |
| **Epic** | A container for related delivery work. | `Backlog -> In progress -> Done` | No |
| **Spike** | A time-boxed investigation that ends in evidence or a decision. | `Open -> In progress -> Ready -> Closed` | No |
| **Story & Bug** | User-visible change, customer-impacting defects, and product behaviour. | `Open -> In progress -> In review -> Ready for QA -> In QA -> To deliver -> Closed` | Yes |
| **Task** | Technical-only work where code review and deployment verification are sufficient. | `Open -> In progress -> In review -> To deliver -> Closed` | No |

The goal is a shared language for work in flight. A status should tell any engineer, QA engineer, product person, or manager what is true now, who owns the next action, and where work may be waiting.

## How to Read This Decision

This is a workflow-state decision, not a delivery-process decision.

| Standardized | Team-owned |
| --- | --- |
| Issue types and their workflows. | Sprint cadence, planning, estimation, and ceremonies. |
| State names and transition meaning. | Board layout, swimlanes, quick filters, labels, and components. |
| Split QA queue for Story & Bug work. | QA staffing model and team-level testing practice. |
| Explicit rework transitions. | Retrospective format and local improvement experiments. |

If a team needs a local state, the default answer is not to add one locally. First ask whether the need is already covered by an existing state, a board filter, or a custom field. If the need is genuinely shared, raise a workflow-change proposal.

## Choosing the Right Workflow

Use the work's output and risk, not its size:

| Question | If yes |
| --- | --- |
| Is this a container for several child issues? | Use **Epic**. |
| Is the expected output learning, evidence, a decision, or a recommendation? | Use **Spike**. |
| Does this change user-visible behaviour, customer experience, business rules, displayed data, or production risk that QA should verify? | Use **Story & Bug**. |
| Is this technical-only work where code review and deployment verification are enough? | Use **Task**. |

Useful rules:

* A small user-visible fix is still Story & Bug work.
* A large refactoring can still be a Task if behaviour does not change.
* Mixed work should usually be split: technical cleanup in a Task, user-facing behaviour in Story & Bug.
* Reclassify when reality changes; do not stretch the wrong workflow to avoid moving the issue.

## QA Model

Story & Bug work has two QA states:

| State | Meaning |
| --- | --- |
| `Ready for QA` | Review passed; the issue is waiting for QA capacity. |
| `In QA` | A QA engineer is actively testing. |

This split matters because waiting time and testing time are different constraints. A growing `Ready for QA` queue suggests capacity, scheduling, or handoff problems. Long `In QA` time suggests test-data, environment, acceptance-criteria, or complexity problems.

Failed review and failed QA return explicitly to `In progress`. Comments alone are not enough because the transition is the signal that makes rework visible.

## Transition Discipline

Move an issue when a fact changes:

| Transition | What must be true |
| --- | --- |
| `Open -> In progress` | Someone has started active work. |
| `In progress -> In review` | A reviewable artifact exists. |
| `In review -> Ready for QA` | Code review passed and QA has enough context. |
| `Ready for QA -> In QA` | QA actively started testing. |
| `In QA -> To deliver` | QA passed and the issue is ready for release or production verification. |
| `To deliver -> Closed` | Release and verification are complete. |
| Any active state -> `To refine` | The issue lacks enough clarity to continue safely. |
| Any non-terminal state -> `Abandoned` | The work is intentionally stopped. |

Do not move issues for optics. Do not leave failed review or failed QA hidden in comments. Leave enough evidence for the next person to understand the transition.

## Metrics This Enables

The standard makes metrics comparable because every team uses the same state semantics:

| Metric | Signal |
| --- | --- |
| Lead time | How long work waits from creation or commitment to closure. |
| Cycle time | How long active delivery takes from `In progress` to `Closed`. |
| QA wait time | How long work waits in `Ready for QA`. |
| QA execution time | How long active testing takes in `In QA`. |
| Rework rate | How often review or QA sends work back to `In progress`. |
| Blocked time | Which dependencies repeatedly stop delivery. |

Use these metrics to improve the system of work, not to rank individuals.

## What This Means for Teams

* Pick the issue type before work enters delivery.
* Keep issue state aligned with reality.
* Use `Ready for QA` and `In QA` honestly.
* Route review and QA failures back through explicit rework transitions.
* Keep local customization in boards, filters, labels, fields, and ceremonies.
* Raise recurring workflow gaps through the agreed governance path instead of adding local statuses.

## Anti-Patterns

* **Single `QA` state.** Hides whether work is waiting or actively being tested.
* **Task as a fast path.** Uses a technical workflow to bypass QA for user-visible change.
* **Comment-only rework.** Makes quality feedback invisible to flow metrics.
* **Local workflow drift.** Recreates the cross-team ambiguity this standard is meant to remove.
* **Blocked as prioritization.** Uses `Blocked` for "we are busy" rather than a named external dependency.
* **Spike that never closes.** Leaves investigation work without evidence, decision, or follow-up.

## Scope and Revisiting

This decision applies to engineering delivery work tracked in Jira: epics, spikes, stories, bugs, and technical tasks.

Out of scope:

* Product discovery boards that are not part of engineering delivery.
* Operational ticket queues already managed in a different tool.
* Team-specific estimation, planning, ceremonies, and dashboards.
* Custom fields, components, labels, and board configuration.

Revisit when a recurring work type does not fit the four workflows, when QA metrics show the bottleneck has moved, or when Jira is replaced as the issue tracker.

## Appendix: Vocabulary

This vocabulary is reference material. It should support the workflow decision without turning the record into a Jira manual.

| Term | Meaning in this standard |
| --- | --- |
| **Issue type** | The category that selects the workflow: Epic, Spike, Story, Bug, or Task. |
| **State** | A named phase with clear entry and exit meaning. |
| **Transition** | An allowed state change that records a real event. |
| **Queue** | Work is ready for someone but not yet actively handled. |
| **Active work** | Someone is currently doing the work represented by the state. |
| **Gate** | A quality check that must pass before work moves forward. |
| **Rework** | Work returns to an earlier state because a gate found a problem. |
| **Blocked** | Progress is prevented by a named external dependency. |
| **Abandoned** | Work is intentionally stopped because it is cancelled, superseded, or no longer valuable. |
| **Closed** | The workflow's definition of done is satisfied. |
