> **Status**: DRAFT
>
> **Decision**: Organization uses one logical package name per artifact, `<domain>-<system>-<component>`, and maps it consistently into each package ecosystem: JVM, npm, Python, .NET, and OCI. Library versions follow SemVer. Internal packages publish to internal registries by default. Public publication requires an explicit decision.

## Decision

Adopt one logical naming model for packages:

```text
<domain>-<system>-<component>
```

Map that name into each ecosystem's native shape:

| Ecosystem | Shape | Example |
| --- | --- | --- |
| JVM group/package | `io.organization.<domain>.<system>.<component>` | `io.organization.listings.search.client` |
| Maven artifact | `<domain>-<system>-<component>` | `listings-search-client` |
| npm | `@organization/<domain>-<system>-<component>` | `@organization/listings-search-client` |
| Python distribution | `org-<domain>-<system>-<component>` | `org-listings-search-client` |
| Python import | `org_<domain>_<system>_<component>` | `org_listings_search_client` |
| NuGet | `Organization.<Domain>.<System>.<Component>` | `Organization.Listings.Search.Client` |
| OCI image | `<registry>/organization/<domain>/<system>/<component>` | `registry.example/organization/listings/search-api` |

The logical name should align with GitHub repository names and AWS tags.

## How to Read This Decision

This is a mapping rule, not a packaging tutorial. Each ecosystem keeps its own build tools and metadata files. The standard decides how an Organization artifact is named, versioned, and published.

| Standardized | Ecosystem-owned |
| --- | --- |
| Logical name. | Build tool details. |
| Registry choice. | Dependency declaration syntax. |
| Versioning posture. | Runtime packaging mechanics. |
| Public-publication rule. | Language-specific release automation. |

## Naming Rules

* Use shared `domain`, `system`, and `component` vocabulary.
* Use stable capability names rather than implementation details.
* Avoid team names, individual names, temporary project names, and vendor names.
* Use lowercase kebab-case for logical names.
* Use PascalCase only where the ecosystem requires it, such as NuGet.
* Keep import names predictable from distribution names.

If the same artifact appears in multiple ecosystems, the logical identity should still be recognizable.

## Registry Rules

| Artifact | Default registry |
| --- | --- |
| JVM packages | Internal Maven-compatible registry. |
| npm packages | Internal npm registry or GitHub Packages under Organization control. |
| Python packages | Internal PyPI-compatible registry. |
| NuGet packages | Internal NuGet feed. |
| OCI images | Internal container registry. |

Public publication is an exception. It needs a clear owner, license review, security review, maintenance commitment, and explicit decision record.

## Versioning

Library and package versions follow SemVer:

```text
MAJOR.MINOR.PATCH
```

Rules:

* The published version must match the git release tag.
* Breaking API changes increment `MAJOR`.
* Backward-compatible additions increment `MINOR`.
* Compatible fixes increment `PATCH`.
* Pre-release versions may use SemVer pre-release labels.

Deployable services are versioned by release automation and OCI image tags. Mutable tags such as `latest` may exist for convenience, but they are not sufficient for audit or rollback.

## OCI Images

OCI image names use path structure:

```text
<registry-host>/organization/<domain>/<system>/<component>
```

Required OCI labels should include source repository, revision, build time, owning team, and license where applicable. OCI names should align with AWS resource names and GitHub repositories where possible.

## What This Means for Teams

* Choose the logical package name before publishing.
* Keep package, repository, and cloud vocabulary aligned.
* Publish internal artifacts only to internal registries unless an ADR says otherwise.
* Treat public package publication as a product commitment.
* Do not rename packages casually; consumers depend on names.
* Keep package metadata, ownership, and source links current.

## Anti-Patterns

* **Ecosystem-specific identity.** The same artifact has unrelated names in JVM, npm, and OCI.
* **Team-owned names.** The package name changes meaning when team ownership changes.
* **Public by accident.** A package leaks to a public registry because the default registry was not explicit.
* **Version drift.** Git tags, package versions, and release notes disagree.
* **Ambiguous clients.** Names like `common-client` or `utils` hide the owning system.
* **Mutable release identity.** Consumers cannot reproduce what was deployed.

## Scope and Revisiting

This applies to packages and images produced by Organization engineering in JVM, npm, Python, .NET, and OCI ecosystems. It does not define dependency-management policy, vulnerability scanning, binary retention, or language-specific build standards.

Revisit when a new package ecosystem becomes important, when the domain vocabulary changes, or when registry tooling provides better metadata enforcement.
