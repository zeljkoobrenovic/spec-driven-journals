> **Status**: DRAFT
>
> **Decision**: Substantive changes to this repository are made through an AI coding agent (Claude Code, Codex CLI, or an equivalent successor) rather than by hand. Authors describe intent in a session with the agent; the agent edits files and runs the build; the author reviews the resulting diff and merges it. The workflow exists to preserve reasoning, not to outsource judgement. Trivial fixes (typos, broken links, status flips, dependency bumps) may be hand-edited and noted in the PR.

## Decision

Adopt **AI-mediated authoring** as the default workflow for changing the contents of this repository. Concretely:

* **Mediator**: an AI coding agent (Claude Code, Codex CLI, or a comparable CLI tool that can read and edit the working tree) performs all substantive file edits.
* **Author role**: the human author provides intent, context, judgement, review, and final approval. The author does not normally type into post files directly.
* **Scope**: applies to all journals under `_journals/` - posts, configs, templates, build scripts, conventions in `CLAUDE.md`, and editorial assets.
* **Exception path**: trivial mechanical fixes (typos, broken links, `status:` value flips, version bumps, file renames driven by tooling) may be hand-edited. The pull request description must say so explicitly so reviewers know to apply a lighter check.
* **Learning goal**: authors learn to express intent, constraints, and review criteria clearly; reviewers learn to evaluate both the content change and the reasoning trail.

This decision **does not** dictate which AI tool authors use, **does not** prevent direct editing of code or content outside `_journals/`, and **does not** remove human accountability for the result - see [[ai-policy]].

## How to Read This Decision

Five ideas, easy to confuse:

| Idea | Meaning | Practical implication |
| --- | --- | --- |
| **AI-mediated** | The agent makes the edit; the human directs and reviews. | Authors describe intent; they do not paste finished prose into files. |
| **Tool-agnostic** | Claude Code today, Codex CLI tomorrow, something else next year. | The policy targets the *workflow*, not a specific vendor. |
| **Sessioned** | Each change ships with a session log naming author, intent, journals affected. | The reasoning behind a change survives, not just the diff. |
| **Reviewable** | The agent's diff is reviewed in a PR like any other change. | The human is still accountable for what lands. |
| **Pragmatic** | Trivial fixes can skip the session, with a note in the PR. | The rule guides substantive work; it does not block one-character fixes. |

The point is not to remove humans from authoring. It is to make the *thinking* that produced each change as durable as the change itself.

## What This Should Help You Do

This foundation should help an author or reviewer:

* Explain the difference between AI-mediated authoring and "AI wrote it for me".
* Decide whether a change is substantive enough to require an AI session and session log.
* Run a session that starts from intent, constraints, existing records, and review criteria.
* Write a session log that preserves useful reasoning without becoming a transcript.
* Review a PR by checking the content diff, the session log, the build result, and human accountability.

The skill this workflow teaches is not prompt-writing in isolation. It teaches **traceable authoring**: state intent clearly, let the agent operate on the repository, review the artifact, and preserve the reasoning for the next reader and the next AI session.

## One-Minute Mental Model

Think of the repository as a place where two artifacts are produced together:

1. **The record** - the markdown file under `_journals/<journal>/posts/`.
2. **The conversation** - a session log under `_journals/ai-conversations/`.

The record says *what we decided*. The conversation says *how we got there, what we asked, what we rejected, and what the agent did*. Both are markdown in git. Both are reviewable. Both are AI-readable on the next session.

A change without a conversation is a change whose reasoning was lost the moment the author closed their terminal.

* Claude Code as the standard AI coding agent - see [[claude-code-as-standard]].

This foundation is the missing piece: it states that the *authoring action itself* runs through the agent, not just the engineering work the agent supports. It binds the agent to the artifact at the moment of creation, not after the fact.

## Substantive vs Trivial

The practical boundary is whether the change could affect what a future reader believes, cites, or does.

| Change | AI-mediated session? | Reason |
| --- | --- | --- |
| Add a new foundation, principle, strategy, or ADR. | Yes | New meaning enters the corpus. |
| Rewrite a section for clarity while preserving meaning. | Yes | Reviewers need to see intent and editorial judgement. |
| Add examples, checklists, or learning outcomes. | Yes | The record becomes more directive and educative. |
| Change tags, excerpt, title, or section placement. | Usually yes | Discovery and interpretation change, even if the decision does not. |
| Fix a typo in one sentence. | No, if noted as trivial. | Meaning does not change. |
| Fix a broken `[[link]]`. | No, if noted as trivial. | Mechanical repair. |
| Flip `status:` after the body already reflects the accepted state. | No, if noted as trivial. | Metadata catches up with an already-recorded outcome. |
| Change a decision from proposed to accepted and update rationale. | Yes | Meaning and organisational commitment change. |

When unsure, treat the change as substantive. The cost of a short session log is lower than the cost of losing the reasoning behind a meaningful edit.

## What Belongs in a Session

A session is a single working interaction between an author and an AI agent that ends in one or more commits (or no commits, if the conversation produced advice only).

| Belongs in a session log | Does not belong |
| --- | --- |
| The author's intent in their own words. | A verbatim transcript of every prompt and reply. |
| Decisions made during the session (including ones the author overrode). | Token-level disagreements that were resolved cleanly. |
| Files that changed and a one-line reason for each group of changes. | A copy of the diff (git already has it). |
| Follow-ups, open questions, or records worth writing next. | Speculative ideas the author already discarded. |
| Cross-links to affected journals and records. | Re-stating the content of the records that changed. |

The session log is a *short structured summary*, not a transcript. Think of it as the changelog entry for the reasoning behind a commit.


## Session Log Quality Bar

A good session log lets a future reader understand why the change exists without replaying the whole conversation.

| Quality | Strong session log | Weak session log |
| --- | --- | --- |
| **Intent** | States the author's goal and constraints in plain language. | "Updated docs." |
| **Outcome** | Summarises what actually changed. | Copies the git diff or lists every paragraph touched. |
| **Decisions** | Names choices made during the session and who made them. | Says "the AI decided" or hides the human judgement. |
| **Files changed** | Groups files by journal or purpose. | Dumps a long path list without explanation. |
| **Follow-ups** | Captures real next steps or explicitly says none. | Adds generic "review later" filler. |
| **Cross-links** | Links the records affected or implied. | Leaves future readers to search manually. |

The log should be honest about uncertainty. If the author rejected an agent suggestion, say so when it matters. If a decision was inferred from existing records rather than newly made, say that too.

## Naming

* **Author ID**: short kebab-case identifier of the human author (typically firstname-lastname or a stable handle). Same ID across sessions, so all of an author's logs sit under one folder.
* **Session ID**: `YYYY-MM-DD-short-slug`, where the slug captures the session's intent in two or three words. If an author runs more than one session in a day, append `-2`, `-3`, etc.
* **Permalink**: `<author-id>-<session-id>`. This is what `[[…]]` cross-links resolve to and what appears in the URL.

Author ID and session ID together must produce a globally unique permalink within the journal.

## Workflow

A standard session looks like this:

1. **Pull main**, branch if helpful, and open the AI agent (`claude` or `codex`) in the repo root.
2. **State intent** to the agent in plain language. Reference existing records with `[[name]]` so the agent picks up context.
3. **Let the agent edit**. Review proposed edits as they happen; redirect when something is off.
4. **Run the build** (`python3 _wiring/build.py`) via the agent and inspect the output.
5. **Edit the session log** with the agent until it accurately reflects what happened. Keep it short.
6. **Commit and open a PR**. The PR description should link the session log and the records that changed.
7. **Reviewer reads both** the record change and the session log. The session log is part of the review surface, not an afterthought.

If a session produced advice only and no file changes landed, the session log is still written - it captures why the advice did not turn into an edit, which is itself a decision.

## Worked Examples

### Example 1: Substantive Edit

An author asks:

> Improve `[[architecture-advisory-forum]]` and make it more educative.

This is substantive. The agent reads the record and nearby foundations, edits the source markdown, adds learning outcomes and practical prompts, validates front matter, and reports what changed. The session log should say:

* Intent: make the AAF record more useful for onboarding and facilitation.
* Outcome: added learning sections, prompts, examples, and validation notes.
* Decisions: kept the AAF advisory-only; did not change decision ownership.
* Files changed: the AAF foundation source, generated docs only if intentionally rebuilt.
* Follow-ups: none, or a specific next record if one emerged.

### Example 2: Trivial Fix

An author fixes a broken link from `[[markdown-record]]` to `[[markdown-records]]`.

This can be hand-edited if the PR says: `hand-edited, trivial: broken cross-link repair`. No session log is required because the meaning is unchanged and the reasoning is obvious.

### Example 3: Advice-Only Session

An author asks the agent whether the repository should add build-time enforcement that every PR touching `_journals/` links a session log. The agent analyses options, but the author decides not to edit anything yet.

This still deserves a session log if the advice shaped a real decision. The outcome is "no file changes landed"; the decision is "defer enforcement until compliance gaps appear"; the follow-up may be "revisit after ten sessions".

## What This Means for Teams

* **Authors** should expect to spend more time describing intent and reviewing diffs, and less time typing into markdown files. Treat the agent as a junior editor with full repo context, not a search box.
* **Reviewers** should treat the session log as part of the change. A PR without a session log (and without an explicit "trivial fix" note) is incomplete.
* **Journal owners** should keep their `CLAUDE.md` and `README` up to date so the agent has the context it needs. Bad context produces bad edits.
* **Tooling owners** should keep the AI agent contract stable: the same `_wiring/build.py`, the same metadata shape, the same `[[…]]` resolver. The policy assumes the agent can read and modify the repository safely.

## Reviewer Checklist

Reviewers should inspect both the changed record and the session log. Useful questions:

* Does the diff match the stated intent?
* Does the session log explain the reasoning without becoming a transcript?
* Are human decisions clearly attributed to the author rather than the agent?
* Did the agent run the relevant validation or explain why it did not?
* Are generated files included or omitted deliberately?
* Are links, metadata, status, and read time still coherent?
* Is a "trivial fix" claim credible, or did the change alter meaning?

The reviewer is not reviewing the AI as a person. The reviewer is reviewing a human-authored change made through an AI tool.

## Author Prompts That Work Well

Good sessions start with intent and review criteria. Useful prompt shapes:

* "Improve `[[record]]` for new readers. Preserve the decision, add education and practical examples, and keep edits source-only."
* "Review this journal for inconsistent terminology. Propose a scoped edit before changing files."
* "Make this post more actionable for teams. Add checklists and examples, but do not change policy meaning."
* "Refactor these sections for clarity. Keep `id`, `permalink`, status, and cross-links stable."
* "After editing, parse front matter, check heading structure, and tell me whether generated docs were touched."

Weak sessions start with output-only instructions like "rewrite this better" or "make it sound strategic". The agent needs intent, constraints, and success criteria.

## Anti-Patterns

Avoid:

* **Hand-edits with a fake session log written after the fact.** The log is supposed to capture a real interaction, not justify a manual change.
* **Transcript dumps.** A 4 000-word paste of the chat is not a session log; it is a transcript. Summarise.
* **One mega-session per week.** If a session touches ten unrelated records, split it.
* **Author ID drift.** Pick one identifier per human and stick with it across sessions so logs accumulate under one folder.
* **Skipping the build.** A session that didn't run `python3 _wiring/build.py` did not verify its own output.
* **Treating "trivial fix" as a default.** The exception is for genuinely mechanical edits, not for "I was in a hurry".
* **Letting the agent invent decisions.** The agent edits; the human decides. A session log should attribute decisions to the human, not the tool.
* **Diff-only review.** Looking only at changed files and ignoring the session log loses the reasoning trail.
* **Session log as marketing.** Generic praise about AI productivity is not evidence. The log should say what happened.

## Common Misunderstandings

| Misunderstanding | Correction |
| --- | --- |
| "AI-mediated means AI owns the content." | No. The human author owns intent, judgement, review, and approval. |
| "The session log is an audit transcript." | No. It is a concise reasoning summary, not a verbatim recording. |
| "Using an agent means the PR can be reviewed faster." | Sometimes, but the reviewer still checks meaning, diff, validation, and log. |
| "Trivial fixes never need review." | They still need review; they just do not need a full session log. |
| "The chosen tool matters most." | The durable workflow matters more than whether the agent is Claude Code, Codex CLI, or a successor. |
| "Generated docs must always be committed." | Generated docs are committed when the repository workflow requires it; source-only edits may intentionally leave them untouched. |

## Exceptions

Hand-editing is acceptable, without a session log, for:

* Typo and grammar fixes inside an existing paragraph.
* Broken links and `[[name]]` slugs.
* Front-matter `status:` value flips (e.g. `proposed:orange` → `accepted:green`) where the body already reflects the new status.
* Build-script dependency or version bumps that do not change behaviour.
* File renames driven by tooling (e.g. `_wiring/build.py` enforcing a new naming rule).

In each case the pull request description must say "hand-edited, trivial" and name the category, so the reviewer can apply a lighter check rather than wondering where the session log is.

Anything that *changes the meaning* of a record - even slightly - is not trivial and goes through the agent.

## Considered Options

* **AI-mediated by default with explicit trivial-fix exceptions** (chosen).
* **AI use is optional; authors choose tool per change.** Cleaner for individual freedom, but loses the reasoning trail and makes AI-readiness of records uneven.
* **Hard rule, no exceptions.** Cleaner principle, but creates friction for one-character fixes and tempts authors to dress up trivial work as a session.
* **AI as reviewer only, humans edit.** Inverts the model; humans still type and the agent comments. Misses the point: the agent should be the one operating the keyboard so its context is fresh on the next session.

## Consequences

### Positive

* Reasoning behind each change is captured at the time of the change, not reconstructed later.
* The agent's session log becomes durable context for the *next* session - this repository teaches itself.
* Human accountability is preserved through diff review and PR approval, unchanged from current practice.
* Tool-agnostic phrasing means the policy survives the next CLI tool generation.
* `CLAUDE.md` files become operationally important, not decorative - the agent reads them every session.
* Trivial-fix exceptions keep the rule honest and avoid theatre.

### Negative

* Authors who prefer to type into markdown directly will feel slowed down at first.
* The `ai-conversations` journal will grow quickly; index search and folder hygiene will matter.
* Session logs that are written badly (transcripts, generic filler) will erode the value of the journal.
* The policy assumes a stable AI tool contract. If Claude Code, Codex CLI, or successors change behaviour materially, conventions may need to follow.
* Auditing compliance ("did this change really come from a session?") relies on author honesty until tooling enforces it.

## Scope and Revisiting

In scope: any change to files under `_journals/`, `_templates/`, `_wiring/`, `CLAUDE.md`, and `docs/` insofar as it is regenerated by the build.

Out of scope:

* Code in repositories outside this one.
* Personal scratch notes that never enter a journal.
* Operational fixes to CI, deployment, or hosting infrastructure that happen to live nearby.

Revisit this decision when:

* A successor CLI tool changes the AI-agent contract in a way that materially affects authoring.
* A practical alternative emerges for capturing the reasoning behind a change without a session log.
* Compliance gaps appear that justify build-time enforcement (e.g. requiring every PR touching `_journals/` to link a session log).

## Appendix: Vocabulary



The workflow works better when the words are precise.

| Term | Meaning in this repository | Example |
| --- | --- | --- |
| **Author** | The human accountable for the change. | "Zeljko asked for the foundation to be improved and reviewed the diff." |
| **Agent** | The CLI tool operating on the working tree. | Claude Code, Codex CLI, or a successor tool. |
| **Session** | One bounded interaction with the agent around one coherent intent. | "Improve the AAF foundation and make it more educative." |
| **Substantive change** | A change that alters meaning, structure, guidance, workflow, assets, or generated output. | Adding sections, rewriting a post, changing a workflow, restructuring a journal. |
| **Trivial fix** | A mechanical edit with no meaningful change to content or policy. | Typo, broken link, status flip already supported by the body. |
| **Review surface** | Everything the reviewer should inspect. | Diff, session log, build result, generated docs if relevant. |

The most important distinction is between **the agent operating the keyboard** and **the human owning the judgement**. The first is delegated. The second is not.
