> **Status**: DRAFT
>
> **Decision**: Organization standardizes source-code naming and tagging in GitHub. Repository names use `<domain>-<system>-<component>[-<kind>]` in lowercase kebab-case. Repositories carry required flat topics with namespace prefixes. Branches include Jira keys. Commits follow Conventional Commits with Jira traceability. Release tags use SemVer for libraries and date-based release tags for deployable services.

## Decision

Adopt one source-code naming and tagging model for the Organization GitHub organization.

| Area | Standard |
| --- | --- |
| Repository name | `<domain>-<system>-<component>[-<kind>]` |
| Repository topics | Required flat topics such as `domain-listings`, `system-search`, `lifecycle-active`, `tier-1`, `lang-kotlin`. |
| Branch name | `<type>/<JIRA-KEY>-<short-slug>` |
| Commit message | Conventional Commits plus Jira key in the footer. |
| Library release tag | `vMAJOR.MINOR.PATCH` |
| Service release tag | `release/YYYY.MM.DD[.N]` |
| CODEOWNERS | Required for production repositories. |

The standard makes the source estate queryable. A repository should reveal what it belongs to, who owns it, how active it is, what language it uses, and how work connects to Jira.

## How to Read This Decision

This is a mechanical convention. It should be boring, predictable, and easy to enforce.

| Standardized | Not standardized here |
| --- | --- |
| Names, topics, branch shape, commit shape, release tags. | Team branching strategy beyond the branch-name shape. |
| Required inventory metadata. | Product roadmap or ownership model. |
| Traceability to Jira. | Jira workflow itself; see [[jira-workflow-standards]]. |
| Repository visibility defaults. | Fine-grained access policy for every team. |

## Repository Names

Use:

```text
<domain>-<system>-<component>[-<kind>]
```

Examples:

| Name | Meaning |
| --- | --- |
| `listings-search-api` | Search API in the listings domain. |
| `platform-deployment-runner` | Platform-owned deployment runner. |
| `finance-billing-contracts` | Billing contracts in the finance domain. |
| `shared-design-system-web` | Web package for a shared design system. |

Rules:

* Use lowercase kebab-case.
* Use shared domain and system vocabulary.
* Add `kind` only when it clarifies the artifact: `api`, `worker`, `contracts`, `web`, `mobile`, `infra`, `docs`, `template`.
* Do not use team names, personal names, temporary project names, vendor names, or jokes as primary segments.
* Prefer durable capability names over current implementation detail.

## Required Topics

Every production repository should carry topics from these families:

| Topic family | Example | Purpose |
| --- | --- | --- |
| `domain-*` | `domain-listings` | Business or platform domain. |
| `system-*` | `system-search` | System or product capability. |
| `component-*` | `component-api` | Component type where useful. |
| `lifecycle-*` | `lifecycle-active` | Active, deprecated, archived, experimental. |
| `tier-*` | `tier-1` | Operational criticality. |
| `lang-*` | `lang-kotlin` | Primary language. |
| `runtime-*` | `runtime-ecs` | Runtime or deployment substrate. |

Topics are flat because the GitHub API treats them as flat strings. Namespace prefixes preserve meaning.

## Branches and Commits

Branches use:

```text
<type>/<JIRA-KEY>-<short-slug>
```

Examples:

```text
feature/IW-1234-search-filter
fix/IW-1888-null-price
chore/IW-2020-upgrade-gradle
```

Allowed branch types: `feature`, `fix`, `chore`, `docs`, `refactor`, `test`, `spike`, `release`, `hotfix`.

Commits follow Conventional Commits:

```text
feat(search): add price filter

Refs: IW-1234
```

The Jira key should be present in the branch name and commit footer unless the change is repository maintenance with no Jira issue.

## Release Tags

Use two release tag families:

| Artifact | Tag |
| --- | --- |
| Library or versioned package | `vMAJOR.MINOR.PATCH` |
| Deployable service | `release/YYYY.MM.DD[.N]` |

Do not move release tags. If a release is wrong, create a new tag and record the correction.

## Repository Metadata

Production repositories should have:

* description that names the purpose in one sentence;
* homepage or documentation link when applicable;
* `CODEOWNERS`;
* required topics;
* default branch protection;
* archived status when no longer active;
* link to owning system or runbook where relevant.

## Enforcement

Enforce mechanically where possible:

| Layer | Check |
| --- | --- |
| Repository creation | Name and required metadata exist. |
| Pull request | Branch name, commit message, and CODEOWNERS expectations. |
| Scheduled inventory | Missing topics, stale lifecycle, missing owners. |
| Release automation | Tag shape and immutability. |

Manual exceptions should have an owner, reason, and review date.

## What This Means for Teams

* Pick the domain, system, and component before creating the repository.
* Keep GitHub names aligned with AWS and package names.
* Use Jira keys consistently so work is traceable.
* Keep topics current when ownership, lifecycle, language, or runtime changes.
* Archive rather than abandon repositories.

## Anti-Patterns

* **Team-name repositories.** Ownership changes, but the name stays misleading.
* **Project-code names.** Temporary initiative names outlive their context.
* **Missing topics.** Inventory becomes impossible without cloning or reading every repo.
* **Moving release tags.** Breaks reproducibility.
* **Branch names without work context.** Changes cannot be traced to a decision or issue.
* **Different vocabulary per layer.** GitHub says one system name while AWS and packages say another.

## Scope and Revisiting

This applies to repositories under the Organization GitHub organization. It does not define the full branching model, access-control model, or CI/CD standard.

Revisit when the domain vocabulary changes, when GitHub adds better structured metadata, or when repeated exceptions show that the naming shape is too narrow.

## Appendix: Vocabulary

| Term | Meaning |
| --- | --- |
| **Domain** | Business or platform area. |
| **System** | Product capability or service family. |
| **Component** | Deployable, library, contract, template, or documentation unit. |
| **Kind** | Optional suffix that clarifies repository type. |
| **Topic** | Flat GitHub metadata label used for inventory and search. |
| **Lifecycle** | Current state of the repository: active, experimental, deprecated, or archived. |
