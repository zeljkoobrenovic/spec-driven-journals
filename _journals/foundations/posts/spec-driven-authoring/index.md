> **Status**: DRAFT
>
> **Decision**: Substantive posts in this repository are drafted against a short sibling `spec.md`. The spec states the post's intent, readers, success criteria, boundaries, open questions, decisions, sources, and changelog. The spec is the authoring contract; the post is the published artifact. Trivial mechanical fixes do not need a spec.

## Decision

Adopt **spec-driven authoring** as the lightweight default for substantial post work.

Concretely:

* Each substantive post folder may include a sibling `spec.md` next to `index.md`.
* The spec is rendered as `<permalink>.spec.html`.
* The spec is written before or during the first drafting pass, not after publication.
* The spec is updated when the intent, scope, decision, or success criteria change.
* Trivial fixes can skip the spec when no meaning or structure changes.

The point is not bureaucracy. The point is to make intent reviewable before an AI agent or human author starts rewriting prose.

## When a Spec Is Required

Use a spec for:

| Work | Spec? |
| --- | --- |
| New decision, strategy, principle, or foundation post. | Yes |
| Major rewrite of an existing post. | Yes |
| Split, merge, or lifecycle change across records. | Yes |
| Changes that affect guidance, policy, status, or cross-links. | Yes |
| Typo fix, broken link, image path, formatting cleanup. | Usually no |
| Generated docs rebuild with no source meaning change. | No |

If the change would be hard to review from the diff alone, write or update the spec.

## What a Spec Contains

Every spec should stay short and practical:

| Section | Purpose |
| --- | --- |
| **Intent** | What the post should accomplish. |
| **Audience** | Who should read it and what they need from it. |
| **Success criteria** | Checkable statements a reviewer can verify. |
| **Non-goals** | Boundaries that prevent scope creep. |
| **Open questions** | Known unresolved issues. |
| **Decision log** | Important choices made while shaping the post. |
| **Sources** | Internal and external material the post relies on. |
| **Changelog** | What changed, when, and by whom. |

The spec should be precise enough to steer work and short enough to read before reviewing the post.

## Workflow

---begin mermaid---
flowchart LR
    Intent[Author states intent] --> Spec[Write or update spec.md]
    Spec --> Draft[Draft or rewrite index.md]
    Draft --> Review[Review against success criteria]
    Review --> Build[Build generated site]
    Build --> Publish[Publish or merge]
    Review -. scope changed .-> Spec
---end mermaid---

The spec and post should move together. A post that no longer satisfies its spec is not ready; a spec that no longer matches the post should be updated.

## Spec vs Post vs Session Log

| Artifact | Role |
| --- | --- |
| `spec.md` | The contract for what the post is trying to achieve. |
| `index.md` | The durable published record. |
| AI conversation/session log | Evidence of how the work happened, useful for traceability but not the source of truth. |
| Pull request | Review surface for source and generated output. |

Do not turn the spec into the post. Do not turn the post into a transcript.

## Drafting Routine

A useful first spec can be written in ten minutes:

1. Write the decision or intended change in one paragraph.
2. Name the readers.
3. Add five success criteria.
4. Add three non-goals.
5. List open questions.
6. Add internal records that should be linked.
7. Start the decision log with why this work is happening now.

That is enough to guide an AI-mediated drafting session.

## Review Checklist

Before accepting a substantive post change, check:

* Does the opening decision match the spec intent?
* Can the intended readers find their answer quickly?
* Are success criteria satisfied by visible content?
* Are non-goals respected?
* Are cross-links stable `[[slug]]` links where possible?
* Are examples and templates supportive rather than dominating the main flow?
* Did the build succeed?
* Did generated docs change as expected?

## What This Means for Teams

* Authors get a clearer prompt for AI-mediated work.
* Reviewers can evaluate intent, not just prose.
* Curators can detect scope drift earlier.
* Future agents can understand why a record has its shape.
* Lightweight specs make large edits safer without requiring a heavyweight process.

## Anti-Patterns

* **Spec after the fact.** Backfilling vague intent after the post is already written.
* **Spec as bureaucracy.** Writing a long template that no reviewer reads.
* **Hidden scope change.** Rewriting the post's decision without updating the spec.
* **Checklist theater.** Marking success criteria complete when the post does not actually satisfy them.
* **Session log as spec.** Treating a conversation transcript as the contract.

## Scope and Revisiting

This decision applies to substantive posts in the journals that publish durable engineering writing: foundations, principles, tech decisions, AI decisions, AI strategy, org design, and tech strategy.

Revisit if specs become too heavy for normal edits, if reviewers ignore them, or if the site grows a better mechanism for authoring contracts.

## Appendix: Vocabulary

| Term | Meaning |
| --- | --- |
| **Spec** | A short sibling `spec.md` that states the authoring contract. |
| **Post** | The published `index.md` record. |
| **Success criterion** | A checkable condition the finished post should satisfy. |
| **Non-goal** | A boundary that prevents the post from expanding into adjacent topics. |
| **Decision log** | A dated list of important authoring or scope choices. |
| **Changelog** | A dated list of edits to the post or spec. |
