> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use Bulma as our CSS framework. The choice is reversible — we will revisit it as the framework landscape evolves.

## Context

We need a CSS framework for our web applications. Three goals shape the choice:

- **Performance and reliability** across modern browsers and screen sizes.
- **Fast iteration** on design, layout, and UI/UX.
- **Responsive layouts** that work on mobile, desktop, ultra-wide, and rotatable displays.

A practical constraint sits on top of these: we want to avoid frameworks that hard-depend on jQuery. jQuery is increasingly redundant against modern JavaScript, conflicts with the DOM model used by React, Vue, and Svelte, and breaks when loaded twice — all of which raise its real cost well above its apparent size.

## Decision

We will adopt **Bulma** as our CSS framework. We remain open to revisiting this choice as new jQuery-free frameworks mature or as our component needs grow.

## Rationale

1. **No jQuery.** Bulma is built with modern techniques and has no jQuery dependency. This keeps it compatible with the modern JS frameworks we use elsewhere.
2. **Sufficient component coverage.** Bulma provides the components we currently need — grids, tabs, buttons, forms — without the weight of a kitchen-sink framework.
3. **Lightweight and modular.** We can import only the parts we use, keeping bundle size in check.
4. **Healthy ecosystem.** Third-party extensions cover most of the gaps when we need a component Bulma does not ship.
5. **Rejected alternatives.** Semantic UI was the closest competitor and offered richer components, but it depends deeply on jQuery and its maintainers explicitly rejected a jQuery-free path. That combination — a deep technical dependency on an outgoing library plus a leadership stance against migrating — is a long-term risk we are not willing to take on.

## Implications

1. **Component gaps.** Bulma ships fewer batteries than Semantic UI. Where we need components Bulma does not provide (e.g. richer carousels, advanced data tables), we will pick targeted libraries rather than switch frameworks.
2. **CSS Grid still available.** Bulma does not preclude using native CSS Grid directly; the team should reach for it when the framework's grid system is more friction than help.
3. **Easily reversible.** Bulma's footprint is small and idiomatic enough that switching frameworks later, if needed, is a contained refactor rather than a rewrite.
4. **Testability.** The framework choice does not constrain our test strategy; component tests work the same way regardless.

## Conclusion

Bulma covers what we need today and stays out of the way of the rest of our stack. We will treat the decision as easily reversible and re-evaluate if our component needs outgrow what Bulma plus a few targeted libraries can deliver.
