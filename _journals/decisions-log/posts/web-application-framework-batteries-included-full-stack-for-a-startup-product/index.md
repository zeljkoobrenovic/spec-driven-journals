> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will build the startup product on Django (Python). Among the batteries-included, full-stack frameworks we considered, Django is the only one that puts us inside Python's data and ML ecosystem — which is the deciding factor for this product.

## Context

We are building a web application for paying customers to sign in, upload files, process the data, and view reports. As an early-stage startup, the priorities that matter most are:

- **Agile development.** We need fast iteration on features and product hypotheses.
- **Full-stack coverage.** A single framework should handle both backend and frontend so we are not maintaining a parallel JavaScript stack.
- **AI/ML compatibility.** The product is data-heavy. Easy interop with Jupyter, Pandas, NumPy, and the rest of the Python data ecosystem is core to the product, not a side concern.

What matters less right now is raw runtime speed, massive-scale concurrency, and backwards compatibility with anything we already run.

We evaluated four batteries-included, full-stack candidates: Django (Python), Ruby on Rails (Ruby), Phoenix (Elixir), and Loco (Rust).

## Decision

Django will be the web framework for the product. We will use it as a full-stack framework — server-rendered templates plus light JavaScript where needed — and keep the data/ML code in the same Python runtime where possible.

## Rationale

1. **Python ecosystem alignment.** Django runs on the same language as Jupyter, Pandas, NumPy, scikit-learn, TensorFlow, and PyTorch. Our data and ML code can share types, libraries, and tooling with the web layer with no translation step. None of the other candidates can match this for a data-heavy product.
2. **Batteries included.** Auth, ORM, admin, forms, templating, and routing are all in the box. For a startup, this is exactly the kind of leverage that lets a small team ship a real product.
3. **Agile fit.** Django's conventions, scaffolding, and documentation make it fast to go from idea to running feature. Rails is comparable on this axis; the others are not.
4. **Why not Rails.** Rails is an excellent agile framework with a great ecosystem, but Ruby is a second-class citizen in the AI/ML world. Bridging from Rails to a separate Python data layer would erase most of the velocity advantage we are choosing a full-stack framework to get.
5. **Why not Phoenix.** Phoenix and the BEAM are outstanding for concurrency and real-time use cases, but those are exactly the qualities we said we do not need yet. Meanwhile its data-science ecosystem is too immature for our core workload.
6. **Why not Loco.** Loco is promising, but Rust's web and AI/ML ecosystems are still maturing, and Rust's iteration speed is slower than higher-level languages. Wrong tool for a product still searching for fit.

## Implications

1. **Lean on the framework.** We will use Django's admin, auth, ORM, and forms by default and resist building parallel infrastructure. Conventional Django is fast Django.
2. **Server-rendered first.** The default UI will be Django templates with progressive enhancement, not a separate single-page front-end. We revisit only if a feature genuinely needs it.
3. **Data and ML in-process where possible.** Reports, transforms, and model inference run in Python; we share libraries with our Jupyter notebooks rather than maintaining a separate analysis stack.
4. **Performance is a future problem.** Python and Django are fast enough for our scale today. When we hit limits, the answers will be caching, query tuning, and pulling hot paths into background workers — not rewriting the stack.
5. **Watch the failure case.** If the product turns out to need real-time concurrency or sub-millisecond latency, the framework choice will have to be revisited. We are explicitly betting that will not happen before we have product-market fit.

## Conclusion

Django is the only candidate that satisfies all three of our high-priority criteria — agile development, full-stack coverage, and tight integration with Python's data and ML tooling. Phoenix and Loco are stronger on axes we are not optimising for. Rails is close on velocity but loses on AI/ML. For this product, at this stage, Django is the right choice.
