> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use Svelte as the front-end JavaScript library for our web application. Svelte's compile-time model gives us a smaller runtime, faster rendering, and simpler component syntax than the alternatives we evaluated.

## Context

We are building a new web application and need to pick a front-end library. We want something lightweight, fast, and approachable for the team. We evaluated React, Vue, Angular, and Svelte. The first three are well-established with large ecosystems; Svelte takes a different approach by shifting work from the runtime into the compiler, which is interesting for an application where bundle size and rendering performance matter.

## Decision

We will adopt Svelte as the front-end library. UI components will be authored as `.svelte` files; we will let the Svelte compiler produce the runtime code rather than ship a heavy framework runtime to the browser.

## Rationale

1. **Lightweight output.** Svelte compiles components to small, framework-free JavaScript. Bundles are noticeably smaller than equivalent React or Angular apps, which improves load time on the devices our users actually have.
2. **Fast rendering.** Without a virtual-DOM diff at runtime, Svelte updates the DOM directly from compiled instructions. For UI-heavy screens this translates into less jank and lower CPU use.
3. **Simple authoring model.** Component syntax is close to plain HTML, CSS, and JavaScript, with reactive declarations built into the language. New team members get productive quickly.
4. **Sufficient ecosystem.** While smaller than React's, the Svelte ecosystem and documentation cover the features we need, and SvelteKit handles routing, SSR, and tooling out of the box.
5. **Honest trade-off against the alternatives.** React is powerful but ships more runtime and ceremony than we need; Vue is approachable but does not match Svelte's output size; Angular is heavier and more opinionated than the size of this application warrants.

## Implications

1. **Smaller hiring pool.** Fewer engineers list Svelte than React or Vue. We will need to onboard hires from those communities, which is straightforward but not free.
2. **Smaller third-party ecosystem.** Some niche components we would have pulled off the shelf in React do not exist for Svelte. We will either adapt them or build them.
3. **Compiler-first mental model.** The team will need to internalize how Svelte's reactivity is compiled rather than evaluated at runtime, which differs from the libraries they may have used before.
4. **Maturity risk.** Svelte is younger than the established frameworks. We will track its roadmap and revisit this decision if the project stalls or if our requirements shift toward features it does not cover.

## Conclusion

Svelte gives us a lightweight, fast, and ergonomic foundation that fits this application better than the heavier alternatives. We are accepting the smaller ecosystem and hiring pool in exchange for cleaner code and smaller bundles.
