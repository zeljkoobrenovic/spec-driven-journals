> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will standardize on OpenID Connect (built on OAuth 2.0) for user authentication and API authorization across our web applications. SAML is retained only for enterprise SSO integrations that require it; LDAP remains in use as an internal directory store rather than as a public-facing auth protocol.

## Context

Our web applications need a consistent answer to two distinct questions: *who is this user?* (authentication) and *what are they allowed to do?* (authorization). Today we have a mix of approaches across services, which makes integration, audit, and onboarding harder than it should be.

We surveyed the protocols and technologies in active use in our environment and the wider ecosystem:

- **OAuth 2.0** — delegated access. An authorization server issues short-lived tokens that let a third-party application act on the user's behalf without seeing their credentials. OAuth alone does not tell us *who* the user is.
- **OpenID Connect (OIDC)** — an identity layer on top of OAuth 2.0. Adds an ID token containing user claims (subject, email, name, etc.), turning OAuth into a full authentication flow.
- **SAML** — XML-based assertions exchanged between an identity provider and a service provider. Mature in enterprise SSO, less mobile-friendly, heavier on the wire than OIDC.
- **WS-Federation** — Microsoft-centric SSO from the WS-* family. Largely superseded by OIDC outside legacy estates.
- **LDAP** — a directory protocol, not an auth protocol. Useful as a backing store for user attributes and group membership; typically fronted by another protocol for application sign-in.
- **Social SSO** (Google, GitHub, etc.) — OAuth/OIDC providers aimed at consumer users; reduces signup friction.

## Decision

OpenID Connect is the default for all new and migrated services:

- User authentication uses an OIDC provider; applications consume the ID token to establish identity.
- API authorization uses OAuth 2.0 access tokens issued by the same provider, with scopes mapped to application permissions.
- SAML is permitted only for integrating with external partners or enterprise customers whose IdPs do not speak OIDC.
- LDAP/Active Directory remain as the directory of record for employees, surfaced through the OIDC provider rather than queried directly by applications.
- WS-Federation is not used in new work; existing endpoints are scheduled for migration.

## Rationale

1. **One protocol covers both questions.** OIDC gives us authentication and OAuth 2.0 gives us authorization in a single, coherent stack. Standardizing on one model removes duplicate code paths and per-service auth designs.
2. **Modern web and mobile fit.** OIDC is JSON/HTTP-based and works cleanly with SPAs, mobile apps, and machine-to-machine flows. SAML's XML and browser-redirect ergonomics are awkward outside server-rendered web apps.
3. **Strong ecosystem.** Mature libraries exist for every language we use; managed providers (Auth0, Okta, Cognito, Azure AD, Keycloak) all support OIDC out of the box.
4. **Backwards compatibility where it matters.** SAML support remains available for enterprise SSO scenarios where the partner mandates it, so we keep the deals without imposing legacy on our own codebase.
5. **Directory separation.** Treating LDAP as a data source rather than an auth protocol means applications never see passwords and the directory can be replaced or extended without touching application code.

## Implications

1. **Provider selection.** We will pick a single OIDC provider (or a thin abstraction) and route all applications through it, rather than letting each team integrate directly with upstream IdPs.
2. **Token handling discipline.** Teams need clear guidance on token storage, refresh, scope design, and revocation. We will publish a short reference implementation per stack.
3. **Migration work.** Existing custom auth and WS-Federation endpoints will be migrated to OIDC on a service-by-service basis; no flag day is planned.
4. **SAML as an integration concern, not an architecture concern.** Where SAML is required, it is terminated at the provider, not at the application — apps still see OIDC tokens.
5. **Social SSO is optional per product.** Consumer-facing products may enable Google or GitHub sign-in via the same OIDC provider; internal tools should not.

## Conclusion

Standardizing on OIDC and OAuth 2.0 gives us one auth model to build, document, and audit against, while still letting us meet enterprise customers on their own terms via SAML. The remaining work is migration and consistent implementation, not further protocol selection.
