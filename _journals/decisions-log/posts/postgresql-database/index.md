> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use PostgreSQL as the primary database management system for the upcoming project.

## Context

We evaluated database options for an upcoming project and narrowed the field to two mature open-source RDBMS choices: PostgreSQL and MySQL. The project will rely on advanced query features, structured data with some semi-structured fields, and the ability to scale gracefully as data grows. Both options are widely supported in our stack; the question is which fits the project's needs more naturally.

## Decision

We will use PostgreSQL as the primary database for the project.

## Rationale

1. **Richer feature set.** PostgreSQL provides native JSON support, full-text search, spatial data (PostGIS), and window functions — features the project will use directly rather than re-implement on top of a simpler engine.
2. **Strong performance on complex workloads.** Its query planner handles complex joins and analytical queries well, which matches our expected access patterns.
3. **Standards compliance and data integrity.** Strict SQL conformance and robust constraint handling reduce the number of correctness bugs we have to guard against in application code.
4. **Active community and tooling.** Regular releases, mature drivers across our languages, and a healthy extension ecosystem (e.g. PostGIS, pg_stat_statements) keep operational risk low.
5. **Scalability.** Partitioning, logical replication, and a wide range of managed offerings give us clear paths for growth.

## Implications

1. **Operational expertise.** The team will need familiarity with PostgreSQL-specific operational concerns (vacuum, replication topology, extensions).
2. **Driver and ORM choices.** Some tooling defaults assume MySQL; we will verify driver/ORM compatibility for each service that connects to the database.
3. **Migration cost (if relevant).** Any existing components currently written against MySQL will need adjustments for SQL dialect and type differences.

## Conclusion

PostgreSQL offers the capabilities the project needs without trading away performance, reliability, or community support. We accept a small ramp-up cost for the team in exchange for a database that fits the workload directly.
