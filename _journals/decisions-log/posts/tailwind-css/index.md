> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use **Tailwind CSS** as the styling framework for the client web application. Customization will live in the Tailwind config; component composition will happen in markup using utility classes.

## Context

We are building a web application for a client that needs a responsive, modern, and consistently styled UI. The team has solid custom-CSS experience but wants to reduce the time spent hand-rolling component styles. We need a framework that is easy to pick up, customizable enough to match the client's brand, well supported, and fast enough to ship in.

Candidates considered:

- **Bootstrap** — mature component library, large community, opinionated visual language.
- **Foundation** — mobile-first, component-oriented, slightly less mainstream than Bootstrap.
- **Materialize** — implements Google's Material Design, opinionated look-and-feel.
- **Tailwind CSS** — utility-first, no pre-built components, configuration-driven theming.

## Decision

Adopt **Tailwind CSS**. We will configure it via `tailwind.config.js` to encode the client's brand (colors, spacing, typography) and use utility classes directly in markup. Where a pattern recurs, we will extract it into a Svelte/React/HTML component rather than into a Tailwind `@apply` rule.

## Rationale

1. **Customization without fighting defaults.** Tailwind ships no opinionated component look. Matching the client's brand means editing config, not overriding library styles. Bootstrap, Foundation, and Materialize all encode a design language that we would have to push back against.
2. **Fast iteration.** Utility classes are quicker to write and adjust than custom CSS for the same outcome, especially during early design churn.
3. **Maintainable styling.** Styles live next to the markup they apply to. This makes it easy to see what a component looks like without grepping across stylesheets.
4. **Strong ecosystem.** Active community, frequent releases, broad third-party plugin and template availability. Hiring engineers familiar with Tailwind is straightforward.
5. **Composable with components.** Tailwind is style-only; it does not constrain our component architecture or our choice of UI library on top.

## Implications

1. **Markup verbosity.** Utility classes can pile up on a single element. We will rely on component extraction (one component per recurring pattern) to keep this under control rather than building a parallel `@apply`-based CSS layer.
2. **Design tokens in config.** The `tailwind.config.js` becomes the source of truth for spacing, colour, type, and breakpoints. Designers and developers will treat it as a versioned design contract.
3. **Build pipeline.** Tailwind requires a PostCSS-driven build step with content scanning for tree-shaking unused classes. This must be wired into the project's build before production.
4. **Learning curve.** Engineers new to Tailwind need to learn its class vocabulary. Editor tooling (IntelliSense plugins) and the docs make this fast in practice.
5. **Reversible.** Tailwind is a styling layer; replacing it later would be invasive but contained — no application logic is coupled to it.

## Conclusion

Tailwind matches what we need — speed, customization, and a clean separation from component architecture. The trade-off (busy markup) is well understood and manageable through normal component extraction.
