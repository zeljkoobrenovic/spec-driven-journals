> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use React as our front-end JavaScript library. React's component model, declarative rendering, and ecosystem make it a strong fit for the responsive, interactive UI we need to build.

## Context

We are building a new web application that needs a responsive, interactive user interface and we need to commit to a front-end library to anchor the stack. The main candidates in our shortlist are React, Vue, and Svelte — all viable, but with different trade-offs around ecosystem maturity, team familiarity, and long-term support.

## Decision

We will adopt React as the front-end JavaScript library for the application. All new UI work targets React; existing prototypes in other frameworks will be migrated or sunset.

## Rationale

1. **Component-based architecture.** React encourages composing the UI from small, reusable components. This keeps the codebase modular and makes it easier to evolve features without large rewrites.
2. **Declarative model.** Describing UI as a function of state — backed by a virtual DOM — produces code that is easier to read and reason about than imperative DOM manipulation.
3. **Ecosystem and community.** React has the deepest ecosystem of the three: routing, state management, testing utilities, component libraries, and hiring pool are all well-established.
4. **Performance.** One-way data flow and a mature reconciler keep updates predictable and fast. Server-side rendering is available when we need it for SEO or first-paint.
5. **Flexibility.** React composes well with the rest of our stack and does not lock us into a single set of opinions about routing, data fetching, or state.

## Implications

1. **Learning curve.** Engineers new to React will need ramp-up time on hooks, rendering behavior, and the surrounding tooling.
2. **Build tooling.** React projects bring along a build chain (bundler, transpiler, linters). We will standardize on a common setup so teams don't each reinvent it.
3. **Browser support.** Modern React assumes evergreen browsers. If we need to support older targets, we will configure transpilation accordingly.
4. **Ecosystem choices.** Because React is unopinionated about state and routing, we will need to make and document those choices explicitly to avoid divergence across teams.

## Conclusion

React's component model, declarative rendering, and broad ecosystem outweigh the ramp-up cost and tooling complexity. We are committing to React for the front end and will document the supporting choices (state management, routing, testing) as we make them.
