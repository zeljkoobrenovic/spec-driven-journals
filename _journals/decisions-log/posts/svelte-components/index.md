> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use **SvelteUI** as the default component library for tables, grids, lists, charts, and forms. Where it falls short — notably for Gantt charts and advanced visualizations — we will integrate specialized libraries (e.g. chart.js, d3.js) rather than switch libraries.

## Context

We need a Svelte component library that provides ready-made building blocks for the application's core UI: tables, charts, lists, grids, and Gantt views. The goal is to avoid writing these components from scratch while keeping the door open for specialized libraries on the long tail.

Candidates evaluated: SVAR, Carbon, Flowbite, SkeletonUI, MeltUI, SvelteUI, shadcn-svelte.

Selection criteria: feature coverage (especially tables/grids/charts/Gantt), ease of integration with Svelte, performance, customizability, and community/documentation health.

## Decision

Adopt **SvelteUI** as the default component library. Use specialized libraries — chart.js or d3.js for advanced charts, a dedicated Gantt library for Gantt views — where SvelteUI does not cover the use case.

## Rationale

1. **Breadth of components.** SvelteUI provides tables, grids, charts, forms, and layout primitives in one library. No other Svelte-native candidate covered the same breadth without a major gap.
2. **Charting integration.** SvelteUI has explicit hooks for chart.js and d3.js, so we can extend charting capability without leaving the library's idioms.
3. **Native to Svelte.** Unlike Carbon (IBM, framework-agnostic) it does not require a compatibility shim.
4. **Customization.** Theme support and component overrides are first-class, including light/dark mode out of the box.
5. **Rejected alternatives.**
   - **SVAR** — solid for design systems but weak on Gantt and advanced grids.
   - **Carbon** — polished and accessible but its Svelte integration is awkward and it lacks Gantt support.
   - **Flowbite** — clean Tailwind-based components, but no native charting and minimal grid/Gantt.
   - **SkeletonUI / MeltUI** — too minimal; would force us to build the rich components ourselves.
   - **shadcn-svelte** — utility-first, customizable, but no built-in tables, charts, or grids.

## Implications

1. **Bundle weight.** SvelteUI is heavier than the minimalist alternatives. We will use its tree-shakeable imports and revisit if we see real bundle-size regressions.
2. **Charts and Gantt as separate concerns.** We accept that advanced charting and Gantt will live outside SvelteUI. We will write thin Svelte wrappers around chart.js / d3 / a Gantt library so call sites stay consistent.
3. **Theming once, applied everywhere.** SvelteUI's theme system will define our visual baseline. Any third-party widget we drop in will need to honour that theme.
4. **Accessibility.** SvelteUI handles accessibility for its built-in components; for third-party widgets we add, accessibility is on us. We will audit those integrations as a matter of course.

## Conclusion

SvelteUI is the closest match to what we need — broad enough to cover the everyday UI, customizable enough to extend, and Svelte-native enough to integrate cleanly. Specialized libraries for charts and Gantt remain available without committing us to a different framework.
