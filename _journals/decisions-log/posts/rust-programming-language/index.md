> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will adopt Rust as our primary language for new systems-level work and gradually migrate existing C and C++ code where the cost is justified. Rust gives us memory safety guarantees without giving up the performance we currently get from C/C++.

## Context

A growing share of our defects come from the same family of problems: buffer overflows, use-after-free, data races, and undefined behaviour in our C and C++ codebases. These bugs are expensive to find, expensive to fix, and a recurring source of security incidents. We need a language that closes those classes of error at compile time without forcing us to give up the performance profile that put us on C/C++ in the first place.

## Decision

Rust will be the default language for new systems-level components. Existing C and C++ code will be migrated opportunistically, prioritising modules where memory-safety bugs have cost us the most.

## Rationale

1. **Memory safety by construction.** Rust's ownership model and borrow checker eliminate entire categories of bug — use-after-free, double free, data races — without a garbage collector.
2. **Performance parity with C/C++.** Rust compiles to native code with no runtime overhead and zero-cost abstractions, so adopting it does not cost us the performance we depend on.
3. **Modern toolchain.** Cargo, rustfmt, clippy, and a strong type system give us a developer experience that is materially better than what we have with C++ build tooling.
4. **Healthy ecosystem.** The Rust community is active and growing, and the library ecosystem is mature enough for the systems work we do today.
5. **Interoperability.** Rust links cleanly with C, so we can migrate incrementally rather than rewriting whole systems at once.

## Implications

1. **Training investment.** Engineers experienced in C/C++ or managed languages will need time to internalise ownership and lifetimes. We will run dedicated training and pair new Rust authors with reviewers who already have experience.
2. **Initial slowdown.** Output will dip during the learning period. We will plan delivery accordingly rather than pretending the ramp-up is free.
3. **Migration scope.** Not every C/C++ component is worth rewriting. We will set explicit criteria — defect history, security exposure, maintenance cost — and only migrate modules that clear the bar.
4. **Hiring leverage.** Rust is attractive to a strong segment of systems engineers, which should help recruiting.
5. **Safety dividend.** Over time we expect fewer memory-safety incidents and a smaller security surface, which is the core reason we are making the change.

## Conclusion

Rust gives us the safety properties we have been missing in C and C++ without forcing a performance compromise. The cost is a real learning curve and a multi-year migration, but the reduction in memory-safety incidents is worth the investment.
