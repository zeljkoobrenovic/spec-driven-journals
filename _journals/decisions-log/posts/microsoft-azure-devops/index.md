> **Status**: Rejected (2020-01-21)
>
> **Decision**: We will not adopt Microsoft Azure DevOps. We will continue with the AWS-based incumbent toolchain for build, integration, deployment, and hosting. The choice is reversible if Azure DevOps significantly improves.

## Context

We want DevOps tooling to build, integrate, deploy, and host our projects. Microsoft Azure DevOps was on the candidate list because:

- We are considering Azure as a hosting platform more broadly.
- Microsoft has been investing heavily in developer experience (notably the GitHub acquisition).
- A unified offering from one vendor is operationally attractive if it works well.

Our requirements:

- **Fast builds.** We are willing to pay a premium for short build times because they accelerate the feedback loop.
- **High reliability.** We test high-value workloads (financial and confidential transactions), and reliability is non-negotiable.
- **Composability.** We want to mix and match — bring our own build agents, choose our own VCS, or coordinate with self-hosted CI — without being locked in.
- **Reportable failure modes.** When something breaks, especially something security-relevant, we need a working channel to report it.

## Decision

Reject Microsoft Azure DevOps. We will stay with the AWS-based incumbent toolchain. The decision is reversible if Azure DevOps's situation changes materially.

## Rationale

1. **Hands-on experience was poor.** We ran a real evaluation across Azure DevOps, Azure Pipelines, Azure Repos, and infrastructure spin-up via Terraform. The advertised capabilities did not hold up in practice.
2. **Setup is a tangle of overlapping UIs.** Azure sign-in, microsoft.com sign-in, and live.com sign-in coexist and overlap in confusing ways. Identity alone consumed disproportionate setup time.
3. **Security reporting did not work.** We discovered a minor security issue during evaluation and could not find a working channel to report it. Multiple Microsoft representatives could not route it. We eventually reached Microsoft Security, who responded "won't fix."
4. **Documentation is inconsistent.** Docs are frequently outdated or wrong; Microsoft's own search makes them hard to find even when they exist.
5. **Terraform support lags AWS.** It works, but Microsoft's pattern is to push vendors to maintain Terraform examples rather than maintain authoritative ones themselves. The result is gaps and inconsistency compared with the AWS equivalents.
6. **Peer reports confirm our experience.** Independent reports — Hacker News discussion, Windows Development MVPs, blog write-ups — describe the same UI clunkiness, slow builds, weak pull-request UX, and integration friction. The complaints predate our evaluation and have persisted across UI rewrites.
7. **Cost premium.** Our rough estimate is that choosing Azure DevOps would cost ~3x in time and effort over staying with the incumbent, with no clear corresponding capability gain.

## Implications

1. **Status quo.** We continue with the AWS-based incumbent for build, integration, deployment, and hosting.
2. **Bounded Azure exposure.** Rejecting Azure DevOps reduces the pull toward adopting Azure more broadly. Other Azure services can still be evaluated on their own merits.
3. **Microsoft acquisition strategy.** Microsoft is acquiring developer tools aggressively (GitHub, Citus). If we revisit, we should evaluate the acquired offerings separately and watch for tissue-rejection risk (staff turnover, slow integration).
4. **Easily reversible.** Our DevOps choices live alongside the AWS incumbent; revisiting Azure DevOps in the future is a parallel evaluation, not a migration.
5. **Watch list.** Edward Thomson (Azure PM) and other Azure team members have engaged openly and technically in public discussion. That is a positive signal worth tracking when deciding whether to re-evaluate.

## Conclusion

Azure DevOps's advertised offering does not match its delivered experience today. We are not adopting it. We remain open — explicitly — to revisiting the decision if the platform's integration, performance, and support improve enough to clearly beat what we already have.
