> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use **Vue** as the primary front-end JavaScript library for the web application. The choice favors simplicity, learnability, and incremental adoption over the broader ecosystems of React and Angular.

## Context

The web application needs a front-end library to deliver dynamic, responsive UIs. We evaluated the major options:

- **React** — large ecosystem, JSX-based, ubiquitous in industry; deeper tooling stack to learn.
- **Angular** — opinionated full framework; substantial up-front investment; works well for very large teams.
- **Vue** — progressive framework; smaller surface; easy to introduce incrementally.

The team's primary constraints are time-to-productivity for new contributors, flexibility to adopt the library piece by piece, and a runtime light enough that performance is not a separate concern.

## Decision

Adopt **Vue** as the primary front-end library. We will use the single-file component model, the official router, and the official state management library, adding ecosystem pieces only when a concrete need appears.

## Rationale

1. **Gentle learning curve.** Vue's templates extend HTML in a way that is immediately readable; the reactivity model is straightforward. Engineers coming from any front-end background get productive quickly.
2. **Lightweight runtime.** The runtime footprint is small, and rendering is fast in our reference workloads. We do not need to tune performance to get acceptable behaviour.
3. **Incremental adoption.** Vue can wrap existing pages or be the whole application. This gives us optionality if we ever need to mix it into legacy code.
4. **Composable.** Vue plays well with other libraries — routing, state management, and UI kits are unbundled and replaceable. We are not locked into a particular ecosystem.
5. **Healthy ecosystem.** Documentation is high quality, the community is active, and the official add-ons (router, store, devtools) cover the common needs without third-party churn.

## Implications

1. **Hiring profile.** Vue is widely known but less ubiquitous than React. Hiring should weight transferable front-end skill over Vue-specific experience; ramp-up is short.
2. **State management discipline.** Vue does not force a single state pattern. We will choose one (Pinia or Vuex) early and apply it consistently to avoid an ad-hoc reactive-state sprawl.
3. **TypeScript integration.** Modern Vue has solid TypeScript support but requires some setup care. We will commit to TypeScript from day one rather than retrofitting later.
4. **Component library choice.** Vue's UI kit ecosystem is rich but fragmented. We will pick one component library (or commit to building components ourselves) rather than mixing several.
5. **Reversible at module boundaries.** Vue components are self-contained. A future migration to a different framework is feasible at module boundaries, though non-trivial — same as for any front-end framework choice.

## Conclusion

Vue gives us the simplicity, flexibility, and performance we need without the up-front cost of Angular or the broader ecosystem complexity of React. It is the right default for the application we are building and the team we are building it with.
