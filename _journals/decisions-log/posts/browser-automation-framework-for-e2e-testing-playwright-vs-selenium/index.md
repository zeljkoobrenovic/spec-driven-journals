> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will adopt Playwright as the browser automation framework for our end-to-end (E2E) test pipeline, in preference to Selenium. Selenium remains a future option if mobile-native or legacy-browser coverage becomes a hard requirement.

## Context

We need to choose a browser automation framework for E2E tests that run in our CI/CD pipeline. The tests cover sign-up and sign-in, file uploads, dashboard interactions, and report downloads — a mix of standard form flows and rich, interactive UI with charts.

Constraints and preferences:

- The team works primarily in **TypeScript** and **Python**; tests must be writable in both.
- Development is fast-moving; concise, low-boilerplate test code matters.
- The product UI is highly interactive (dashboards, charts, async updates).
- CI/CD pipeline speed is a secondary concern but not negligible.
- Mobile-native testing, legacy browser support, and multi-monitor scenarios are not current requirements.
- File-upload flows are central to the product and must be straightforward to automate.

The two realistic contenders are **Playwright** (Microsoft) and **Selenium** (Selenium Project).

## Decision

We will use **Playwright** for E2E tests, in both TypeScript and Python, integrated into the existing CI/CD pipeline. Existing Selenium tests, where they exist, will be migrated incrementally rather than via a big-bang rewrite.

## Rationale

1. **Concise API, less to maintain.** Playwright's API is shorter and more declarative than Selenium's. Tests are easier to read, easier to review, and easier for AI-assisted code generation to produce correctly.
2. **First-class TS and Python support.** Both bindings are maintained by the Playwright team itself, so feature parity and quality are consistent across our two main languages.
3. **Better fit for dynamic UIs.** Auto-waiting, locator engines, and built-in handling of shadow DOM, iframes, and async content make tests against our charts and dashboards far less flaky than the equivalent Selenium code.
4. **Simpler file-upload story.** `setInputFiles` is a one-liner; the Selenium equivalent requires more setup and is more brittle in headless mode.
5. **Faster in CI.** Playwright runs faster than Selenium in headless mode and parallelizes cleanly across browsers, which shortens pipeline feedback loops.
6. **Selenium's edge isn't ours.** Selenium's main advantages — Appium-backed mobile, legacy browsers (IE), multi-monitor — are not on our roadmap.

## Implications

1. **Adoption.** New E2E tests are written in Playwright; the existing pipeline is reconfigured to run Playwright suites for Chromium, WebKit, and Firefox.
2. **Skills.** Developers and QA will need short onboarding on Playwright's locators, fixtures, and trace viewer. We will publish an internal starter template and example suite.
3. **Migration.** Legacy Selenium tests are ported opportunistically — when a test is touched, it gets rewritten. We avoid keeping two frameworks alive long-term.
4. **Tooling.** CI runners need Playwright's browser binaries cached; reports and traces are wired into the existing test artifact storage.
5. **Revisit conditions.** If we add mobile-native testing or need broad legacy-browser coverage, we will reassess and may introduce Selenium (or Appium) alongside Playwright for those scenarios specifically.

## Conclusion

Playwright is the better fit for the product we are actually building: modern, TypeScript- and Python-driven, with interactive UIs and no current mobile-native or legacy-browser obligations. We are committing to it as the default and keeping Selenium on the table only for scenarios it uniquely covers.
