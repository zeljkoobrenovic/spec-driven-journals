> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use TypeScript on the front-end and Rust on the back-end. Both choices prioritise compile-time safety; Rust additionally avoids the operational complexity and GC pauses of a VM-based runtime.

## Context

We need to pick two languages: one for typical browser-facing front-end work, and one for a back-end with higher-than-typical requirements for reliability, security, and predictable latency. The back-end must be deployable on major cloud function platforms (e.g. AWS Lambda). We expect the front-end codebase to evolve quickly and the back-end to be longer-lived.

We considered a broad set of candidates: C, C++, Clojure, Elixir, Erlang, Elm, Flow, Go, Haskell, Java, JavaScript, Kotlin, Python, Ruby, Rust, and TypeScript.

## Decision

The front-end will be written in TypeScript. The back-end will be written in Rust.

## Rationale

1. **Compile-time safety on both tiers.** TypeScript adds a sound-enough type system to JavaScript; Rust's type system plus borrow checker eliminates a large class of runtime bugs. We are explicit that we will trade compile time for runtime safety.
2. **No VM on the back-end.** A VM-based stack (JVM, BEAM, CLR) buys capabilities we do not currently need and adds operational complexity and GC pauses we prefer to avoid. Rust gives us native performance with no runtime to babysit.
3. **TypeScript is the path of least resistance for the front-end.** It builds on JavaScript, the most widely supported browser runtime; the type system is good enough to prevent the bugs we care about; and migration from plain JavaScript is well-trodden.
4. **Why not the alternatives.** Briefly: C and C++ are rejected on safety grounds — Rust does the same job with stronger guarantees. Go has good ergonomics but a track record of language decisions we disagree with. Haskell and Elm are strong but have smaller production track records than we want. Clojure, Java, Kotlin, and Elixir are honourable mentions but pull us back onto a VM. Ruby and Python are excellent for prototyping but slower and harder to package for our deployment targets. Flow is losing momentum to TypeScript.
5. **Cloud function compatibility.** Both languages are first-class targets for the cloud function platforms we plan to deploy on.

## Implications

1. **TypeScript ramp-up is light.** Front-end engineers already working in JavaScript should transition quickly.
2. **Rust ramp-up is heavier.** Engineers coming from C/C++ face a moderate curve; engineers coming from managed languages (Java, Python, Ruby) face a steeper one. We will plan training and pairing accordingly.
3. **Tooling gaps.** Some DevOps tools we use do not yet ship first-class examples for TypeScript or Rust; we will need to invest in pipeline templates ourselves.
4. **Compile times.** Both languages have slow-ish compilers. We will mitigate with incremental compilation, build caches, and parallelism, and accept the trade-off for the safety we are getting in return.
5. **IDE support.** Tooling for Rust in particular is not yet at parity with mature languages. We will standardise on the best available options and revisit as the ecosystem matures.
6. **Ecosystem alignment.** Downstream choices (front-end framework, web server library, deployment targets) should favour ones that prioritise TypeScript and Rust support.

## Conclusion

TypeScript and Rust together let us prioritise compile-time provability and runtime predictability without taking on a VM-based stack. We accept slower compile times and a steeper Rust learning curve as the cost of that choice, and we expect the trade-off to pay off in fewer production incidents and lower operational overhead.
