> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use Java as the primary programming language for the project. It best matches our needs for security, concurrency, large-codebase ergonomics, and ecosystem breadth.

## Context

We are building a web-based application that must run on multiple operating systems and is expected to grow into a sizeable codebase. The project needs strong security primitives, first-class concurrency, and an ecosystem that can support both web and back-office workloads over several years. We evaluated several languages before settling on Java.

## Decision

We will use Java as the primary programming language for the project. New services and shared libraries will target the JVM unless an explicit exception is granted.

## Rationale

1. **Security model.** Java's well-defined security policies, access control, and bytecode verification give us a strong baseline for running untrusted code paths and exposing services on the network.
2. **Concurrency.** Built-in multithreading and a mature memory model let the application serve multiple requests in parallel without bolting on a separate runtime.
3. **Codebase scale.** Object-oriented features, encapsulation, and a strong static type system make large codebases easier to navigate and refactor as the team grows.
4. **Ecosystem breadth.** Java has a deep catalogue of libraries and frameworks for web, persistence, messaging, and observability, which reduces the amount of infrastructure we need to write ourselves.

## Implications

1. **Hiring and skills.** We will need engineers who are proficient in Java; this is generally easy in our market but will shape the hiring profile.
2. **JVM constraint.** All shipped code must run on the JVM. Native dependencies will need to be wrapped or replaced.
3. **Evolution cost.** Introducing new features or swapping frameworks within the Java ecosystem can be heavier than in more dynamic languages.
4. **Effort for large changes.** Significant architectural changes in Java tend to take more time and more code than in lighter-weight stacks; we accept that trade-off in exchange for the safety guarantees.

## Conclusion

Java gives us the security posture, concurrency model, and ecosystem we need for a long-lived multi-platform web application. The cost is some verbosity and a heavier toolchain, which we judge a fair price for the durability the language provides.
