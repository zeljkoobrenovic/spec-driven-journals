> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use a document database as the primary data store for the new application.

## Context

We are designing a new application that needs to store and retrieve data at scale and with low latency. Three classes of database technology are on the table:

- **Relational databases** (MySQL, PostgreSQL, Oracle) — fixed schemas, strong transactional guarantees, good for complex relationships.
- **Document databases** (MongoDB, Couchbase, DynamoDB) — schema-less JSON documents, horizontal scaling, flexible models.
- **Event databases** (Kafka, Pulsar, Kinesis) — append-only event streams, suited to auditing, event sourcing, and stream processing.

The application's data shape is still evolving, the access pattern is dominated by document-level reads and writes, and we do not expect strong cross-entity transactional needs.

## Decision

We will adopt a document database as the primary data store for this application. The specific product will be chosen in a follow-up decision based on operational fit and managed-service availability.

## Rationale

1. **Flexible schema.** The data model will keep changing as the product matures. A document store lets us add and reshape fields without coordinated schema migrations.
2. **Horizontal scalability.** Native sharding and replication give us a clear path to handling growth in both data volume and request rate.
3. **Fast document-level retrieval.** Indexing and query capabilities are well matched to our read patterns, which are dominated by lookups and small aggregations rather than complex joins.
4. **Workload fit.** The application does not need cross-entity ACID transactions or complex relational integrity, so the trade-offs of a relational store are not justified here.
5. **Event store isn't a fit either.** We need a system of record for entities, not an append-only event log; event databases would solve a problem we do not have.

## Implications

1. **Tooling and skills.** The team will need to invest in learning the specific document database we adopt — its query language, indexing model, and operational characteristics.
2. **Data modelling discipline.** Schema-less doesn't mean schema-free; we will need conventions and validation at the application layer to keep documents consistent.
3. **Transactional limits.** Cross-document transactions are limited compared to RDBMS; we will design workflows to avoid relying on them or use the document database's transactional features sparingly.
4. **Future flexibility.** If a future component genuinely needs relational guarantees, we should not force it into the document store — a polyglot setup is acceptable.

## Conclusion

A document database is the best match for this application's data shape and growth profile. The cost is a one-time investment in team familiarity and good modelling discipline, which we judge to be well worth it.
