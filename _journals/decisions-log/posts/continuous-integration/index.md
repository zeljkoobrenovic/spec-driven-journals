> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will adopt continuous integration across all active projects. Every change merged to the main branch must build, run automated tests, and deploy to a delivery environment without manual intervention.

## Context

Our current development process leans on manual builds, manual testing, and manual deployment. The cost is now visible: integration issues surface late, releases are painful and infrequent, defects slip into the delivery environment, and stakeholders cannot predict when work will land. The pain is not located in any single project — it is a systemic property of how we ship.

## Decision

We will implement continuous integration as the default workflow for all active projects. Concretely:

- Every commit to a feature branch triggers an automated build and test run.
- Every merge to the main branch triggers a build, test run, and automated deployment to a delivery (pre-production) environment.
- Pipeline failures block merges; broken builds are treated as a priority-one problem.

## Rationale

1. **Faster feedback.** Issues surface within minutes of a change rather than during a release. The cost of fixing a regression is sharply lower when the author still has the change in their head.
2. **Predictable releases.** A repeatable, automated pipeline replaces a fragile manual process. Releases become a non-event rather than a coordinated effort.
3. **Quality gates.** Tests, linters, and security scans run on every change, not just before a release. Quality becomes a property of every merge.
4. **Collaboration.** Frequent integration forces small, reviewable changes and reduces long-lived branches that drift from main.
5. **Cost over time.** The automation cost is paid once and amortized across every future release; the manual cost recurs forever.

## Implications

1. **Up-front investment.** We will need to invest in pipeline infrastructure, test environments, and the time to write the automation. The first projects will absorb the largest share of that cost.
2. **Test coverage.** CI only delivers value if there are tests to run. Each team will need to grow and maintain an automated test suite — unit, integration, and a small layer of end-to-end checks.
3. **Discipline.** A red build blocks the team. We will need shared norms about responding to and fixing broken pipelines quickly rather than working around them.
4. **Tooling and skills.** Engineers will need familiarity with the pipeline platform, build scripts, and artifact management. We will provide reusable templates so each team does not start from scratch.
5. **Operational risk during rollout.** Migrating live projects to CI introduces transitional fragility. We will roll out per project rather than flip everything at once.

## Conclusion

The cost of standing up CI is finite; the cost of *not* having CI compounds. We accept the up-front investment because the alternative — continuing with manual processes — is more expensive, less reliable, and harder to scale than the practice we are adopting.
