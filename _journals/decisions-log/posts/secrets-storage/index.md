> **Status**: Accepted (2020-01-21)
>
> **Decision**: Bitwarden for user-oriented secrets; HashiCorp Vault for system-oriented secrets.

## Context

We need a consistent way to store secrets — passwords, private keys, authentication tokens — across two distinct use cases:

- **User-oriented secrets.** A developer needs to look up a credential from a laptop or phone, share a password with a teammate, or autofill a login in a browser.
- **System-oriented secrets.** A CI/CD pipeline needs to fetch cloud credentials at deploy time; services need to retrieve API keys at runtime.

Today the team relies on a mix of personal habits, shared documents, and ad-hoc notes. That is neither secure nor sustainable. We want a solution that is easy to adopt, doesn't lock us into a single cloud provider, and can be reversed cheaply if we change our minds.

Options considered:

- User-oriented password managers: Bitwarden, 1Password, LastPass, Dashlane, KeePass, `pass`, GPG.
- System-oriented secret managers: HashiCorp Vault, AWS KMS, EnvKey, Confidant (Lyft), Devolutions Password Server, Thycotic Secret Server.
- Sharing via shared documents, Slack channels, or network folders.
- Ad-hoc: memorisation, sticky notes, "figure it out yourself".

## Decision

We will use **Bitwarden** for user-oriented secrets and **HashiCorp Vault** for system-oriented secrets.

## Rationale

1. **Right tool per job.** The two use cases have different ergonomics and threat models. A user-facing password manager and a programmable secret store are different products; we treat them as such.
2. **Bitwarden for users.** It matches the feature set of LastPass, 1Password, and Dashlane (desktop and mobile apps, browser extensions, team sharing) while being open source, peer-reviewable, and well-regarded by security-minded developers.
3. **Vault for systems.** Vault has strong reviews, a serious operational community, and HashiCorp's track record on tooling. It supports the dynamic-secret and policy-driven access patterns CI/CD pipelines and services need.
4. **No cloud lock-in.** We avoid AWS KMS as a primary choice because it ties us to a single provider; both Bitwarden and Vault work regardless of where we host.
5. **Rejected alternatives.** Shared docs and channels do not provide the security properties we need. Ad-hoc approaches are not a viable long-term path. KeePass, `pass`, and GPG are fine for individuals but raise the bar for team adoption.

## Implications

1. **Two tools, not one.** Developers will track user-oriented secrets in Bitwarden and reference system-oriented secrets via Vault. We accept this split as the cost of fitting each tool to its use case.
2. **Vault operational investment.** Vault has a steep initial learning curve and its documentation is uneven; we should stand it up in a test environment first and budget time for the helm/Consul setup. Consider Goldfish as a UI to lower the bar for team adoption.
3. **Policies still to define.** Rotation policies, organisation/team structure, onboarding/offboarding, auditing, and compliance handling are follow-up work.
4. **CI/CD compatibility.** The choice of CI/CD server must include first-class Vault integration.
5. **Reversibility.** Both tools are easy to leave: Bitwarden can export vaults, and Vault's data can be re-keyed or migrated. The decision is cheap to revisit.

## Conclusion

Splitting the problem along the user/system boundary lets each tool do what it is good at without compromise. Bitwarden gives the team a low-friction, open-source password manager; Vault gives our infrastructure a serious, programmable secret store. We expect this pairing to hold for the foreseeable future, and we remain open to revisiting it as the landscape changes.
