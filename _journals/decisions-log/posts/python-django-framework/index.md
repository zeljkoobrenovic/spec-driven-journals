> **Decision**: We will use Django as the web framework for the new customer-data application. Django gives us a complete, opinionated stack with strong defaults for ORM, security, and admin tooling — a good fit for the application we are building.

## Context

We are building a web application that manages customer data. Python is already settled as our primary language. The remaining question is which Python web framework best fits a customer-data application that needs an ORM, sensible security defaults, and an admin surface for internal users. The realistic alternatives are Django, Flask, and Pyramid, plus the option of assembling a stack from libraries (SQLAlchemy, Flask-RESTful, etc.).

## Decision

Django will be the web framework for the application. We will use it end-to-end rather than treating it as one component among many.

## Rationale

1. **Built-in ORM.** Django's ORM lets us model customer data declaratively and migrate it safely without hand-rolled SQL. For data-management workloads, this is the largest single source of leverage.
2. **Clear architecture.** Django's model–template–view layout enforces a separation between data, behaviour, and presentation, which keeps the codebase navigable as it grows.
3. **Security defaults.** CSRF protection, SQL-injection-safe queries, XSS escaping, and clickjacking protection are on by default. For an application handling customer data, secure-by-default beats secure-by-vigilance.
4. **Admin interface.** Django's auto-generated admin gives us a usable internal tool from day one, which is disproportionately valuable for a customer-data application.
5. **Maturity and community.** Django has been production-hardened for well over a decade, has clear release and security-patch cadences, and has a large community to draw on.
6. **Scope vs. alternatives.** Flask and Pyramid are excellent but require us to assemble the same pieces ourselves. For this application, the integrated package is the right trade.

## Implications

1. **Learning curve.** Django is larger than micro-frameworks; engineers new to it will need ramp-up time before they are productive.
2. **Follow Django's conventions.** The framework rewards apps that fit its model and punishes those that fight it. We will stay on the well-trodden path unless we have a strong reason not to.
3. **Performance work later, not now.** Django is fast enough by default; if and when we hit limits, the answer will be caching, query tuning, and async views rather than rewriting.
4. **Reduced ongoing maintenance.** Because so much functionality lives inside Django, we have fewer third-party dependencies to track for security and compatibility.

## Conclusion

Django covers the shape of this project — ORM-driven customer data, secure defaults, internal admin tooling — better than any of the smaller Python frameworks would. The cost is the framework's learning curve and its opinionatedness; both are acceptable in exchange for the velocity and safety guarantees Django gives us.
