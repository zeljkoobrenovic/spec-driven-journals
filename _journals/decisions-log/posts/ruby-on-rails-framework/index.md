> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use Ruby on Rails as the web framework for the new application. Rails fits the project — complex business logic, several third-party integrations, and a team that already knows it.

## Context

We are building a web application with non-trivial business logic and multiple integrations with third-party services. The team already has Rails experience. We need a framework that supports rapid development now, and gives us a clear path to scale and maintain the code over the next several years.

## Decision

Ruby on Rails will be the web framework for the application. We will follow Rails conventions and idioms by default, and treat departures from them as decisions that need a written justification.

## Rationale

1. **Maturity.** Rails has been used at scale for nearly two decades and the failure modes are well understood. We will not be discovering basic framework bugs.
2. **Convention over configuration.** Rails' opinionated structure (MVC, ActiveRecord, naming conventions) saves us from inventing project-level conventions and keeps new contributors productive quickly.
3. **Batteries included.** ActiveRecord, ActionMailer, ActionCable, scaffolding, and the routing DSL cover most of what the application needs without bolting on a separate stack of libraries.
4. **Testing culture.** Rails' first-class support for TDD and BDD, plus tools like RSpec and Capybara, make it natural to ship tested code rather than retrofitting tests later.
5. **Team fit.** Our team already writes Rails. Choosing it means we ship sooner and avoid the cost of training the team on an unfamiliar stack.
6. **Ecosystem.** A deep catalogue of gems covers common needs (auth, background jobs, payments, integrations), which is exactly the kind of leverage a third-party-heavy project benefits from.

## Implications

1. **Stay on the Rails happy path.** The framework rewards conventional design. We will resist building bespoke patterns where a Rails idiom exists.
2. **Ruby fluency required.** Productivity in Rails depends on real Ruby fluency, not just framework knowledge. New hires without that background will need time and mentoring.
3. **Performance and scaling.** Ruby's runtime is slower than compiled alternatives. We will plan for caching (HTTP, fragment, Russian-doll), database query tuning, and background processing from the start rather than retrofitting them under pressure.
4. **Operational footprint.** Rails apps are heavier to operate than minimalist stacks. We will invest in deployment automation, observability, and CI early.

## Conclusion

Rails matches the shape of this project: business-logic-heavy, integration-heavy, and time-to-market sensitive. We accept the runtime cost in exchange for the velocity the framework and the team's existing expertise give us, and we will plan for performance work as the application grows.
