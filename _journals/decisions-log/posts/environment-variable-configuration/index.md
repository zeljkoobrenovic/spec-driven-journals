> **Status**: Accepted (2020-01-21)
>
> **Decision**: All applications will read configuration from environment variables loaded from `.env` files at deploy time, with a checked-in `.env.defaults` for non-secret defaults and a `.env.schema` enumerating the keys an application expects.

## Context

We want one build of an application to behave differently in different environments — development, test, demo, production — without rebuilding the binary or branching the source. We also want our configuration approach to:

- Keep secrets out of the source code management system.
- Be discoverable: a developer should be able to see what an app can be configured with, and what the defaults are.
- Match the 12-factor (and 15-factor) conventions our teams already follow on other projects.

## Decision

We will configure all applications via environment variables, populated from `.env` files. Each application will ship with:

- A `.env.schema` file that lists every supported configuration key (checked in).
- A `.env.defaults` file with non-secret default values (checked in).
- An environment-specific `.env` file containing the actual values and any secrets (not checked in; managed by deploy tooling).

Secrets in particular never live in `.env` files inside the repo.

## Rationale

1. **Matches industry practice.** The 12/15-factor app pattern is widely adopted, well understood, and broadly supported across our language runtimes (e.g. `dotenv` for Node, similar libraries elsewhere).
2. **Team familiarity.** Our teams have used `.env`-style configuration on many previous projects successfully. Standardizing on it removes a per-project decision and reduces ramp-up cost.
3. **Discoverability via schema.** A `.env.schema` makes the contract explicit: anyone touching the app can see at a glance what knobs exist. A missing key is a build-time question, not a runtime mystery.
4. **Defaults via a separate file.** Keeping defaults in `.env.defaults` lets developers run the app locally without copying values around, while leaving the real `.env` for environment-specific overrides and secrets.
5. **Simplicity.** Files are easier to reason about than a network call to a config server. We accept the trade-off (no central audit trail, no live reconfiguration) for now.

## Implications

1. **Secrets handling.** We need a separate mechanism for secrets — a vault, a secrets manager, or environment injection by the deploy platform. The `.env` mechanism only ever carries the values, never the secrets themselves in source.
2. **DevOps integration.** We will add hooks, validators, and CI checks that compare a deploy's environment against the `.env.schema` so missing or extra keys fail fast.
3. **Training.** All developers need to learn the convention — schema, defaults, and the rule about secrets — so the pattern is applied consistently across applications.
4. **Existing applications.** Apps using a less capable approach (hardcoded config, in-source constants) will be migrated. Apps using a more capable approach (a dedicated config or licensing server) will be left alone unless there is a separate reason to change.
5. **Reversibility.** The choice is easily reversible: moving from `.env` files to a config server later is a contained change at the boundary, not a rewrite.

## Conclusion

`.env` files plus a schema and defaults file is the simplest approach that meets our requirements, matches industry practice, and aligns with how our teams already work. We accept its limits — no centralized audit, no live reload — as fair trade-offs for now.

---

Example `.env`:

```env
NAME=Alice Anderson
EMAIL=alice@example.com
```

Example `.env.defaults`:

```env
NAME=Joe Doe
EMAIL=joe@example.com
```

Example `.env.schema` (keys only):

```env
NAME
EMAIL
```
