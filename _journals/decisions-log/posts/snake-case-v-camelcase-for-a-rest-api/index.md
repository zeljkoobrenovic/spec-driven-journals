> **Status**: Accepted (2020-01-21)
>
> **Decision**: REST API endpoint names and JSON field names will use **snake_case**.

## Context

REST APIs need a naming convention for paths and payload fields. Two candidates dominate:

- **snake_case** — words separated by underscores (`first_name`, `/order_items`).
- **camelCase** — first word lowercase, subsequent words capitalized (`firstName`, `/orderItems`).

Picking one and enforcing it matters more than which one we pick — but the choice still shapes how the API reads, integrates with client tooling, and matches the rest of the codebase.

The decision drivers are:

- Consistency with existing naming conventions in the project.
- Readability for anyone working on the API.
- Alignment with common practice in the REST ecosystem.
- Cost of implementing and maintaining the choice.

## Decision

REST API endpoint paths and JSON field names will use **snake_case**.

## Rationale

1. **Consistency with existing code.** Our existing endpoints already use snake_case. Standardizing on it avoids a costly migration of code, documentation, and integrations.
2. **Readability.** Underscores produce a clear word boundary, which makes long compound names (`shipping_address_postal_code`) easier to scan than their camelCase equivalent.
3. **Common in the REST ecosystem.** snake_case is widely used in REST APIs and JSON payloads, particularly in the Ruby, Python, and Rails-influenced parts of the ecosystem. Engineers picking up the API will find it familiar.
4. **Lower implementation friction.** Keeping the convention means we do not need to update existing endpoints, clients, or documentation. Switching to camelCase would generate work without changing what the API does.

## Implications

1. **Client interop.** Some client ecosystems (notably JavaScript/TypeScript) idiomatically use camelCase internally. Clients in those languages will need a small adapter layer that converts between snake_case (over the wire) and camelCase (in their code). This is a routine pattern.
2. **Onboarding.** New engineers more familiar with camelCase will need a brief heads-up that the API uses snake_case. The risk of confusion is small because the convention is widely understood.
3. **Tooling defaults.** Any code generator, ORM, or serialization library used on the server side must be configured to emit snake_case to keep responses consistent. We will treat that as part of the project's standard setup.
4. **Documentation.** API documentation should call out the convention explicitly so external integrators are not left to infer it.

## Conclusion

snake_case is consistent with what we already have, reads well, and matches a large segment of the REST ecosystem. The choice is low-cost, low-risk, and removes a recurring conversation. We commit to it and document it.
