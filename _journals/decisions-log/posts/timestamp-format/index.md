> **Status**: Accepted (2020-01-21)
>
> **Decision**: All timestamps we serialize will use **ISO 8601 in UTC with nanosecond precision**: `YYYY-MM-DDTHH:MM:SS.NNNNNNNNNZ` (e.g. `2024-04-12T15:30:45.123456789Z`).

## Context

We need a single timestamp format that works consistently across our internal systems and integrates cleanly with the third-party systems we exchange data with. Today we sit between several pressures:

- **JSON has no native timestamp.** We have to choose how timestamps serialize to strings and parse back.
- **Local vs UTC time.** Some systems are configured for local time (useful when triggering events in the user's time zone), others for UTC. Mixing the two causes silent bugs.
- **Precision varies.** Different systems use seconds, milliseconds, or nanoseconds. The Linux `date` command defaults to seconds; financial-grade systems (e.g. Nasdaq) require nanoseconds.

## Decision

Use **ISO 8601 in UTC with nanosecond precision** as the canonical wire format:

```
YYYY-MM-DDTHH:MM:SS.NNNNNNNNNZ
```

The `Z` denotes Zulu time (UTC). All timestamps in JSON, logs, and database fields that store text timestamps will use this format.

## Rationale

1. **Human-readable.** A developer can read the timestamp directly — no epoch math, no time-zone arithmetic. This matters for log inspection, sample data, and grepping.
2. **Standards-based.** ISO 8601 is a widely supported standard. Every modern language has a parser and formatter that handles it.
3. **UTC by default.** Anchoring on UTC eliminates an entire class of time-zone bugs at the boundary. Local-time presentation is a UI concern, not a storage concern.
4. **Nanosecond headroom.** Even where we only need millisecond precision today, the format reserves space for higher precision without requiring a future migration.
5. **Easy to convert.** ISO 8601 strings convert trivially to any internal representation a language prefers (Unix epoch, native datetime types).

## Implications

1. **Platform precision limits.** Some OS tools (e.g. macOS `date`) cannot natively produce or print nanoseconds. We will accept truncated output from those tools and rely on application-level formatting for canonical timestamps.
2. **Local-time handling stays at the edge.** Any system that needs to act on a user's local time will store UTC and convert to local time at the boundary.
3. **Conversion convention.** We will document how to serialize (datetime → string) and deserialize (string → datetime) in each language we use. Where libraries differ, we standardize on the one that round-trips ISO 8601 cleanly (e.g. `date-fns` for JavaScript, `chrono` for Rust).
4. **Duration tracking.** Durations are not timestamps; for time deltas we will use the language-native duration type or a numeric millisecond/nanosecond count, not subtract ISO strings.
5. **Logging compatibility.** Most log aggregators (Splunk, Sumo, ELK) ingest ISO 8601 natively. If a specific tool requires a different stamp, we adapt at the log shipper, not by changing the source format.
6. **Reversible.** Moving to a different format later (e.g. Unix epoch for performance-sensitive paths) is a contained change — the canonical format defines the wire, not the application's internal representation.

## Conclusion

ISO 8601 in UTC with nanosecond precision is the best balance of readability, interoperability, and future-proofing. We accept the small inconveniences (verbose strings, partial OS-tool support) in exchange for a single, unambiguous, broadly understood timestamp format across the stack.
