> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use a single git **monorepo** for all front-end, middleware, and back-end code while the team and project remain small enough that rapid iteration matters more than long-run stability. We will plan a transition to a polyrepo layout when scale demands it.

## Context

The project has three categories of software to develop:

- Front-end GUIs
- Middleware services
- Back-end servers

Our version control system is git, and we need to decide how to organize the code across repositories. The top-level options are:

- **Monorepo** — all components in one repository.
- **Polyrepo** — each component in its own repository.
- **Hybrid** — some mix of the two.

All the code we are developing is for our own organization; we are not running an open-source community project that benefits from independent component repos.

## Decision

- **Monorepo** while the organization, team, and project are relatively small and rapid iteration is the top priority.
- **Polyrepo** when the organization, team, and project grow large enough that sustaining stability outweighs iteration speed.

The current state matches the first condition, so we are starting with a monorepo.

## Rationale

1. **Coordinated changes.** A monorepo lets a single change touch front-end, middleware, and back-end together — a property that disappears when each lives in its own repository.
2. **Lower overhead at small scale.** Tooling for build, test, and dependency management is simpler when everything lives in one place. We get to focus on the product, not on managing inter-repo coordination.
3. **Proven at scale.** Google, Facebook, and others run very large monorepos successfully. The scaling concerns are real but well in the future for us, and the practices we'd need then are already documented.
4. **Customer obsession.** Getting the product to customers fast is the priority. A monorepo gets us there quicker than a polyrepo and lets us iterate faster once we are there.
5. **Easily reversible.** If the monorepo turns out to be the wrong choice — or leadership prefers a polyrepo — we can split the repository later. The transition is significant work, but it is feasible.

## Implications

1. **CI/CD adjustments.** Any existing per-component pipeline needs to be reorganized to test multiple projects within one repository.
2. **Build times.** A naive CI configuration will build the whole monorepo on every change. We will need selective-build tooling (path-based triggers, affected-project detection) before this becomes painful.
3. **Tooling on the horizon.** As we grow, monorepo-specific tools (Bazel, Nx, Turborepo) become more relevant. We will create separate decisions for these when needed.
4. **Branching and access.** A monorepo concentrates access. We will need clear conventions for branching, code review, and CODEOWNERS-style ownership so unrelated changes do not collide.
5. **Migration cost ahead.** We accept that a future split to polyrepo is non-trivial DevOps work. We are choosing to defer that cost in exchange for short-term velocity.

## Conclusion

A monorepo is the right structure for the project's current size and velocity needs. The decision is explicitly time-bounded: when scale and stability become the dominant concerns, we will plan and execute a transition to polyrepo rather than try to keep stretching the monorepo past its useful range.
