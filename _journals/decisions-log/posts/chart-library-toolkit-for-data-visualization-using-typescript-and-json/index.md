> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use Apache ECharts as the chart library for our TypeScript/JSON-driven visualizations of financial, scientific, and government data. D3.js remains available as an escape hatch for visualizations ECharts cannot express.

## Context

We are building a startup product that visualizes financial, scientific, and government datasets. The visual surface needs to cover advanced and somewhat unusual chart types — doughnut, radar, area-with-time-axis, candlestick, Nightingale, geo SVG maps, and clustering-process charts — and to do so quickly enough for fast iteration on the MVP.

Three properties drive the choice:

1. **Coverage of advanced interactive chart types**, especially financial and geographic.
2. **Agile development velocity** — we need to ship and reshape charts quickly, not write a visualization framework.
3. **Fully free and open source**, so licensing does not gate growth.

Runtime speed, large-scale performance, and backwards compatibility are explicitly *not* priorities at this stage.

We evaluated eight candidates: Apache ECharts, Chart.js, ApexCharts, AG Charts, Highcharts, Carbon Charts, Layer Cake, and D3.js.

## Decision

We will adopt **Apache ECharts** as the default charting toolkit. **D3.js** is approved as a secondary tool for bespoke visualizations that fall outside ECharts' built-in chart catalogue.

## Rationale

1. **Coverage.** Apache ECharts natively supports every chart type on our must-have list — doughnut, radar, area-with-time, candlestick, Nightingale, geo SVG maps, and clustering. None of the other open-source candidates cover the full set without significant custom work.
2. **License fit.** ECharts is Apache-licensed and fully open-source. AG Charts and Highcharts have commercial editions for the features we need; we want to avoid the cliff.
3. **Interactivity.** ECharts ships with the zoom, pan, brush, and dynamic-update primitives we expect to lean on heavily for financial dashboards.
4. **Speed of iteration.** Configuration is JSON-driven, which composes well with TypeScript types and lets us treat chart definitions as data we can store, diff, and template.
5. **Escape hatch.** When ECharts cannot express a visualization, D3.js remains the obvious fallback — at the cost of significantly more development time per chart.

## Implications

1. **Learning curve.** ECharts is powerful but has a broad API surface. We will invest in a small internal style guide and reusable wrappers so chart code stays consistent across the product.
2. **Bundle size.** ECharts is sizeable; we will use its tree-shakable build to include only the chart types we ship.
3. **TypeScript typing.** Community types exist but are imperfect; we will keep a thin typed wrapper that hardens the call sites we use most.
4. **D3 as a second tool.** Adding D3.js to the toolbox means engineers will eventually encounter both. We will keep custom D3 work behind clearly labelled internal components so it does not sprawl.
5. **Rejected alternatives.** Chart.js, ApexCharts, and Carbon Charts lack the advanced chart types we need. AG Charts and Highcharts hit the licensing constraint. Layer Cake is too low-level for routine charting.

## Conclusion

ECharts is the only candidate that meets all three of our hard requirements — chart coverage, open source, and quick iteration — without forcing us into D3-scale custom work or a commercial licence. We adopt it as the default, with D3.js held in reserve for the long tail of bespoke visualizations.
