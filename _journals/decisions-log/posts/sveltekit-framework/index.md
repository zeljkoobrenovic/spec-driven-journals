> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will use SvelteKit as the application framework. SvelteKit gives us file-based routing, server-side rendering, and a single configured toolchain on top of Svelte components.

## Context

We have already decided to use Svelte for UI components. We now need a framework around Svelte that handles routing, server-side rendering, build configuration, and deployment targets, rather than wiring those together ourselves. The realistic candidates are SvelteKit (the official Svelte meta-framework), or assembling a custom stack from Vite, a routing library, and a separate SSR layer. We also briefly considered staying with another framework (Next.js on React, Nuxt on Vue) at this layer, but switching the UI library at this point would undo a recent decision.

## Decision

We will build the application on SvelteKit. New routes, layouts, and endpoints follow SvelteKit conventions; existing prototypes will be ported into the SvelteKit project structure.

## Rationale

1. **Official framework.** SvelteKit is maintained by the Svelte core team. Aligning with it minimizes the risk that future Svelte changes break our stack.
2. **Routing and SSR included.** File-based routing, layouts, server-side rendering, and static site generation all ship in the box. We avoid stitching these together from independent libraries.
3. **Configured tooling.** SvelteKit comes with a sensible Vite-based build, dev server, hot reloading, and adapters for common deployment targets. We do not have to design the toolchain from scratch.
4. **TypeScript support.** First-class TypeScript across routes, endpoints, and components keeps our code typed end to end.
5. **Adapter model.** The same application can target Node, edge runtimes, or static hosting by swapping an adapter, which keeps deployment options open as the product grows.

## Implications

1. **Convention adoption.** Folder structure, route files, and `load` functions follow SvelteKit's conventions. Engineers need to learn them rather than invent their own.
2. **Framework lock-in at the routing layer.** Moving off SvelteKit later would mean re-doing routing and SSR. The component code is largely portable; the framework wiring is not.
3. **Maturity.** SvelteKit's API has been stabilizing but is younger than Next.js or Nuxt. We will track breaking changes and plan migrations as the framework evolves.
4. **Hosting choices.** Some deployment targets work better than others depending on which adapter we use. We will validate the adapter for our chosen host early.

## Conclusion

SvelteKit is the natural framework to pair with our Svelte component decision. It gives us routing, SSR, and a configured toolchain without bespoke glue, in exchange for adopting its conventions and tracking its evolution.
