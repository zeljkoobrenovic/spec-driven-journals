> **Status**: Accepted (2020-01-21)
>
> **Decision**: We will structure the programming code editor as a modular platform with four clearly separated components — Core, UI, Plugins, and Integrations — prioritizing performance, extensibility, and broad language compatibility.

## Context

A code editor is one of the few tools developers use continuously throughout the day, and expectations vary widely across languages, workflows, and teams. We need an architecture that lets us deliver a fast, focused editing experience out of the box while still allowing individual developers and teams to shape the tool to their work without forking it.

The qualities that matter most:

- **Modularity** so the editor can be customized and extended without changes to the foundation.
- **Performance** so editing always feels immediate.
- **Usability** so the interface stays out of the developer's way.
- **Extensibility** through third-party plugins for features the core does not need to ship.
- **Compatibility** across the programming languages and toolchains our users work in.

## Decision

The editor is built as four cooperating components with explicit boundaries:

- **Core** — text editing, file management, syntax highlighting, and the editing primitives every user depends on.
- **UI** — menus, toolbars, panels, keyboard shortcuts; the surface developers interact with.
- **Plugins** — a stable extension API that third parties use to add features such as completion, linting, and debugging.
- **Integrations** — connectors to external tools and systems: version control, build systems, debuggers, language servers.

The Core stays small and performance-critical. Everything optional ships as a plugin or an integration.

## Rationale

1. **Separation of concerns.** A clear split between Core, UI, Plugins, and Integrations means changes in one area do not ripple through the others, keeping the codebase maintainable as the editor grows.
2. **Performance through restraint.** Keeping the Core minimal protects responsiveness; heavier features live behind the plugin boundary where users only pay for what they enable.
3. **Extensibility without forks.** A stable plugin API lets the ecosystem add capability we do not need to own, so users with niche workflows can satisfy them without modifying the editor itself.
4. **Language and toolchain reach.** Integrations (language servers, build tools, VCS) decouple the editor from any single language or stack, broadening who can use it productively.
5. **UI as a layer, not a hard-coded surface.** Treating the UI as its own component lets us evolve presentation — themes, layouts, keybindings — without disturbing editing logic.

## Implications

1. **API stewardship.** The plugin and integration APIs become contracts. They must be versioned, documented, and changed carefully.
2. **Performance budget on the Core.** Anything added to the Core needs to justify itself against the responsiveness bar; features that fail that test belong in a plugin.
3. **Plugin ecosystem.** We need discovery, install, and update mechanisms for plugins, plus a baseline of quality and security expectations.
4. **Integration surface.** Each integration (LSP, Git, build, debug) is a sustained engineering commitment, not a one-off connector.
5. **Configurability has limits.** Modularity is a means, not an end — defaults should still be opinionated enough that a new user has a productive experience without configuration.

## Conclusion

A modular Core/UI/Plugins/Integrations architecture lets the editor stay small and fast at its heart while still being reshaped by users and extended by a wider ecosystem. The discipline is in keeping the boundaries clean and the Core lean over time.
