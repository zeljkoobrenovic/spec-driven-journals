> **Status**: Accepted
>
> **Decision**: When an option appears in multiple places — Considered Options, Pros and Cons, Decision Outcome — repeat its name in full each time. Do not introduce short identifiers like `[A]`.

## Context

An option is mentioned at "Considered Options", repeated under "Pros and Cons of the Options", and named again under "Decision Outcome". We need to decide whether to spell out the full option name each time or use a short identifier.

## Decision Drivers

- **Easy to read.**
- **Easy to write.**
- **Avoid copy-and-paste errors.**

## Considered Options

- Repeat all option names when they occur.
- Assign an identifier to each option (e.g. `[A] Use gradle as build tool`).

## Decision

**Repeat the full option name** wherever an option is referenced.

## Rationale

1. **No Markdown standard for identifiers.** There is no idiomatic way to attach a short label to a section that survives rendering.
2. **Multiple options become hard to track.** A reader following the document linearly should not have to remember which short identifier maps to which option.

## Implications

1. **Option names should stay short.** Long option names become noisy when repeated. Pick names that compress well.
2. **Find-and-replace works.** Renaming an option is a search-and-replace operation across the document.

## Conclusion

Repetition costs a few extra words; aliases cost the reader's working memory. We optimize for the reader.
