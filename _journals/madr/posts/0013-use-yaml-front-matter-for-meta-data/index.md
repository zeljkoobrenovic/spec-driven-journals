> **Status**: Accepted
>
> **Decision**: ADR metadata fields — status, decision makers, date — are written in a YAML front-matter block at the top of the ADR, not inside the body.

## Context

MADR offers metadata fields like `Status`, `Decision Maker(s)`, and `Date`. These describe *about* the decision rather than being part of the decision itself. We need to decide where to put them.

## Decision Drivers

- **Easy to read.**
- **Easy to write.**

## Considered Options

- Use YAML front matter.
- Use plain Markdown everywhere.

## Decision

Use **YAML front matter** for metadata. Example:

```markdown
---
status: accepted
decision-makers:
date:
---

### Context and problem statement

We want to record architectural decisions made in this project.
Which format and structure should these records follow?
```

## Rationale

1. **Shorter body.** The essence of the ADR — context, decision, rationale — is not pushed below a wall of metadata.
2. **Easier tooling.** Tools (Jekyll, static-site generators, indexers) parse front matter natively and can filter, sort, or badge ADRs without prose parsing.
3. **Visual de-emphasis.** Front matter signals "this is metadata" — lower importance than the decision narrative.

## Implications

1. **Renderer support required.** Renderers must strip front matter from the body. Most modern tooling does; a few do not.
2. **Schema is not enforced.** Front matter pretends to be more structured than it really is — there is no enforced vocabulary for status values, for example. Authors should follow conventions.
3. **Not part of CommonMark.** Some renderers will show the YAML block as literal text. Where that happens, choose a renderer that supports front matter.

## Rejected alternative

- **Plain Markdown everywhere** — all parsers handle it, but metadata is then indistinguishable from content and requires bespoke parsing.
