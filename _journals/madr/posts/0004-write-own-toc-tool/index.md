> **Status**: Accepted
>
> **Decision**: We will write and maintain `adr-log` as the canonical MADR TOC tool, rather than rely on `adr-tools`' TOC functionality.

## Context

ADRs need to be indexed somehow — for example, to display a list of all decisions on a project website.

Options considered:

- Write our own tool, `adr-log`.
- Use `adr-tools`' built-in TOC functionality.

## Decision

Write and maintain **`adr-log`** as the canonical MADR TOC tool.

## Rationale

1. **Desired TOC format.** We want entries in the form `ADR-0001 - Title`. `adr-tools` only emits the title.
2. **Friendlier installation.** `adr-log` installs via `npm install -g adr-log`, which is more accessible than installing `adr-tools` for many users.
3. **Upstream contribution is feasible but not free.** `adr-tools` already supports header/footer injection ([example](https://github.com/npryce/adr-tools/blob/master/tests/generate-contents-with-header-and-footer.sh)), but contributing would leave us coordinating releases with a third party.

## Implications

1. **Another tool to maintain.** `adr-log` needs to keep up with MADR changes.
2. **Independent release cadence.** TOC behaviour can change in step with the format without waiting on an upstream merge.
3. **Different distribution.** Users install via `npm` rather than the shell-script install used by `adr-tools`.

## Conclusion

A small dedicated tool is cheaper to maintain than the indirection of upstream contribution, given the specific output format we want.
