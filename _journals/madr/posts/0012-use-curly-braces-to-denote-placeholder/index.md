> **Status**: Accepted
>
> **Decision**: Template placeholders in MADR templates use **curly braces** (`{option 1}`). The exception is the `bare` template, which uses HTML comments because its target audience reads Markdown source directly.

## Context

When writing an ADR from a template, authors replace placeholders with real values. We need a convention for marking those placeholders that is unambiguous, easy to find-and-replace, and doesn't conflict with Markdown's own syntax.

## Considered Options

- Curly braces — `{option 1}`.
- Square brackets — `[option 1]`.
- Angle brackets — `<option 1>`.
- HTML comments — `<!-- option 1 -->`.

## Decision

Use **curly braces** in all templates. Use HTML comments in the `bare` template only.

## Rationale

1. **Consistent with mustache templates.** `{placeholder}` is a familiar pattern across templating systems ([reference](https://krasimirtsonev.com/blog/article/markdown-smart-placeholders)).
2. **No conflict with Markdown links.** Square brackets collide with link syntax `[text](url)` and produce confusing rendering.
3. **Clearly distinct from template comments.** The placeholder marker should be different from any "instructions to the author" markers — curly braces achieve that.

## Implications

1. **Author convention.** Authors learn `{...}` as the placeholder marker across every template.
2. **`bare` template exception.** The `bare` variant uses HTML comments because its readers are advanced users editing raw Markdown, not consumers of rendered HTML.
3. **Rejected alternatives.**
   - **Square brackets** — used in MADR 1.x and 2.x, but conflicts with link syntax and users often forgot to remove them (e.g. `Date: [2021-03-12]`).
   - **Angle brackets** — clean in source but easy to confuse with an HTML element.
   - **HTML comments** — natural for direct Markdown editing but invisible in rendered HTML, which causes confusion when an author skips replacement.

## Conclusion

Curly braces are the clearest "this is a placeholder" signal that does not collide with Markdown's own syntax.
