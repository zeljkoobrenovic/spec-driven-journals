> **Status**: Accepted
>
> **Decision**: ADR filenames follow the pattern `NNNN-title-with-dashes.md`.

## Context

We need a consistent filename pattern for ADR files. The choice affects identifiers, referencing, and command-line ergonomics.

Options considered:

- `NNNN-title-with-dashes.md` — used by [adr-tools](https://github.com/npryce/adr-tools).
- `YYYY-MM-DD Title.md` — see [Joel Parker Henderson's conventions](https://github.com/joelparkerhenderson/architecture_decision_record#adr-file-name-conventions).

## Decision

Adopt **`NNNN-title-with-dashes.md`** as the canonical filename pattern.

## Rationale

1. **Stable identifier.** `NNNN` is a unique number that can be referenced as `ADR-0001` in plain text and as `@ADR(1)` in Java code (via [e-adr](https://adr.github.io/e-adr/)).
2. **Creation dates lose meaning over time.** Once an ADR is updated, the original date becomes historical trivia. (See the discussion in [Does Git have keyword expansion?](https://git.wiki.kernel.org/index.php/GitFaq#Does_Git_have_keyword_expansion.3F).) Version control already records when each line was written.
3. **No spaces.** Dash-separated names are friendlier on the command line than `YYYY-MM-DD Title.md`.
4. **Compatible with existing tools.** This is the format `adr-tools` already produces, so existing ADR collections drop in without renaming.

## Implications

1. **Dates of creation, if relevant, live in the ADR body or in version control** — not in the filename.
2. **Renaming to fix a typo** requires updating number-based references, or keeping the old number stable.

## Conclusion

`NNNN-title-with-dashes.md` is the lowest-friction format that aligns with the broader ADR ecosystem.
