> **Status**: DRAFT
>
> **Decision**: Organization will use AVIV Group's current native mobile baseline for mobile work: Swift and SwiftUI on iOS, Kotlin and Jetpack Compose on Android. Organization will copy the inherited iOS and Android app source code and operate those copies as Organization-owned product codebases from day one. Alignment with AVIV is a starting point, not a standing commitment.

## Decision

Adopt **AVIV Group's current native mobile baseline** as the starting point for Organization mobile applications:

* iOS uses Swift and SwiftUI.
* Android uses Kotlin and Jetpack Compose.
* Organization takes source copies of the inherited AVIV iOS and Android applications.
* The copied apps become Organization-owned product codebases.
* Organization owns build pipelines, release trains, app store submissions, SDK upgrades, dependency upgrades, observability, support, documentation, and roadmap decisions.
* Future AVIV mobile stack changes require an explicit Organization decision. They do not flow automatically into Organization apps.

This is a **starting-point decision**. It avoids rebuilding inherited apps immediately while preserving Organization's right to diverge when product, operating-model, or technical needs justify it.

## How to Use This Decision

Use this ADR before starting or reshaping mobile work:

| Question | Default answer |
| --- | --- |
| Is this new or inherited native mobile work for Organization? | Start from the inherited native baseline. |
| Is the change platform-specific iOS work? | Use Swift and SwiftUI unless an app-local constraint says otherwise. |
| Is the change platform-specific Android work? | Use Kotlin and Jetpack Compose unless an app-local constraint says otherwise. |
| Is the proposal a cross-platform framework, rebuild, or deep divergence from the copied apps? | Treat it as an exception requiring an explicit decision. |
| Is AVIV changing its mobile stack? | Review deliberately; do not auto-adopt. |

The practical question is not "do we like AVIV's stack?" The practical question is whether Organization should spend scarce time rebuilding working native apps before it can operate, release, and improve them. The answer is no.

## Why This Default

The inherited apps already exist on a mainstream native stack. Swift, SwiftUI, Kotlin, and Jetpack Compose are credible choices even without AVIV alignment. Starting from them gives Organization:

* a working app foundation instead of a blank-page rebuild;
* access to existing AVIV patterns, modules, examples, and operational knowledge;
* native integration with Apple and Google platform capabilities;
* faster ownership of release, support, observability, and product differentiation;
* a stable baseline from which later divergence can be evaluated with evidence.

The decision also makes ownership explicit. Taking copied source gives autonomy, but it also means Organization carries the maintenance burden for two native applications.

## Ownership Boundary

Owning the copied apps means Organization is accountable for:

| Area | Organization responsibility |
| --- | --- |
| Source code | Review, merge, adapt, and refactor the copied iOS and Android codebases. |
| Release | Signing, store submission, staged rollout, rollback, release notes, and release readiness. |
| Platform change | Apple and Google SDK upgrades, OS compatibility, dependency upgrades, and compliance changes. |
| Product behavior | Cross-platform parity where users expect it, and deliberate platform-specific differences where they make sense. |
| Production health | Crash monitoring, analytics, incident response, support triage, and operational runbooks. |
| AVIV alignment | Selective porting of fixes or patterns when there is a clear Organization reason. |

AVIV source history and future changes are inputs, not governance. Organization should neither ignore useful AVIV changes nor pull them blindly.

## Consequences

### Positive

* Inherited iOS and Android apps can be used and extended without immediate re-platforming.
* Teams start from mainstream native stacks with broad market support.
* Organization can reuse AVIV patterns and modules while building local ownership.
* Product teams can focus first on release capability, operational stability, and Organization-specific differentiation.
* Organization controls its own app store identity, release cadence, support model, and roadmap.

### Negative

* Organization inherits the technical debt, assumptions, bugs, and architectural constraints present at copy time.
* Two native apps require ongoing platform-specific maintenance and staffing.
* AVIV fixes and features do not arrive automatically; selective alignment becomes work.
* iOS and Android behavior can drift unless product and engineering teams manage parity deliberately.
* The inherited architecture may encode AVIV-specific assumptions that require later reshaping.

## What This Means for Teams

* Start new native mobile work on the copied Organization-owned apps.
* Treat inherited modules as starting assets, not untouchable constraints.
* Make release ownership visible before shipping: signing, stores, staged rollout, rollback, crash monitoring, and support.
* Document meaningful divergence from AVIV with reason, owner, expected maintenance cost, and migration impact.
* Keep product behavior aligned across iOS and Android where users expect consistency.
* Check analytics, consent, privacy, deep links, notifications, and regulatory requirements early because they often shape mobile architecture.

## Exception Criteria

A proposal to diverge from the native baseline should explain:

* the user, product, or engineering problem that the baseline cannot solve well;
* whether the issue is caused by the technology stack, inherited architecture, local implementation quality, or product requirements;
* the migration impact on existing features, analytics, consent flows, notifications, deep links, app store operations, and support;
* who will own the new stack or architecture over time;
* how iOS and Android behavior will remain coherent during and after the change;
* when the exception will be revisited.

Valid exceptions are expected to be significant. A preference for a different mobile framework is not enough.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Adopt AVIV's native baseline and own source copies** | Chosen. Fastest path to running inherited apps while preserving Organization autonomy. |
| **Build independent Organization apps immediately** | Gives maximum autonomy, but delays delivery and discards useful inherited assets. |
| **Partial adoption** | Can be useful later, but creates integration and parity risk if used as the initial model. |
| **Cross-platform rewrite** | May reduce some duplicated UI work, but would add migration cost, native escape hatches, and release risk before Organization has stabilized ownership. |

## Scope and Revisiting

In scope:

* inherited iOS and Android app codebases;
* new native mobile work at Organization;
* mobile release and support ownership;
* deliberate divergence from AVIV's future mobile direction.

Out of scope:

* web frontend stack decisions, covered by [[web-tech-stack]];
* backend service stack decisions, covered by [[backend-tech-stack-framework]];
* detailed app architecture, module boundaries, and release runbooks.

Revisit this decision if:

* the inherited apps become more expensive to maintain than to replace;
* app store, regulatory, privacy, or platform requirements cannot be met cleanly;
* product differentiation is blocked by inherited assumptions;
* iOS and Android drift in user-visible behavior;
* AVIV makes a major mobile stack change that Organization may want to adopt or reject explicitly.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Native mobile app** | An app built directly for iOS or Android using platform-native languages, tools, SDKs, and release channels. |
| **Swift** | The default iOS implementation language for Organization mobile work. |
| **SwiftUI** | The default iOS UI framework for new Swift-native UI work, while inherited UIKit code may still exist. |
| **Kotlin** | The default Android implementation language for Organization mobile work. |
| **Jetpack Compose** | The default Android declarative UI toolkit for new Kotlin-native UI work, while inherited Android View code may still exist. |
| **Source copy** | An Organization-owned copy of inherited AVIV code, not an automatically synchronized upstream dependency. |
| **Divergence** | A deliberate Organization change that makes its app or stack differ from AVIV's future direction. |
