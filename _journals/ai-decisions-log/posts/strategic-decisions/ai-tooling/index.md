> **Status**: DRAFT
>
> **Decision**: Organization will adopt Claude Code as the default AI developer tool for 2026. This is a platform decision, not a model leaderboard call: it selects the shared surface where CLAUDE.md files, Skills, MCP servers, subagents, hooks, settings, templates, permission modes, registries, and audit controls can compound across teams. Codex, Gemini, and future agentic coding tools remain available by documented exception.

## Decision

Adopt **Claude Code as the default AI developer tool** across Organization for 2026.

Concretely:

* New AI-assisted software-development work uses Claude Code by default.
* Shared AI developer-platform assets are built first for Claude Code: repository guidance, Skills, MCP servers, subagents, hooks, settings, permission modes, registries, templates, and audit controls.
* Codex, Gemini, and future agentic coding tools remain available through documented, time-bounded exceptions.
* AI Engineering Enablement owns the default platform and exception process, aligned with [[ai-operating-model]].
* The default is reviewed every six months. The next planned review is **2026-11-15**.

This decision chooses the shared **tooling platform surface**. It does not claim Claude Code is permanently the best model, ban alternative tools, or reduce review/testing obligations.

## How to Use This Decision

Use this ADR when starting or changing AI-assisted development workflows:

| Question | Default answer |
| --- | --- |
| Is a team starting AI-assisted coding in a repository? | Use Claude Code and the shared baseline. |
| Does the work need repository guidance, Skills, MCP, subagents, hooks, or settings? | Build on the Claude Code platform surface first. |
| Does another tool fit a specific workflow better? | Request a time-bounded exception with evidence and controls. |
| Is a model benchmark better this month? | Treat it as review input, not an automatic platform change. |
| Is the task high-risk? | Follow [[ai-policy]] and the repository's local controls regardless of tool. |

The decision should move conversation from "which assistant is most exciting?" to "which platform lets Organization make safe, reusable AI-assisted workflows normal?"

## Why This Default

AI developer tooling is not only chat. Once a tool can inspect repositories, run commands, call external tools, edit files, and delegate work, it becomes part of the engineering platform.

Claude Code is the 2026 default because it gives Organization a single place where shared investment can compound:

* repository-local guidance through CLAUDE.md;
* reusable workflows through Skills;
* controlled external context through MCP;
* role-specific work through subagents;
* executable policy through settings, hooks, permission modes, and CI integration;
* shared evidence through registries, templates, and audit trails.

The main value is not that every individual task is best in Claude Code. The value is that standards, policy, enablement, review evidence, and reusable workflows have one default implementation surface.

## Exception Criteria

A non-default AI coding tool may be justified when it has a concrete workflow advantage, such as:

* better support for a specific language, IDE, operating system, or local toolchain;
* stronger model or tool behavior for a bounded task;
* required integration with an ecosystem not covered by Claude Code;
* a security, data-residency, or enterprise constraint that makes the default unsuitable;
* a pilot intended to inform the six-month review.

An exception request should include workflow, repositories, expected actions, data classification, requested tool capabilities, blocked capabilities, audit evidence, success measures, owner, and review date. Preference for another UI is not enough.

## Consequences

### Positive

* Enablement investment compounds on one platform surface.
* Policy in [[ai-policy]] has a primary enforcement path.
* Onboarding, support, and internal mobility benefit from a shared toolchain.
* Shared Skills, MCP servers, subagents, hooks, and settings can be reused across repositories.
* The exception path keeps alternatives available without fragmenting the default.
* The six-month review prevents the default from becoming permanent by inertia.

### Negative

* Organization accepts vendor concentration risk for the default tooling surface.
* Teams already invested in other tools pay a switching or adaptation cost.
* Claude-specific governance assets must be translated if the default changes later.
* Exception-approved tools need their own control envelope, which adds governance work.
* A six-month review may lag short-lived market shifts, but shorter cycles would create churn.

## What This Means for Teams

* Start from the shared Claude Code baseline for repository onboarding.
* Keep repository guidance current so the agent sees local standards before editing.
* Turn repeated workflows into maintained Skills or subagents, not copied prompts.
* Use MCP only when live external context or tools are genuinely needed.
* Keep high-risk work under the controls defined in [[ai-policy]].
* Record material AI-assisted changes with enough evidence for review.
* Bring useful local patterns back to AI Engineering Enablement for possible shared catalog inclusion.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Claude Code as default, others by exception** | Chosen. Best current fit for shared governance, repository guidance, reusable workflows, and controlled execution. |
| **Codex as default** | Credible alternative, but would require rebuilding the shared control and enablement surface around a different default. |
| **Gemini CLI as default** | Credible alternative for some workflows, but not selected as the main 2026 platform surface. |
| **No default** | Maximizes local choice, but fragments policy, onboarding, security review, and reusable workflow investment. |

## Scope and Revisiting

In scope:

* AI-assisted software development at Organization;
* developer-agent default tooling;
* shared Skills, MCP servers, subagents, hooks, settings, templates, and registries;
* exception-approved coding agents.

Out of scope:

* end-user product AI features;
* internal non-engineering AI tools;
* model-level choices for production inference;
* procurement terms except where they materially affect the tool default.

Revisit this decision on **2026-11-15**, or earlier if:

* a competing platform shows a durable advantage in Organization's real workflows;
* Claude Code regresses materially in governance, security, cost, or enterprise readiness;
* exception pilots produce strong evidence that the default should change;
* Organization's modernization strategy changes enough that another tool is clearly better.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Model** | The underlying LLM used by a tool. |
| **Tool** | The developer-facing agent, CLI, IDE integration, and execution environment. |
| **Platform surface** | Files, settings, hooks, Skills, MCP servers, subagents, registries, and templates that teams share. |
| **CLAUDE.md** | Repository or directory guidance read by Claude Code. |
| **Skill** | Reusable instruction package for a specific capability or workflow. |
| **MCP server** | Model Context Protocol server that exposes external tools, data, or services to the agent. |
| **Subagent** | Specialized agent with narrower task framing, tools, and output contract. |
| **Hook** | Automation triggered by agent events, such as tool use or stop events. |
| **Permission mode** | Approval model for tool use and actions. |
| **Registry** | Central source of truth for approved tools, MCP servers, Skills, subagents, and exceptions. |
| **Exception** | Approved use of a non-default tool or non-standard configuration. |
