> **Status**: DRAFT
>
> **Decision**: Organization will standardize backend compute on Docker containers running on AWS ECS with the Fargate launch type, using Ubuntu as the foundational base image layer. ECS Fargate is the default because it is a simple, mainstream, AWS-native compute pattern that Organization can own without operating Kubernetes clusters, EC2 fleets, or broad serverless diversity by default.

## Decision

Adopt **Docker containers on AWS ECS with the Fargate launch type** as Organization's default backend compute platform.

Concretely:

* Backend services are packaged as Docker container images.
* Service images start from the approved Ubuntu-based foundation.
* Long-running services run on AWS ECS.
* Fargate is the default launch type, so Organization does not own EC2 worker nodes for general service compute.
* Lambda, EKS, ECS on EC2, standalone EC2, and other compute models are exceptions, not parallel defaults.

This decision builds on the AWS cloud decision, [[infra-tech-aws]]. It defines the default compute pattern; it does not define every service template, task parameter, autoscaling rule, or deployment module.

## How to Use This Decision

Use this ADR when choosing how a backend workload should run:

| Question | Default answer |
| --- | --- |
| Is this an HTTP API, internal service, background worker, or scheduled backend job? | Package it as a container and run it on ECS Fargate. |
| Does the workload need long-running service behavior, health checks, predictable deployment, or normal service observability? | ECS Fargate is the default. |
| Is the workload very short-lived, event-triggered, and stateless? | Lambda may be a narrow exception. |
| Does the workload depend on Kubernetes APIs, operators, or ecosystem components? | EKS may be an exception, but only with explicit ownership. |
| Is the workload large, stable, and highly utilized? | ECS on EC2 may be reviewed for durable cost reasons, with node-ownership accepted. |

The decision should move discussion from "which compute platform do we prefer?" to "does this workload have a concrete reason to leave the default?"

## Why This Default

Organization's expected backend workload shape is ordinary service compute: HTTP APIs, internal services, Kafka consumers, workers, and scheduled jobs. ECS Fargate fits that shape because it gives a mainstream container model without making Organization operate Kubernetes or EC2 fleets as the default.

The strategic value is concentration:

* one packaging model;
* one orchestrator;
* one launch type for normal services;
* one base-image policy;
* one set of templates, deployment guidance, dashboards, alerts, and runbooks;
* one exception path for workloads that genuinely do not fit.

Docker improves packaging consistency, but it does not make workloads cloud-independent by itself. Real services still depend on IAM, networking, logs, secrets, load balancing, deployment automation, and managed data services.

## What the Default Includes

| Area | Default expectation |
| --- | --- |
| Image | Built from the approved Ubuntu-based base image, tagged immutably, scanned, and stored in the approved registry. |
| Runtime | ECS task definitions describe image, CPU, memory, environment, secrets, IAM role, logging, and networking. |
| Service model | ECS services keep long-running tasks available and integrate with load balancing, health checks, and scaling. |
| Identity | Task roles grant only the permissions each service needs. |
| Networking | Subnets, security groups, load balancer exposure, and outbound access follow approved patterns. |
| Operations | Logs, metrics, traces, dashboards, alarms, rollback, and task-draining behavior exist before production use. |
| Maintenance | Base-image, runtime, and dependency updates have owners and expected cadence. |

These are not optional extras. They are part of what it means to run on the compute platform.

## Consequences

### Positive

* Organization gets one mainstream compute pattern for most backend services.
* Fargate removes EC2 node ownership, patching, and capacity management from the default operating model.
* ECS integrates directly with AWS IAM, VPC, ALB, CloudWatch, ECR, and Secrets Manager.
* A shared Ubuntu base-image foundation simplifies scanning, patching, vulnerability response, and runtime assumptions.
* Platform investment can focus on one narrow path instead of ECS, EKS, Lambda, EC2, and custom variants at the same time.
* The default aligns with the AWS decision in [[infra-tech-aws]] and the backend service stack in [[backend-tech-stack-framework]].

### Negative

* Organization accepts AWS-specific orchestrator lock-in to ECS.
* Fargate can cost more per requested CPU and memory than EC2-backed compute for stable high-utilization workloads.
* Some very short-lived event handlers may fit Lambda better than ECS Fargate.
* Engineers familiar with Kubernetes need to learn ECS concepts directly.
* Holding one default requires active exception review; otherwise compute fragmentation will return.
* A future move away from AWS would carry meaningful re-platforming cost.

## What This Means for Teams

* Start new backend services from the ECS Fargate path.
* Do not choose Lambda, EKS, EC2, or another compute model because it is familiar or fashionable.
* Treat CPU and memory settings as design choices; Fargate cost follows requested resources.
* Keep health checks meaningful enough for ECS and load balancers to act on.
* Keep image rebuilding, base-image updates, dependency patching, and vulnerability response in the service ownership model.
* Use approved templates or modules where possible so IAM, logging, networking, health checks, and deployment behavior remain consistent.
* Record exceptions with owner, reason, support model, cost model, and review trigger.

## Exception Criteria

An exception to ECS Fargate should explain:

* the workload characteristic that ECS Fargate does not handle well;
* the alternative compute model and why it is materially better;
* who owns operations, upgrades, incident response, security patching, observability, and cost management;
* how logs, metrics, traces, secrets, deployment, rollback, and networking integrate with Organization standards;
* what migration path exists if the exception becomes too expensive or too complex;
* when the exception will be revisited.

Potentially valid exceptions include:

* Lambda for short-lived, event-driven functions with clear timeout boundaries;
* EKS for workloads that genuinely depend on Kubernetes-native operators or APIs;
* ECS on EC2 for large, predictable, high-utilization workloads where FinOps analysis justifies node ownership;
* standalone EC2 only for legacy or infrastructure cases that cannot practically be containerized yet.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Docker on ECS Fargate with Ubuntu base images** | Chosen. Simple, mainstream, AWS-native, and sufficient for Organization's service shape. |
| **EKS / Kubernetes as default** | Powerful and portable at the orchestrator layer, but adds operating complexity Organization does not currently need. |
| **Lambda-first** | Strong for narrow event functions, but a poor default for long-running services, workers, and steady APIs. |
| **Pattern per service** | Optimizes locally, but recreates platform fragmentation across tooling, operations, security, and hiring. |

## Scope and Revisiting

In scope:

* backend HTTP services;
* internal APIs;
* background workers;
* Kafka consumers and producers running as services;
* scheduled backend jobs;
* base-image and compute-platform default.

Out of scope:

* public cloud provider selection, covered by [[infra-tech-aws]];
* AWS foundation and account tooling, covered by [[infra-tech-stack-aws-foundation]];
* AWS networking, covered by [[infra-tech-stack-networking]];
* data-plane managed services such as databases, object storage, and message brokers;
* one-off local operational tooling that does not become a production service.

Revisit this decision if:

* Organization workloads outgrow ECS's natural operating envelope;
* Fargate's cost profile changes materially relative to alternatives;
* AVIV Group changes its AWS operating model in a way that affects ECS as a default;
* a durable majority of Organization workloads are better served by another compute pattern;
* platform ownership cannot sustain the ECS Fargate templates, base-image policy, and upgrade path.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Container image** | The packaged application artifact, including runtime dependencies and the approved Ubuntu-based foundation. |
| **Base image** | The common operating-system layer from which service images start. |
| **ECS task definition** | The versioned runtime description for one or more containers: image, CPU, memory, IAM role, logging, networking, and configuration. |
| **ECS service** | The ECS construct that keeps long-running tasks available and integrates with scaling and load balancing. |
| **Fargate task** | A containerized workload launched without Organization managing the EC2 host fleet. |
| **Task role** | The IAM role assumed by a running ECS task. |
| **Launch type** | The ECS capacity model used to run tasks, such as Fargate or EC2. |
| **Exception** | A workload that uses Lambda, EKS, ECS on EC2, standalone EC2, or another compute model instead of the default. |
