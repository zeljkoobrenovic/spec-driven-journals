> **Status**: DRAFT
>
> **Decision**: For the next 6-12 months, Organization will inherit AVIV Group's central AWS networking setup for VPCs, subnets, route tables, transit, DNS, perimeter controls, and inter-account connectivity. Organization will use that period to simplify deliberately: consolidate accounts, distinguish internet-exposed paths from internal-only paths, and evaluate Amazon VPC Lattice with ECS and ECS Service Connect as simpler service-to-service networking defaults.

## Decision

For the next **6-12 months**, Organization will use **AVIV Group's existing central AWS networking setup** as its baseline.

Concretely:

* VPC design, subnet layout, route tables, transit connectivity, DNS, perimeter controls, and cross-account connectivity remain configured through the central AVIV Group networking setup.
* Organization will not stand up parallel networking tooling or fork central tooling during this period.
* Non-trivial changes to shared topology flow through the central AVIV Group networking function.
* Organization will use the period to build knowledge, consolidate accounts, simplify internal connectivity, and prepare for more direct ownership.
* Organization will evaluate Amazon VPC Lattice and ECS Service Connect as simpler service-to-service connectivity patterns.

This decision sits under [[infra-tech-aws]] and next to [[infra-tech-stack-aws-foundation]]. It is a time-bounded sequencing decision, not an endorsement of current network complexity as the desired end state.

## How to Use This Decision

Use this ADR before requesting network changes or creating new service-to-service connectivity:

| Question | Default answer |
| --- | --- |
| Does the change affect shared routing, shared DNS, transit, account-level topology, or internet exposure? | Treat it as a networking/foundation change and use the central path. |
| Is this ordinary service-to-service communication inside an approved pattern? | Keep ownership close to the workload team. |
| Can account consolidation remove the need for cross-account connectivity? | Prefer removing the network need over adding new routes. |
| Is the destination internet-exposed, partner-exposed, shared-internal, or team-internal? | Match review depth to exposure class. |
| Can VPC Lattice, ECS Service Connect, PrivateLink, or a VPC-local pattern solve the need narrowly? | Prefer service-level connectivity over broad network reachability. |

Network changes have high blast radius. A route, DNS association, perimeter rule, or transit change can affect many services at once.

## Why This Default

Inherited workloads already run inside the central AVIV network. Diverging immediately would create migration risk while the AWS foundation is also changing. The expected group direction is toward simpler and more delegated country ownership, so waiting and aligning is cheaper than diverging now and reconverging later.

The simplification thesis is practical:

* account consolidation removes many cross-account paths entirely;
* internet-exposed traffic needs stronger review than internal-only traffic;
* service-to-service needs should use service-level patterns where possible;
* custom central tooling should not become a deeper Organization dependency;
* VPC Lattice and ECS Service Connect may replace some older custom connectivity patterns after evaluation.

## Boundary: Foundation, Networking, Workload

| Question | Default owner |
| --- | --- |
| Should this AWS account exist, and which OU should it belong to? | AWS foundation governance, covered by [[infra-tech-stack-aws-foundation]]. |
| Which VPC, subnet, transit, DNS, and perimeter patterns are available? | Networking foundation. |
| Is a service internet-exposed or partner-exposed? | Product team plus security/network review. |
| Which internal services may call this service inside an approved pattern? | Product team within platform guardrails. |
| Which protocol-level authentication is required? | Product team and security. |
| How is the service observed and supported? | Product team with platform standards. |

If a change modifies shared routing, shared DNS, shared transit, account-level topology, or internet exposure, it belongs in networking/foundation review. If it only changes which approved internal service can call another approved internal service, ownership should stay closer to the workload.

## Commitments During the Inheritance Period

The 6-12 month window should produce concrete simplification:

* Reduce the roughly 200-account footprint into a smaller set of meaningful accounts where account boundaries still serve ownership, environment, compliance, and blast-radius needs.
* Classify service paths as internet-exposed, partner-exposed, shared-internal, or team-internal.
* Publish a small connectivity pattern catalog for common cases: same-account ECS calls, cross-account internal calls, private AWS service access, shared platform service access, and internet ingress.
* Evaluate VPC Lattice for selected cross-account or cross-VPC service-to-service needs.
* Evaluate ECS Service Connect for ECS-native service discovery and communication.
* Inventory central custom networking scripts and workflows that Organization depends on.
* Build operational literacy around route tables, DNS associations, security groups, VPC Flow Logs, Reachability Analyzer, and service-connectivity metrics.

These commitments make the ADR more than "wait for the central team".

## Operating Guardrails

During the inheritance period:

* No broad route or security group changes without source, destination, protocol, owner, and review date.
* No direct public exposure of workload services outside approved perimeter patterns.
* No new VPC peering mesh as a shortcut around the inherited transit model.
* No private hosted zone association changes without DNS ownership and rollback notes.
* No cross-account connectivity for accounts planned for retirement unless migration-scoped.
* No service-to-service pattern becomes a default until observability and debugging guidance exist.
* No Organization workflow should depend deeply on central custom scripts expected to be replaced.

The intent is not to slow ordinary delivery. It is to prevent hidden coupling that becomes expensive during account consolidation.

## Consequences

### Positive

* Inherited workloads stay on the network they already use.
* Organization avoids duplicate investment in tooling that is expected to change.
* Account consolidation can remove whole categories of cross-account networking work.
* Separating internet-exposed from internal-only paths reduces friction without weakening the perimeter.
* VPC Lattice and ECS Service Connect can be tested against real Organization workloads before becoming defaults.
* The approach stays coherent with the AWS foundation decision in [[infra-tech-stack-aws-foundation]].
* The 6-12 month horizon forces an explicit revisit.

### Negative

* Organization remains dependent on the central AVIV networking function for non-trivial changes.
* Lead times and prioritization are not fully under Organization control.
* Organization must deliberately build networking expertise.
* Workflows built on custom central tooling may need rework later.
* Account consolidation is itself a program with workload moves, IAM changes, cost/reporting updates, and migration risk.
* VPC Lattice and ECS Service Connect may not satisfy all cross-account or cross-VPC needs.

## What This Means for Teams

* Before asking for new connectivity, ask whether account consolidation, co-location, or an existing pattern removes the need.
* Classify traffic exposure explicitly: internet-exposed, partner-exposed, shared-internal, or team-internal.
* Prefer service-level connectivity patterns over broad network-level reachability.
* Treat DNS names, service discovery, authentication, authorization, observability, and ownership as part of the network design.
* Include rollout, rollback, review date, and incident owner in non-trivial network requests.
* Do not use custom central scripts as stable Organization platform APIs.

## Exception Criteria

A network exception should explain:

* the business capability that needs connectivity;
* source and destination accounts, VPCs, subnets, services, and owners;
* exposure class and data classification;
* whether the path is temporary migration support or permanent dependency;
* why an approved pattern, VPC Lattice, ECS Service Connect, PrivateLink, or account consolidation is not sufficient;
* authentication, authorization, DNS, observability, rollout, rollback, and incident ownership;
* review or expiry date.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Inherit central networking for 6-12 months and simplify deliberately** | Chosen. Minimizes near-term migration risk while preparing for a simpler model. |
| **Stand up Organization-specific networking tooling now** | More autonomy, but duplicates effort during a period when the group model is expected to change. |
| **Fork central networking tooling** | Local control, but long-term maintenance of custom non-differentiating tooling. |
| **Adopt conventional AWS-native networking immediately** | Aligns with the future direction, but moves too early before account consolidation and foundation changes reduce complexity. |

## Scope and Revisiting

In scope:

* VPCs, subnets, route tables, transit, DNS, perimeter controls, and cross-account connectivity;
* the tooling that manages shared AWS networking;
* service-to-service connectivity pattern evaluation;
* account consolidation as a network simplification lever.

Out of scope:

* cloud provider choice, covered by [[infra-tech-aws]];
* account vending and access foundation, covered by [[infra-tech-stack-aws-foundation]];
* compute platform choice, covered by [[infra-tech-stack-compute]];
* application-level authentication, authorization, and business protocol design except where they affect network pattern selection.

Revisit this decision in the planned **2026-11-01 to 2027-05-01** window, or earlier if:

* AVIV Group's networking foundation moves materially toward a simpler, less centralized model;
* custom central networking tooling is replaced or materially changed;
* account consolidation removes a significant share of cross-account complexity;
* VPC Lattice or ECS Service Connect evaluations produce a clear recommendation;
* the AWS foundation or cloud decisions materially change.

At the revisit, Organization should have a smaller account estate, a published internal connectivity default, clearer rules for perimeter exposure, and evidence on whether transit, PrivateLink, VPC Lattice, ECS Service Connect, VPC sharing, or VPC-local communication should be defaults for common cases.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **VPC** | A logically isolated AWS network in one Region. |
| **Subnet** | A slice of a VPC CIDR in one Availability Zone. |
| **Route table** | Rules that send traffic for a destination CIDR or prefix to a target. |
| **Security group** | Stateful resource-level allow rules for inbound and outbound traffic. |
| **Network ACL** | Stateless subnet-level allow/deny rules. |
| **Transit Gateway** | A regional network hub for connecting VPCs and on-premises networks. |
| **VPC peering** | Direct routing between two VPCs. |
| **AWS RAM** | AWS Resource Access Manager, used to share supported resources across accounts or OUs. |
| **Private hosted zone** | Route 53 DNS zone visible only to associated VPCs. |
| **PrivateLink** | Private connectivity to services through interface endpoints or endpoint services. |
| **VPC Lattice** | AWS application networking across services, VPCs, and accounts. |
| **ECS Service Connect** | ECS-native service discovery and service-to-service communication. |
| **Internet-exposed** | Reachable from the public internet through an approved public ingress path. |
| **Internal-only** | Not directly reachable from the public internet, but still requiring authorization and observability. |
