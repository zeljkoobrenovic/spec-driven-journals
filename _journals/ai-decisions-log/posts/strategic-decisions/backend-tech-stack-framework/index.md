> **Status**: DRAFT
>
> **Decision**: Organization will use .NET with C# as the default platform for core backend services. The default applies to domain services, stable HTTP APIs, Kafka producers and consumers, and long-running workers running in containers on AWS ECS Fargate. Node.js with TypeScript remains available for narrow frontend-edge use cases, but it is not the default for durable backend business logic. This is a platform-convergence decision, not a claim that .NET is best for every workload.

## Decision

Adopt **.NET with C#** as the default backend services platform for Organization.

Concretely:

* New core backend services use C# on the approved .NET LTS runtime.
* HTTP APIs use the approved ASP.NET Core service template.
* Background workers and Kafka consumers use the approved .NET worker pattern.
* Services run as Docker containers on AWS ECS Fargate, aligned with [[infra-tech-stack-compute]].
* Kafka producers and consumers target Amazon MSK, aligned with [[infra-tech-stack-messaging]].
* Node.js with TypeScript is allowed only where the component is genuinely frontend-edge: small BFFs, thin UI-specific aggregation, or simple glue with no durable business ownership.
* Java, Kotlin, Go, Python, and other backend stacks are not defaults for new core backend services. Exceptions need an explicit ADR or Architecture Advisory Forum advice.

This decision defines the default **language, runtime, and service model** for backend services. It does not define every NuGet package, service template detail, migration sequence, or training plan.

## How to Use This Decision

Use this decision to answer three questions before creating or modernizing a backend component:

| Question | Default answer |
| --- | --- |
| Is this a core backend service with durable business logic, owned data, stable APIs, Kafka processing, or long-running work? | Use .NET with C#. |
| Is this a thin frontend-edge component for one UI surface, with shallow aggregation and no durable business rules? | Node.js with TypeScript may fit, with review. |
| Is this a data-platform job, ML workload, one-off script, or operational tool? | Use the relevant domain standard; do not treat it as a backend-service exception by default. |

The decision is intended to stop recurring language debates for normal backend work. It should move discussion from "which stack do we like?" to "what service shape are we building, and does it fit the default?"

## Why This Default

Organization's strategic backend shape is:

* business-logic-heavy;
* API-serving;
* Kafka-driven;
* containerized;
* long-running;
* owned and operated by product/domain teams.

.NET with C# fits that shape because it gives one coherent service model for APIs, workers, dependency injection, configuration, logging, health checks, graceful shutdown, and strongly typed domain code. It also aligns with Organization's existing C#/.NET estate and concentrates platform investment in one primary backend stack.

The choice is not based on syntax preference. It is based on reducing platform fragmentation and making service ownership easier to teach, template, review, operate, and staff.

## What .NET Means Here

This ADR means **modern .NET**, not preservation of every historical .NET pattern.

| Modern expectation | Legacy pattern to avoid |
| --- | --- |
| Cross-platform .NET on an approved LTS baseline. | Treating old .NET Framework assumptions as the default. |
| Containerized services on Linux-based images. | Server-bound deployments with hidden machine configuration. |
| ASP.NET Core, Generic Host, and Worker Services. | Custom process wrappers and ad hoc startup code. |
| Typed configuration, dependency injection, health checks, and structured logging. | Scattered environment reads, service locators, static globals, and local-debug logs. |
| Explicit API and event contracts. | Sharing hidden DTOs or duplicating business rules across runtimes. |

The decision should therefore guide modernization work: when legacy C#/.NET systems are touched meaningfully, teams should move toward current .NET service patterns rather than copy old structure into a new runtime.

## Node.js Boundary

Node.js with TypeScript remains useful at the frontend edge. It is not a second strategic backend platform.

| Component | Default |
| --- | --- |
| Core domain service | .NET with C#. |
| Stable backend API | .NET with C#. |
| Kafka consumer with business side effects | .NET with C#. |
| Long-running worker | .NET with C#. |
| Shared backend library or platform capability | .NET with C#, unless a separate platform decision says otherwise. |
| Small BFF for one frontend | Node.js with TypeScript may fit. |
| Thin API aggregation close to UI code | Node.js with TypeScript may fit. |
| One-off operational script | Case by case; do not let scripts become hidden services. |

The boundary test is simple: if the component owns durable business behavior, data, events, or operational responsibility, it belongs on the backend platform. If it only shapes backend APIs for one frontend surface, the Node.js edge can be appropriate.

Choosing .NET also means frontend/backend integration should rely on explicit contracts rather than shared runtime code:

* HTTP APIs publish OpenAPI or equivalent contracts.
* Kafka topics use versioned schemas and compatibility rules.
* Clients are generated or maintained from published contracts.
* Durable validation and business rules live in backend services, not in BFFs or UI code.

## Consequences

### Positive

* Backend platform investment can concentrate on one default stack: templates, base images, analyzers, CI, observability, package policy, runbooks, and examples.
* Engineers get a consistent service model for APIs, workers, Kafka consumers, configuration, logging, health checks, and shutdown behavior.
* Kafka-driven services align with the MSK direction in [[infra-tech-stack-messaging]].
* Containerized services align with the ECS Fargate direction in [[infra-tech-stack-compute]].
* Existing C#/.NET knowledge becomes an asset, while legacy patterns can be retired deliberately.
* Node.js remains available where it is strongest: frontend-adjacent edge work.

### Negative

* Engineers whose backend experience is mainly TypeScript need a deliberate C#/.NET ramp-up path.
* Frontend/backend direct code sharing becomes less natural; contract-first integration becomes more important.
* The Node.js exception boundary needs active review, or it will grow into a second backend platform.
* .NET LTS does not remove maintenance work. Runtime versions, SDK feature bands, NuGet packages, and base images still need ownership and upgrade discipline.
* Some legacy systems may need pattern cleanup before teams can experience the benefits of modern .NET.

## What This Means for Teams

For product and domain teams:

* Start new core backend services from the approved .NET template.
* Treat Kafka consumers as durable backend services, not plumbing scripts.
* Publish API and event contracts explicitly.
* Keep runtime, dependencies, base images, dashboards, alerts, and runbooks current.
* Use Node.js only when the component is genuinely frontend-edge and intentionally thin.

For platform and enablement teams:

* Maintain the approved .NET runtime baseline, SDK guidance, base images, service templates, and package policy.
* Provide examples for HTTP APIs, workers, Kafka producers and consumers, health checks, observability, and graceful shutdown.
* Support teams moving from legacy .NET patterns to modern .NET service patterns.
* Review Node.js exceptions for scope creep.

For reviewers and the Architecture Advisory Forum:

* Challenge new backend services that do not start from the .NET default.
* Ask whether a proposed Node.js service is truly edge-shaped or is carrying durable backend responsibility.
* Keep exceptions explicit, owned, observable, and revisited.

## Exception Criteria

An exception to the .NET default should explain:

* the workload shape that makes .NET a poor fit;
* why the alternative is materially better for this component;
* whether the component is edge, data, operational tooling, or core backend;
* who owns runtime patching, security scanning, deployment, observability, and support;
* how the component integrates with .NET services, frontends, Kafka topics, and AWS infrastructure;
* when the exception will be revisited.

Valid exceptions are expected to be narrow. A broad exception is usually evidence that this ADR needs to be revisited rather than bypassed repeatedly.

## Considered Options

| Option | Assessment |
| --- | --- |
| **.NET with C# as default** | Chosen. Best fit for Organization's backend shape, existing skills, Kafka direction, and ECS Fargate deployment model. |
| **Node.js with TypeScript as default** | Strong for frontend-edge work, but would create a second core backend platform and weaken convergence. |
| **No default / team choice** | Maximizes local autonomy, but spreads templates, libraries, operations, hiring, and review across too many stacks. |
| **JVM stack as default** | Technically credible, but adds a major skills and platform migration without a clear Organization-specific advantage over .NET. |
| **Go as default** | Strong for some infrastructure and small services, but not the best strategic fit for Organization's business-logic-heavy backend estate. |

## Scope and Revisiting

In scope:

* new core backend services;
* backend APIs;
* Kafka producers and consumers;
* long-running workers;
* meaningful modernization of actively changed backend services.

Out of scope:

* frontend stack decisions, covered by [[web-tech-stack]];
* mobile stack decisions, covered by [[mobile-tech-stack]];
* data engineering and analytics workloads;
* one-off operational scripts that do not become production services;
* legacy systems that are not being actively modernized.

Revisit this decision if:

* Organization's backend workload shifts away from long-running containerized services;
* the .NET ecosystem materially weakens for containers, Kafka, AWS operation, or support lifecycle;
* the engineering skills base shifts materially away from C#/.NET;
* Node.js edge exceptions become common enough to indicate that the boundary no longer matches reality;
* platform ownership cannot sustain the .NET baseline, templates, and upgrade path.

## Appendix: Vocabulary

This vocabulary is included as reference material. It should help readers share the same language without turning the ADR body into a framework tutorial.

| Term | Meaning in this ADR |
| --- | --- |
| **C#** | The default programming language for new core backend services. |
| **.NET** | The runtime and platform used to build and run those services. |
| **.NET LTS** | The approved Long Term Support runtime baseline used for production services. |
| **ASP.NET Core** | The default .NET framework for HTTP APIs and web-hosted backend services. |
| **Generic Host** | The common .NET application host for startup, dependency injection, configuration, logging, and shutdown. |
| **Worker Service** | The .NET service shape for long-running background work such as Kafka consumers. |
| **Hosted service** | A background component managed by the .NET host lifecycle. |
| **Dependency injection** | The standard mechanism for composing application dependencies and managing lifetimes. |
| **Options pattern** | The .NET convention for binding and validating typed configuration. |
| **Health check** | A liveness or readiness signal used by runtime and operational tooling. |
| **NuGet** | The package manager and dependency distribution mechanism for .NET. |
| **BFF** | Backend-for-Frontend: a thin UI-specific API layer, not a place for durable backend business rules. |
| **Frontend edge** | Code close to one frontend surface that shapes or aggregates APIs without owning durable domain behavior. |
| **Core backend service** | A service that owns durable business behavior, data, events, APIs, or operational responsibility. |
