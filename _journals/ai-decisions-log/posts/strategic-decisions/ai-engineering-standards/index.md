> **Status**: DRAFT
>
> **Decision**: Organization will express engineering standards as governance-as-code through the AI developer platform. CLAUDE.md files, shared fragments, settings, hooks, Skills, MCP servers, subagents, pull request templates, and CI gates will encode how modernization work is done here. Standards are versioned with the code, applied across repositories, and reviewed every six months alongside the AI tooling and policy decisions.

## Decision

Adopt **governance-as-code** for Organization engineering standards in AI-assisted development.

Concretely:

* Repository `CLAUDE.md` files capture local architecture direction, commands, standards, data classification, high-risk areas, and review evidence expectations.
* Shared fragments and templates maintained by AI Engineering Enablement carry organization-wide defaults.
* Settings and hooks enforce the parts of the standard that can be made executable.
* Skills encode repeated modernization workflows.
* MCP servers expose approved live context only when needed.
* Subagents package specialized repeatable work with limited tools and defined output.
* Pull request templates and CI gates capture evidence and block non-negotiable failures.
* Deviations are allowed, but must be visible, owned, justified, and reviewed.

This decision operationalizes [[ai-tooling]], [[ai-policy]], and [[ai-operating-model]]. It does not list every concrete repository standard.

## How to Use This Decision

Use this ADR when deciding where an engineering standard should live:

| Standard shape | Preferred artifact |
| --- | --- |
| Repository context, architecture direction, commands, boundaries, and local risk notes. | `CLAUDE.md`. |
| Organization-wide default repeated across repositories. | Shared fragment or template. |
| Tool permission, command approval, required reminder, or automatic check. | Setting, hook, or CI gate. |
| Repeatable modernization procedure. | Skill. |
| Live external context or tool access. | MCP server. |
| Specialized bounded review or analysis. | Subagent. |
| Human-readable review evidence. | Pull request template. |
| Non-negotiable machine-verifiable rule. | CI gate. |

The standard is incomplete if it lives only in prose that the agent never sees and reviewers cannot verify.

## Why This Default

AI-assisted coding moves faster than standards stored only in wiki pages. The agent needs to see standards before it edits, and reviewers need evidence that standards were applied.

Governance-as-code gives Organization:

* standards that travel with the repository;
* shared patterns that can be reused across teams;
* local deviations that are visible in version control;
* hooks and gates for rules that must be enforced;
* Skills and subagents for repeated modernization work;
* review evidence that humans can evaluate.

The goal is not to automate all judgment. It is to put standards into the developer loop where AI-assisted work actually happens.

## Minimum Repository Standard Package

Every repository using AI-assisted development should have:

* maintained repository guidance;
* accurate build, test, lint, and format commands;
* architecture boundaries and high-risk areas;
* data classification notes for code, fixtures, logs, and test data;
* links to relevant ADRs and local standards;
* approved local Skills, subagents, and MCP servers, if any;
* required pull request evidence for material AI-assisted changes;
* owner and review date for the local standard;
* visible deviations from the shared baseline.

Shared defaults should reference the relevant strategic decisions, including [[backend-tech-stack-framework]], [[infra-tech-stack-compute]], [[infra-tech-stack-messaging]], [[infra-tech-aws]], [[web-tech-stack]], and [[mobile-tech-stack]] where applicable.

## Deviation Model

Local deviations are allowed when they are visible and justified. A good deviation includes:

| Field | Meaning |
| --- | --- |
| Name | Stable identifier for the deviation. |
| Repository | Where the deviation applies. |
| Standard area | What baseline it changes. |
| Reason | Why the shared default is not appropriate. |
| Owner | Team accountable for the deviation. |
| Accepted until | Review date or expiry. |
| Compensating controls | Tests, reviews, monitoring, or restrictions that reduce risk. |
| Review trigger | Event that forces re-evaluation. |

A deviation with no written reason is not customization. It is hidden drift.

## Consequences

### Positive

* Standards are visible at the developer's seat and inside the agent's context.
* Modernization patterns become reusable Skills and subagents instead of tribal knowledge.
* Deviations from shared defaults are reviewable.
* Standards travel with the code through team or ownership changes.
* CI gates and hooks enforce what can be automated.
* Reviewers get better evidence for AI-assisted changes.

### Negative

* Maintaining shared fragments, Skills, subagents, hooks, and templates is real work.
* Standards expressed through Claude Code are strongest when Claude Code remains the default.
* Some standards are judgment-based and still need human review.
* `CLAUDE.md` sprawl is a risk if teams turn guidance into wiki dumps.
* Over-enforcement can push developers to bypass controls instead of improving them.

## What This Means for Teams

* Keep repository guidance accurate and short enough to be useful.
* Put repeated local prompts into Skills only when they become real workflows.
* Use MCP for live approved context, not for static documentation.
* Mark high-risk areas clearly: authentication, authorization, customer data, infrastructure, deployment, and dependency changes.
* Include AI-assistance evidence in pull requests when changes are material.
* Record local deviations with owner and review date.
* Retire standards, Skills, hooks, or subagents that no longer carry their weight.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Governance-as-code through the AI developer platform** | Chosen. Puts standards where AI-assisted work happens and makes deviations visible. |
| **Wiki-based prose standards** | Familiar, but weak at generation time and hard to verify during review. |
| **Linter/CI-only standards** | Strong for machine-checkable rules, but too narrow for architecture intent, workflow guidance, and review context. |
| **No central standards** | Maximizes local autonomy, but loses compounding modernization patterns and common review evidence. |

## Scope and Revisiting

In scope:

* engineering standards as expressed through AI developer tooling;
* repository guidance, shared fragments, settings, hooks, Skills, MCP servers, subagents, PR templates, and CI gates;
* visible deviations and six-month standards review.

Out of scope:

* language-level formatting fully handled by formatters;
* end-user product AI features;
* internal non-engineering AI use;
* detailed production inference standards;
* every concrete repository-specific rule.

Review standards every six months, aligned with [[ai-tooling]] and [[ai-policy]]. The next planned review is **2026-11-15**. Revisit earlier when a strategic platform decision changes, the default AI tool changes, deviations become common, or an AI incident requires a stronger standard.
