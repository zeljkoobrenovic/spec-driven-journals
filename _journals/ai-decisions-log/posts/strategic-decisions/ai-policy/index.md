> **Status**: DRAFT
>
> **Decision**: Organization will adopt a unified AI coding policy that starts with human accountability and makes it enforceable through executable controls. The policy covers data classification, approved tool registries, Claude Code permission modes, hooks, settings, command allowlists, MCP and subagent controls, audit trails, CI gates, and incident handling. Developers remain accountable for AI-assisted changes they accept.

## Decision

Adopt a unified **AI coding policy** for AI-assisted software development at Organization.

The policy has seven required elements:

* **Human accountability**: a human developer accepts and owns AI-assisted work.
* **Data classification**: public, internal, confidential, and regulated data have explicit AI-use rules.
* **Approved registry**: tools, MCP servers, Skills, subagents, and exceptions are listed in a central registry.
* **Executable controls**: permission modes, hooks, settings, command allowlists, and CI gates enforce policy where possible.
* **Least-capability access**: agents receive only the tools, context, data, credentials, and network access needed for the task.
* **Audit trails**: material AI-assisted work leaves evidence of tool, scope, context class, commands, changes, and human verification.
* **Incident handling**: prompt injection, data leakage, unsafe action, tool compromise, or excessive-agency incidents have a containment and disablement path.

This policy is implemented primarily through the Claude Code default in [[ai-tooling]] and operated by AI Engineering Enablement in [[ai-operating-model]].

## How to Use This Decision

Use this ADR before starting non-trivial AI-assisted work:

| Question | Default answer |
| --- | --- |
| Who owns the AI-generated change? | The human who accepts it. |
| Can private repository code be used? | Yes, as internal data in approved tools under enterprise controls. |
| Can secrets, credentials, or production data be pasted into the agent? | No, unless a specific approved workflow exists. |
| Can an agent run broad commands or use broad MCP tools? | Only when scoped, approved, logged, and necessary. |
| Is the task high-risk? | Apply stronger review, tests, owners, and evidence. |
| Is a tool or workflow outside policy? | Request a documented, time-bounded exception. |

The policy should move teams from trust-by-enthusiasm to trust-by-controls-and-evidence.

## Data and Risk Rules

| Area | Default rule |
| --- | --- |
| Public data | Allowed with normal review. |
| Internal code and documentation | Allowed in approved tools under enterprise controls. |
| Confidential data | Restricted by default; use redaction or specific approval. |
| Regulated data | Prohibited by default unless a specific approved workflow exists. |
| Secrets and credentials | Never prompt, paste, expose through MCP, or write into guidance files. |
| Production logs | Treat as confidential or regulated until scrubbed and approved. |
| Dependencies | AI-added dependencies require license, maintenance, and vulnerability review. |
| Infrastructure, IAM, networking, CI/CD | High-risk; require owner review and stronger controls. |
| Authentication, authorization, payments, privacy, crypto | High-risk; require stronger review and test evidence. |

The task's risk tier is determined by impact, not by file count. A one-line IAM change can be higher risk than a large documentation edit.

## Control Boundary

| Control layer | Examples | Default owner |
| --- | --- | --- |
| Identity | Enterprise account, SSO, user attribution. | IT / Security / AI Engineering Enablement. |
| Tool approval | Approved tool registry and exceptions. | AI Engineering Enablement. |
| Context scope | Repository boundaries, CLAUDE.md scope, MCP scopes. | Repository owners and AI Engineering Enablement. |
| Data handling | Classification rules, redaction, no-secrets checks. | Security and repository owners. |
| Action permissions | Permission modes, command allowlists, write boundaries, network rules. | AI Engineering Enablement. |
| Automation | Hooks, preflight checks, post-edit checks, CI gates. | AI Engineering Enablement and repository owners. |
| Review | Pull request review, tests, threat review for risky changes. | Product teams. |
| Audit | Tool logs, registry references, exception records, incident records. | AI Engineering Enablement and Security. |

No single control is sufficient. The expected posture is defense in depth without making ordinary low-risk work slow.

## Incident Handling

AI-related incidents follow normal engineering seriousness. The response path is:

1. **Contain**: stop the session, disable relevant tools or scopes, revoke exposed credentials if needed.
2. **Preserve evidence**: tool logs, prompts where available, changed files, command history, pull requests, and registry entries.
3. **Classify**: determine whether data exposure, unsafe action, production impact, vulnerability, or policy bypass occurred.
4. **Notify**: involve Security, AI Engineering Enablement, repository owners, and Legal/Privacy when required by data class.
5. **Remediate**: revert unsafe changes, rotate secrets, patch controls, update registry entries, or retire risky assets.
6. **Learn**: add or adjust a control, template, Skill, hook, or review rule.

The ability to disable a tool, MCP server, Skill, subagent, hook, or permission scope quickly is part of the policy.

## Consequences

### Positive

* Human accountability remains clear.
* Data classification limits accidental exposure and regulated-data risk.
* Approved registries give developers, security, and audit one source of truth.
* Settings, hooks, allowlists, and CI gates make policy enforceable.
* Incident response is faster because risky tools or scopes can be disabled centrally.
* Developers can move quickly inside a known policy envelope.

### Negative

* Building and maintaining executable controls is real work.
* Strong controls can block legitimate work if the exception path is slow.
* Exception-approved tools need their own control envelope.
* Audit trails create operational and storage overhead.
* Some risks remain judgment-based and cannot be fully automated.

## What This Means for Teams

* Review AI-assisted changes as owned code, not as tool output.
* Know the data class before giving context to an agent.
* Keep secrets, credentials, production data, and regulated data out of ordinary AI workflows.
* Use approved tools, Skills, MCP servers, subagents, and settings.
* Include material AI-assistance evidence in pull requests.
* Escalate suspected prompt injection, data leakage, unsafe action, or tool compromise immediately.
* Use exceptions for real workflow needs, not silent local bypasses.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Unified policy plus executable controls** | Chosen. Keeps human accountability while making policy operational. |
| **Policy-as-prose only** | Easy to write, but weak in practice and hard to audit. |
| **Team-by-team AI rules** | Locally flexible, but fragments security, audit, and incident handling. |
| **Disallow AI-assisted coding** | Removes official risk surface, but gives up modernization value and may create shadow use. |

## Scope and Revisiting

In scope:

* AI-assisted software development;
* tool, context, data, action, audit, and incident controls;
* Claude Code default workflows and exception-approved alternatives;
* MCP servers, Skills, subagents, settings, hooks, and CI gates used for engineering work.

Out of scope:

* end-user product AI features;
* internal non-engineering AI use;
* model-level decisions for production inference;
* detailed legal interpretation of AI regulation.

Review this policy every six months, aligned with [[ai-tooling]]. The next planned review is **2026-11-15**. Revisit earlier after a material AI incident, regulatory change, default-tool change, or evidence that controls are routinely bypassed.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Accountability** | A human accepts and owns AI-assisted work. |
| **Policy** | The rules that govern safe AI-assisted development. |
| **Executable control** | A setting, hook, allowlist, registry, CI gate, or review mechanism that enforces or proves a rule. |
| **Least-capability access** | Giving an agent only the context and tools needed for the task. |
| **Prompt injection** | Untrusted content instructing the agent to ignore policy, misuse tools, or exfiltrate data. |
| **Excessive agency** | The agent acts too broadly for the task or risk envelope. |
| **MCP server** | External tool or data interface exposed to the agent through Model Context Protocol. |
| **Registry** | Central record of approved tools, Skills, MCP servers, subagents, and exceptions. |
| **Audit trail** | Evidence showing tool, scope, context class, actions, reviewer, and verification. |
| **Exception** | A documented, time-bounded deviation from the default policy. |
| **Incident** | A failure or suspected failure of AI tooling, controls, data handling, or policy. |
