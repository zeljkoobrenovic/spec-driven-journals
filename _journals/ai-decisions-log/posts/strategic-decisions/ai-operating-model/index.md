> **Status**: DRAFT
>
> **Decision**: Organization will establish a dedicated AI Engineering Enablement function to own the shared AI developer platform: Claude Code defaults, CLAUDE.md templates, Skills, MCP and subagent registries, policy controls, exception process, adoption support, incident coordination, and the six-month review cycle. Product teams retain delivery accountability; enablement owns shared primitives, not product features.

## Decision

Establish a dedicated **AI Engineering Enablement** function inside Organization engineering.

The function owns:

* the Claude Code default platform defined in [[ai-tooling]];
* shared CLAUDE.md templates, settings, hooks, Skills, MCP servers, subagents, examples, and onboarding material;
* approved registries for tools, MCP servers, Skills, subagents, exceptions, and review dates;
* translation of [[ai-policy]] into executable settings, hooks, allowlists, CI gates, registries, and review guidance;
* the exception process for non-default tools and non-standard configurations;
* adoption support, office hours, contribution paths, incident coordination, and disablement paths;
* the six-month review cycle for tooling, policy, standards, incidents, exceptions, value, and cost.

Product teams retain accountability for code, services, tests, pull requests, local guidance, production outcomes, and AI-assisted changes they accept.

This is a platform operating model, not an AI center that does product teams' work.

## How to Use This Decision

Use this ADR when deciding who owns an AI developer-platform concern:

| Question | Default owner |
| --- | --- |
| Shared Claude Code baseline, settings, hooks, templates, and registries? | AI Engineering Enablement. |
| Repository-local CLAUDE.md additions and local checks? | Product team, within the shared baseline. |
| Shared Skills, MCP servers, and subagents? | AI Engineering Enablement, with contributors from teams. |
| Local one-off Skill or guidance for one repository? | Product team, unless it should become shared. |
| AI-assisted pull request quality and merge decision? | Product team. |
| Tool exception, MCP intake, or high-risk shared capability? | AI Engineering Enablement with Security/Legal/Privacy where needed. |
| AI incident involving tool disablement or policy controls? | AI Engineering Enablement coordinates; affected teams own service impact. |

The function should make the safe path easier than local invention.

## Why This Default

The AI tooling and policy decisions need an owner. Without one, Skills, MCP servers, hooks, templates, and exceptions drift across teams; policy becomes inconsistent; reviews are anecdotal; and incident response is slow.

Dedicated enablement is chosen because shared AI developer tooling behaves like an internal platform. It needs:

* maintained defaults;
* authoritative registries;
* executable controls;
* contribution intake;
* support and onboarding;
* incident disablement paths;
* regular evidence-based review.

A guild can help, but part-time coordination is not enough for controls, registries, incidents, and six-month reviews.

## Operating Commitments

The function should keep a narrow, useful service catalog:

| Service | Expected output |
| --- | --- |
| Repository onboarding | Baseline CLAUDE.md, settings, hooks, approved Skills/subagents, PR evidence guidance. |
| MCP intake | Review, scope, owner, registry entry, audit behavior, and disablement path. |
| Skill/subagent review | Acceptance criteria, naming, input/output contract, tool scope, and owner. |
| Policy-control update | Settings, hooks, allowlists, CI checks, and documentation. |
| Exception review | Time-bounded approval or rejection with reason and review date. |
| Team enablement | Office hours, examples, onboarding help, and first-workflow support. |
| Incident support | Disablement, evidence collection, control updates, and post-incident learning. |
| Six-month review | Evidence pack, recommendation, accepted changes, retired assets. |

Enablement should measure adoption, reuse, exception lead time, incidents, cost, developer feedback, and retired assets. It should not optimize for generated-code volume.

## Consequences

### Positive

* There is one owner for the shared AI developer platform.
* Skills, MCP servers, subagents, hooks, and templates can compound across teams.
* Policy becomes executable faster because one function owns the translation from rule to control.
* Exceptions have a known owner and review date.
* Six-month reviews happen with evidence rather than market noise.
* Product teams get a paved path instead of designing controls from scratch.

### Negative

* Dedicated headcount is required.
* The function can become a bottleneck if it behaves as an approval queue.
* Central assets can become stale without ownership and retirement discipline.
* Boundaries with product teams, Security, Legal, Privacy, and architecture need active maintenance.
* Fast AI tooling changes may outpace the function unless exception pilots and review evidence stay healthy.

## What This Means for Teams

* Use the shared baseline unless a local deviation is documented.
* Own the quality, tests, review, and production outcome of AI-assisted changes.
* Contribute repeated local workflows back to the shared catalog when they are broadly useful.
* Request MCP servers and non-default tools through the published intake path.
* Escalate unsafe tool behavior, suspected data leakage, prompt injection, or control bypass immediately.
* Do not wait for enablement approval for ordinary work inside the approved envelope.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Dedicated AI Engineering Enablement function** | Chosen. Provides accountable ownership for shared platform primitives, controls, registries, and reviews. |
| **Embedded ownership in each product team** | Maximizes local fit, but fragments policy, registries, Skills, MCP governance, and review evidence. |
| **Federated guild** | Useful as a community layer, but too weak for operational ownership of controls and incidents. |
| **Vendor-managed enablement** | Can accelerate setup, but puts policy and platform learning outside Organization. |

## Scope and Revisiting

In scope:

* ownership of the shared AI developer platform;
* registries, policy controls, exception process, onboarding, incident coordination, and six-month review;
* contribution paths for shared Skills, MCP servers, subagents, hooks, and templates.

Out of scope:

* end-user product AI features;
* internal non-engineering AI tools;
* operating model for production model inference;
* detailed team-by-team staffing plan;
* product-team delivery ownership.

Review this operating model every six months, aligned with [[ai-tooling]] and [[ai-policy]]. The next planned review is **2026-11-15**. Revisit earlier if enablement becomes a bottleneck, an AI incident exposes an ownership gap, the default tooling changes, or Organization engineering is materially restructured.

## Appendix: Vocabulary



| Term | Meaning in this ADR |
| --- | --- |
| **Enablement** | Shared defaults, support, patterns, and self-serve paths that help teams move safely. |
| **Governance** | Rules, registries, controls, evidence, and exception paths. |
| **Shared primitive** | A reusable platform asset such as a template, Skill, MCP server, subagent, hook, or setting. |
| **Registry** | Authoritative record of approved tools, Skills, MCP servers, subagents, and exceptions. |
| **Exception process** | Time-bounded review path for non-default tools or configurations. |
| **Disablement path** | Operational route for quickly turning off a risky tool, Skill, MCP server, subagent, hook, or scope. |
| **Six-month review** | Evidence-based review of default tooling, policy, standards, exceptions, incidents, and value. |
| **Product-team ownership** | Accountability for repository code, tests, review, service operation, and production outcomes. |
