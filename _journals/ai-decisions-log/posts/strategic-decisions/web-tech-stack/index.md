> **Status**: DRAFT
>
> **Decision**: Organization will use AVIV Group's current Web Golden Path as the baseline for web work: React, TypeScript, Gemini Design System, and Unified Frontend. Organization will copy the Gemini Design System and Unified Frontend source code and operate those copies as Organization-owned platforms from day one. Alignment with AVIV is a starting point, not a standing commitment.

## Decision

Adopt **AVIV Group's current Web Golden Path** as the starting point for Organization web delivery:

* React is the default UI framework.
* TypeScript is the default implementation language.
* Gemini Design System is the starting component and design foundation.
* Unified Frontend is the starting platform for micro-frontend composition and routing.
* Organization owns source copies of Gemini and Unified Frontend.
* Future AVIV web stack changes require an explicit Organization decision. They do not flow automatically into Organization platforms.

This is an **own-the-fork decision**. Organization starts from AVIV assets to move quickly, but local ownership protects Organization from being blocked by another roadmap.

## How to Use This Decision

Use this ADR before starting or reshaping web work:

| Question | Default answer |
| --- | --- |
| Is this new or inherited Organization web work? | Start from React, TypeScript, Gemini, and Unified Frontend. |
| Is the work about shared components or composition? | Improve the Organization-owned Gemini or Unified Frontend copies rather than creating local one-offs. |
| Is the proposal to replace React, TypeScript, Gemini, or Unified Frontend? | Treat it as an exception requiring an explicit decision. |
| Is AVIV changing its Web Golden Path? | Review deliberately; do not auto-adopt. |

The decision is meant to stop immediate re-platforming debates while the inherited web estate is being absorbed. It does not freeze Organization's web architecture forever.

## Why This Default

The inherited White Label components already use AVIV's Web Golden Path. Rebuilding that foundation before Organization can operate and evolve it would create avoidable cost and delay.

React and TypeScript are mainstream, well-supported choices. Gemini and Unified Frontend are more specific assets: they are valuable because they carry existing design, composition, tooling, and community knowledge. Their value depends on Organization treating them as owned internal products rather than passive copied code.

## Ownership Boundary

| Area | Organization responsibility |
| --- | --- |
| Gemini copy | Component quality, accessibility, brand fit, releases, documentation, contribution rules, and migration support. |
| Unified Frontend copy | Routing, composition, runtime dependencies, platform compatibility, monitoring, releases, and support. |
| Product teams | Use the shared stack by default and feed reusable improvements back into the owned platform copies. |
| AVIV alignment | Review upstream fixes and changes deliberately; port only when useful for Organization. |
| Divergence | Record reason, owner, expected maintenance cost, and migration impact. |

The fork is not a license for uncontrolled local variation. The point is to create a clear Organization platform boundary.

## Consequences

### Positive

* Inherited web assets can be used and extended without immediate re-platforming.
* Teams start from mainstream React and TypeScript foundations.
* Organization can reuse AVIV patterns, documentation, examples, and engineering knowledge while building local ownership.
* Product teams can prioritize Organization-specific needs without waiting for every AVIV platform decision.
* A shared design system and composition model reduce avoidable variation across web surfaces.

### Negative

* Organization inherits the technical debt, assumptions, bugs, and constraints present in Gemini and Unified Frontend at copy time.
* Maintaining copied platform assets requires staffing, release discipline, documentation, dependency upgrades, browser compatibility work, and support.
* AVIV fixes and features do not arrive automatically.
* Over time, AVIV examples and Organization copies may diverge, reducing the value of shared knowledge.
* Teams may be tempted to bypass the shared copies with local component or composition variants unless ownership is active.

## What This Means for Teams

* Use React and TypeScript for new Organization web work by default.
* Treat Gemini and Unified Frontend as Organization internal platforms with owners, backlogs, releases, and support expectations.
* Prefer improving shared platform assets over creating one-off team variants.
* Review AVIV security and dependency updates deliberately; neither ignore them nor pull them blindly.
* Make divergence visible with reason, owner, expected maintenance cost, and review trigger.
* Keep accessibility, design consistency, browser support, and runtime observability as platform responsibilities, not afterthoughts.

## Exception Criteria

A proposal to diverge from the web baseline should explain:

* the user, product, or engineering problem that the current baseline cannot solve well;
* whether the issue is caused by React, TypeScript, Gemini, Unified Frontend, inherited design, or local implementation quality;
* the migration impact on inherited White Label components and active product teams;
* who will own the new capability, support it, and upgrade it over time;
* whether the change reduces total complexity or only moves complexity into another layer;
* how success will be measured and when the exception will be revisited.

## Considered Options

| Option | Assessment |
| --- | --- |
| **Adopt AVIV's Web Golden Path and own source copies** | Chosen. Fastest path to use inherited assets while preserving Organization autonomy. |
| **Build an independent Organization stack immediately** | Maximizes autonomy, but discards useful inherited platform work and delays delivery. |
| **Partial adoption** | Can be useful later, but risks an unclear platform model if used as the initial stance. |
| **Follow AVIV automatically** | Reduces local decisions, but makes Organization dependent on another roadmap. Rejected. |

## Scope and Revisiting

In scope:

* inherited Organization web assets;
* React and TypeScript as the default web implementation baseline;
* Organization-owned Gemini and Unified Frontend copies;
* deliberate divergence from AVIV's future Web Golden Path.

Out of scope:

* native mobile stack decisions, covered by [[mobile-tech-stack]];
* backend service stack decisions, covered by [[backend-tech-stack-framework]];
* detailed web application architecture, state management, testing, and release runbooks.

Revisit this decision if:

* maintaining the Organization-owned copies becomes disproportionate;
* divergence from AVIV materially reduces productivity;
* Organization product strategy requires capabilities the inherited stack cannot support without significant rework;
* AVIV makes a major web stack change that Organization may want to adopt or reject explicitly.
