> **Status**: DRAFT
>
> **Decision**: Move from today's fragmented estate toward a Germany-scale converged engineering landscape in six phases. Phase 0 establishes control and visibility. Phase 1 stabilises the delivery path. Phase 2 clarifies ownership and contains legacy. Phase 3 converges backend patterns. Phase 4 reduces infrastructure fragmentation. Phase 5 executes priority legacy retirement waves. The sequence reduces the highest-friction fragmentation first and avoids creating more complexity while simplifying. The roadmap is a frame; exact dates and projects come from the wave plan in [[prioritization]] once first targets are validated.

## Decision

This roadmap turns [[strategy]], [[diagnosis]], [[policy]], and [[operate]] into a practical sequence of change. It is **designed to reduce fragmentation and improve delivery speed and stability without attempting a full redesign all at once**.

The roadmap follows these principles:

* Reduce the **highest-friction fragmentation first**.
* Improve **delivery path and ownership clarity early**.
* **Avoid creating more complexity while trying to simplify**.
* Sequence legacy retirement and convergence in **realistic increments**.
* Prefer **visible reduction of active complexity** over abstract target-state discussion.

## How to Use This Roadmap

Use the roadmap to sequence work, not to claim dates prematurely. A candidate initiative should be placed by asking:

* Which phase does this work belong to?
* What prerequisite visibility, ownership, or delivery-path work must exist first?
* Does the initiative reduce active complexity in the estate, or mostly produce a plan?
* Which signal will show the phase has progressed?
* Which later phase becomes easier because this work is done?

The roadmap is a dependency map. It should prevent the organisation from jumping straight to large backend or legacy programmes before control, delivery path, and ownership basics are in place.

## What This Should Help You Do

This roadmap should help a reader:

* Explain why visibility and control must come before major convergence work.
* Place a proposed initiative into the right phase.
* Identify dependencies between delivery path, ownership, backend convergence, infrastructure simplification, and legacy retirement.
* Distinguish progress that reduces active complexity from planning activity that only describes it.

Common misreadings to avoid:

| Misreading | Correction |
| --- | --- |
| "The phases are a fixed calendar." | They are a dependency order; phases can overlap where prerequisites are met. |
| "Backend convergence should start first because it matters most." | It matters most technically, but it depends on delivery and ownership basics. |
| "Phase 5 is optional because it is last." | Legacy retirement is the payoff; earlier phases prepare it. |

## Phase Map at a Glance

| Phase | Focus | Main outcome |
| --- | --- | --- |
| 0 | Control and visibility | Stop fragmentation from growing further. |
| 1 | Delivery path and operating defaults | Fewer ways to ship code. |
| 2 | Ownership clarity and legacy containment | Clear who owns what; legacy stops expanding. |
| 3 | Backend convergence | Smaller set of active backend patterns. |
| 4 | Infrastructure fragmentation | Account estate matches Germany's scope. |
| 5 | Priority legacy retirement waves | Material reduction in active complexity. |

The phases are **not strictly sequential**. Phase 0 must come first; Phase 5 must come last; Phases 1-4 overlap as readiness allows. The numbering reflects priority and dependency, not calendar.

## Phase 0. Establish Control and Visibility

**Objective:** create the minimum visibility and governance needed to stop fragmentation from growing further.

**Focus:**

* Assign **strategy ownership** ([[operate]]).
* Establish the **strategy review forum**.
* Create a **legacy register**.
* Create an **exception register**.
* Create an initial **inventory** of AWS accounts, delivery pipelines, key systems, and accountable owners.

**Expected outcomes:**

* Major sources of active complexity become visible.
* Exceptions stop being hidden.
* Teams know where strategic decisions are reviewed.

**Success signals:**

* Named strategy owner exists.
* Exception and legacy registers are in use.
* Major systems and estates have accountable owners.
* No high-impact architecture divergence happens without review.

**Why Phase 0 matters most.** Without visibility, every later phase is operating blind. The diagnosis already tells us what the shape of the estate looks like; Phase 0 makes that shape **maintained**, not snapshotted once.

## Phase 1. Stabilise Delivery Path and Operating Defaults

**Objective:** reduce day-to-day delivery friction and improve operational consistency.

**Focus:**

* Identify the **preferred code-to-production path** for Germany. (See [[delivery-path]].)
* Reduce manual builds and manual deployments.
* Define a **narrower set of supported CI/CD and deployment patterns**.
* Publish first operational standards and blueprints.
* Make new fragmentation in tooling and pipelines require explicit justification (P8 in [[policy]]).

**Expected outcomes:**

* Developers face fewer delivery variants.
* Delivery becomes easier to understand and support.
* Teams have clearer defaults for how software should be built and shipped.

**Success signals:**

* Preferred CI/CD path is explicitly documented.
* Manual delivery paths are inventoried and decreasing.
* New projects follow default delivery tooling.
* Repeated delivery exceptions are reviewed and reduced.

**Why this is Phase 1.** Delivery friction is the most universally visible symptom. Improvements here are felt across every team, build a track record of convergence working, and create the operating muscle that later phases depend on.

## Phase 2. Clarify Ownership and Contain Legacy

**Objective:** make the current estate easier to reason about and operate before deeper convergence work.

**Focus:**

* Clarify ownership boundaries for **major systems, services, accounts, and platforms** ([[ownership-clarification]]).
* Identify shared components with **weak or ambiguous ownership**.
* Classify legacy into categories: **retire, consolidate, contain, or tolerate temporarily**.
* Stop unbounded spread of legacy tooling and platforms.
* Define **containment rules** for systems that cannot yet be retired.

**Expected outcomes:**

* System responsibility becomes easier to understand.
* Legacy stops expanding by default.
* Teams can distinguish strategic systems from tolerated historical burden.

**Success signals:**

* Major systems have named accountable owners.
* Legacy register includes a declared status for priority items.
* Shared ownership hotspots are explicitly identified.
* New work avoids extending legacy by default.

**Why ownership before backend convergence.** Convergence reorganises systems. Reorganising systems with unclear ownership produces louder versions of the existing problem. Clarifying ownership first makes Phase 3 tractable.

## Phase 3. Converge Backend Patterns

**Objective:** reduce backend fragmentation by shrinking the number of active architectural patterns in use for new and evolving work.

**Focus:**

* Define the **preferred backend patterns** for Germany ([[backend-convergence]]).
* Limit introduction of additional backend styles (P2 in [[policy]]).
* Identify areas where monolith, event-driven services, and serverless patterns **overlap unnecessarily**.
* Start convergence in **high-friction or high-change areas first**.
* Align backend modernisation with ownership simplification and delivery-path simplification.

**Expected outcomes:**

* Backend direction becomes clearer.
* Teams stop adding new architectural variance.
* Modernisation decisions become easier to evaluate against a known direction.

**Success signals:**

* Preferred backend patterns are documented in blueprints.
* New backend work follows a smaller set of defaults.
* Overlapping backend patterns are reduced in targeted areas.
* Architecture reviews spend less time debating direction from first principles.

**Why backend is Phase 3, not Phase 1.** Backend convergence is the **most important** focus area (implication 4 in [[implications]]) but it is also the **most expensive**. Doing it before delivery and ownership are stable risks reorganising backend systems through fragmented pipelines and unclear ownership - which produces a new generation of the current problem.

## Phase 4. Reduce Infrastructure Fragmentation

**Objective:** simplify the infrastructure estate to match Germany's actual operational scope.

**Focus:**

* Analyse the **AWS account estate** and define principles for consolidation, containment, and future creation ([[aws-accounts]]).
* Reduce **unnecessary account sprawl** where ownership and isolation needs do not justify it.
* Align infrastructure boundaries with **clearer operational ownership** (output of Phase 2).
* Reduce **parallel platform surfaces** that create operational burden without Germany-specific value.

**Expected outcomes:**

* Infrastructure becomes easier to govern and understand.
* Account structure better reflects Germany's real delivery and ownership model.
* Operational overhead from fragmented estates starts to decrease.

**Success signals:**

* Account inventory has clear status and ownership.
* Account creation rules are explicit.
* A subset of low-value fragmentation is consolidated or retired.
* Infrastructure exceptions are becoming rarer, not more common.

**Why infrastructure follows backend.** Account boundaries should follow operational ownership and backend topology. Consolidating accounts before backend direction is settled risks consolidating around a shape that the next phase will move away from.

## Phase 5. Execute Priority Legacy Retirement Waves

**Objective:** convert strategy from constraint and guidance into **measurable reduction of active complexity**.

**Focus:**

* Execute selected retirement and consolidation waves against the **highest-value targets** ([[prioritization]]).
* Prioritise legacy that creates significant **delivery friction, quality risk, or ownership confusion**.
* Use each wave to **remove active technologies, pipelines, platforms, or estates** from the supported footprint.
* Avoid migration work that adds new layers without removing old ones (P4 in [[policy]]).

**Expected outcomes:**

* Active complexity materially decreases.
* The strategic technology and delivery footprint becomes smaller.
* Engineering effort shifts from compensating for fragmentation to building and operating product capabilities.

**Success signals:**

* Supported technology/tooling set is **smaller than at the start**.
* Legacy register shows **real retirements**, not only new plans.
* Delivery and stability metrics move in the right direction.
* Onboarding and ownership clarity improve in affected areas.

**Why retirement is Phase 5.** Earlier phases create the conditions in which retirement can actually happen: delivery pipelines that work, owners who can authorise removal, backend direction that defines what the new home looks like, infrastructure that does not depend on the legacy estate. Without those conditions, "retirement" is a slide deck.

## Cross-Cutting Rules

These rules apply across all phases:

* **Do not add new fragmentation** unless there is explicit approval. (P8 in [[policy]].)
* **Every modernisation effort should state what existing complexity it removes.** (P4 in [[policy]].)
* **Prefer standards and defaults that teams can realistically adopt.** A blueprint that is too aspirational to use is worse than a blueprint that constrains.
* **Favour incremental convergence over large-batch redesign.** The diagnosis warns against the pattern of adding new layers; large-batch redesigns are how that pattern usually starts.
* **Review roadmap progress against delivery speed, stability, ownership, and quality** - not against activity counts.

## Suggested Prioritisation Heuristic

When choosing what to tackle first within a phase, prefer work with these properties:

1. **High delivery friction today.**
2. **High ownership confusion.**
3. **High stability risk.**
4. **Clear retirement or consolidation path.**
5. **Broad benefit across multiple teams.**

The full scoring method and the first wave plan are in [[prioritization]].

## Dependencies Between Phases

The phases are not strictly sequential, but they do have load-bearing dependencies:

| Later phase | Depends on output of |
| --- | --- |
| Phase 1 (delivery) | Phase 0 (visibility into pipelines) |
| Phase 2 (ownership) | Phase 0 (visibility into systems and estates) |
| Phase 3 (backend) | Phase 1 (working delivery), Phase 2 (clear ownership) |
| Phase 4 (infrastructure) | Phase 2 (ownership), Phase 3 (backend direction) |
| Phase 5 (retirement waves) | Phases 1-4 (the conditions that make retirement feasible) |

If a later phase starts before its dependencies are ready, expect rework. The most common version of this failure mode is starting Phase 3 (backend) before Phase 2 (ownership) - which produces backend reorganisation through unclear accountability, which is exactly the diagnosis pattern in [[diagnosis]].

## What This Roadmap Does **Not** Yet Define

The roadmap does **not** yet define:

* Exact dates.
* Named projects.
* Team-by-team sequencing.
* Effort estimates.
* Funding or staffing model.

Those should be added once Organization Germany decides which specific systems, platforms, and legacy estates are the first migration targets. [[first-targets]] names the first target areas; [[prioritization]] turns them into a wave plan.

## Anti-Patterns

* **Treating the phases as a strict gantt.** They are sequenced by dependency, not calendar.
* **Starting Phase 3 before Phase 2.** Backend convergence under unclear ownership produces louder versions of the existing problem.
* **Counting initiatives as progress.** P9 in [[policy]] is explicit: progress is measured by outcomes, not by activity.
* **Letting Phase 0 run forever.** Visibility is a prerequisite, not a destination. If Phase 0 work is not converging, the operating model ([[operate]]) is failing.
* **Adding new technologies in service of "modernisation"** while in Phase 5. Retirement waves that introduce new platforms are not retirements.
