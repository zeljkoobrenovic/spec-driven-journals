> **Status**: DRAFT
>
> **Decision**: One preferred source-control path per codebase, one preferred CI path, one preferred deployment path per runtime class, with automated build, automated deploy, and a standard rollback path. No new manual build paths, no new manual deployment paths, no new CI/CD tool without approval. GitHub + CircleCI is the current best candidate for the preferred modern path. GitLab + GitLab pipelines may remain transitional while services are classified, with explicit owner, purpose, and review intent for each use. TFS + manual builds and Bitbucket + Jenkins legacy combinations are retirement targets where they duplicate modern capability.

## Decision

This document is the **delivery path blueprint** (the default code-to-production shape for new and evolving work) and the **delivery convergence plan** (how the current mixed-generation delivery estate moves toward the default).

[[policy]] P3 makes delivery-path simplification part of the **core strategy**, not a tooling-hygiene concern. This blueprint is the operational form of that policy.

## Why This Matters

The current delivery estate spans:

* **GitHub + CircleCI** for newer services.
* **GitLab + GitLab pipelines** for mixed legacy and transitional services.
* **TFS + manual build paths** for older legacy services.
* **Bitbucket + Jenkins** for legacy and mixed-generation runtimes.

That is **too much variation for a single organisation** of Organization Germany's current scope and business expectations. Every delivery path:

* Has its own conventions.
* Has its own failure modes.
* Has its own knowledge requirements for engineers.
* Has its own operational support cost.

The diagnosis in [[diagnosis]] is unambiguous: fragmented delivery is one of the strongest contributors to slow delivery and weak stability.

## How to Use This Blueprint

Use this blueprint when creating, migrating, or reviewing a code-to-production path. It should answer:

* Which source-control, CI, and deployment path is the preferred one for this codebase and runtime class?
* Is any manual build or deployment step still part of normal work?
* Who owns the pipeline and its rollback or recovery path?
* Is this delivery path transitional, strategic, or legacy?
* What old delivery path becomes retireable when this migration is complete?

The goal is not one identical pipeline for every artifact. The goal is one understood, owned, automatable path per runtime class, with legacy paths shrinking rather than multiplying.

## What This Should Help You Do

This blueprint should help a reader:

* Explain why delivery path fragmentation is a strategic delivery problem.
* Classify a delivery path as strategic, transitional, or legacy.
* Identify the minimum standard for source control, build, deploy, visibility, rollback, and ownership.
* Challenge manual build or deployment paths that have become normal practice.

Common misreadings to avoid:

| Misreading | Correction |
| --- | --- |
| "One preferred path means one pipeline for everything." | Runtime classes may differ; unexplained variation within a class should shrink. |
| "Transitional paths can stay indefinitely." | Transitional paths need owner, purpose, and review intent. |
| "CI/CD is tooling hygiene." | Delivery path simplification is part of the core strategy. |

## Default Delivery Shape

### One Preferred Path Per Codebase, Per Runtime

* **One preferred source-control path** per codebase.
* **One preferred CI path.**
* **One preferred deployment path per runtime class.**
* **Automated build.**
* **Automated deployment.**
* **Standard rollback path.**

"Runtime class" matters: a long-running service and a static frontend bundle may justifiably have different deployment paths, but two long-running services should not.

### Required Characteristics

Every production delivery path should have:

* **Source control as the system of record.**
* **Automated build** - no manual build as normal practice.
* **Automated deploy** or one controlled approval gate.
* **Standard logging and deployment visibility.**
* **Clear production approval rule** where required.
* **Rollback or recovery path** defined.
* **Named owner.**

The owner is part of the delivery path, not separate from it. A pipeline without an owner cannot meet the strategy-compliance bar.

### Minimum Delivery Standard

A path is **strategy-compliant** when:

* Source control is the system of record.
* The build is automated.
* The deploy is automated (or has one explicit approval gate).
* Production state is visible without tribal knowledge.
* A rollback or recovery path exists.
* The pipeline has a named owner.

Anything that fails one or more of these is **legacy delivery** for the purposes of [[policy]] P4 and needs a declared status: retire, consolidate, contain, or tolerate temporarily.

## Allowed and Not Allowed

### Allowed

* **Temporary legacy pipelines** with explicit owner.
* **Transitional dual-running** while migrating to the default.
* **Runtime-specific deployment variations** where justified.
* **Temporary coexistence** of old and new pipelines during migration.

### Not Allowed by Default

* **Introducing new CI/CD tools** without approval through [[operate]].
* **Team-specific permanent delivery paths.**
* **Production deployment that depends on tribal knowledge.**
* **Silent exceptions** to the preferred delivery path.

"Silent exceptions" matters: this is the failure mode the operating model ([[operate]]) is built to prevent.

## Transitional Direction

While the estate is moving toward the default:

* **No new manual build paths.**
* **No new manual deployment paths.**
* **No new CI/CD tool** should be introduced without approval.
* **Legacy pipelines may remain temporarily**, but only with named owner and migration intent.

These four rules are how the current estate stops getting worse while convergence work happens.

## First Migration Priority

Within the current estate, prioritise removal of:

* **Manual builds.**
* **Manual deployments.**
* **Opaque pipelines** with no clear owner.
* **Duplicate paths** that serve the same runtime class.

These are the categories most directly responsible for delivery friction, and the ones whose retirement is most realistic in the strategy horizon.

## Minimum Questions to Ask of Any Delivery Path

When designing or reviewing a delivery path:

* Can **a new engineer understand** how this reaches production?
* Is the pipeline **automated end to end**?
* Is there a **standard rollback path**?
* Is this **converging with the preferred delivery model**, or diverging from it?

The fourth question is the one most often skipped. The strategy is explicit that "neutral" is not an option: every change either converges or diverges.

## Convergence Plan

The blueprint above is the **target shape**. The convergence plan below is how the current estate moves toward it.

### Convergence Goal

Move toward:

* **One preferred modern path** for newer services.
* **Explicit transitional status** for tolerated legacy paths.
* **Retirement** of manual and opaque legacy delivery flows.

### Proposed Direction

#### Preferred Modern Path

* **GitHub + CircleCI** is the current best candidate for the preferred modern path.

Reason:

* It is **already active** in the estate.
* It is **easier to standardise around a modern automated flow** than around mixed legacy tooling.
* The newer-service portion of the estate is already substantially on this path.

This is a **proposal**, not a finished decision. It should be validated through the [[operate]] review forum and through the first wave of standardisation work.

#### Transitional Paths

* **GitLab + GitLab pipelines** can remain transitional while services are classified.

Condition for transitional status:

* Each use must have **owner, purpose, and review intent**.
* GitLab paths without those are not "transitional" - they are unclassified, which is a failure of [[policy]] P5 (ownership) and P8 (no hidden divergence).

#### Retirement Targets

* **TFS + manual build paths.**
* **Bitbucket + Jenkins** legacy combinations where they duplicate modern capability.

These are scored as Wave 1 retirement candidates in [[prioritization]].

### Current Delivery Estate at a Glance

The known delivery paths in the current estate, with their candidate action:

| Path | Source Control | Build / Deploy | Runtime Class | Candidate Action |
| --- | --- | --- | --- | --- |
| GitHub + CircleCI | GitHub | CircleCI / runtime-specific | Newer services | **Standardise** |
| GitLab + GitLab pipelines | GitLab | GitLab pipelines | Mixed legacy and transitional services | **Investigate** |
| TFS + manual build | TFS | Manual build / legacy tooling | Older legacy services | **Retire** |
| Bitbucket + Jenkins | Bitbucket / mixed | Jenkins / mixed | Legacy and mixed-generation runtimes | **Consolidate** |

The full inventory and scoring are in [[prioritization]].

### Wave 1 Actions

The first concrete steps to start the convergence:

1. **Inventory all known delivery paths** by owner, service, and runtime.
2. **Identify every manual build and manual deployment path.**
3. **Identify every TFS and Bitbucket / Jenkins path still active.**
4. **Confirm whether GitHub + CircleCI** can serve as the preferred path for the main modern runtime classes.
5. **Mark each path** as preferred / transitional / retire / investigate.

The output of these five steps is the first version of the **delivery convergence wave plan** that [[operate]] reviews on the monthly cadence.

### Minimum Decision Rules

| Rule | Why |
| --- | --- |
| No new manual build path. | Manual is exactly the anti-pattern the diagnosis names. |
| No new manual deployment path. | Same reason. |
| No new CI/CD tool without explicit approval. | Prevents recreating the fragmentation. |
| No legacy delivery path without owner and review intent. | Defeats P5 and P8 of [[policy]] otherwise. |

These four rules are the bare minimum the operating model should enforce through the strategy review forum.

### Success Criteria

The convergence is working when:

* **Fewer active delivery patterns** exist than at the start.
* **No unowned pipeline path** remains.
* The **manual delivery footprint** is shrinking.
* The **default path for new services** is clearer than it is today.

These are restatements of [[policy]] P9 in delivery-specific form.

## What This Blueprint Deliberately Does Not Specify

| Not specified | Why |
| --- | --- |
| One single deploy target across all runtimes | Runtime-specific variation is allowed; team-specific is not. |
| Specific monitoring / alerting product | The requirement is *standard* observability, not a specific tool. |
| One single review/approval gate model | Production approval rules can vary by runtime class. |
| Whether GitHub Actions or CircleCI | The current proposal is CircleCI because it is already in use; if GitHub Actions becomes the better fit, that is a future blueprint update, not a per-team decision. |

These are blueprint-extension questions that [[operate]] resolves over time.

## Anti-Patterns

* **"Each team picks its own delivery path."** The blueprint exists specifically to remove that as a default.
* **"This pipeline is transitional."** Without owner, purpose, and review intent, "transitional" is a euphemism for "unmanaged".
* **"Manual deploy is fine for this one service."** Manual delivery as a normal operating model is explicitly not allowed.
* **"We need a new CI tool because the current one is annoying."** Convenience is not a strategic reason. The exception process exists for genuine misfit, not for tooling preference.
* **"GitHub Actions, CircleCI, GitLab pipelines, and Jenkins - we'll support them all."** This is the diagnosis in [[diagnosis]] restated as a strategy. It is not the strategy.

## Scope and Revisiting

Revisit the blueprint when:

* The first wave of standardisation work shows that GitHub + CircleCI is not in fact the right preferred path.
* A new central AVIV Group direction makes a different CI/CD platform the default.
* A specific runtime class genuinely cannot be served by the preferred path.

In the absence of those triggers, the blueprint should be **stable**.
