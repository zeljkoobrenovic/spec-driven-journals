> **Status**: DRAFT
>
> **Decision**: Use a conventional microservice model based on standard AWS patterns as the default backend direction for new and evolving product work in Organization Germany: clear bounded context per service, explicit API contract, explicit service owner, service-owned data, synchronous API interaction by default, and selective async messaging only where clearly justified. Converge by reducing backend overlap zones, simplifying shared messaging surfaces, and removing shared-database coupling across logical service boundaries. No new backend paradigm beyond the approved direction, no shared-database coupling between services by default, no serverless-first product backend development by default, no new messaging pattern without explicit reason.

## Decision

This document is the **backend blueprint** (the default shape for new and evolving backend work) and the **backend convergence plan** (how the current estate moves toward the default).

It is the most important per-area blueprint in the strategy, because backend fragmentation is the **strongest architectural mismatch** in [[diagnosis]] and the main technical focus of [[policy]] P2.

## Why This Matters

The current backend landscape combines:

* **Monolithic systems** around SQL Server and C#.
* **Event-driven Java microservices** on self-hosted Kafka.
* **Newer serverless-based services.**

These three eras coexist, often within the same business domain. The cost is overlap, unclear target state, and high cognitive load - all of which directly push back on the business expectation of faster delivery and stability.

The job of this blueprint is to give teams **a clear default** to reach for, so that future backend decisions stop adding to the three-era problem.

## How to Use This Blueprint

Use this blueprint when starting, changing, or reviewing backend work. It should answer:

* Is this backend decision moving toward the default shape or preserving another active paradigm?
* Does the service have a bounded context, explicit owner, explicit API contract, and owned data?
* Is async messaging used because the domain needs it, or because the current estate already has it?
* What legacy coupling, overlap, or shared database dependency becomes easier to retire?
* If the proposal diverges from the default, which ADR explains the reason and review trigger?

The blueprint is not a demand to migrate every backend immediately. It is the default direction for new and evolving work, and the test for whether changes are converging or fragmenting.

## What This Should Help You Do

This blueprint should help a reader:

* Describe the default backend shape in terms of bounded context, ownership, contracts, and service-owned data.
* Explain why conventional microservices are the direction for new and evolving product backend work.
* Identify backend decisions that add another active paradigm rather than converging the estate.
* Distinguish justified async messaging from inherited messaging habit.

Common misreadings to avoid:

| Misreading | Correction |
| --- | --- |
| "Everything must become a microservice now." | The blueprint guides new and evolving work; migration still follows prioritization and risk. |
| "Synchronous by default means async is bad." | Async is allowed where the domain need is explicit and justified. |
| "Private data means no integration." | Services integrate through contracts, not shared databases. |

## Default Backend Shape

### Architecture

* **Conventional microservice architecture.**
* **One service per business capability**, with clear bounded context.
* **Explicit service owner.**
* **Service-owned data** - no shared databases across services.
* **API-first synchronous interaction by default.**
* **Selective async messaging** only where clearly justified.

### Runtime Expectations

* **Long-running service runtime** for core product backends.
* **Explicit API endpoint** or message contract.
* **Explicit deployment unit.**
* **Explicit production owner.**
* **Explicit operational dashboarding and alerting.**

### Required Characteristics

A service is **strategy-compliant** when it has all of:

* Independently deployable.
* Explicit API contract.
* Explicit data ownership.
* Standard observability.
* Standard CI/CD integration (per [[delivery-path]]).
* Clear runtime model.
* Minimal hidden coupling to legacy.

### Preferred Internal Structure

A conventional layered structure inside each service:

* **API / controller layer** — accepts requests, validates input.
* **Application service layer** — orchestrates use cases.
* **Domain logic layer** — business rules, owned by the service.
* **Repository / integration layer** — persistence and outbound integration.
* **Private service database** — not shared with other services.

The blueprint does not mandate a single language or framework. It mandates the **shape**.

## Allowed and Not Allowed

### Allowed

* Synchronous service-to-service APIs.
* Selective event-driven integration.
* Incremental migration from monolith or legacy service boundaries.
* Temporary containment of legacy dependencies (behind an explicit adapter).
* One service owning one business capability, even if some legacy integration remains.

### Not Allowed by Default

* **Shared-database integration** between services.
* **Serverless-first** product backend design.
* **Event-driven interaction as the automatic default.** Use sync API unless there is a reason.
* **Custom per-service platform patterns** without approval through [[operate]].
* **Introducing a new backend style** beyond the approved direction.

The "by default" matters. None of these is permanently forbidden; all of them require an explicit exception through the [[operate]] process. The point is to make them visible rather than invisible.

## Transitional Rules

While the estate is moving toward the default:

* **New backend work** should not introduce a new service style beyond the approved direction.
* If a **legacy component cannot yet be removed**, the new service should isolate that dependency behind an **explicit adapter**.
* If an **event-based integration already exists**, keep it only where it serves a clear purpose; do not extend it by default.
* If a **synchronous API is sufficient**, prefer it over adding new messaging complexity.

These rules give teams a way to make progress without waiting for the entire convergence plan to finish.

## Minimum Definition of Done

A backend service is **not strategy-compliant** unless it has:

* Named owner.
* Explicit API contract.
* Explicit data ownership.
* Deployment path through the preferred delivery model ([[delivery-path]]).
* Standard logging, monitoring, and alerting.
* Documented dependency on legacy, if one exists.

A service that exists in the estate but does not have all six of these is **legacy** for the purposes of [[policy]] P4 - it needs a declared status: retire, consolidate, contain, or tolerate temporarily.

## Decision Questions

When designing or reviewing a backend service, work through:

* Does the service have a **clear bounded context**?
* Is **ownership** explicit?
* Does it **own its data**?
* Can **another team** understand how to run and evolve it?
* Does it **reduce or increase** backend fragmentation?

The last question is the one most often skipped. P2 of [[policy]] is explicit about it.

## Convergence Plan

The blueprint above is the **target shape**. The convergence plan below is how the current estate moves toward it.

### Convergence Goal

Move toward:

* A **smaller number** of supported backend patterns.
* **Clearer** service and data ownership.
* **Fewer overlapping** integration styles.
* **Less hidden coupling** to legacy.

### Preferred Direction

Use a conventional microservice model as the default direction for new and evolving product backend work, based on standard AWS patterns. (Restated from the blueprint above; this is the **convergence direction**, not a one-time target.)

That means:

* Clear bounded context per service.
* Explicit API contract.
* Explicit service owner.
* Service-owned data.
* Synchronous API interaction by default.
* Selective async messaging only where clearly justified.

### First Convergence Targets

#### 1. Backend Overlap Zones

Areas where:

* **Monolith and microservices overlap.**
* **Microservices and serverless overlap.**
* **Multiple integration styles** coexist without clear reason.

Goal: reduce parallel backend models in the same functional area.

Candidate zones from the current estate (subject to validation):

| Area / Domain | Current shapes | Candidate direction |
| --- | --- | --- |
| Core classifieds / listing flow | Monolith + services + newer distributed components | **Converge** — strong candidate due to business centrality |
| Search and discovery | Monolith + services | **Investigate** — high value, needs clearer mapping |
| User / account / CRM | Monolith + services | **Investigate** — likely ownership and integration hotspot |
| Messaging and shared integration | Services + legacy shared integration | **Contain** — critical substrate; not first direct convergence target |
| Data / analytics / DWH | Legacy data platform + distributed integrations | **Investigate** — heavy historical estate, slower-moving |

The scoring and wave assignment for these candidates is in [[prioritization]].

#### 2. Shared Messaging Surfaces

The current messaging estate includes both **self-hosted Kafka** and **RabbitMQ**, with overlapping use cases and unclear retirement decisions.

Goal:

* Determine where messaging is **essential**.
* Determine where messaging can be **simplified, contained, or retired**.

Action principles:

* Treat the messaging surface as **shared infrastructure**, not as a per-service choice.
* No new messaging pattern without explicit reason and approval.
* When a synchronous API would do, use it; messaging is the heavier option.

#### 3. Shared-Database Coupling

The current estate still contains components that depend on **shared-database access across logical service boundaries**. This is the canonical anti-pattern that defeats service ownership.

Goal:

* Move toward **explicit API or owned-data boundaries**.
* When a service owns a capability, it owns the data behind that capability.

The transitional rule is: if a shared-database dependency cannot be removed yet, **wrap it behind an explicit adapter** owned by the new service. The legacy database stays; the dependency becomes explicit.

## Minimum Rules

| Rule | Why |
| --- | --- |
| No new backend paradigm beyond the approved direction. | Prevents recreating the three-era problem. |
| No shared-database coupling between services by default. | Defeats the core ownership property of the blueprint. |
| No serverless-first architecture for new product backend development by default. | Adds a fourth pattern on top of three; serverless is permitted selectively, not by default. |
| No new messaging pattern without explicit reason. | Messaging is the heavier option; the bias is toward sync APIs. |

These four rules are the bare minimum the operating model ([[operate]]) should enforce through the strategy review forum.

## First Wave Actions

Concrete first-wave work to start the convergence:

1. **Identify the top backend overlap zones** (using the table above as a starting point, validated by domain owners).
2. **Classify each zone** as monolith-heavy / mixed / service-oriented / serverless-heavy.
3. **Identify where Kafka and RabbitMQ both contribute to fragmentation** rather than clear value.
4. **Identify where shared-database coupling still defines integration.**
5. **Select the first backend area where convergence is both high-value and realistic.** (Likely candidate: core classifieds / listing flow.)

Steps 1-4 are **mapping**, not migration. Step 5 is the first migration decision. Splitting them is intentional: convergence work that starts without mapping has a high rate of producing yet another parallel pattern.

## Questions to Answer

Open questions for the first wave:

* Which backend areas combine multiple paradigms today?
* Which messaging flows are essential, and which are historical leftovers?
* Which service boundaries are clear, and which are still implicit?
* Which backend area offers the best first convergence candidate?

These are not rhetorical. They go on the agenda of the strategy review forum ([[operate]]) once Phase 0 visibility work has produced enough data to answer them.

## Success Criteria

The convergence plan is working when:

* The number of **active backend patterns** is decreasing.
* The **default direction** is documented and visible to engineers in their normal work.
* **Reliance on hidden shared-data or unclear messaging surfaces** is reducing.
* **Ownership and cognitive load** are easier for teams.

These are restatements of [[policy]] P9 in backend-specific form.

## What This Blueprint Deliberately Does Not Specify

| Not specified | Why |
| --- | --- |
| Single backend language | The shape matters more than the language. Multiple sensible language choices remain inside the default. |
| Single API framework | Same reasoning. |
| Specific database technology per service | The principle is service-owned data, not a specific database product. |
| Whether REST or gRPC | The sync-API default does not force a specific protocol. |
| Specific messaging product | Containment of Kafka and RabbitMQ is the active question. |

These are blueprint-extension questions that the [[operate]] mechanisms can resolve over time as patterns stabilise.

## Anti-Patterns

* **"Microservices for everything."** The blueprint says conventional microservices are the default *shape*. It does not say split every capability into the smallest possible service. The size-related concerns are upstream in the engineering principles, not in this blueprint.
* **"We already have a serverless pattern, so this new thing should also be serverless."** Serverless is not the default. Reusing it because it exists is exactly how a fourth backend style becomes a permanent fixture.
* **"Sync APIs are old-fashioned."** The default is sync. Async messaging is the special case. The bias matters.
* **"This service is technically a microservice."** Technical labels do not satisfy the minimum definition of done. Ownership, data, observability, and CI/CD integration do.
* **"We'll fix the shared database later."** Shared-database coupling without an adapter is the most expensive of the not-allowed-by-default patterns. Wrap it now.

## Scope and Revisiting

Revisit the blueprint when:

* The first backend convergence wave produces evidence that one of the not-allowed-by-default patterns is in fact essential.
* AVIV Group golden paths change materially in a way that affects the default direction.
* A new technology or pattern appears that genuinely simplifies the estate (not just adds a new flavour).

In the absence of those triggers, the blueprint should be **stable**. Frequent rewriting suggests the diagnosis is unsettled rather than the blueprint being wrong.
